# JFrog 收购 CloudMunch 是为了更好地测量 DevOps 流程

> 原文：<https://thenewstack.io/jfrog-acquires-cloudmunch-boost-data-driven-devops/>

为了给 [DevOps](/category/devops/) 过程带来更多的洞察力， [JFrog](https://www.jfrog.com/) 收购了 [CloudMunch](https://www.cloudmunch.com/) ，将 CloudMunch 的 DevOps 智能平台纳入 JFrog 旗下。

JFrog 的业务开发副总裁 [Kit Merker](https://www.linkedin.com/in/kitmerker/) 表示，此次收购将提供 JFrog 承诺的端到端 DevOps 堆栈的关键部分，该堆栈以 [Artifactory](https://www.jfrog.com/artifactory/) 二进制库开始，以 [Mission Control](https://www.jfrog.com/mission-control/) 结束，这是一个用于监控软件整体健康状况的监控管理仪表板。

CloudMunch 专门处理隐藏在开发运维及持续集成流程下的数据。CloudMunch 首席执行官兼创始人[帕拉德普·普拉布](https://www.linkedin.com/in/pradeepprabhu/)说，与任务控制相比，它可以提供对开发运维流程及其各种组件的更深入的数据洞察。

收购[还将帮助](https://jfrog.com/press/jfrog-acquires-cloudmunch-roi-speedometer-containers-devops/) JFrog 在[数据驱动的 DevOps](https://www.pagerduty.com/wp-content/uploads/2016/02/PagerDuty-DataDriven-DevOps-White-Paper.pdf) 上取得进展，该公司认为这是 DevOps 总体发展的下一步。该公司的观点是，开发过程中更好的智能和可见性、持续集成以及更新的二进制文件的分发将为软件开发公司带来更好的总体底线。

Prabhu 说:“开发人员或管理人员将获得关于基础设施、他们的发布渠道以及他们所有不同工具的更新。“他们将有一个仪表板，实时显示流量是如何发生的。我认为拥有一个单一平台将是非常标准的，从代码到基础设施，都连接在一起。”

JFrog 的基础设施正朝着单一窗口的方向发展，该窗口可以显示补丁和更新是如何被分发出去的。Merker 说这是 JFrog 的主要最终目标之一。

“我们正在推动的事情之一是软件更新的连续性，我们希望它以一种流动的方式流动，”Merker 说。“我认为版本就像宠物:版本号会更像牛。人们不会考虑他们正在运行的版本。它只是更新。有很多关于管理的细节——我们可以改进软件的很多过程和流程——减少人工干预。目标是让开发者专注于高价值的创造性任务，而不必担心流程，”Merker 说。

任务控制中心将是 CloudMunch 计划的着陆平台。Merker 说，它是在功能上接近 CloudMunch 的 JFrog 产品，并且已经保存了关于 Artifactory 部署的数据。

“当我们审视围绕数据驱动开发运维的战略时，我们的第一批客户是早期采用者和开发者。他们不需要证明，也不需要数据来证明。我们现在遇到的情况是，客户必须向他们的老板证明这样做是值得的。“此外，我需要随着时间的推移进行微调，使我的产品变得更好。对于第二个级别的性能改进，你需要更多的数据，更多的洞察力，”Merker 说。
