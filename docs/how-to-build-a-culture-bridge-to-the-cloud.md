云原生技术的承诺是明确的:提高敏捷性、增加可靠性和扩大创新视野。

但是像任何范式转变一样，向云的过渡面临一些严重的障碍。八月[来自](https://www.cncf.io/blog/2018/08/29/cncf-survey-use-of-cloud-native-technologies-in-production-has-grown-over-200-percent/)[云原生计算基金会](https://www.cncf.io/)的一项调查发现，采用云原生技术的努力面临着任何新技术都会遇到的一些障碍:联网、复杂性和缺乏培训。但受访者面临的最大挑战是“开发团队的文化变革”，占 41%。

有效使用开源、云原生技术所必需的 DevOps 文化已经从根本上改变了软件和团队流程。它扩展了我们工作和思考的方式。对一些人来说，这是一个令人兴奋的机会。其他人则更加战战兢兢地接近它。

一般来说，创业公司都在船上。他们没有需要维护和升级的固定技术。他们也能够雇佣那些技能组合非常适合新技术的人。

对于企业来说，有点难。他们在诸如 Java 和 WebLogic 等主力技术和平台上有大量投资。但他们的 IT 团队在构建、部署、运行和维护应用程序方面拥有深厚的传统和运营知识。

可以理解的是，他们的开发者不一定想成为基础设施和 Kubernetes 等项目的专家。他们可能看不到让新手摆弄它的价值。

只要开发团队和运营团队保持分离，他们每个人都有一定程度的权力和舒适。让这两个群体一起工作并接受 DevOps 不仅是为了提高人们的技能，也是为了让每个人都理解并接受更具协作性的方法。它是关于从整体上理解一个系统，关于从整体上思考，关于在具有非常不同的训练和背景的人之间进行富有成效的交流。换句话说，这是关于文化的。如果企业要实现云技术的全部优势，这正是必须要做的事情。

幸运的是，有一些方法可以让这种转变变得更容易。正如 Bitnami 的工程经理 Araceli Pulido 在最近的 KubeCon+CloudNativeCon 2018 的一个小组讨论会上指出的那样，你不能将新技术强加给不愿意的 IT 员工。相反，种下吃苦耐劳的 DevOps 文化的种子的最好方法之一是找到一个愿意前进的冠军团队。你的一些开发人员可能已经参加了会议，并且正在使用开源软件。如果他们也异乎寻常地合作，并且喜欢记录他们所学到的东西，他们是第一个团队的好候选人。

让他们从他们知道的开始。您可以将一个应用程序移动到一个容器中，而不必将其分解为微服务。第一步将让您在新环境中启动并运行应用程序。这将更容易将工作流分解成可组合的部分，让您可以添加其他微服务，如日志功能。

之后，尝试将应用程序逐步分解为微服务。然后，可能是时候为该应用程序构建一个新的前端了，新的前端可以驻留在云中。下一步可能是寻找可以是云原生的新项目和模块。当你的冠军团队开发了那个应用程序，也许他们可以开始与一个更传统的团队合作。

作为一个组织，你需要从一种由内向外的文化转向一种更加由外向内的文化。参加会议。去聚会。向你的行业和更广泛的开源社区学习。你会发现自己在为招募新人才搭建桥梁，这些人也乐于从事新技术工作。

最重要的是:你不需要自己做所有这些，冠军团队也不需要。期望开发人员成为基础设施方面的专家，或者成为能够帮助他们更有效地使用云的每个开源项目的专家，这是一个沉重的负担。

这就是托管服务发挥巨大作用的地方。例如，Kubernetes 是开发人员必须学习的第一批主要技术之一。但是有了托管服务，如 [Oracle 的 Kubernetes (OKE)](https://cloud.oracle.com/containers/kubernetes-engine) 容器引擎，组织不再需要建立自己的 Kubernetes 集群，维护它并跟上版本。OKE 是一个容器原生的和企业就绪的托管 Kubernetes 服务。它与云中的其他托管服务相集成，如存储、应用程序开发工具和负载平衡。您的开发人员不必是 Kubernetes 专家和应用程序专家，他们只需专注于应用程序即可。

借助托管服务，您的团队可以跨越云学习曲线的很大一部分，直接进入云原生生态系统和社区，并立即开始为自己、企业和客户产生成果。

所有这些都是说，即使您还不是云原生用户，您也可以享受云的转型优势。理解文化及其价值是第一步。拥有更全面的思维方式会培养同理心。理解系统的模块化部分可以提高效率。最重要的是，有了托管服务的切入点，您就可以开始为创新让路。

云原生计算基金会和 Kubecon+CloudNativeCon 是新堆栈的赞助商。
