Kubernetes (K8S) [现在似乎是管理和部署基于微服务和容器的应用程序的事实上的标准](https://logz.io/blog/rise-kubernetes-2017/)——很容易理解为什么。Kubernetes 是最大的开源社区，它由[云本地计算基金会(CNCF)](https://www.cncf.io/) 提供支持，对开发人员友好，并提供混合云的优势。什么叫*不*去爱？

虽然在最近的调查中有 69%的受访者表示 Kubernetes 是他们进行容器编排的首选，但是部署和管理 Kubernetes 并不一定是在公园里散步。尽管 Kubernetes 非常灵活，但运营工作流非常复杂，其中许多涉及应用性能管理(APM)，需要有效管理，以便 Kubernetes 实现其承诺的优势。

## 重新思考您的 Kubernetes 监控策略

最近的一项 CNCF 调查显示，38%的受访者认为监控是他们采用 Kubernetes 的最大挑战之一——随着企业规模的扩大，这一比例甚至上升到 46%。那么，现代 IT 领导者如何简化 Kubernetes 监控以提高效率，同时优化性能呢？

当前的 Kubernetes 监控方法有几个不同的原因，从缺乏端到端的可见性到容易出错的迁移。以下是您在监控 Kubernetes 时可能会遇到的四个常见挑战，以及如何解决它们的技巧。

## 挑战 1:缺乏端到端可见性

Kubernetes 中传统监控最常见的挑战之一是缺乏对客户接触点和分布式应用程序的端到端可见性。

因此，IT 团队无法洞察终端用户体验和应用程序性能如何影响业务 KPI，从而无法知道要修复或改进什么。

为了克服这一点，重要的是利用 Kubernetes 监控解决方案,该解决方案为正常性能设定基准，并通过机器学习的力量，在出现问题时智能地提醒 it 团队。

## 挑战 2:警报风暴

虽然拥有对所有应用程序问题的可见性*看起来*像是一件好事，但当多个问题同时出现时，它可能会迅速失控并成为一个障碍。毕竟，您真的需要在每次作业完成或新容器启动时发出警报吗？

如果不能对警报进行优先级排序，IT 团队通常必须对每个问题的*根源进行响应和分类，从而导致糟糕的用户体验和收入损失。*

强大的 Kubernetes 监控解决方案可以帮助您识别和解决确切的底层问题——深入到代码行、单个设备、Kubernetes 服务甚至单个容器级别。

## 挑战 3:解决困境

应用程序停机的[成本可能会非常高](https://blog.appdynamics.com/engineering/idc-devops-cost-downtime/) —一个关键的应用程序故障每小时的成本可能高达 100 万美元*。时间就是金钱，在检测问题的根本原因时，IT 团队不应该浪费时间。*

 *问题在于，当今的许多监控工具缺乏在 Kubernetes 环境中执行自动化根本原因分析的能力，这使得故障排除成为一场耗时的噩梦，通常会导致高 MTTR 和减少停机时间。

为了避免这种情况，请确保您的 Kubernetes 监控解决方案通过比较迁移前后的用户体验，提供应用程序依赖性的可见性和迁移成功的验证。

## **挑战 4:向 Kubernetes 的易错迁移**

将传统应用程序迁移到 Kubernetes 可能容易出错且耗时。将现有整体应用迁移到部署在 Kubernetes 上的微服务的公司缺乏对 Kubernetes 环境的了解，因此无法看到每个微服务或传统应用的实时交互。

借助通过单一控制台提供一步安装和统一监控的解决方案，IT 团队可以利用他们现有的技能组合、流程和工具。

## 在 Kubernetes 上提供完美的应用性能

使用 Kubernetes 来改变公司在分布式、多云环境中部署和运行应用程序的方式已经变得越来越受欢迎，而且没有放缓的迹象。但是对于在 Kubernetes 上运行传统或基于微服务的应用程序的组织来说，传统监控方法的缺点是显而易见的。

因此，组织必须重新思考他们在 Kubernetes 中的监控方法，以简化复杂的企业工作流，提高成本效益和生产力。借助对整个 Kubernetes 堆栈和 Kubernetes 编排的应用程序的端到端统一可见性，IT 团队可以提供完美的应用程序体验— *和*确保他们的 Kubernetes 投资推动更好的业务成果。(下载我们的电子书了解更多:[如何放心地监控 Kubernetes】。)](https://www.appdynamics.com/lp/how-to-monitor-kubernetes-with-confidence/)
