# 如何通过硬件信任链扩展安全性

> 原文：<https://thenewstack.io/how-to-scale-security-with-a-hardware-chain-of-trust/>

众所周知，CISOs 规避风险，注重合规性，为 IT 和开发团队提供执行策略。相比之下，在服务业务成果的名义下，应用程序开发领导者希望尽可能消除 DevOps 摩擦。什么方法可以满足这些相互冲突的需求，同时实现最终目标:大规模安全性？

## 建立信任链

基于硬件的信任链验证云平台中每个相关组件的完整性，为您提供灵活集成到 DevOps 管道中的安全自动化。真正的信任链始于主机芯片固件，并通过容器引擎和编排系统建立起来，在应用的生命周期内保护所有关键数据和工作负载。

硬件是理想的基础，因为它植根于硅，黑客很难改变。

信任链将从这个根使用测量和验证安全模型构建，每个组件测量、验证和启动下一个级别。该过程将扩展到容器引擎，创建信任边界，测量结果存储在主机上的可信平台模块(TPM)中。

到目前为止，一切顺利——但是现在您必须将这个过程扩展到主机信任边界之外的容器编排级别。你必须继续提高安全性。

不同服务器上的证明软件可以对照已知的正确值来验证当前测量值。container orchestrator 与证明服务器通信，以验证工作主机的完整性，进而设置和管理部署在其上的容器。超出主机信任边界的所有通信都被加密，从而形成高度自动化的可信容器系统。

## 如何通过信任链提升安全性

一个完全实现的信任链会给你带来什么？

*   **增强的透明度和可伸缩性:**因为信任链促进了自动化的安全性，DevOps 团队可以不受阻碍地自由工作。他们只需要管理安全策略，可信容器系统根据这些策略来评估其度量。
*   **地理工作负载策略验证:**智能集装箱编排仅将移动限制到批准的位置。
*   **容器完整性保证:**当容器被移动时，证明者检查以确保在该过程中没有发生篡改。系统验证移动的容器是否与最初创建的容器相同。
*   **敏感数据的安全性:**加密的容器只能在批准的服务器上解密，保护传输中的数据不被暴露和滥用。
*   **简化的合规性控制和报告:**元数据审计跟踪提供了可见性和可审计的证据，表明关键容器工作负载正在可信服务器上运行。

信任链架构旨在满足对安全性和快速创新的迫切需求。安全官员可以制定安全策略，自动应用于每个被创建或移动的集装箱。除了在清单中维护策略本身，序列中的每个步骤都是自动化的，使开发运维团队能够快速构建和部署应用，而无需手动管理安全性。

当您的团队评估云平台时，请供应商解释他们如何建立和维护对托管您组织的应用程序的技术的信任。有明确的预期会有所帮助。
