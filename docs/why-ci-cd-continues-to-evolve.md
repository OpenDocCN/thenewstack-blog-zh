# 为什么 CI/CD 会继续发展

> 原文：<https://thenewstack.io/why-ci-cd-continues-to-evolve/>

在过去的几年中，持续集成和交付(CI/CD)已经经历了一些根本性的变化，特别是对于持续交付。虽然不久前，应用程序开发和交付还只是针对单片堆栈，但为微服务和容器环境交付软件是一种非常不同的方式。

在 10 月份录制于 [KubeCon+CloudNativeCon](https://www.cncf.io/kubecon-cloudnativecon-events/?utm_content=inline-mention) 的 New Stack Maker 播客中，嘉宾 [Rob Zuber、](https://www.linkedin.com/in/robzuber/)首席技术官、 [CircleCI](https://circleci.com/?utm_content=inline-mention) 从 CircleCI 十多年经验的角度讨论了 CI/CD 的演变。

《新书库》的创始人兼发行人亚历克斯·威廉姆斯主持了这次播客。

[CI/CD 为什么继续进化](https://thenewstack.simplecast.com/episodes/why-ci-cd-continues-to-evolve)

CI/CD 正在“变得相当主流——人们认识到这是我们交付软件的方式或者我们渴望交付软件的方式。但在 CircleCI 的 10 年生命周期中，我们部署软件的环境已经发生了显著变化，”Zuber 说。“当我们创办公司时，每个人基本上都在构建[单片]，现在每个人都在谈论微服务、容器和流程编排。我们在一个非常不同的世界中交付产品。”

适应这种新的环境是 CircleCI 作为软件提供商不断发展的过程。“确保我们生产的产品能够在人们改变生产方式时为他们提供支持，这一点非常重要，我们看到的一个发展趋势是生产的复杂性正在增加。我们正在做一些事情，以降低开发人员的个体复杂性——就像你可以在单个服务上工作，但整个系统变得越来越难以预测，因为你可以在我试图改变我的服务时改变你的服务，我们不一定知道这两种改变同时发生，这很好，因为它给了我们快速移动的自由，”Zuber 说。

“但我们需要能够理解这些事情可能会相互影响，所以我们看到的人们遇到的一些问题直到他们的所有软件都投入生产后才会发生。因此，我们扩展了我们如何帮助客户理解他们的代码在生产中始终是好的，并在他们将功能展示给客户时，为他们提供管理发布过程的工具。”

DevOps 团队对 CI/CD 的需求应该继续发展和变化，这将取决于 CircleCI 和其他 CI/CD 提供商的适应能力。

“人们会根据我们的交付方式在不同的环境中构建新工具吗？我们的目标不是拘泥于一种特定的方法，而是理解客户正在努力做什么，也就是为他们的客户带来价值，这是他们关心的事情，”Zuber 说。“因此，随着他们周围的环境不断变化，我们专注于花时间找出如何帮助他们有效地做到这一点。”
