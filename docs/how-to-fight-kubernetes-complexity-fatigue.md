# 如何对抗 Kubernetes 的复杂性疲劳

> 原文：<https://thenewstack.io/how-to-fight-kubernetes-complexity-fatigue/>

在这一集由 TNS 创始人兼出版商 [Alex Williams](/author/alex/) 主持的 [The New Stack Makers](/podcasts/makers) 播客中，我们将讨论 Kubernetes 的复杂性疲劳，更重要的是，我们可以为此做些什么。嘉宾是持续集成和交付系统提供商 [Harness](https://harness.io/products/continuous-integration?utm_content=inline-mention) 的布道者[拉维·拉赫曼](https://www.linkedin.com/in/ravilachhman)，以及数据存储提供商 [DataStax](https://www.datastax.com/?utm_content=inline-mention) 的高级技术工程经理[弗兰克·莫利](https://www.linkedin.com/in/frankmoley)。

[拉维·拉赫曼和弗兰克·莫利——如何对抗库伯内特复杂疲劳怪物](https://thenewstack.simplecast.com/episodes/ravi-lachhman-and-frank-moley-how-to-fight-the-kubernetes-complexity-fatigue-monster)

Kubernetes 的复杂性疲劳在很大程度上可以归因于 DevOps 团队在迁移到云原生架构时必须如何扩展他们的专业领域，通常超出他们的舒适区。基础设施团队成员必须更加适应开发人员的需求，而开发人员的工作量越来越多地涵盖与基础设施相关的任务。

拉赫曼说，Kubernetes 正在“跨越两个鸿沟”。开发人员需要更多地了解基础设施问题，另一方面，“运营、基础设施或系统工程人员越来越接近开发人员，因为你如何编写 Kubernetes 资源或 Kubernetes YAML 有时会模仿软件开发，因为其中涉及到迭代，”拉赫曼说。“因此，双方都不得不学习比他们的舒适区(允许)多得多的技能，我认为这是疲劳所在。”

类似于一般的生活，帮助 DevOps 团队避免 Kubernetes 的复杂性疲劳是一个认识到您的限制是什么，以及如何以您能够实现您的目标的方式最好地管理差距的问题。

“我可能会赌很多钱说，真的不可能知道关于任何技术工具的所有事情，不仅仅是 Kubernetes——但就这一点而言，你可以知道足够多的东西，然后你可以找出其余的东西，”Moley 说。“所以，我认为这是管理对你所知的期望，以及你如何建立一个战略来学习你所不知道的东西……所以，对我来说，管理复杂性涉及管理你自己对如何工作的个人想法和感受，就像管理你所知道的一样。”

拉赫曼说，还必须意识到，一旦 Kubernetes 被采用——当然，这对大多数组织本身来说是一项艰巨的任务——你组织的所有软件问题都不会得到解决。拉赫曼说:“(Kubernetes)实际上是一个相当复杂的系统，对于一些投身其中的人来说，随着它变得越来越成熟，开始褪色，闪亮的便士现在变得不那么闪亮了。”。

拉赫曼说，Kubernetes 组件——尽管它们非常复杂——是组织明智地“小批量”购买的东西。“对于 Kubernetes 生态系统来说，最重要的是它实际上非常可插拔，所以没有非常严格的定义:如果你将其与[平台即服务(PaaS)](https://en.wikipedia.org/wiki/Platform_as_a_service) 进行比较，这给了你一个非常强烈的观点，Kubernetes 的设计是可插拔的。”
