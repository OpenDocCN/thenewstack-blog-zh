# 一个开源项目的资本分叉背后的故事

> 原文：<https://thenewstack.io/story-behind-capital-ones-fork-open-source-project/>

对于大型开源项目，管理来自不同方的所有[拉请求](https://yangsu.github.io/pull-request-tutorial/)可能会很困难，因此，自然地，自动批准拉请求的需求导致了几个开源项目的创建。 [LGTM(在我看来不错)](https://github.com/lgtmco/lgtm)就是这样一个自动化系统，围绕 GitHub 构建。在收到给定数量的批准之前，它锁定拉请求，防止其被合并到上游。

LGTM 没有预先配置为被固定到某些实现者的现有工具链中。现在，由于它是一个通用项目，也许没有人会期望它被这样配置。但是如果不向实现者开放集成能力，那么开源开发过程的目的是什么呢？

在最近的 OSCON 会议上，Capital One 首席软件工程师 [Jon Bodner](https://www.linkedin.com/in/jonbodner/) 讲述了 LGTM 的首席开发人员如何祝福他的团队开发出一个 fork，该 fork 添加的功能可能不仅针对 Capital One 或另一家金融机构，而且针对其规模的任何组织。

博德纳向新堆栈的 Alex Williams 解释说:“当你有拉请求时，你会希望确保有某种代码审查，有人确实检查过，以确保你合并的东西已经过验证。“人们经常会看到拉式请求，他们会将其合并，然后就完成了。但是你真的不知道谁在做合并。可能是同一个人首先发出了拉取请求。在像 Capital One 这样的大公司，你希望对流程有某种控制。”

在本期来自得克萨斯州奥斯汀最后一届 OSCON 的 [The New Stack Makers](https://thenewstack.io/tag/the-new-stack-makers) 播客中，了解更多有关 Bodner 的努力如何导致更基于角色、面向策略的请求管理系统的信息。
