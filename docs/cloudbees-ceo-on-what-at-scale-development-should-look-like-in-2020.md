# CloudBees 首席执行官谈 2020 年大规模开发应该是什么样子

> 原文：<https://thenewstack.io/cloudbees-ceo-on-what-at-scale-development-should-look-like-in-2020/>

CloudBees 赞助了这个播客。

对持续交付(CD)的关注转移是 2019 年出现的主要趋势之一。对于那些采用了云原生平台并经历了 Kubernetes 平台部署和管理的陡峭学习曲线的组织来说尤其如此。

在最新一集的 [The New Stack Makers](https://thenewstack.io/podcasts/makers) 播客中，[CloudBees](https://www.cloudbees.com/)CEO[Sacha laboury](https://www.linkedin.com/in/sachalabourey/)讨论了 2020 年 CD 和软件交付管理的复杂性，以及 Jenkins X 和 cloud bees 在大规模开发中的角色。

对于许多组织来说，2019 年是关键的一年，因为 Kubernetes 采用的雪球效应，机会和复杂性都出现了。

[SACHA laboury-cloud bees 首席执行官谈 2020 年的大规模开发应该是什么样子](https://thenewstack.simplecast.com/episodes/sacha-labourey-cloudbees-ceo-on-what-at-scale-development-should-look-like-in-2020)

“让开发人员感到惊讶的是，他们如何利用从内部到多云的真正不可知的层，”Labourey 说。对于多云环境，无论您是否想要，“只有一个层是真正的标准，它使开发人员能够在内部或任何云上工作，这就是 Kubernetes——故事结束。”

然而，如上所述，组织也正在觉醒并学习如何更好地管理与 Kubernetes 相关的复杂性。“我们看到了许多掩盖这种复杂性的方法，”Labourey 说。

一个这样的解决方案由 [Jenkins X](https://github.com/jenkins-x) 组成，既适用于新的云原生部署，也适用于传统部署。这一概念在很大程度上取决于在应用程序和运营管理级别简化和自动化 CI/CD。

“显然，在 CloudBees，我们是 Jenkins X 这种方法的支持者。但 Jenkins X 不仅仅是关于 CI/CD，它还提供了一种抽象，关于如何理解这些层，”Labourey 说。“但我认为我们正处于成熟期，在这一阶段，it 被视为事实上的云计算基础架构层。但是，要真正让普通开发人员变得更加高效和多产，需要做些什么呢？”

因此，在许多方面，裁谈会“只是旅程的一部分”，劳雷说。随着 CD 反馈回路从抽象概念转变为今天的现实，衡量开发和部署“影响”的能力变得至关重要。“当我们谈论监控和可观察性时，让我非常兴奋的不是概念本身，而是因为我们可以开始测量具体的影响，并根据我们想看到的和不想看到的来自动化一些行为，”Labourey 说。“因此，对我来说，这真的是持续交付旅程中不可或缺的一部分。”

随着 CloudBees 快速向前扩展其 Jenkins X 平台和用于新技术和平台的工具套件，重要的是不要让那些可能刚刚开始采用 CI/CD 和 DevOps 的组织落在后面。这也是 Jenkins X 适应遗留平台的能力如此重要的原因之一。

“我们必须认识到，实际上许多组织甚至还没有开始这一旅程，”Labourey 说。“我们[有时]认为这一切都是既定的，但我们要记住，今天并非所有的开发人员和组织都有 CI。所以，我的心和他们在一起。”
