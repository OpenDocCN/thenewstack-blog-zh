现在是 2019 年，世界各地的 It 组织都应该生活在 DevOps 和 DevSecOps 中。但事实是，很多都不是。数据显示，仅有 17%的[组织完全采用了 DevOps。其余的可能仍然停留在与](https://www.statista.com/statistics/673505/worldwide-software-development-survey-devops-adoption/)[敏捷](https://agilemanifesto.org/)相关的相对低效、缓慢的交付过程中。

可以肯定的是，敏捷软件交付在本世纪初出现时是创新的。但那是近二十年前的事了。如果您的组织今天仍然主要依赖于敏捷方法，那么它就错过了重要的效率，并且应该专注于升级到 DevOps 实践。一旦你做到了这一点，你就可以通过实现 DevSecOps 来更进一步，这将 DevOps 的好处扩展到了软件安全性。

这篇文章解释了如何从敏捷到 DevOps，然后到 DevSecOps。

敏捷是从项目经理的角度来看的软件开发。虽然它让团队以更疯狂的速度工作，但团队之间仍然存在孤岛，当事情出错时，这是一个无休止的指责游戏，而不是找到解决方案。意识到这些缺点，生态系统转向了 DevOps，一种寻求将三个功能(开发、质量保证和运营)集成到一个单一的内聚单元的方法。

DevOps 是对敏捷的改进，因为它更关注人而不是过程和工具。此外，DevOps 鼓励团队之间使用[“左移”](https://www.twistlock.com/resources/shift-left-security/)方法进行协作，其中 QA 和 Ops 团队是应用程序设计和开发阶段的积极参与者。此外，DevOps 方法将自动化置于手动操作之上。无论是构建自动化、测试自动化还是发布自动化，自动化的每一步都会带来开发速度的增量收益，并缩短应用程序的上市时间。

虽然 DevOps 是敏捷思维的一个巨大转变，但是如果你已经开始采用 DevOps 实践，那么 [DevSecOps](https://www.twistlock.com/resources/guide-modernizing-traditional-security/) 并不是一个大的改变。从本质上讲，DevSecOps 强调了开发流程中经常被忽略的安全性方面。关于自动化、协作和 DevOps 方法的其他核心支柱已经说了很多，在这个过程中，安全性可能会被遗忘。DevSecOps 希望通过给予安全性应有的优先权来改变这种情况。

DevSecOps 并不需要一个独立的安全团队，而是希望在每一个步骤和每一个合作者身上注入安全原则，包括开发人员和 QA。它要求开发人员在从 Git 存储库协作时遵循安全性最佳实践。这包括在提交新代码之前运行自动化的“模拟”测试。QA 团队应该将自动安全检查作为构建自动化过程的一部分，以便在不良代码进入生产环境之前阻止它们。在开发和测试之间建立一个即时的反馈循环是实现“左移”好处的关键

## 云本机安全性

值得一提的是，这种软件交付管道的现代化只有借助云的力量才有可能实现。然而，仅仅在云平台上运行几个虚拟机是不够的。这将需要使用云供应商提供的多种云服务，以及云供应商不提供的利基服务，这些服务由专注于一个利基的云初创公司提供更好。这带来了工具的大规模变化，从供应商特定的工具到[开源](https://www.twistlock.com/2018/11/13/open-source-cloud-discovery-tool/)和基于云的工具。对此最好的描述是，它不仅仅是云计算，而是云原生计算。

容器是云原生基础设施的新构建块。使用容器打包应用程序时，有些东西和以前一样，但有些东西是新的。例如，基于角色的访问控制(RBAC)、加密密钥管理和最小特权原则并不是新的实践，它们可能只是在云原生环境中以不同的方式实施。另一方面，像用非根特权运行容器、容器映像扫描和为网络配置服务网格这样的实践都是容器革命带来的软件交付的新方法。云原生安全性应该注意系统中的这些新组件，并建立新的实践来保护使用这些组件构建的应用程序。

由于应用程序和基础设施的分布式特性，有大量数据需要出于安全目的进行分析，还有一个大的(尽管是分布式的)攻击面需要保护。这是人工审查无法做到的，需要机器学习算法来分析系统性能和健康指标，以识别和防止内部漏洞和外部威胁。运行时威胁检测是一种现代的软件安全方法，在云环境中运行时至关重要。

## 结论

作为一个离别的想法，让我提出一个观点，关于采取一个现实的方法来进行敏捷到 DevOps 到 DevSecOps 的旅程。不要期望改变会在一夜之间发生。相反，关注从敏捷到开发运维，最后到开发运维的缓慢而稳定的进展。您希望确定您的组织的需求和目标，并根据您的情况选择合适的解决方案。

有时，解决方案是将一个单一的应用程序“提升并转移”到云，并将其打包在容器中，而在其他情况下，弃用旧应用程序并替换为更新、更灵活的应用程序可能是有意义的，该应用程序旨在从根本上实现云原生。在每种情况下，都应该从一开始就考虑安全性，而不是事后才考虑。这样做会让你进入开发团队的思维模式，帮助你实现你所追求的目标。
