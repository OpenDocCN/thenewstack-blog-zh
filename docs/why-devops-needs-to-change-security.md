# 为什么 DevOps 需要改变安全性

> 原文：<https://thenewstack.io/why-devops-needs-to-change-security/>

[帕洛阿尔托网络公司的 Prisma](https://www.paloaltonetworks.com/prisma/cloud) 在 2020 年 2 月 11 日举行的[云本地安全直播，2020 年虚拟峰会](https://vshow.on24.com/vshow/Palo_Alto_Networks/registration/16700)之后，赞助了这篇文章。

采用 DevOps 方法后，应用程序的开发方式发生了巨大的变化。然而，将安全性添加到这种方法中，并不是开发人员思考过程中最重要的事情。这可能会在部署的应用程序中造成许多缺口。团队可以很容易地通过在构建期间引入安全性来增加可见性，同时添加协作点。

但是需要做哪些改变，对安全架构和操作会有什么影响？什么需要保持不变，什么需要改变？

## 快速安全

即使工作流中内置了手动审批点，传统的安全运营模式也将成为瓶颈。为了提高效率，安全团队需要采用 DevOps 模型并与之集成，以将测试和控制作为管道的一部分。这将需要采用一些新的工具和技能，并改变运营做法。在 DevOps 驱动的业务中，这是完成团队保护企业使命的唯一方式。

“[将安全向左转移](https://blog.paloaltonetworks.com/2019/07/4-practical-steps-shift-left-security/)”既有定义又有解释。在核心上，这意味着在软件交付生命周期的早期插入安全性考虑。这是有意义的，因为在应用程序开发的构建阶段，一些安全弱点比在软件部署后更容易检测到，并且修复起来更便宜。

然而，这并不意味着将应用程序和运行时安全性的责任完全转移给开发团队。安全和开发团队需要合作，以便更早地识别威胁和控制，并将安全测试插入到软件交付工作流中。

好消息是，尽管开发团队自动化安全测试可能需要的特定工具可能还没有到位，但是它们是可用的。

## 威胁分析

最后，开发人员对运行时堆栈承担了更多的责任，他们的代码将在运行时堆栈中执行，通过使用基础设施即代码来定义整个运行的应用程序环境，或者使用 Dockerfiles 来定义他们的应用程序容器。反过来，安全团队需要了解这些不断发展的开发环境中可能存在的威胁，并提供开发团队可以在应用程序编码的最早阶段集成的工具。这将允许两个团队都认识到不安全的配置，因此它们甚至可以在第一次代码提交之前就被修复。

## 拥抱发展

随着 DevOps 启发的软件交付变得越来越普遍，IT 的其他部分——特别是安全性——将需要适应更快的开发周期和高度自动化软件交付管道中的新攻击媒介。除此之外，还要实施安全最佳实践，跟上不断变化的威胁和危害技术。可以肯定地说，唯一正在缩小的风险是无所事事的风险。
