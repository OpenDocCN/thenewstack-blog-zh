# 开发商对控制权的争夺

> 原文：<https://thenewstack.io/the-developers-struggle-for-control/>

GitLab 赞助了这个播客。

今天的开发人员体验无疑为软件工程师提供了在高度分布式微服务环境中创建大规模应用程序的自由。但是有了这种自由度，开发人员面临着提供更快节奏的压力。

在这个[的新堆栈制造商](/podcasts/makers)播客中，记者 [B. Cameron Gain](/author/bruce-gain/) 采访了 GitLab 开发副总裁 [Christopher Lefelhocz](https://www.linkedin.com/in/christopher-lefelhocz-8900133) 和 [Lightstep](https://lightstep.com/) 的首席执行官兼联合创始人 [Ben Sigelman](https://www.linkedin.com/in/bensigelman) ，讨论了开发人员如何利用弹性和其他流程和工具来确保软件从代码上传到 GitLab 的存储库开始，以及在整个部署和使用周期中保持弹性和安全。

[开发商对控制权的争夺](https://thenewstack.simplecast.com/episodes/the-developers-struggle-for-control)

DevOps 团队(包括开发人员和运营人员)面临的长期挑战是如何跟上日常部署和更新的步伐，同时维护安全性并确保一切正常。Lefelhocz 解释说，这些印刷机还需要保持可靠性和持续可用性。

“我们看到了创新和快速行动的巨大压力，”Lefelhocz 说。“我们发现平衡这两种努力对于获得良好的 DevOps 体验非常重要，我们发现在这种情况下一致性非常重要。”

当代码出现问题时——正如 DevOps 团队所熟知的，这种情况经常发生——开发人员通常会承担责任，并且必须找出如何修复通常隐藏在堆栈几层中的问题。

“我认为作为一名开发人员压力很大，因为压力的定义是有责任却没有控制，”Sigelman 说。“如今，开发人员肩负着巨大的责任——不仅仅是他们的服务，还有他们下面的一切——这通常是整个系统的一个巨大部分，尤其是如果你在堆栈的顶部。”

因此，开发人员“要对所有这些东西负责”，西格曼说。“如果它坏了，你会收到传呼，”他说。

Sigelman 说:“微服务和 DevOps 的全部意义在于对每个部分进行独立控制，所以你不能控制你所依赖的这些东西，但你要对它们负责。”“这就是为什么我认为 CI/CD 中非常好的卫生不仅仅是一种最佳实践，而且是帮助开发人员保持理智的唯一方法，在这种环境中，您的责任远远大于您实际控制的一项服务。”

同时，回滚配置或功能部署将解决大多数事件，但不是在所有情况下。Sigelman 说:“当事情变得非常棘手，出现某种干扰效应或下游云依赖中断之类的情况时，我很遗憾地认为，我们仍然需要那些已经在公司工作了五年以上的优秀人才。”。“他们可以帮助诊断和修复这些问题，这不是我们想要的情况，但我认为这就是我们的现状。”
