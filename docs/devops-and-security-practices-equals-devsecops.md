# 开发运维与安全实践等于开发运维

> 原文：<https://thenewstack.io/devops-and-security-practices-equals-devsecops/>

CloudBees 赞助了这个播客。

[Devops 和安全惯例等于 DevSecOps](https://thenewstack.simplecast.com/episodes/devops-and-security-practices-equals-devsecops)

在今天的《新堆栈制造商》节目中，TNS 创始人 Alex Williams 与 tCell 联合创始人兼工程副总裁 Boris Chen[进行了一次讨论，以进一步了解现在所谓的 DevSecOps 对当今企业的影响。统一 DevOps 和安全性并向 DevSecOps 的趋势发展是许多工程团队正在接受的事情，因为他们发现自己不仅跨分布式系统工作，而且跨分布式团队工作。](https://www.linkedin.com/in/boris-chen-4b118a1/)

Chen 有 25 年的企业软件工作经验，有 J2E 背景，之前在 Splunk 担任工程副总裁。在他职业生涯的早期，他做了大量的 QA 和性能测试，这极大地影响了他开发 tCell 的方式。“监控方面似乎是应用程序安全性中的一个普遍缺失的漏洞，”Chen 说。这些观察是他们如何到达 tCell 的，其建筑灵感来自 APM 空间中的 New Relic 和 AppDynamics。陈指出，使用能够插入应用程序进程的代理在云服务中是不可行的，他补充说，如果你在云服务中的每台服务器上插入一个代理，那么这个代理就成为软件的一部分。

tCell 后来模仿了这种方法，使用代理“从安全角度收集数据”。有了收集点和执行点，我们就可以通过我们的云服务来协调和管理，这样他们就不必在本地安装任何东西或类似的东西。他们可以登录、管理他们的所有代理，并监控所有可能与安全相关的数据。”

Chen 接着指出，代理架构是通过安装了 tCell 的 CI/CD 管道来实现的，允许安全团队审查和制定策略以匹配正在部署的软件。Chen 补充说，采用 DevOps 方法来实现安全性，可能会有相当模糊的界限。这就是 DevSecOps 的用武之地。“我们不只是谈论开发和运营部门如何推倒那堵墙，我们还会推倒安全方面的墙，并将它们融入对话中。”
