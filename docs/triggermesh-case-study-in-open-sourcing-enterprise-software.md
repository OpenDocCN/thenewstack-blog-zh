# TriggerMesh，开源企业软件案例研究

> 原文：<https://thenewstack.io/triggermesh-case-study-in-open-sourcing-enterprise-software/>

基础设施和系统管理软件开发的未来是开源驱动的，不仅是供应商，还有技术用户。

 [马克·辛克尔

Mark 在新兴技术和开源领域有着悠久的历史。在共同创立 TriggerMesh 之前，他是 Node.js 基金会的执行董事，也是 Citrix、Cloud.com 和 Zenoss 的高管，在那里他领导了开源工作。](https://www.linkedin.com/in/markrhinkle/) 

Kubernetes 就是一个最好的例子。它已经成为每个主要云供应商支持的事实上的云原生平台。有许多理由认为，在未来十年，自由和开源软件将会持续增长。

看看那些决定全力以赴使用企业开源解决方案的公司，已经有了令人印象深刻的结果。我们见证了 Confluent、Kong、蟑螂实验室和 HashiCorp 以开源方式提供系统管理和基础设施软件，它们都作为私营公司赢得了它们的独角兽。

Elastic 和 MongoDB 都已成为价值数十亿美元的上市公司，IBM 以 340 亿美元收购开源供应商 Red Hat 是历史上最大的一笔交易。这就是为什么我仍然认为这是交付企业基础设施软件的最有吸引力的方式。

## TriggerMesh 向开源 iPaaS 的过渡

在 TriggerMesh，我们很早就知道我们希望开源我们的云原生集成平台。我们甚至建立了一个团队，作为领导者、贡献者和维护者，他们拥有丰富的开源经验，尽管许多人质疑我们为什么不开始让我们的平台开源。因为没有更好的理由，收养。不是我们不知道如何做开源，而是我们知道我们没有资源来做好它。仅仅为了开源软件而开源，然后看着它在架子上积灰，这与我们的愿景相去甚远。

2018 年， [Sebastien Goasguen](https://twitter.com/sebgoa) 和我用我们的一毛钱共同创立了 TriggerMesh，然后在 2019 年 12 月获得了一笔非常适度的资金来打造核心产品。2021 年 6 月，我们筹集了额外的资金，将我们的产品推向市场，这意味着我们现在有资金支持我们的开源计划。

这就是为什么我们[在 Kubecon North America 2021 上推出了](https://www.triggermesh.com/blog/introducing-triggermesh-open-source)我们的 TriggerMesh 云原生集成平台，作为一个开源项目。这是一个艰难的决定，因为我们和我们的投资者必须考虑什么对我们公司最有利。经过多次讨论，我们决定全力以赴开发源代码和一个补充我们的开源战略的商业模式。

## 选择开源许可

我们非常清楚，我们需要选择一个开源倡议批准的许可证，这样软件才能被认为是开源的。然而，我们想向那些提出许可证的人学习，这些许可证在很大程度上包含了开源的价值，但有一些限制。

例如，我们评估了 MongoDB 和后来的 Elastic NV 最初使用的服务器端公共许可证。我们看到 Confluent 选择许可其围绕 Apache Kafka 构建的产品，但有一个条款禁止供应商将其自由代码作为服务提供。

我们还考虑了 Business Source 许可证，这是一种延迟即时开源的替代方案，由几家著名的供应商使用。最初由 MariaDB 和后来的蟑螂实验室支持，它还包括一项条款，禁止供应商提供商业数据库服务。

我理解其中的原因，因为开源许可可能允许其他供应商，尤其是 SaaS 供应商，提供与原始开发者相同的技术，而原始开发者已经完成了大部分的开发工作。考虑因素是看到公司如何花费大量资源只让云供应商提供软件即服务，并切入初始开发者的商业机会。

然而，我们最终选定了 Apache License，Version 2.0，因为我们想确保我们交付的是开源软件，而不是一些现代的共享软件变体。

## 开源是一种开发方法，而不是一种商业模式

我们相信开源是将基础设施软件推向市场的最佳方式。我们是在这样的假设下运作的:到 2021 年，每个大型企业都将拥有某种开源战略。这就是为什么我们使用开源开发来开发我们的技术。虽然销售软件不是我们的商业模式，但解决企业技术问题才是。

我们还认为，公司需要非常小心地让他们的商业模式依赖于开源许可证或类似的许可证，就像前面提到的那些。这就是我们选择 Apache 许可证 2.0 版的原因。

这种许可被广泛接受，并为潜在客户降低了风险，这些客户可能不愿意冒险投资一家业绩记录不佳的新兴公司。我们还确保注册和保护我们的商标，因此虽然我们的代码是免费的，但我们的商标只能在我们指定的指导原则下使用。

## 建设一个社区

我们没有同意“梦想的领域”开源战略，简单地说，“如果你开源它，他们就会来。”相反，我们希望将基础设施放在一起，以交付我们的项目并发展它。这也意味着对我们的流程和程序做出一些改变。例如，我们需要考虑新用户和潜在的未来开发者的体验。

我们做的第一件准备工作是将我们在 GitHub 上的存储库重组为一个单独的存储库。巩固我们的代码库可以让用户快速找到他们想要的东西。

我们还为用户创建了一个单一的 Kubernetes 控制器来轻松安装产品。我们聘请了一位经验丰富的开源贡献者来管理开发者关系，他在开源和云集成方面有着长期的记录。

我们还更新了我们的网站，关注开源和社区参与。此外，我们花费了大量的精力来更新和添加我们的文档，以帮助新用户和开发人员更加成功。

最大的问题是:你如何定义成功？这与你管理业务的方式没有什么不同，但我们关注的是用户、贡献和拉动请求等指标，而不是收入。也许最肤浅的衡量标准是星星的数量，这是很好的第一步，我们希望这是社区成员漏斗的开始。

## 我们的商业计划，不是开源计划

尽管许可发生了变化，但我们的商业计划基本上保持不变。TriggerMesh 入门级产品涉及 TriggerMesh 开源集成平台的支持和服务。我们为内部使用提供此版本，并在云中为用户提供托管版本。我们提供补充开源项目的产品和服务。我们可能会开源更多的软件，并增加对 Knative 等上游项目的贡献，Knative 刚刚达到了 1.0 的里程碑。
