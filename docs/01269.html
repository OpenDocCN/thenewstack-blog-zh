Docker定义了以开发者为中心的API集成路线图

原文：https://thenewstack.io/docker-defines-roadmap-with-developer-focused-api-integrations/#0001-01-01

自从 Docker向Mirantis出售Docker企业技术平台和所有相关知识产权，以及四分之三的员工和所有企业合作关系以来，已经过去了将近四个月。今天，该公司提出了一项关于如何向前发展的计划，产品副总裁 Justin Graham 写道，重点将是帮助开发人员完成工作，从源代码管理“以最简单、最有效和不可知云的方式实现公共云运行时”当然，这在一定程度上是公司在收购时所说的，但是今天他们提供了更多的细节。

格雷厄姆写道，这种关注将“通过Docker桌面关注开发者体验，与生态系统合作，并使Docker Hub成为构成应用和微服务的应用组件的所有集成、配置和管理的纽带。”

Docker Hub是一个基于云的库和相关的容器映像社区； Docker Desktop 是一个用于构建容器的集成开发环境(ISE)。

在接受New Stack采访时，Docker首席执行官Scott Johnson解释说，自从Docker上任以来的七年里，开发人员和开发团队发生了很多变化，在出售了其企业部门后，公司现在可以专注于这些问题。

“随着集装箱化的发展，以及部署团队现在希望快速发货，您需要一个新的自动化主干来让您的工具链做到这一点。尽管围绕源代码管理、持续集成和持续部署已经出现了许多单点产品，但它们都非常专注于解决工具链的这一阶段，”Johnson解释道。“现在我们不得不自己用胶带、打包线和口香糖把它们串起来。我们想要一致的体验，仍然允许我们选择，选择我们最喜欢的配置管理工具，选择我们最喜欢的竞争情报工具，但要以一种方式来做，让它融入一个框架，为他们的应用带来更好的用户体验和更快的吞吐量。”

根据这篇博文，第一步将是扩展Docker Desktop中的工具和体验，在Docker Hub提供的Docker CLI和Docker Desktop UI中提供有关成员入职、团队协作和通信的功能，以防止上下文切换。接下来，格雷厄姆写道，Docker将“扩展Docker Hub，以帮助您管理作为开发和部署的一部分而生成的所有应用程序组件”，并“使Docker Hub成为与我们合作的工具生态系统的中心点。”

在这一点上，约翰逊很快指出，Docker并不打算重新发明轮子。他指出，如果已经有了GitHub、GitLab和BitBucket，就没有必要再构建一个源代码管理工具。相反，Docker的发展方向将是为用户和生态系统成员提供一种编程方式来构建集成。

“这将全部由API驱动。我们的目的是让所有的表面区域都暴露给我们将提供给生态系统的公共API。围绕这一点将会有SDK和程序，这样他们就可以获得SDK，注册，获得他们的密钥，访问API，开始构建他们的插件，开始构建自动化，然后开始。我们没有对上一代Docker投入足够多的资金，我们已经从中吸取了教训。激活生态系统的方法是提供伟大的接口和伟大的SDK，因此合作伙伴可以抓住它就走。”

最后，Graham写道，该公司将继续关注开源，Compose、Engine和公证人将继续为Docker产品做出巨大贡献，尤其是Docker Desktop。Docker将继续以软件即服务的订阅模式交付这些工具，Johnson表示，Docker Hub上已经有
“成千上万的付费客户正在这样做。”

为了进一步了解细节，Docker还推出了一个公共路线图，并将在3月26日举办一个“问我任何事”(AMA)虚拟论坛。
