# 蜂窝监控现在结合了跟踪和事件分析

> 原文：<https://thenewstack.io/honeycomb-monitoring-now-combines-tracing-with-event-analytics/>

根据 Honeycomb 的 Aneel Lakhani 的说法，大多数追踪工具的问题是知道将追踪的焦点放在哪里，并且必须在系统之间切换以查看整体情况和粒度活动。

可观察性平台供应商[蜂巢](https://www.honeycomb.io/)旨在通过发布追踪软件来解决这些问题。

总部位于旧金山的初创公司 Honeycomb 的软件旨在帮助工程师[解决最棘手的基础设施问题](https://thenewstack.io/honeycomb-addresses-flawed-systems/)，该软件从头开始设计，旨在调试实时生产软件，使用任何数据模型从任何来源消费事件数据，并鼓励协作解决问题。客户包括[挡泥板乐器](https://honeycomb.io/blog/2018/04/honeycomb-at-fender-faster-answers-happier-customers/)、 [CarGurus](https://www.cargurus.com/) 、[对讲机](https://www.intercom.com/)。

Lakhani 说，它最初的支柱是事件分析，但如果没有跟踪，它就不是一个完整的包。

他说:“这不仅仅是一个中断/修复平台，它还涉及了解整个系统的行为，并能够随着时间的推移优化性能。”

它使用户能够回答诸如“哪个东西最慢？”“哪个对用户影响最大？”

他说，大多数产品不提供来回查看整体情况和你真正想深入了解的具体地方的能力。

“对我们来说，差异是这些不同的看待事物的方式之间的整合程度，而不会破坏任何东西或让你在系统之间跳来跳去，”他说。

他是这样描述的:

有人在抱怨我们的一个 API 端点速度慢，但是你不知道是哪个。因此，您可以查看到达端点的所有内容、命中次数和最慢的命中次数。按缓慢排序，您可以看到一个 API 经历了非常高的延迟。您可以深入到该端点。它上升了半个小时，然后又下降了，所以肯定有事发生。这是一个特殊的要求还是导致这一切的一切？你看所有的请求。你可以看到这段时间我所有的请求都变慢了。

如果你有一个日志或监控产品，这通常是你能做到的最大限度。在世界上的大多数地方，你会打开你的追踪工具，在这个时间段开始采样，寻找匹配的东西，直到你偶然发现它。有了蜂巢，你可以直接跳到它上面。你可以通过点击来查看正常时期的痕迹。然后就可以看变态了。结果是一个 MySQL 查询一次又一次地命中端点。您可以准确地知道是哪个服务——如果您编写了这个服务，您就可以准确地知道应该查看哪个代码块。而有了蜂巢，你可以从用户 ID 的角度来看待一切。Honeycomb 不仅可以查明服务、代码块，还可以查明发出奇怪查询的用户。

今年 3 月，该公司宣布了分层存储，让客户决定他们希望获得高性能的数据量，以及将哪些数据放在较低的层。

这项新服务将成为蜂巢的一部分，免费提供给所有用户。

它完全兼容 [OpenTracing](https://thenewstack.io/opentracing-open-standard-distributed-tracing/) 标准和 [Zipkin](https://thenewstack.io/meet-zipkin-tracer-debugging-microservices/) ，微服务的开源追踪器，没有改变任何东西。

对于想要超越这一点的人，它提供了 Java、go、Ruby 和 Python 版本的 SDK。

对于那些没有工具的人来说，它创建了所谓的[直线](https://www.honeycomb.io/beelines/)，类似于 APM(应用程序性能监控)供应商为 Go、Node、Ruby 和 Python 提供的代理，Java 正在工作中。

“如果你什么都没有，我们给你一个开源软件，你可以从那里开始构建，”他说。

据该公司称，将事件和跟踪结合起来将可观察性放在每个软件工程师的手中，使他们能够检测和诊断自己代码的行为，并更快地交付软件。

Lakhani 说，它已经从主要与软件工程师互动发展到也与中层管理人员和主管合作，他们看到了蜂巢在帮助团队更快行动和做出更好决策方面的作用。

在这方面，未来 it 将致力于在平台中建立更多的协作。

“需要一种更好的方式让(团队成员)互相解释他们在蜂巢内部做什么，”他说。“你不仅需要跟踪你自己的面包屑，还需要跟踪你团队的面包屑，这样别人就可以说，‘我上周也遇到了类似的问题。去看看我的查询。"
