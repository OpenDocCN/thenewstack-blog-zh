# 敏捷编程:DevOps 的最后一英里

> 原文：<https://thenewstack.io/devops-plumbing-last-mile/>

随着 [DevOps](/category/devops/) 逐渐成熟，像 Chef 这样的 IT 自动化公司已经使自动化和管理发布管道变得更加简单。在 ChefConf 2016 上，Chef 公布了新工具，其中包括 [Chef Automate](https://thenewstack.io/chef-rolls-management-ingredients-one-package-chef-automate/) ，它将 Chef 的所有 IT 自动化工具整合在一个包中。随着 Slack、HipChat 等工具和 [ChatOps](https://thenewstack.io/hpes-chatops-aims-to-replace-dev-collaboration-platforms-with-slack/) 等流程的兴起，DevOps 团队在业务中与他人沟通的方式也发生了变化。

在这一集的[新的 Stack Makers](https://thenewstack.io/podcasts/) 播客中，我们将探索 Chef Automate 和 ChatOps 如何让 DevOps 团队更高效地工作，敏捷开发实践如何塑造了 DevOps，以及随着企业使用软件的方式发生变化，DevOps 文化如何演变。[电云](https://thenewstack.io/electric-cloud-automates-rolling-deployments/)首席技术官 [Anders Wallgren](https://www.linkedin.com/in/anderswallgren) 和 ChatOps 软件提供商 [VictorOps](https://victorops.com/) DevOps 布道者 [Jason Hand](https://www.linkedin.com/in/jasonhand24) 在 ChefConf 2016 上与 TNS 咨询工程师 [Lee Calcote](https://twitter.com/lcalcote) 和 TNS 执行主编 [Joab Jackson](https://www.twitter.com/Joab_Jackson) 就此播客进行了对话

[DevOps:管道最后一公里](https://thenewstack.simplecast.com/episodes/devops-plumbing-the-last-mile)

汉德解释说，虽然《厨师长栖息地》是这次活动的热门话题，但《自动化厨师长》却抢了 VictorOps 的风头。作为实时事件管理和基础架构时间线，Chef Automate 的工作流可见性功能尤其令人感兴趣。“我们可以帮助理解这一点，并提供更多的背景信息，为待命人员提供他们需要立即响应和采取行动的一切信息。“减少修复问题所需的时间是 DevOps 的重要话题之一，”Hand 说。

对于 Wallgren 来说，Chef 提供的各种集成对 Electric Cloud 的客户至关重要。由于电云是在 AWS 或 RackSpace 堆栈上本地运行的，Wallgren 指出，“我们必须与许多定制的本土系统合作。往往是因为十年前你买不到或者开源不了这个东西。人们建造了它，那些东西还在。”

随着企业改变了他们在自己的堆栈中查看和利用软件的方式，DevOps 也发生了变化。虽然敏捷软件开发实践已经可以大规模地创建应用程序，但是在 [DevOps 社区](https://thenewstack.io/tns-research-devops-factions-software-development-world/)中，敏捷运动还没有被广泛采用。Wallgren 对此给出了特别的理由，“我认为敏捷是我们在行业中所做的最后一英里。”

Hand 继续解释道，DevOps 这个术语本身就是从敏捷软件开发的概念中衍生出来的，并将其应用于基础设施。因此，正是敏捷将 DevOps 带到了今天的位置。Wallgren 用一段关于软件的强有力的结束语进行了总结，许多开发人员可能会对此产生共鸣，

“这么多年来，软件最大的问题之一就是我们试图将十磅重的垃圾装进一个五磅重的垃圾桶里。”
