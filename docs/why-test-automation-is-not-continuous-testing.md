# 为什么测试自动化不是连续测试

> 原文：<https://thenewstack.io/why-test-automation-is-not-continuous-testing/>

Tricentis 赞助了这篇文章。

就像 20 世纪 50 年代美国经典电影《我爱露西》中露西和埃塞尔[在巧克力工厂](https://www.youtube.com/watch?v=8NPzLBSBzPI)努力跟上步伐一样，许多软件测试人员都在努力跟上加速的过程——然后主管宣布，“你做得非常好……加快速度！”

随着与测试相关的期望的改变，遗留测试平台就跟不上了。遗留测试平台采用“繁重”的测试方法。它们依赖脆弱的脚本，交付缓慢的端到端回归测试执行，并产生大量的误报。结果，组织在测试自动化方面取得了有限的成功。企业的总体测试自动化率平均为 8%。在一个民意测验问题中，我们在行业网络研讨会和贸易展览中询问，绝大多数的回答者报告说测试自动化的结果到目前为止“一般”

整个行业最近的变化对测试提出了更高的要求，同时使得测试自动化更加难以实现。这有几个原因:

*   应用架构越来越分散和复杂，包括云、API、微服务等。并且在单个商业交易中创建不同协议和技术的几乎无限的组合。
*   由于敏捷、DevOps 和持续交付，许多应用程序现在每两周发布一次，甚至一天发布数千次。结果，可用于测试设计、维护，尤其是执行的时间急剧减少。
*   既然软件是业务的主要接口，那么应用程序故障就是业务故障。这是真的，即使是看似很小的故障，如果它影响了用户体验，可能会有严重的后果。因此，与应用程序相关的风险甚至已经成为非技术型企业领导的主要担忧。

鉴于软件测试人员正面临越来越复杂的应用程序，他们被期望以现代商业的新速度交付可信的去/不去决策。更多相同的传统测试方法不会让我们达到目的。我们需要像改造开发过程一样，有意识地、显著地改造测试过程。这需要更有效的测试自动化…等等。这需要一种叫做持续测试的不同方法。

## 什么是持续测试？

[持续测试](https://www.tricentis.com/products/what-is-continuous-testing/)是执行自动化测试的过程，作为软件交付管道的一部分，以便尽快获得与软件发布相关的业务风险的反馈。连续测试不需要任何特定类型的测试方法(左移，右移…)或测试工具。但是，它要求:

*   在正确的时间向正确的利益相关方提供可操作的反馈。
*   测试发生在软件交付管道的所有阶段。

测试自动化旨在产生一组与用户故事或应用程序需求相关的通过/失败数据点。另一方面，持续测试关注业务风险，并提供关于软件是否可以发布的洞察力。为了实现这一转变，我们需要停止问“我们完成测试了吗”，而是集中于“发布候选具有可接受的业务风险水平吗？”

以下是持续测试的五个关键属性:

*   评估业务风险范围是其主要目标。
*   建立一个安全网，帮助团队保护用户体验。
*   需要一个稳定的测试环境，以便随时可用。
*   无缝集成到软件交付渠道和 DevOps 工具链中。
*   提供适用于交付渠道每个阶段的可操作反馈。

## 比较连续测试和测试自动化

持续测试和测试自动化之间的主要区别可以分为三大类:风险、广度和时间。

## 风险

今天的企业不仅向最终用户公开了他们的许多内部应用程序，他们还开发了大量的附加软件来扩展和补充这些应用程序。例如，航空公司已经不仅仅是公开他们曾经的内部订票系统。他们现在让客户计划和预订完整的假期，包括酒店、租车和活动。向用户展示越来越多的创新功能现在是一个竞争优势，但它也增加了潜在故障点的数量、种类和复杂性。

大规模的“软件故障”具有如此严重的业务影响，以至于与应用程序相关的风险现在是企业公开财务申报的重要组成部分。鉴于[显著的软件故障](https://www.tricentis.com/software-fail-watch/)导致股价平均下跌 4.06%(这相当于市值平均亏损 25.5 亿美元)，业务领导者注意到这一点并期待 it 领导者采取行动就不足为奇了。

如果您的测试用例没有考虑到业务风险，您的测试结果将不会提供评估风险所需的洞察力。大多数测试被设计来提供关于用户故事是否正确地实现了需求的低级细节——而不是一个发布候选是否风险太大而不能发布的高级评估。你会根据测试结果自动停止发布吗？如果没有，您的测试就没有与业务风险恰当地结合。

需要澄清的是:我们并不是说低粒度测试没有价值；我们声明需要更多的措施来阻止高风险的候选人进入野外。

以下是测试人员应对风险的一些方法:

*   了解与整个应用程序组合相关的风险。
*   将风险映射到应用程序组件和需求(然后映射到测试)。
*   使用一个测试套件，用最少的测试用例达到尽可能高的风险覆盖率。
*   始终报告从业务、技术、性能和合规性角度显示风险暴露的状态。

## 宽度

即使一家企业设法避开了成为头条新闻的大规模软件故障，如今即使看似很小的故障也可能造成麻烦。如果用户体验的任何一部分没有达到预期，你就有可能被竞争对手抢走。如果用户决定将问题暴露给社交媒体，你也面临品牌受损的风险。

仅仅知道一个单元测试失败了或者一个 UI 测试通过了，并不能告诉你整体用户体验是否受到了最近应用程序变化的影响。为了保护最终用户的体验，运行足够广泛的测试，以检测应用程序更改何时会无意中影响用户已经开始依赖的功能。

以下是解决测试广度的一些方法:

*   定义并执行完整的端到端测试，从用户的角度测试应用程序
*   为关键用户交易中涉及的所有技术(web、移动、消息/API 层、SAP 和打包应用程序等)提供集成支持。).
*   为执行完整的端到端事务所需的依赖组件模拟服务虚拟化，这些组件对于重复测试既不可用也不可配置。
*   确保每次执行测试时，测试和服务虚拟化资产都填充了真实有效的数据。
*   执行探索性测试，以发现自动化测试范围之外的用户体验问题(例如，可用性问题)。

## 时间

既然组织交付软件的速度已经成为一个竞争优势，绝大多数组织都转向敏捷和开发运维来加速他们的交付过程。

当自动化测试出现时，它专注于测试根据瀑布开发过程构建和更新的内部系统。系统都在组织的控制之下，在测试阶段准备开始时，一切都已完成并准备好进行测试。既然敏捷过程正在成为规范，测试必须与开发并行开始；否则，用户故事不太可能在极度压缩的迭代时间框架内(通常是两周)被测试并被认为“完成了”。

如果您的组织已经采用了 DevOps 并且正在执行连续交付，软件可能每小时发布一次——甚至更频繁。在这种情况下，过程中每个阶段的反馈都不能只是“快”；它必须几乎是瞬间的。如果质量不是您的应用程序最关心的问题(例如，如果在产品中发现缺陷时进行回滚的影响很小)，那么在每个版本上运行一些快速的单元测试和冒烟测试就足够了。然而，如果企业想要最小化错误软件到达最终用户的风险，您需要一些方法来实现必要的风险覆盖水平和测试广度——快速。

对于测试，有几个重要的影响:

*   测试必须成为开发过程的组成部分(而不是开发完成后附加的“卫生任务”)。
*   相关功能一实现，测试就必须准备好运行。
*   组织必须有一种方法来确定在交付管道的不同阶段执行的正确测试(签入时的冒烟测试、集成后的 API/消息层测试以及系统级的端到端测试)。
*   每组测试必须足够快地执行，以便不会在软件交付管道的相关阶段产生瓶颈。
*   需要一种稳定测试环境的方法来防止频繁的更改导致大量的误报。

基于上述标准，以下是测试人员应对时间压力的一些方法:

*   确定哪些测试用例对于解决顶级业务风险至关重要。
*   随着应用程序的不断变化，定义和发展测试。
*   重新平衡测试金字塔，使大多数测试在 API 层执行，这比 UI 测试执行至少快 100 倍。
*   将测试集成到交付管道中。
*   根据需要，在多个虚拟机、网络计算机或云中运行分布式测试。
*   利用服务虚拟化和合成数据生成/TDM，以便测试不需要等待数据或环境供应。

## 持续测试>测试自动化

如果您只从这一部分中得出一个想法，我们希望是这个:

测试自动化≠连续测试

持续测试>测试自动化

当他们的组织采用现代的架构和交付方法时，即使是使用传统的测试自动化工具取得了相当程度的成功的团队也会遇到严重的障碍:

*   他们不能足够快或足够频繁地创建和执行真实的测试。
*   应用程序的不断变化会导致大量的误报，并且需要永无止境的测试维护。
*   他们不能提供关于发布候选是否风险太大而不能通过交付管道的即时洞察力。

重要的是要认识到，没有任何工具或技术可以立即“给予”您持续的测试。像敏捷和 DevOps 一样，持续测试需要人员、过程和技术的变化。然而，当你的技术不能胜任这项任务时，试图在人员和流程上发起相关的改变，从一开始就是一场艰苦的战斗，而且最终会失败。
