# 为什么 Dynatrace 说人工智能是云原生“单面板”可观察性的答案

> 原文：<https://thenewstack.io/why-dynatrace-says-ai-is-the-answer-for-single-panel-observability-for-cloud-native/>

如今，DevOps 团队成员的生活经常充满了对大量 bug 和安全漏洞的追踪和优先排序。除了不得不花费宝贵的时间来筛选大量误报，同时匆忙决定哪些需要首先修复之外，您还经常不得不使用单独的应用程序性能管理(APM)、可观察性和监控界面，这往往只会增加混乱。

检测只是补救开始前流程的第一部分。最终，许多 DevOps 团队成员可以将大部分时间花在检测和修复 bug 和安全漏洞上。

DevOps 困境的一个潜在解决方案是由人工智能(AI)控制的系统，由 [Dynatrace](https://www.dynatrace.com/) 开发，作为单一面板进行监控和观察，并自动化应用程序管理流程。

在上周的 [Perform 2020 大会](https://www.dynatrace.com/perform-vegas/)期间，Dynatrace 宣布，它已经扩展了对在 Kubernetes 上部署[这些新功能](https://www.dynatrace.com/#)的支持。该公司表示，Dynatrace 的人工智能引擎现在可以自动吸收 Kubernetes 事件和指标，从而能够检测和处理 Kubernetes 集群、容器和工作负载的性能问题和异常。

企业管理协会(EMA) 的分析师 [Torsten Volk](https://www.linkedin.com/in/torstenvolk) 说，这些功能对开发人员来说可能很重要，尤其是在 CI/CD 场景中，因为他们花了太多时间对代码进行监测和跟踪。“Dynatrace 自动为移动应用程序和 Kubernetes 注入所需的工具，这样开发人员就不必再担心为了监控而修改和管理代码了，”Volk 说。

此外，新的自动移动应用程序工具帮助开发人员“更容易地调试代码”，Volk 说。“开发者现在可以看到故障的背景——可能故障只发生在 LTE 高速行驶的拥挤列车上，也可能只在特定版本的 Android 或 iOS 上出现，”Volk 说。“这种在上下文中访问单个故障或故障组并接收完整堆栈跟踪的能力，使得基于实际用例的应用程序调试和增强变得更加容易。”

如上所述，Dynatrace 一直在开发其[平台](https://www.dynatrace.com/news/blog/perform-2020-dynatrace-software-intelligence-platform-updates/)，重点强调可观察性、监控和应用管理的人工智能技术，特别是针对微服务和容器环境。例如，Dynatrace 的平台涵盖自动仪器和全栈监控，同时利用人工智能自动解决操作和开发问题，Volk 说。“大约一年前，Dynatrace 基本上改写了他们的整个投资组合，”Volk 说。

单面板模型是平台中的一个关键组件。“拥有一个集中的仪表板，让您可以跟踪延迟问题的影响和根本原因，这将让您根据完整的信息做出合理的决定，无论您是否应该将稀缺的资源用于解决问题，”Volk 说。“例如，仪表盘显示应用程序是否正在使用，您甚至可以准确地看到谁正在使用它，并决定该用户群是否值得立即关注，因为他们因为对额外的重量感到沮丧而放弃了高价值的购物车，或者他们是否可以称重，因为他们不是您的目标客户。”

有时，DevOps 发现应用程序延迟可能是由“昨晚的软件发布”中的错误查询引起的，因此使用 Dynatrace 的平台，“你可以简单地回滚，让开发团队优化他们的查询代码，”Volk 说。

“也许红灯源于外部微服务的暂时减速或中断，例如支付网关，在那里你所能做的就是打一个愤怒的电话，等待你的支付提供商解决他们的问题，但绝对没有理由在半夜叫醒开发人员来帮助解决问题。或者可能是由于合法的基础设施问题，如磁盘性能或 CPU 利用率或内存不足，”沃尔克说。“如果能知道您是应该购买更多硬件还是硬件很好，这不是很好吗？但是问题源于新的开发团队，他们正在使用大量临时请求来测试 API 的负载测试。你只需告诉他们按照不同的时间表进行测试，所有的灯就会再次亮起。”

Dynatrace 是新堆栈的赞助商。
