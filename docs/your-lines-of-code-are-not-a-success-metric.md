# 你的代码行不是成功的标准

> 原文：<https://thenewstack.io/your-lines-of-code-are-not-a-success-metric/>

*每一行代码与其说是资产，不如说是负债。*

当你买房时，你是在投资一个正在进行的项目，这个项目会让你忙于渐进式的改进、持续的升级和不经常但必不可少的大规模翻新。开发和维护软件类似于买房，因为你处于不断变化的状态。

一所房子提供最基本的庇护和吸引居民的建筑设计——这两者的结合使房子值得居住。你必须保持房子的结构完整性，让它屹立不倒，但你不能忽视让房子独特迷人的美学。

同样，软件提供了关键的结构、工具和连接，它必须不断更新新的特性和错误修复，以保持其完整性，并继续使其对用户具有独特的价值。

当买家投资买房时，他们必须权衡房产是负债还是资产。使用一系列软件代码*的公司应该*以大致相同的方式对待它。然而，这并不总是反映在开源社区和软件开发中——如果你问我，一行软件代码更像是负债而不是资产。

多余的软件代码不是成功的标准。

一行代码并不是一个社区在开源领域成功的衡量标准。所有类型的企业都需要了解，软件中的代码行越多，无论是专有代码还是开源代码，改变你的项目就越复杂和昂贵。

使用代码行这样的虚荣心度量来跟踪和分享软件的健康状况是一个糟糕的方法。虚荣度量与正确的软件工程是对立的，正确的软件工程是关于降低复杂性和减少代码行。(事实上，我认为只有在你积极尝试减少而不是增加代码行的时候，跟踪代码行才是合理的。)

为什么一行代码与其说是资产，不如说是负债？这是一个持续的工作来源。开发人员、团队、公司或社区必须维护的每一行代码都会引发额外的工作，必须完成这些工作才能使代码保持最新。

*钻头腐烂，就像木头腐烂一样，会不被发现，直到为时已晚。*

当软件代码被忽略，并且一行代码所依赖的操作系统或库被更新时，它会导致一系列事件，这些事件会破坏与其他系统的集成。这种现象——“比特腐烂”——往往不被发现，直到为时已晚。

仍然停滞不前的软件，尤其是在网络空间或现代软件架构中，将很快经历比特腐烂。在分布式 web 系统的蛮荒西部，你周围的一切都在变化，所以如果你不去管那些代码，它最终会腐烂并停止工作。

多余的代码会带来短期和长期的挑战。代码行数越多，出现比特错误的机会就越多，这使得开发人员很难进行修改。

软件之所以有价值是有原因的，但是如果你不给予它所需要的持续关注，你的整个操作可能会在一条死胡同里走向毁灭。想象一下，一栋房子刷了一层全新的油漆，但木头腐烂侵蚀了它的地基。开发人员和企业理所当然地为他们构建的软件感到自豪，但是当难以放弃您的代码时，这种自豪可能会被误导——即使它需要更少的维护，因此维护责任也更少。

拥有所有权的自豪感必须得到锻炼。

这种心理表现在几个方面。一种方式是不愿意定期剔除代码以降低复杂性。另一种被称为“不是在这里发明的”综合症，在这种综合症中，软件工程师团队认为外部选项“不够好”，并决定自己构建它。这困扰着企业和开源社区，迫使他们花费更多的资源来构建可以声称是他们自己的东西。

如果软件不是独一无二的，或者不是你组织需求的核心，可以从其他地方获得，那么你应该毫不犹豫地采用它。开发人员可能不得不在内部构建软件来启动项目，但是他们需要愿意在以后放弃那些代码，以获得同样足够的软件，更重要的是，由其他人集体维护。

如果一行代码不是你的核心任务，你应该把它外包给能更有效地构建和维护它的人，让你的团队专注于更重要的内部工作。如果通用软件项目中的代码重用可以适应您的架构并解决这一需求，为什么不利用它呢？

放开代码是这个旅程中极其重要的一部分。我们大多数人甚至不再考虑构建自己的服务器——或者自己的房子。我们买了一栋已经建好的房子，如果我们想扩建，我们会雇佣承包商来做。当必要但与任务无关的功能作为服务可用时，企业会抢购这些功能。同样的思维模式应该适用于软件层。

额外的代码行不是进步的标志。没有人会对十倍的代码增长感到惊讶。进步就是放开我们的束缚，尽可能减少软件中的代码行。将它视为软件健康发展的一部分。代码行是一种负担——让我们开始这样对待它们。
