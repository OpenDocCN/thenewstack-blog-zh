当人们听到“原生云”这个术语时，他们通常会想到公共云。这种隐含的公共云偏见延伸到了术语“无服务器”和“基础设施即代码”但是有了 [Knative](https://knative.dev/) 和其他开源软件，如 [Kubeless](https://kubeless.io/) 和 [OpenFaaS](https://www.openfaas.com/) ，您可以获得无服务器的所有成本降低和灵活性提升的优势，而没有“提升和转移”(将工作负载从内部迁移到云的过程，很少或没有修改)的风险和成本。

当前的经济环境点燃了许多 it 和应用团队积极向云迁移的热情。这些举措在新冠肺炎会议之前就已经在进行了，但现在它们被加速了。

然而，提升和转移策略带来了巨大的成本和风险。另一种方法——称之为混合转移或“翻新”——利用 Kubernetes 和 Knative 的可移植性优势，让您能够快速地逐个应用程序、逐个服务地实现现代化。这种方法的优势包括降低您选择云的风险并延长传统系统的使用寿命，同时仍然享受云的所有固有优势，如更低的计算成本和将应用程序转向事件驱动的基础。

以下面的场景为例。您已经有了一个现有的内部电子商务应用程序，并希望对其进行现代化改造，以利用新的 SaaS 参与平台，如 Amazon Lex、Slack 或 Mattermost。不需要提升和移位来实现这一点；不再是了。

## 降低云选择的风险

云原生和无服务器领域的创新数量掩盖了领导者的市场份额增长。但是云迁移是一个很大的战略赌注。理想的场景是测试几个服务来评估性能、特性和支持。但是，当每个供应商的学习曲线陡峭而独特时，这可能是不切实际的。这是 Kubernetes 如此重要的主要原因之一。它已经成为云的一致结构，能够使用任何支持 [CloudEvents 规范](https://cloudevents.io/)的云来运行工作负载和构建事件驱动的无服务器应用。

## 延长传统系统的使用寿命

在收入增长不确定的时期，从现有系统中挤出额外的年数很有吸引力。也就是说，只要它不损害您通过云计算和基础设施代码获得的成本节约和更高的响应能力。在 Kubernetes 出现之前，这是企业面临的不幸权衡。然而今天，你不需要把婴儿和洗澡水一起倒掉。

## 降低计算成本并提高利用率

通过从虚拟化或裸机迁移，您可以提高硬件的利用率；通过添加 Knative，您可以获得无服务器零扩展的优势，从而进一步提高利用率。您显然希望在云中构建新的应用程序，但是许多组织已经在核心遗留系统上进行了数年甚至数十年的投资。现在，通过使用 Kubernetes 从虚拟机迁移到容器，然后使用 Knative 迁移到无服务器，您可以看到同样的基础架构节省—所有这些都在您的防火墙之后。

请放心，你会在这条道路上得到很好的支持，Knative 的[顶级贡献者就是证明:](https://knative.dev/community/annual_reports/)

## 将应用转向事件驱动

对不断变化的消费者偏好的实时响应意味着它需要由事件驱动。只有这样，您才能在应用程序架构中构建必要的灵活性，以适应不确定的未来。传统的云原生思维会让你认为你的应用程序必须 100%在云中才能做到这一点。并非如此。Knative 通过在不同架构之间共享事件，能够从其他云或您的传统数据中心应用程序中触发功能和/或 Kubernetes 工作负载。
