日益复杂的体系结构加上客户对一致可用性的需求，加速了对随时可用、可靠的软件和基础架构的需求。客户期望高质量和可靠的服务，但他们也希望创新。

持续集成和持续交付(CI/CD)是一个基本的 DevOps 策略，用于构建可靠和一致的产品交付生命周期。一个良好的 CI/CD 管道允许敏捷团队拥有快速的反馈循环，以更快的速度部署应用程序，并允许一天发布几个版本。

在本文中，您将了解 CI/CD 管道可观察性的含义，如何监控您的 CI/CD 管道，以及您需要记录的指标。最后，您将发现 CI/CD 数据提供的价值。

## 为什么可观察性很重要

一个典型的软件团队由来自不同学科的人组成，他们跨职能地工作来构建、交付和部署新的服务和特性。随着与构建软件相关的不断变化，快速了解 CI/CD 管道中发生的事情是一项挑战。

复杂的软件通常需要包含多种服务、工具和集成平台的复杂管道。管道中某个服务的失败或错误可能会中断您发布新功能的能力。作为一名 IT、SRE、运营工程师或经理，重要的是你的团队能够从 CI 渠道了解正在发生的事情，以便他们能够识别代码中的问题并快速解决它们。

持续的可观察性对于理解预生产中可能发生的问题至关重要。可观察性允许您收集指标，跟踪 CI/CD 管道的每一步，并保持对出现的任何风险或问题的控制。通过可观察的 CI/CD 管道，软件团队可以在整个开发和运营生命周期中确保其基础设施和应用的可靠性、性能和健康。

## CI/CD 管道可观测性的好处

当管道中出现问题时——这种情况经常发生——您需要了解问题所在。持续可观察性对于任何想要了解 CI/CD 管道中的问题的软件团队来说都是一个一站式解决方案。

没有 CI/CD 管道，一个组织就不能有效地运作；通过观察 CI/CD 管道，软件团队可以快速识别并响应软件开发过程中遇到的问题。最终，一个完全可观察的管道会给你的代码带来更多的信心，并能更快地修复。

以下是可观察到的 CI/CD 渠道的一些主要优势:

*   **更好的故障检测和隔离:**通过分析和汇总日志和指标，软件团队可以快速识别和修复系统缺陷，发布安全补丁，并优化管道以实现快速发布和更好的软件交付。
*   **快速确定最佳指标:**可观察的管道帮助您确定最能代表应用程序性能的指标，并避免您的团队盲目行动。
*   **提高客户满意度:**通过自信地构建和部署满足需求的可靠应用程序，您将提高客户满意度。
*   **更短的反馈循环:**管道中的可观察性允许你在铺开之前得到反馈，而不是等到你的代码投入生产。

可观察性的三个支柱之一是度量。没有正确的指标，就没有可观察的 CI/CD 渠道。在下一节中，我们将深入研究 CI/CD 可观察性的最重要指标。

## 要跟踪的关键 CI/CD 指标

真正的 DevOps 成功意味着增强您的团队的敏捷性，并使他们能够快速交付高性能、可靠的软件，从而提供卓越的客户体验。为了实现这些目标，您需要在整个 CI/CD 管道中跟踪正确的指标集。

尽管您可以收集多种指标来跟踪 CI/CD 管道的性能，但以下指标类别是一个很好的起点:

### 质量指标

在努力快速创新和频繁发布的过程中，组织往往最终部署了不符合标准的应用程序。随着组织向敏捷软件发布周期的过渡，构建和部署高质量产品的需求增加了。毕竟，没有一个团队想在部署一个应用程序后追逐生产大火。

这些质量度量将帮助您的团队建立对代码的信心:

*   缺陷逃逸率
*   bug 数量
*   部署成功率
*   测试通过率

### 基于时间的指标

基于时间的指标对于评估 CI/CD 管道的生产力和工作流程非常有用。通过记录和跟踪基于时间的指标，您可以了解构建和部署新功能和产品需要多长时间。

一些基于时间的观察指标包括:

*   部署时间和频率
*   缺陷解决时间
*   上市时间

跟踪这些基于时间的指标还将使您能够估计解决一个问题需要多长时间，并确定 CI/CD 管道中的瓶颈。

### 自动化指标

软件团队需要量化他们的自动化测试过程的性能、价值和影响，尤其是在部署过程中。

组织可以通过跟踪自动化度量来确定加速他们的测试自动化工作的更好的方法，例如:

*   部署频率
*   失败的部署
*   部署规模

[CI/CD 自动化](https://thenewstack.io/3-ways-to-use-automation-in-ci-cd-pipelines/)指标还允许您评估每个管道阶段的技术债务、部署成功率和总错误率。

现在，您已经了解了质量、基于时间和自动化指标，以及每个类别中的指标，下一节将让您了解促进开发运维成功的关键因素。

## DevOps 成功的关键驱动因素和 KPI

软件开发团队旨在加速软件开发过程，同时交付客户喜爱的高质量应用程序。设置度量标准来衡量您的业务目标将让您知道您的团队是否正在实现这些目标。如果没有明确定义的绩效指标，您将不会知道您的团队做得有多好，也不会知道您从 DevOps 投资中获得的价值。

通过跟踪以下 DevOps KPIs，您的团队可以准确衡量绩效并做出数据驱动的决策，以改善 CI/CD 渠道、应用或基础设施。

### 商业成功

敏捷宣言[的第一个原则](https://agilemanifesto.org/principles.html)声明每个企业都应该致力于“通过早期和持续交付有价值的软件来满足他们的客户”你可以把有价值的软件看作是一种有助于降低客户服务成本、增加收入或扩大客户群的产品。

根据公司的成熟度、行业或市场份额，你可能会关注不同的绩效指标来衡量企业的成功。这些关键绩效指标包括:

*   增量或经常性收入
*   客户获取成本
*   每个用户的平均收入
*   转化率

跟踪业务成功 KPI 还会增强你的信心，让你相信你在做正确的事情。

### 客户体验

在开发和部署应用程序时，您需要确保您的产品是可靠的，从而带来更好的用户体验和更高的客户满意度。如果达不到这些目标，你就有可能将客户流失到生产更好产品或提供更好客户体验的竞争对手那里。

了解消费者是否喜欢使用你的产品的唯一方法是测量和跟踪 KPI，比如:

*   用户增长率
*   客户满意度
*   用户在应用上花费的总时间
*   您的软件中每个用户每周的访问次数

### 应用和基础设施性能

跟踪应用程序和基础设施中的性能 KPI 是您的团队在性能问题出现之前捕捉它们的最佳方式。

要了解您的应用和基础架构的性能，您需要跟踪以下性能指标:

*   资源利用
*   应用程序响应时间
*   第三方服务绩效
*   数据库响应时间

应用和基础设施性能、业务成功和客户体验是 DevOps 成功的关键驱动因素。如果您不监控和衡量这些 KPI，您的团队将不会知道他们做得对，以及他们的 DevOps 策略哪里需要改进。

所有这些都指向一个逻辑问题:如何让 CI/CD 管道变得可观察？

## 使 CI/CD 可观察

为了使您的管道可见，您的团队需要协作并建立操作手册、警报系统和监控工具，以便更好地监控 CI/CD 管道和事件检测。正确的检测策略将允许您的团队快速检测代码中的问题、事件或漏洞。

此外，确定谁负责什么使您的团队能够明确任务的所有权并及时解决问题。许多组织遵循这样的原则，即提交变更的人负责提供修复，而不是让一个团队来修复从 CI 中观察到的问题。

一旦事件得到解决，您需要进行一次彻底的、无可指责的事后事件回顾，以检查什么没有起作用，什么起作用。之后，您可以实施新的系统，更新您的操作手册，并为将来更快的补救创建资源。

最后一步是为你的团队提供正确的观察工具(如[桑德拉 APM](https://www.thundra.io/apm) )来识别潜在的问题，并处理你的 CI/CD 管道中未知的未知。

## 最后

在这篇文章中，我们看到了 CI/CD 管道可观察性的重要性，以及为什么每个团队都需要建立可观察性驱动的文化。随着技术领域的快速变化，可观察的 CI/CD 管道对于 DevOps 的成功至关重要。

无论您是 IT 或工程经理、CTO 还是 CIO，CI/CD 管道可观察性都可以帮助您。通过记录度量和跟踪本文中提到的 KPI，您可以快速评估团队工作的成功程度。这些实践将培养同理心和协作文化，有助于确保在您的组织内成功实施 DevOps。
