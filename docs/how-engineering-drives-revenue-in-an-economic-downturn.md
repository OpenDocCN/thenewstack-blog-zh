# 工程如何在经济低迷时期推动收入

> 原文：<https://thenewstack.io/how-engineering-drives-revenue-in-an-economic-downturn/>

作为站点可靠性工程(SRE)实践的一部分，软件即服务提供商 BetterCloud 在 [重振其事件管理](https://www.blameless.com/case-study/bettercloud) 后，客户流失率降低了 10%。

担心通货膨胀和可能的经济衰退的公司正不惜一切代价在这个充满挑战的时期保持良好的信誉。

工程部门通常被视为成本中心，大多数公司会说削减预算是公平的，假设他们还能“生存”

然而，事实远非如此。

一个健康的软件工程团队通常是公司兴旺和公司失败的区别。虽然很难，但最好抵制削减成本，因为这样做会抑制实现更显著收益的潜力。我想起了一句谚语“小事聪明，大事糊涂”。

当您考虑产品可靠性时，尤其如此，因为它需要工程所有领域的投资，并且它对商业成功变得越来越重要。这种方法被称为站点可靠性工程，致力于优化正常运行时间，减少错误或缓慢的性能，这两者都会转化为糟糕的客户体验。

## **投资增加工程人才的工具**

你不想让更多的工程师来解决这个问题，考虑到“大辞职”或“安静辞职”以及普遍的软件工程人才短缺，这已经很难了。衰退的市场是有远见的公司明智地投资于工具的绝佳机会，这些工具通过更简化的工作流程提高生产率，从而增加人力资源投资并减少倦怠。

简而言之，可靠性工程帮助公司提供更好的用户体验。满意的客户保持忠诚，增加收入，并证明工程部门是一个利润中心。

没有一家公司承诺 100%的正常运行时间(也没有一个理性的用户期待它)。客户忠诚度和增加收入的潜力源于对你品牌的信任，这种信任是日积月累建立起来的。它有助于服务可靠性水平的透明化。归结起来就是如何处理和解决事故，以及在事故期间与客户的沟通。有弹性的工程团队应该很快知道事故何时发生，最好是在事故影响到用户之前。通过快速行动，他们可以立即组建一个团队来处理这种情况，并提供持续的更新。更进一步，有弹性的工程师总是想从事件中学习。通过研究回顾性报告，他们可以为未来改进系统。

## **当前播放状态**

技术领导者必须向组织的其他成员传达服务可靠性对其业务的影响。如何证明更可靠的产品对成功至关重要？您可以从证明不太可靠的产品是成功的重大障碍开始。

**事件激增:**日益增加的复杂性、云原生的数字服务以及在竞争激烈的市场中部署新功能的压力，令人头脑发热。在某些情况下，可靠性是不稳定的，因为较大的组织正在从传统的技术堆栈迁移到现代的原生云，这可能会导致下游问题或与 API 和其他第三方系统和环境连接相关的挑战。根据行业领域的不同，更多的工程团队也将面临严格的合规性法规，或者面临越来越多的网络安全威胁。

**找工程师的麻烦:**虽然“大辞职”已经影响了所有类型的行业和工作职能，但工程师尤其是成群结队地离开。有些人选择了新的职业；其他人仅仅是因为压力增加以更高的速度输送而筋疲力尽。其他人厌倦了淹没在技术债务和/或陷入调试周期中，这不是最有趣或最有利于职业发展的工作。

缺乏品牌忠诚度:疫情加速了一个企业已经开始注意到的趋势:消费者和 B2B 买家的品牌忠诚度比以往任何时候都低。品牌转换可能是成本驱动的，但许多买家也会转换，因为他们对产品承诺感到失望。停机次数越来越多的 SaaS 供应商，尤其是那些损害客户业务的供应商，将很难留住他们。BetterCloud 的云工程和优化副总裁 Stephen M. Dick 亲眼目睹了可靠性工程对客户维系的影响。

【BetterCloud 实施“无可指责”一年后，通过 [重振其事件管理](https://www.blameless.com/case-study/bettercloud) 并采用 SRE 的实践，他们看到客户流失率降低了 10%。使用“无可指责”最终帮助 BetterCloud 对产品可靠性和他们新的事件响应流程充满信心。实施无可指责导致了更多记录的事件，为更有意义的度量铺平了道路，并确保了回顾(事后)报告的完成。该平台建立了一个跨组织的集中沟通方法，因为每个人都在一个标准化的流程上保持一致。

企业不仅可以解决上述问题，还可以在衰退期间创造创收机会。对于资源匮乏的工程团队来说，最好的解决方案是确定哪些流程可以实现自动化，以造福员工和整个公司。通过这种方式，工程团队始终专注于数字服务的性能、可用性以及被认为是可接受的可靠性水平。一旦很好地理解并同意这一点，工程师就可以努力改进如何通过技术工具和熟练资源的组合来提供该级别的性能。

以下是工具、自动化和正确的数据洞察在面临这些挑战时如何提供帮助:

### **创建一个行动手册:简化你如何管理事件、学习并避免重复**

通常，事件管理响应是一个拼凑的过程，有些已经过时，并且通常由特定的团队成员或拥有部落知识的“英雄”来运行。最终，当他们离开公司时，如果这个过程没有被整理成文，这些知识将永远消失。一个编码的系统加上一个共享的工具可以体现一个通用的过程，这个过程比一些主题专家的任期还要长。它还加快了新员工的入职流程，因为该流程是可发现的。

### **在影响业务的事件中，在全公司范围内分享有用的数据见解**

系统停机会影响整个公司，而不仅仅是工程或开发团队。当发生严重事故时，工程师应该集中精力解决它。不幸的是，向高管和客户等利益相关者发送信息和更新会分散他们的注意力。Comms automation 通过预构建的消息模板和为各种情况保存的收件人解决了这一问题。这不仅大大加快了解决问题的速度，而且还节省了工程师大量的精神带宽。BetterCloud 的 Dick 定期与 BetterCloud 董事会分享可靠性数据。他的同事， [Clark Polo 是 SRE 和 DevOps 的主管，他说](https://info.blameless.com/bettercloud-case-study) “我们将事件管理流程作为改善 BetterCloud 的一种方式。如果我们不知道我们今天在哪里，我们就不知道我们需要去哪里。”

### **让工程师保持新鲜和专注**

没有自动化事件管理的公司倾向于在问题上投入更多资源，这意味着他们可能会在晚上叫醒一批工程师来解决棘手的问题。AIM 使用高级智能进行交流，并指派该领域的专家工程师。智能处理可以获得解决问题所需的正确资源，从而降低员工倦怠的风险和潜在的更多成本，而不是使用每个工程师都有责任回复的全员电子邮件。

### **创造一种可靠的文化**

当系统出现故障时，这从来都不容易。事件响应的紧迫性需要集中注意力。您可以通过将管理任务交给可信的自动化工具来实现这一点。让每个人都专注于任务，更新并与集体愿景保持一致。让顾客满意，让生意兴隆。正如 Machinify 运营高级副总裁大卫·莱文杰(David Levinger)在谈到他的团队时所说，“我们雇佣了非常优秀的工程师，他们勤奋努力，希望帮助系统改进。我们团队中遇到事故的人没有一个不知道该做什么、去哪里以及如何解决问题。”

一家公司如何沟通和应对突发事件可能是失去业务和建立更牢固的关系以带来更多业务的区别。系统可靠性对于组织来说太重要了，不能依赖临时或手动流程，尤其是当自动化可以减轻工作量、快速响应并保持一定程度的一致性时。当问题最好用正确的技术解决时，不再需要雇佣额外的工程师来解决。允许工程师自由地处理重要的事情，并观察智能自动化处理其余的事情。
