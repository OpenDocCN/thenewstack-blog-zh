# 加起来:自动化安全测试的现实检查

> 原文：<https://thenewstack.io/add-it-up-reality-check-on-automated-security-testing/>

有限的自动化和无效性是应用安全测试市场面临的两个问题。

正如我们在前面的中提到的[，工具并不能解决大多数正在进行的安全测试。](https://thenewstack.io/testing-automation-perspectives)[网络安全内部人士 2018 年应用安全报告](https://crowdresearchpartners.com/portfolio/application-security-report/)提供了更多细节，显示只有 43%的信息安全专业人士的组织在其软件发布生命周期中进行了自动化安全测试。尽管大多数组织都有一些开发运维流程，但实际上，自动化安全测试并没有部署在大多数 CI/CD 管道中。此外，即使在拥有自动化测试的组织中，自动化测试的数量仍然有限。

【2018 年 DevSecOps 社区调查发现，67%拥有成熟 DevOps 流程的组织在软件开发生命周期(SDLC)的 QA/测试步骤阶段实现了自动化应用安全分析，而只有 13%拥有不成熟 DevOps 流程的组织在该阶段实现了自动化。这些发现排除了处于发展过程中间的公司。我们认为 DevSecOps 报告夸大了实际自动化的程度，因为自动化单一测试不同于进行信息安全团队提供的完整渗透和漏洞测试。除了开发和运营之外，网络安全内部人士报告指出，46%的自动化测试发生在正式测试阶段，25%发生在代码开发阶段，30%在生产中对应用进行自动化监控。

当安全测试做得正确时，它会极大地改善组织的安全状况。将安全性转移到 SLDC 流程中有助于降低安全漏洞的补救成本。扫描开源组件的已知漏洞也是一个胜利。然而，安全测试并不是万能的。在最近的 [DZone 调查中，](https://dzone.com/guides/security-defending-your-code)48%的受访者声称安全测试是不充分的，即使对于实际上已经测试过并且没有被忽略的应用程序也是如此。

安全性测试的有效性受到许多因素的限制，这些因素超出了本专栏的范围。也就是说，尽管对这个问题有口头承诺，但由于时间限制，开发人员并没有优先考虑安全性。根据 DevSecOps 的调查，48%的开发人员知道安全性很重要，但没有足够的时间花在这上面。这产生了真正的后果，因为去年 [DZone 调查的 60%的开发者说发布时间表已经超越了他们组织的安全顾虑。](https://dzone.com/guides/proactive-security-apps-environments-and-messaging)

在充分[将安全性集成到开发运维流程中](https://thenewstack.io/add-it-up-devops-security-needs-more-tooling/)之前，公司还有很多工作要做。不要以为你的工作已经完成了，仅仅是因为你已经自动化了一些测试，并且咨询了 Infosec 团队。
