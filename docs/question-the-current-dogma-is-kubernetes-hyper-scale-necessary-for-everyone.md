2020 年的 Kubernetes 已经成为原生云一词的同义词，也经常被供应商和 IT 组织用作宣传他们正在转变或现代化其工作负载的工具。但是他们实际上在改变什么呢？Kubernetes 本身实际上提供了什么？

对于那些转向 Kubernetes 和实现应用现代化的人来说，最大的驱动力是增加价值实现时间的承诺和更快创新的能力。但事实是，许多组织正在将应用程序从已经很复杂的现有基础架构上迁移到另一个复杂的平台上。迁移到 Kubernetes 和容器是为了解耦应用程序，使它们更加敏捷。但是，如果您首先必须以一种现代的云原生方式重写应用程序逻辑，并且必须找出一套全新的基础架构，那么您将面临双倍的挑战。

> 雇佣更多的人来管理和理解不必要的管道工程是没有意义的。最成功的组织专注于提供差异化。

虽然 Kubernetes 是一个强大的平台，但这种强大伴随着巨大的复杂性和更大的责任。它将比以往更多的权力和控制权交给了开发人员，并且不仅需要您的技术堆栈的彻底转变，还需要组织和开发过程的文化的彻底转变。所有这些复杂性不一定会增加商业价值。毕竟，当您的 it 员工花费的时间比以往任何时候都多的时候，很难进行创新。

当你只有一把锤子时，一切看起来都像钉子。Kubernetes 当然是一个大锤，但是在云原生工具箱中还有其他工具可用于应用程序现代化。如果业务目标是上市时间和以可管理、安全和可伸缩的方式快速交付应用程序，那么大多数时候 Kubernetes 对您的组织来说是错误的选择。

## 无服务器实用主义

如果目标是时间价值和快速创新，那么 Kubernetes 不是答案。

与其相信库伯内特的教条，不如想想你在努力做什么。

如果你的目标是通过交付应用程序或服务来发展你的业务，那么处理应用程序的管道不是你的用户想要支付的。更进一步，雇佣更多的人来管理和理解不必要的管道工程是没有意义的。最成功的组织专注于提供差异化。

如果你计算一下，复杂性就显而易见了。例如，在传统世界中，一个组织可能有 100 台服务器，可以使用大约三年。然后出现了虚拟机，每台服务器有 10 台虚拟机，每台持续约 6 个月。现在，对于每个虚拟机，同一个组织至少有 10 个容器，可能有数百个(如果不是数千个)微服务相互通信，每个微服务都是短暂的，这意味着要管理 1010 件事情。没有比我们过去所做的更简单的方法了。

业务已经从在几年内管理 100 件事情发展到在同一时间段内管理数十亿件事情。虽然 Kubernetes 方法有很多功能，但它最终只是一种只做基本计算的机制，而且它的复杂性高得惊人。

## 透过 Kubernetes 的炒作来看是可以的

Kubernetes 和集装箱不会消失。然而，我认为，不管你是谁，拥抱 Kubernetes 复杂性的教条是不正确的。成本和复杂性的现实将玷污 Kubernetes 的金色光环。

那么，没有 Kubernetes，如何构建和运行现代应用程序呢？。使用无服务器，没有长时间运行的服务器、虚拟机或容器。功能在需要时作为服务执行，后端基础架构全部由服务提供商在云中管理。无服务器基本上允许您只关注业务和应用程序逻辑，因此您可以优雅地重构应用程序，而不必担心重新构建底层管道来运行一组新的服务。

当今最著名的无服务器例子是 AWS Lambda 服务。，尽管需要注意的是，serverless 不仅仅是 Lambda，它还包括 API Gateway、AppSync、Cognito、DynamoDB、GraphQL、S3、Secrets Manager 等资源。AWS(和其他地方)有二十多种无服务器服务，使用正确的管理工具，可以将单一应用程序分解为构建块，并优先考虑更智能的增量现代化战略，该战略侧重于首先解析出最关键的工作负载和组件。我们看到它适用于各种规模的组织，包括 MasterStream 和 Branch Insurance，其中复杂的业务逻辑已经通过无服务器实现了现代化。

例如，在 MasterStream，该公司能够通过采用无服务器方法专注于业务逻辑，并且不再将时间和资源花费在数据库管理、基础架构供应和其他管道任务上。Branch Insurance 能够利用无服务器解决方案，通过一致的控制环境实现法规遵从性，而无需复杂的额外基础架构。

采用无服务器方法使您的组织能够将时间花在使您从竞争对手中脱颖而出的事情上，这最终将为您的客户提供更好的体验。对于组织来说，这是实现应用程序现代化并真正受益于云经济的绝佳方式:规模和弹性、速度、安全性以及对应用程序的关注，而不是不必要的基础架构复杂性。

无服务器使企业能够专注于为其用户、客户和业务带来真正的影响，而无需担心 Kubernetes 容器基础设施的复杂性或寻找合适的人才来管理和维护它。
