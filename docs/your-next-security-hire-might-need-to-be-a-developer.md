最近，一位在大公司工作的同事告诉我，他的 CISO 老板不会雇用任何新的安全人员，除非他们花时间开发软件。这严重冒犯了这位朋友，他是完美的安全专家，有多年保护最大的公司免受网络攻击的经验。虽然我不原谅告诉你最好的员工他们可能缺乏胜任自己工作的资格，但理解这位老板鼓舞士气的话背后的动机是值得的。

长期以来，安全人员一直试图避开他们保护的应用程序。安全团队中流行的理论是，开发人员是鲁莽的、不可靠的，他们有数据支持这一点。造成安全风险的开发人员错误的频率在大约 15 年的时间里几乎没有变化。我们在这方面并没有真正变得更好，因为无数的原因我不会在这里推测。

开发人员的这种观点给安全团队留下了两个基本策略。尝试了解应用程序内部发生的事情，或者在不知道内部发生了什么的情况下，尝试从外部保护应用程序。您可能已经猜到，后一种策略远比前者更常见。

## 每个人的沙盒

我们的安全策略通常归结为试图将应用程序与自身的愚蠢隔离开来。我们在应用程序前面设置了一个 web 应用程序防火墙(WAF)和一些网络安全措施，在应用程序下面的操作系统中设置了某种代理，并且可能在应用程序后面设置了一个数据库防火墙。如果它失控，我们认为我们可以控制损失。

这种策略通常不是特别有效，但随着我们转向云原生应用和无服务器计算，它变得越来越糟糕。这是因为总的来说，我们已经将这些层的所有权、控制权和管理权转移给了云提供商。我们不再像过去那样处理网络配置。我们不再管理操作系统。我们的应用程序由代码和 API 组成，如果我们要保护它们，我们最好开始理解它们。

## DevSecDevOps

故事中的 CISO 可能有些直言不讳，但我能理解这种情绪的来源。作为安全人员，我们需要参与应用程序，而不是回避它。我们需要了解应用程序内部发生了什么，这样我们才能保护自己免受风险。安全团队需要理解设计和编程语言选择的结果。他们需要能够与开发人员一起进行安全代码审查。他们需要能够根据这些信息正确配置安全工具，并根据风险在代码中的位置以及被利用的可能性来确定风险的优先级。

简而言之，安全已经无法避免代码。说实话，这是好事。当安全性变得以代码为中心时，它将变得更加有效。与此同时，如果你雇佣一群开发人员来处理你的安全问题，你将会猛然醒悟。仅仅因为做好安全工作需要理解开发人员的思维方式和世界，并不意味着我们在如何做好安全工作上积累的所有集体智慧都会付诸东流。为了平衡，你需要两者。

## 我可以克隆我的安全团队吗？

不幸的是，知道如何阅读 python 和 rust 的安全人员的田园诗般的解决方案实际上并不能独自工作。这是因为在代码和 API 级别上实现安全性不仅仅需要更好地理解编程堆栈。这也转化为需要做出的不可逾越的决策数量。制定代码级的安全决策，例如提供 IAM 角色，确保您的代码不包含有风险的流，以及确保第三方组件没有风险，是一项永无止境(且费力不讨好)的任务。

这就是自动化不仅仅是“有就好”的地方，而是一个全面的要求。如果你不能在 CI/CD 和生产过程中实现这些过程的自动化，你将永远跟不上，即使你的开发人员都是莱纳斯·托沃兹和吉多·范·罗苏姆。但如果可以，你就不会回头。
