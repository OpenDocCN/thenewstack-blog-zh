# 为什么 Swagger 使 API 更容易描述和使用

> 原文：<https://thenewstack.io/why-swagger-makes-apis-easier-to-describe-and-consume/>

编者注:Swagger 展示了开发人员向业务用户展示 API 是如何被消费的是多么容易。

在过去的几个月里，有一些发展显示了 API 是如何抽象的，从设计和消费的角度来看变得不那么复杂。特别是 [Swagger](https://helloreverb.com/developers/swagger) 的出现，这是一个用 JSON 描述 REST APIs 的开放标准。在旧金山的 API Days 上，我坐下来与一些 API 专家讨论如何使用 Swagger 为客户端和服务器端代码创建文档。和我一起的还有史蒂夫·威尔莫特， [3Scale](http://www.3scale.net/) 的创始人；艾德·阿努夫， [Apigee](http://apigee.com) 的公共战略副总裁；[第 7 层](http://www.layer7tech.com/)的迈克阿蒙森 [API 布道者](http://apievangelist.com/)的金莱恩。3Scale、Apigee、Layer 7 都是 API 管理公司。

[Swagger:让 API 更容易描述和使用](https://thenewstack.simplecast.com/episodes/swagger-making-apis-easier-to-describe-and-consume)

这个故事以前也有过以前的版本。WADL 是一个经常被提及的例子。Swagger 的不同之处在于它是由用户而不是供应商创建的。 [Wordnik](http://wordnik.com) 在 developer.wordnik.com 和底层系统的开发过程中，开发了 Swagger 供自己使用。

根据 Wordnik 的说法，它始于 2010 年初——发布的框架目前由 Wordnik 的 API 使用，它支持内部和外部 API 客户端。正如 Wordnik 所指出的，Swagger 有一个声明性的资源规范，允许客户端在不了解服务器实现或不访问服务器代码的情况下理解和使用服务。

它有一个 UI 框架，供开发人员和非开发人员在沙盒 UI 中与 API 进行交互，该框架提供了对 API 如何响应参数和选项的清晰洞察。Swagger 既会说 JSON，也会说 XML，还会说其他格式。安诺夫说:商业用户，以及谁是商业用户，已经发生了很大的变化。

您现在需要发布供外部使用，但要描述移动应用程序将采取的交互。这不是前几代 API 描述语言所考虑的用例。业务用户希望了解当用户将东西放入购物车时进行的 API 调用。这些信息可以用来对人们正在使用的流行产品做出决策。Swagger 更容易做到。
