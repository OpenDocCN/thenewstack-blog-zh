随着各组织争相对其业务进行数字化转型，全球疫情进一步加速了这一趋势，[无服务器应用架构](https://thenewstack.io/category/serverless/)变得越来越受欢迎。它们允许开发团队快速将新的面向客户的应用程序推向市场，而没有与管理和维护底层基础设施相关的开销。

当今的现代微服务和无服务器架构提供了真正的优势。独立发展服务更容易，因为它们的状态封装在 API 后面。您可以轻松地实施安全性和模式约束，并且数据库硬件可以基于每个服务进行扩展。但是也有不利的一面。

如果一个 API 需要改变，它需要保持向后兼容。客户端需要与之对话的服务和 API 越多，应用程序就变得越慢，越不可靠。使用多个数据库会导致数据在许多服务中以不同格式重复，并很快失去同步。

这可能会直接影响用户体验，并导致电子商务应用程序中的库存检查和结账流程失败，客户支持工具中的消息变慢或丢失，银行和加密货币应用程序中的市场信息过时等。正如布鲁克斯定律所描述的:在一个系统或组织中，潜在的沟通途径越多，沟通开销就越高。

当应用程序很复杂时，开发人员的生产力也会受到影响。为了节省开发人员的前期和后续时间，有必要考虑一种回归过去开发模式的策略——即通过共享数据库集成这些服务。

为什么？跨应用程序共享的数据不需要额外的集成服务层。但是这种简化增加了风险的传统观点仍然成立吗？我认为答案是否定的，尤其是在无服务器架构方面。

在过去，一个常见的模式可能是这样的:客户端和服务永远不会相互通信，而是通过一个低延迟的本地网络访问数据库。数据库管理员(DBA)管理访问控制，并验证性能和正确性。这释放了本地计算资源来运行不安全的业务和表示逻辑。

随着系统的发展，关系数据库管理系统(RDBMS)模型成为了一个操作瓶颈，并且朝着[微服务](https://thenewstack.io/category/microservices/)的方向发展。但是快进到今天，我们实际上已经克服甚至消除了导致趋势偏离“通过数据库集成”模式的限制

在现代数据库、无服务器操作模型、API、现代查询语言和 web 原生安全实践之间，我们现在正处于传统的面向数据库的架构再次变得有意义的时刻。围绕现代数据库(如我们在 Fauna 开发的数据库)构建的集成系统可以在完全事务化、高度一致的环境中运行一切。

越来越多地采用无服务器架构并围绕数据库进行集成的客户通常拥有移动或单页动态 web 应用程序和用 GraphQL 或数据库本地查询语言实现的业务逻辑的某种组合。应用程序和后端只与数据库和任何附加的 API 服务对话。

当团队通过专门为现代应用模式设计的现代数据库集成他们的服务和客户端时，我们看到了开发人员生产力的提高和更好的应用可用性、一致性和性能。我鼓励所有开发人员重新评估他们的假设，并在他们自己的应用程序中利用这些新功能。
