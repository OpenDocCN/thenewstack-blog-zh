# 了解 Palo Alto Networks 即将举办的代码到云峰会

> 原文：<https://thenewstack.io/a-look-at-the-palo-alto-networks-upcoming-code-to-cloud-summit/>

从云安全提供商到开源软件，信任已经成为构建组织安全的基础。但随着容器和基础设施即代码(IaC)等云原生技术的兴起，it 迎来了构建应用程序和需求的新方法，挑战着传统的安全方法。云原生环境不断变化的本质要求跨技术堆栈的更广泛的安全覆盖，以及对环境的更多上下文感知。但是像 Infosec、DevOps 这样的团队应该如何反思他们的安全方法呢？

在本期 New Stack Makers 播客中， [Bridgecrew](https://bridgecrew.io/) 的联合创始人兼产品副总裁 [Guy Eisenkot](https://www.linkedin.com/in/guy-eisen-3012a597/) ，[Palo Alto Networks](https://www.linkedin.com/in/barakschoster/?originalSubdomain=il)和 [Ashish Rajan](https://www.linkedin.com/in/ashishrajan/?originalSubdomain=au) 的高级总监兼首席架构师 Barak Schoster Goihman ， 在 3 月 23 日至 24 日举行的[帕洛阿尔托网络的代码到云峰会](https://start.paloaltonetworks.com/code-to-cloud-summit.html)上，安全和合规负责人和[云安全播客](https://cloudsecuritypodcast.tv/)的制作人和主持人[预览即将到来的内容，包括与开发运维、云服务提供商、软件供应链、SBOM(软件材料清单)和 IBOM(基础设施材料清单)相关的安全和信任的作用。](https://start.paloaltonetworks.com/code-to-cloud-summit.html)

《新书库》的创始人兼发行人亚历克斯·威廉姆斯主持了本期播客。

[用信任保护现代企业:看看即将召开的代码云峰会](https://thenewstack.simplecast.com/episodes/securing-the-modern-enterprise-with-trust-a-look-at-the-upcoming-code-to-cloud-summit)

据 Gartner 称，到 2025 年，公司将在云原生平台上部署 [95%的新数字工作负载。随着这一趋势继续被接受，它引入了更多的复杂性，使其难以保护，并回避了有关信任的问题。“如果你真的想晚上睡个好觉，并且信任你的工程团队，你应该找到最好的方法来获得许多问题的答案，比如我在使用什么开源包？我的配置中有什么样的基础架构？Goihman 说:“我很早就在使用第三方云提供商，而且每一步都是如此。](https://www.gartner.com/en/newsroom/press-releases/2021-11-10-gartner-says-cloud-will-be-the-centerpiece-of-new-digital-experiences)

像 Log4j 这样的严重漏洞给许多企业带来了严重的风险，云服务提供商和组织之间的共同责任模式也考验了信任线。“亚马逊(Amazon)和谷歌云(Google Cloud)等公司不得不拿出服务来应对他们向我们展示的内容。他们现在有一个脚本，可以持续检查 Log4j。所以现在信任的问题来了:你能运行这个脚本吗？停机是对信任边界的一种考验，”Rajan 说。

尽管针对新的威胁，安全实践正在变得更加尖锐，但要创造世界级的文化、传播意识和增强强大的网络卫生，仍有大量工作要做。“我认为我们必须更加深思熟虑如何对待我们的公司代码库。我认为我们需要更好地了解那里有什么；那些公司方储存库连接到什么；他们从哪里传输数据；他们正在全面使用什么样的环境和环境变量，”艾森科特说。

由许多组件构建的现代应用程序可以在内部开发，也可以是现成的，但这让许多人重新思考云安全。“拥有所有这些不同种类的资产:基础设施、代码、容器映像、开源包以及代码库中交付管道的工作流，可以让我们了解供应链的全貌。我们可以为基础设施代码的开源包建立一个软件材料清单，这就像一个运行时材料清单，帮助我们优先处理坏代码，”Goihman 说。

随着安全性的转变，开发人员有责任保护代码，许多团队不堪重负，努力跟上现代软件开发的步伐。但是，全面了解潜在漏洞或错误配置的位置也有助于确定优先级，并提供漏洞在云架构各层中的位置。Rajan 说:“开发人员希望产生代码，但仅仅给他们一个漏洞是不够的，因为他们不知道他们试图解决什么以及为什么他们需要解决这个问题。”。
