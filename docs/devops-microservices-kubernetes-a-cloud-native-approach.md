# DevOps、微服务、Kubernetes:云原生方法

> 原文：<https://thenewstack.io/devops-microservices-kubernetes-a-cloud-native-approach/>

DevOps 实践在现代应用程序架构中深入运行。DevOps 实践帮助开发人员和工程师创建了一个空间，通过持续交付实践来构建优化资源和横向扩展应用架构的新方法。

云原生技术利用容器的效率来创建比组合或单一环境更有用、更具适应性的微服务架构。组织在构建云原生的、基于微服务的应用时，正转向 DevOps 原则。DevOps 和云原生架构的结合有助于组织通过培养能够快速适应市场变化的精简、精益的产品开发流程来实现其业务目标。

## 开发运维支持云原生应用

罗布·斯科特

Rob Scott 是 ReactiveOps 的一名现场可靠性工程师，他在查塔努加的家中工作。他帮助为多个客户构建和维护高度可扩展的、基于 Kubernetes 的基础设施。自 2016 年以来，他一直与 Kubernetes 合作，一路为官方文档做出贡献。当他不建设世界一流的基础设施时，Rob 喜欢与家人共度时光，探索户外，并就 Kubernetes 的所有事情发表演讲。

云原生应用程序基于一组松散耦合的组件或微服务，大部分运行在容器上，并通过编排引擎(如 [Kubernetes](https://kubernetes.io/) )进行管理。然而，它们也开始在无服务器架构中作为一组独立的功能运行。服务或功能由开发人员和工程团队定义，然后由越来越多的跨职能团队不断构建、重建和改进。运营现在不太关注基础架构，而是更关注运行轻负载的应用程序。综合效果是形成自动化流程，从而提高效率。

事实上，有些人会认为应用程序不是真正的云原生的，除非它背后有 DevOps 实践，因为云原生架构是为 web 级计算构建的。DevOps 专业人员需要构建、部署和管理安全、弹性和高性能的声明式基础架构。用传统的孤岛式方法满足这些需求是不可行的。

然而，为了在开发运维中取得成功，企业必须有意识地决定构建一个云原生应用。只有当业务团队愿意投资开发运维实践时，将开发运维付诸实践所需的组织转型才会发生——转型伴随着产品团队在应用开发中的调整。这些团队共同创建了将技术开发持续细化为精益、简化的工作流所需的环境，这些工作流反映了基于 DevOps 原则构建的持续交付流程。

## 云原生开发运维需要转型

对于使用容器编排技术的组织，产品方向是通过开发微服务架构来定义的。只有当组织了解 DevOps 和持续开发过程如何支持最终用户真正发现有用的应用程序的创建时，这才是可能的。

这就是挑战所在:你必须确保你的组织准备好改变产品团队所有成员的方式。最终，DevOps 是一个关于你为什么要首先进行精简、精益的产品开发的故事——这与你要迁移到基于 Kubernetes 的微服务架构的原因是一样的。

在容器出现之前，传统的 DevOps 模式需要不同的流程和工作流。容器技术是从 DevOps 的角度构建的。随着微服务的出现，传统的架构开发发生了变化，抽象容器对我们如何看待 DevOps 产生了影响。这意味着遵循在 Kubernetes 上运行容器的最佳实践，并将 DevOps 扩展到 GitOps 和 SecOps 实践中。
