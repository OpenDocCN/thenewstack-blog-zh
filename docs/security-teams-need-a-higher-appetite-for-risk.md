# 安全团队需要更高的风险偏好

> 原文：<https://thenewstack.io/security-teams-need-a-higher-appetite-for-risk/>

帕洛阿尔托网络公司的 Prisma Cloud 赞助了这个播客。

安全团队需要更高的风险偏好。虽然接受甚至拥抱风险在 IT 领域之外被广泛接受，但风险也经常在开发运维、开发人员和 SRE 团队文化中扮演角色。然而，安全团队通常还没有以这种方式接受和管理风险。

本期 [The New Stack Makers](/podcasts/makers) 播客由 New Stack 的创始人兼发行人 [Alex Williams](/author/alex/) 主持，预告了我们即将推出的电子书[The Best of DevSecOps:Trends in Cloud Native Security Practices](/ebooks/devsecops/best-of-devsecops-trends-in-cloud-native-security-practices/)，讨论了安全团队如何以及为什么需要重新思考风险，目的是提高弹性并实现许多组织迄今仍难以实现的其他好处。

特邀嘉宾都是电子书的特约作者:[帕洛阿尔托网络](https://www.linkedin.com/in/mattchiodi/)[公共云首席安全官马特·奇奥迪](https://www.paloaltonetworks.com/cloud-security?utm_content=inline-mention)、[产品管理高级总监米拉·拉奥](https://www.linkedin.com/in/meerasubbarao)、[新思科技](https://www.synopsys.com/software-integrity.html)和[首席营销官塔尔·克莱因](https://www.linkedin.com/in/talmklein)、[雷兹利翁](https://www.rezilion.com/)。

[为什么安全团队需要更高的风险偏好](https://thenewstack.simplecast.com/episodes/why-security-teams-need-a-higher-appetite-for-risk)

Klein 指出，目前，组织通常在不知道“漏洞实际上是否代表风险”的情况下，发布并分配一个作业单来修复发现的漏洞

克莱恩说:“我认为，在我们克服这一事实，建立能够抵御企图入侵的基础设施，而不必担心我们的代码中是否存在漏洞或零日或任何这些事情之前，我们只需说，‘看，它就在那里，它正在运行——如果出现问题，我们会将其恢复到上次已知的良好状态。’。“除非我们接受这种风险偏好，变得更像保险公司的精算师，而不是像美国海军陆战队的第一道防线，否则在安全和开发方面，我们将一直处于这种奇怪的状态。”

弹性因素在 DevSecOps 团队如何改变他们的风险评估和风险管理文化中也起着巨大的作用。Rao 解释说，通过这种方式，弹性意味着软件出现缺陷的可能性更小。

“那么，当我们说缺陷时，它仅仅指质量吗？还是意味着安全？”饶说。“在我们的情况下，我认为两者都是:你的软件的质量和安全缺陷越少，你的软件就越有弹性。无论您是使用 DevOps 实践，还是使用 SREs，目标之一都是构建具有高质量和安全性的更好软件，这反过来又会转化为更具弹性的软件。”

Chiodi 说，从表面上看，更高水平的风险偏好可能看起来“像是一种矛盾，因为大多数安全团队会说‘我们的工作是做相反的事情:帮助管理，并在我们可以的地方，减少组织中的风险’。

Chiodi 说，安全团队可以采取的更好地管理风险的具体步骤是访问“七个或八个不同的指标，组织可以查看这些指标，它们都关注有效性和效率”。

Chiodi 说:“我认为，如果安全团队有这种信心，他们正在寻找正确的指标，他们就可以培养这种信任和欲望，‘好吧，让我们加快速度，因为我们有这种内置的早期预警系统，可以知道是否会发生什么事情。’。“‘我们不会等到出现漏洞，我们实际上是在持续主动地寻找和跟踪这些东西。’"
