# 开发人员和应用程序安全性之间的脱节

> 原文：<https://thenewstack.io/the-disconnect-between-developers-and-application-security/>

作为一名开发人员，您的职责范围很广——例如，看看一些人所说的神话般的“全栈开发人员”的工作描述。通常，“安全专业人员”是一个可以出现在列表中角色。你已经在考虑开发一个高性能的应用程序，满足质量目标并易于使用。怎么能指望你也成为一名安全专家呢？

应用安全公司 [StackHawk](https://www.stackhawk.com/) 的联合创始人兼首席安全官 Scott Gerlach 表示，你不能也不应该这么做。Gerlach 提出了三点中的第一点，说明为什么开发者要与 AppSec 斗争，AppSec 是他在上周 Snyk 举办的[All Talks](https://www.allthetalks.org/)虚拟会议上的小组的名称。

“当高管团队说‘嘿，我们需要模拟价格上涨或改变我们现有产品的成本结构’时，你永远不会看到会计师对高管这样做。”会计们不喜欢。我们很乐意帮助你，但是让我们先教你总账。他们不需要了解会计是如何工作的，总分类账是如何在组织中建立的，会计人员正在做什么的所有细节。他们可以很快做出决定。输入信息，做决定。作为安全团队，我们应该多做些这样的事情。我们应该为组织提供工具和信息，特别是当我们谈论 AppSec 时，这样他们就可以做出决策，这样他们就可以快速做出决定，并开始将该产品投入生产。"

虽然安全团队可能希望开发人员关心安全性，但这通常是他们最关心的问题，而不是将安全性添加到他们的责任列表中，为开发人员提供开发人员本机工具有助于将安全性引入工作流。这些工具可以帮助指导开发人员在考虑安全性的情况下做出决策，而不是期望开发人员了解作为安全专业人员的详细情况。

Gerlach 的第二点和第三点都提到了角色期望以及企业不同部分如何与安全团队互动方面的错位。例如，AppSec 团队可以非常努力地发现安全问题，并以此过程为荣，但随后只是简单地将它扔给开发人员，而不是试图一起找到解决方案。

“我们以错误的方式接近了它的结尾部分。我们回到工程领域，说‘嘿，我把你的东西弄坏了，是不是很棒？’没人认为这很棒。没有人认为他们所付出的努力需要被妖魔化，因为我们为我们所做的事情感到骄傲。这是另一个桥梁建设功能，我们可以与工程团队合作说，“嘿，让我们聚在一起，谈谈这次攻击是如何进行的。”"

> “对我们来说，说我们发现的所有问题都应该解决是不真诚的。我们的业务不是修复、修补，而是为客户提供服务。”

Gerlach 说，不仅经常缺乏沟通，而且安全团队与开发生命周期相距甚远，他们发现的漏洞可能会追溯到开发团队几周前(如果不是几个月前)所做的更改。他说，这里的部分解决方案是更紧密地合作，以确保开发人员和 AppSec 团队的努力都花在有价值的工作上，这就把我们带到了最后一点——app sec 方面的期望错位，要求开发人员解决所有问题。

“对我们来说，说我们发现的所有问题都应该解决是不真诚的。Gerlach 解释说:“我们的业务不是修复和修补，我们的业务是向客户提供服务，我们应该优先考虑真正非常重要的事情，并能够了解在组织的环境中哪些事情是真正非常重要的。

在他的结束语中，Gerlach 提出，安全需要通过一个组织，通过使用工具和信息来实现民主化，但经常重复的“安全是每个人的责任”更多的是一种逃避。他在这一点上做了总结，提到了他最初的观点，即会计师不要期望高管了解会计的来龙去脉来做出决策。

“实际情况并非如此。会计不是每个人的工作，会计是你的工作——上交你的费用报告——但是会计是会计部门的责任。很有可能，不是每个人都有责任打扫厕所，而是清洁工有责任。我们有责任捡起我们的小垃圾。说安全是每个人的责任可能太宽泛了。”

在这种情况下，开发人员再次关心代码质量、性能和效率，如果 AppSec 团队希望他们关心安全性，他们需要为他们提供工具和信息，以便以符合他们能力和知识的方式更容易地做到这一点。
