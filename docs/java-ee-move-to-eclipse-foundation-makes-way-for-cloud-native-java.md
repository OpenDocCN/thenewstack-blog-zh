# Java EE 迁移到 Eclipse Foundation 为云原生 Java 让路

> 原文：<https://thenewstack.io/java-ee-move-to-eclipse-foundation-makes-way-for-cloud-native-java/>

随着 Jakarta EE 8 规范的发布，Eclipse Foundation 最终完成了 Java 企业版(EE)向 Jakarta 的过渡。尽管 Jakarta EE 8 规范没有引入任何新功能，但它确实保证了与 Java EE 代码和 ide 的完全兼容。根据 Eclipse 基金会执行董事 Mike Milinkovich 的说法，这一举动远远超出了名称，真正的好处是在 Eclipse 基金会的指导下社区进步的潜力，Eclipse 基金会是 350 多个其他开源项目的家园，包括运行时、工具和框架。

“我们已经将所有这些非常核心的技术从单一供应商控制转移到 Eclipse Foundation 和 Jakarta EE 工作组的供应商中立位置。米林科维奇在接受 New Stack 采访时说:“我们正在推动围绕这些方面的创新。”。“Java EE 已经停滞了很多年，主要是因为没有足够的资源来推动规范变得与云原生开发者更加相关。展望未来，我们有机会真正释放社区围绕这些技术的才能和想法，并使 Java 成为未来 20 年大规模应用程序的首选平台。”

除了 Jakarta EE 8 规范，Eclipse 基金会还发布了 [Eclipse GlassFish 5.1](https://projects.eclipse.org/projects/ee4j.glassfish/downloads) ，这是 Jakarta EE 8 平台的一个开源兼容实现，已经在 Jakarta EE 8 技术兼容性工具包(TCKs)下针对兼容性和合规性进行了全面测试，以及 [Open Liberty](https://openliberty.io/) ，这是 Jakarta EE 8 概要文件的一个完全认证的实现。Milinkovich 说，虽然这两个产品可以立即使用，但 Red Hat、Oracle、Fujitsu、Tomitribe、Apache Tomcat 和 Payara 也在实施过程中。

Tomitribe 的创始人兼首席执行官 David Blevins 在一份声明中表示:“在从事开源和 Java 标准工作的 20 年中，我从未见过像发布 Jakarta EE 8 这样真正多样化和平等共享的努力。“源代码已被捐赠，但出现的是一个勇敢接受变化、创新开放流程并反复选择共享胜利而非个人成功的社区。当我们回顾 10 年来在雅加达取得的成就时，每个人都会同意这是塑造我们未来的独特时刻。”

## 云原生 Java

至于未来，Milinkovich 解释说，Java EE 的结构已经适应了过去的单一应用程序，将会投入一些努力来帮助 Jakarta EE 实现云原生应用程序的现代化。以前，Java EE 应用程序不能挑选特定应用程序所需的 API，这意味着它们可能非常大，这是 Jakarta EE 将关注的一点。

“你可以想象，在如何推动这个平台向前发展方面，有很多被压抑的能量和兴趣。同样重要的是，人们对摆脱某些东西也很感兴趣。Milinkovich 说:“JavaEE 平台实际上已经有 20 年的历史了，它的整个生命周期都是完全向后兼容的，所以有些 API 在今天的云原生世界中并不那么重要。“我们希望实现一种 API 更加松散耦合的场景，您可以挑选想要用来服务特定应用的 API，这将使平台更加敏捷、更加轻量，并帮助 Java 在基于微服务的世界中更加相关。”

就走向未来而言，Milinkovich 再次强调，在 Eclipse 基金会的照顾下，Jakarta EE 将享有一个更加开放和协作的未来。

“这是一个全新的、比 Java 社区流程(JCP)更加开放和协作的流程。在 JCP 下，每个规范都有一个规范负责人，通常是一家公司，该公司获得所有贡献的特殊知识产权。一个奇怪的巧合是，几乎所有的时间规范的领导者都是甲骨文，”米林科维奇说。“不再有 spec lead，也不再有任何一方拥有特殊知识产权的概念。参与规范项目的每个人都有能力对规范进行修改。我们预计这可能会导致更多的争吵或争论，但我们坚信这种开放的合作方式是未来的发展方向。”

最后，基金会发布了一本关于“实现开源、云原生 Java 的愿景”的电子书，探讨了云原生 Java 的理念和 Jakarta EE 的未来。

Red Hat 和 Oracle 是新堆栈的赞助商。
