通常开源项目是一项热爱的工作，是开发人员在空闲时间的个人激情。偶尔，他们是一个公司的营销噱头，以赚钱的知名度。相比之下，内容管理软件 Drupal 两者都不是。相反，Drupal 是大约 10，000 名不同贡献者的心血——其中最大的贡献者贡献的代码不到总代码的 5%。

这有点荒谬。而且相当惊人。

这个开源独角兽的核心人物是 Dries Buytaert，他在 2000 年创建了 Drupal，后来在 2007 年创建了 Acquia，这是 Drupal 最大的贡献者，但还不是主导者。尽管 Buytaert 和 Acquia 在 Drupal 的发展中发挥了核心作用，但他们缺乏压倒性的引力。那是故意的。没有人希望看到 Buytaert 被众所周知的公共汽车撞上，但如果它发生了，Drupal 会继续下去。这就是社区主导的开源应该如何工作，在 Drupal 的例子中，它工作得非常非常好。

但事情并不是这样开始的。

## “……看看我能否建立一个小网站”

2000 年，网络还很年轻，还有点新奇。开源也是如此，PHP 和 MySQL 提供了构建网站的新方法。对于学生 Buytaert 来说，这是一个学习的机会。“接下来的几个晚上，我打算涉猎一下 PHP 和 MySQL，”他当时决定，“看看能不能建一个小网站。”

或者会，如果他能上网的话。

然而，他的一个学生朋友做到了。他有 ADSL，但是住在对面。“我在这里，渴望建立网站，但没有互联网，因为我负担不起，”他说，“同时，我的朋友有超快的互联网，只有少数人有。”

这位朋友签署了一份协议，不共享他的互联网服务。“所以我们做的第一件事是问，‘我们如何在我们的房子之间建立一座无线桥梁，以便我们可以共享它？’”Buytaert 说。

他们试图让阿尔卡特朗讯和其他几家公司为他们建造一个——但成本大约为 10 万美元，这不在他们的学生预算之内。他们在英国找到了一家公司，为他们提供 WLAN 卡，但由于 Buytaert 拒绝使用 Windows，他不得不自己编写 Linux 驱动程序，并定期更新。幸运的是，在 Linux 内核社区中已经有了一个现有的项目，他开始积极地参与其中。

随着这项工作的开展，他遇到了其他问题:每个人都想加入他们的非法互联网连接。保持它的运行也被证明是有问题的。如果他使用他的微波炉，或者如果一辆卡车在街上经过，连接就会中断。此外，由于有近 10 个人使用 ADSL 连接，它只是因为过度使用而不断下降。因此，Buytaert 建立了一个网站来报告他们的无线互联网连接状态，这成为了一个内部留言板，供朋友们组织一次麦当劳之夜或只是在考试期间发泄。

二十年后，那个“小网站”已经成为 Drupal，地球上最令人印象深刻的开源项目之一。“我预计可能会有五个人下载并使用它，”他说。

## 不断发展的平台

随着时间的推移，Buytaert 尝试了一系列的网络功能，比如投票系统(比 Digg.com 早几年)，RSS 订阅和“公共日记”(博客)。在 Drupal 存在的前七年，它只是 Buytaert 在晚上和周末摆弄的东西。“那是为了好玩，对吗？”他说。

在这个早期 Drupal 的一年中，Buytaert 开始获得贡献，为了让其他人更容易试验他，他开源了 Drupal。“我把我的实验平台开源，让它成为你自己的实验平台怎么样？或许你可以帮助改进投票算法或他们反馈的任何东西？”他记得当时想。

从对话到编码——Drupal 的社区诞生了。

Buytaert 在 Drupal 上玩了大约七年后，获得了计算机科学和工程的博士学位，他必须做出选择。他说，学术界本来是“容易的选择”，但他决定将 Drupal 作为自己的全职工作。于是他开始了 Acquia。Buytaert 游说了当时的许多开源商业领袖——包括 Marten Mickos、Larry Augustin 和其他人——试图找出正确的模型。

在这个研究过程中， [AWS](https://aws.amazon.com/) 启动了，Buytaert 大脑中的灯泡熄灭了。“我们看到开源支持模式将会很艰难，”他说。"我们把农场押在云上."

也许并非巧合，Buytaert 的博士学位是性能和可伸缩性，这帮助他看到了云的前景。他说，那时扩展 Drupal 很难，因为你必须调整 MySQL 和其他组件中的上百个小旋钮。但是在 AWS 上构建时呢？“我们可以让 Drupal 变得如此可扩展——你不知道，”他说。

所以云帮助 Buytaert 让用户更容易访问 Drupal，但是那些最初帮助 Buytaert 构建 Drupal 的人呢？

## “我会亲自邀请每一个贡献者”

虽然编写 Drupal 很难，但吸引贡献者可能更难。他说，Buytaert 不知疲倦地努力说服早期的潜在贡献者为项目贡献时间，亲自邀请每一位贡献者。不仅如此，他继续说道，“我不遗余力地让人们做出贡献。”他会检查他们的代码，并以友好和尊重的态度给出反馈。“我会‘把他们裹在毯子里’，然后很快赋予他们权力，”他说。正如他所描述的那样，在 Drupal 中给人们一种主人翁的感觉会让他们愿意留下来，为这个项目尽自己最大的努力。

他的方法奏效了。

Buytaert 说，他的方法演变成了一种非常合作的治理模式。早在 GitHub 之前，Drupal 社区就构建了类似 GitHub 的工具来支持协作。Drupal 今天仍然使用同样的工具。“GitHub 对我们来说还不够好，”他说。Buytaert 并不意味着对 GitHub 的轻视。但正如他指出的，“GitHub 不适合与 1000 人合作一个项目”，更不用说 10000 人了。Drupal 社区需要能够以“Drupal 规模”工作

这本身就足够令人印象深刻了，但是 Drupal 真正令人惊奇的一点是 Acquia 不能运行所有的东西，甚至是大多数东西。

## “许多开源公司对控制没有安全感”

Acquia 的方法与大多数开源公司不同。“许多开源公司都非常不安全。他们想要掌控一切。”例如，不能绝对控制路线图的想法:组织应该如何向客户解释？还是他们的董事会？“这是他们头脑中最疯狂的事情，”Buytaert 说。“只是太洋气了。他们甚至无法想象那个世界是什么样子。”

对于 Acquia 来说，它自己的许多工程决策都归结于问题的难度。当试图决定是将代码作为开源贡献给 Drupal 还是构建专有功能时，一个通用的试金石是少数工程师能否在六个月的时间内构建该功能。如果是这样的话，“我们为什么还要费事让它成为专利呢？”他问道。

Drupal 社区中有 10，000 名活跃的贡献者，这个石蕊测试很重要，因为该社区非常有能力构建几乎任何东西，所以 Acquia 必须专注于解决真正复杂的问题，并作为解决方案出售。

相比之下，对于其他开源公司，Buytaert 说他们从一开始就不喜欢这种模式。他们在开源精神深入人心之前筹集资金，因此他们很快屈服于压力，自己编写所有代码并控制他们的路线图。他说，到那时，转行几乎是不可能的。

## “商业模式颠覆者”

Buytaert 说，开源是“商业模式的颠覆者”，它颠覆了几十年前的技术供应商，也颠覆了最近的开源公司。虽然 Buytaert 很快指出其他模式也可以工作，但他不太相信企业主导的开源从长远来看是可持续的。

这将我们带回了 Drupal 的 10，000 强社区这一不可抗拒的事实。Buytaert 创建了社区，但他并不控制它。没人知道。围绕该项目的开源社区是其成功的秘密，也是像 Pfizer 这样的 Drupal 重度用户每年投入数百万美元的工程时间来构建 Drupal 的工作流引擎的原因——其竞争对手 Johnson & Johnson 使用的是同一引擎，尽管它贡献了自己的功能。

没有人控制 Drupal。每个人都受益。这就是社区驱动模式的美妙之处，Buytaert 并没有开始创造，但还是创造了。他只是想建一个小网站。
