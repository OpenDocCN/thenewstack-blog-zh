# Pancake 播客:Kubernetes 和服务网格哪个先出现？

> 原文：<https://thenewstack.io/pancake-podcast-which-comes-first-kubernetes-or-the-service-mesh/>

关于服务网格还有很多问题。因此，在 VMware 赞助的新堆栈的清晨煎饼早餐小组讨论中，一个好奇、活跃和饥饿的 KubeCon + CloudNative Con 北美 2019 与会者提出了这么多问题。Kubernetes 和服务网哪个先出现？每个应用程序有多少个边车才是正确的边车数量？小公司应该从服务网开始，还是等到业务繁荣和运营增长时再引入？

幸运的是，我们已经召集了一个来自整个行业的精明的服务网格专家小组来解决这些问题，即:

新堆栈的创始人兼发行人 Alex Williams 主持了讨论，这是为最新一集的[新堆栈制作者](https://thenewstack.io/podcasts/makers)播客录制的。听着:

[Kubernetes 和 Service Mesh 哪个先来？](https://thenewstack.simplecast.com/episodes/which-comes-first-kubernetes-or-service-mesh)

对于有多少边车就够了的问题，赵承熙和 Bie 告诉观众:这是资源利用和管理效率之间的权衡。Lyft 有 5 个 sidecars，而 Pinterest 有 10 个。Kubernetes 和 service mesh 哪个应该先来？Envid 建议，这实际上取决于企业的需求，尽管 Cho 提到，在 Lyft 的情况下，服务网格是第一位的，它有助于使 Kubernetes 的加入更加容易。

总的来说，Atelsek 总结道，企业对运行服务网格和拥有统一管理层有很多担忧，但许多云原生公司已经将其投入生产。Monclus 指出，他们正在享受好处，因为服务网格提供了端到端的连接，提高了微服务的安全性和管理。
