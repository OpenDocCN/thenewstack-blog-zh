# 使用蒂姆·伯纳斯·李固体的开发者案例

> 原文：<https://thenewstack.io/the-developer-case-for-using-tim-berners-lees-solid/>

万维网的发明者蒂姆·伯纳斯·李爵士在虚拟穿戴开发者世界大会的主题演讲中介绍了可靠的开源数据协议，距今已有一年多了。新的 Stack 最近检查了 Solid 的进展，与程序员兼顾问 [Jackson Morgan](https://www.linkedin.com/posts/jacksoncmorgan_solid-decentralization-activity-6865890583311896576-coQ1/) 进行了交谈，他是 [Solid 团队](https://solidproject.org/team)的管理员，也是 [o.team](https://www.o.team/) 的自由 Solid 开发人员。

Solid 提供了一个机会，让终端用户可以控制他们的个人信息，从而解决很多在线数据的问题。它建立在使用数据存储或 pods 的 HTML 协议之上，可以由个人[创建和自托管](https://solidproject.org/developers/tutorials/getting-started#own-server)——或者至少拥有，如果你使用第三方进行部署——个人可以随意授予或撤销访问权限。如果我们要解开前 Twitter 联合创始人兼首席执行官、现 Block(前 Square)创始人兼首席执行官[杰克·多西](https://www.britannica.com/biography/Jack-Dorsey)在他的 [Web 5.0](https://docs.google.com/presentation/d/1SaHGyY9TjPg4a0VNLCsfchoVG1yU3ffTDsPRcU99H1E/mobilepresent?slide=id.g11b904107df_0_1) 愿景中描述的数字身份解决方案，我们将需要这种数字身份解决方案，该愿景呼吁拥有自己的数字标识符和分散的网络节点。

“这听起来在概念上像固体。他们可能正在创建自己的分散式网络节点，”摩根谈到多尔西的大纲时说。"但问题是，这不一定是什么新概念."

Solid 呼唤 pod，它既是一个标准化的身份，也是一个任何人都可以访问但只有个人拥有的个人数据的存储库。从开发人员的角度来看，所有这些编码都是为了获取关于用户的数据？你可以用固体来减少它。

至少，一个用例是用 Solid 的开源协议取代 Gmail 或苹果 iTunes 或任何其他专有的基于应用的登录协议。一个更强大的用例将是一个个人医疗数据的存储库，这些数据由个人最终用户拥有和维护，但所有系统都可以通过许可进行访问——坦率地说，对于我们这些厌倦了让医生发传真的人来说，这一天来得太快了(*喘息*！)医疗记录交给另一位医生。

## 美国的意识问题。

当然，我们中最担心这种可能性的人往往是在美国，因为它是高度分散的，应用程序孤岛，以及*敢说敢做*——陈旧的医疗记录系统。摩根说，我很幸运，事实证明，美国的可靠采用率非常低。

“如果不管你在世界上的什么地方，你可以走进医院或诊所，然后说，‘这是我的医疗记录’，这不是很棒吗，”摩根说。为什么，是的，摩根。绝对会的。

显然，美国有一个“意识问题”，当谈到固体。

“美国的科技文化对寻求集中数据的公司不那么怀疑，因为他们可以从中找到赚钱的方法，”摩根说。“欧洲文化对此非常怀疑。因此，无论是在公共还是私人生态系统中，美国大多数人甚至不认为这是一个问题，或者至少美国大多数企业管理者或政府管理者不认为这是一个问题。”

## 欧洲试点稳固采用

与此同时，早期采用正在发生，但主要是在欧洲，那里的消费者和政府对大公司收集个人数据更加谨慎，摩根说。虽然平心而论，这些大公司往往总部设在美国，因此被视为对欧洲公民的更大威胁。《通用数据保护条例》是欧洲试图监管美国大型科技公司影响力的一种方式。摩根认为，对于 GDPR 提出的问题，可靠是技术上的答案。

## 开发者的问题是

美国的问题也许不应该是“我们为什么要这样做”，而是为什么开发者不想这样做？目前，应用程序请求的许多数据都是相同的数据，获取这些数据非常昂贵，而且会带来安全风险。不过，它仍然是一个新兴的协议，正在开发的工具将使部署更容易，因此可能对开发人员更有吸引力。

“当谈到前端开发人员和 web 开发人员时，我至少有这样的印象，如果好的工具被构建，那么他们就可以开始构建东西了，”Morgan 说。“拥有自己的身份并不难想象。总而言之，[……]我不是把东西存储在另一个人的数据库中，而是把我的身份、我的 pod 或我的数据保存在我控制的数据库或我信任的云主机上。”

对于前端开发人员来说，这基本上消除了后端问题。摩根说，他们可以把它看作是连接到一个全球的、尽管是分散的数据库，而不是一个内部数据库。

“基本概念相对容易学习，”他说。“像我这样的人，或者可能是想做阅读规范之类事情的超级热心的前端开发人员，有义务构建让每个人都易于使用的工具。”

Morgan 甚至创建了一个名为 LDO(链接数据对象)的[开发工具，试图让开发人员仅用 JSON 就能“尽可能容易地”浏览 RDF 图。开发人员可能想从这个创建 JavaScript 应用程序的](https://medium.com/@JacksonMorgan/building-solid-apps-with-ldo-6127a5a1979c)[教程开始。摩根也写了一个使用 React](https://solidproject.org/developers/tutorials/first-app) 的 Solid】教程。

但是现在，这个项目似乎有点先有鸡还是先有蛋的问题。Solid 将在有用户的时候被使用…这可能不会发生，直到最终用户需要有一个 pod 来连接到应用程序，到目前为止，这是很少的。
