# 将遗留数据仓库迁移到云中？SAP 的回答

> 原文：<https://thenewstack.io/moving-a-legacy-data-warehouse-to-the-cloud-saps-answer/>

随着企业接受 SaaS 服务的操作简单性和便利性，有一个主要的抵制。传统的后台 ERP 系统在很大程度上被证明对云迁移有抵抗力，因为存在如何处理所有定制的问题？回到 2000 年使后端系统现代化的热潮中，典型的做法是将所有的更改放在核心应用程序代码中。这就是为什么，几年前，我们假设传统的企业系统将是云应用的最后前沿。

这就是我们得到 SAP 的地方。当谈到 ERP 和类似的后台应用程序时，没有人触及 SAP 的足迹。根据该公司自己的数据，按收入计算，全球 77%的交易数据涉及 SAP 系统。虽然这并不一定意味着 SAP Business Warehouse (SAP BW)会触及所有这些数据，但它仍然会在运营报告方面留下巨大的足迹，而且有几十年的代码更改起来成本太高，而且会造成中断。因此，如果您是 ECC 或 BW 用户，想要放弃您的数据中心，您的选择是在 AWS EC2 等中管理您自己的部署。

## 客户群的答案

去年秋天， [SAP](https://sap.com/) 推出了 [SAP BW Bridge，](https://blogs.sap.com/2021/11/17/announcement-introducing-sap-bw-bridge-for-data-warehouse-cloud/)推出了经典 SAP 业务仓库(BW)客户群的 SaaS 答案。SAP BW Bridge 提供了一种向云重新搭建平台的方法，同时保持所有根深蒂固的业务逻辑和定制不变。它在 [SAP 数据仓库云(DWC)](https://www.sap.com/products/data-warehouse-cloud.html) 上创建了一个新的工作区(“空间”)，目标是让庞大的 SAP BW 安装群能够提升和转移，利用云部署的运营优势，减少中断，或者至少减少 It 的冲击。

BW Bridge 支持 BW 7.3 和更高版本，是 DWC 内部的一个环境，支持 BW 中使用的经典 ABAP 代码和工件，直接连接到客户 DWC 租户运行。好消息是 BW Bridge 利用现有的 DWC 来建造:[空间](https://help.sap.com/viewer/be5967d099974c69b77f4549425ca4c0/cloud/en-US/268ea7e3e8d448deaab420219477064d.html)。DWC 公司成立时的想法是，不同的生产线组织需要自己的工作空间，因此传统的 BW 环境也是其中之一。

因此，这将允许传统 SAP BW 实施中的典型定制，如定制的收入报告期，保持不变。它避免了重新架构定制的任务，否则将 BW 应用程序移动到 DWC 是必要的。在传统 SAP 应用(包括 BW)中，它们是在核心 ABAP 代码中实现的，而在 SAP 云 SaaS 服务(如 DWC)中，基本代码保持不变，定制作为配置在其上实现。

毫不奇怪，鉴于这曾经被称为 BW 桥，它保持 ABAP 代码完好无损。客户既可以转换和迁移其现有 BW 实施的元数据(“外壳”转换)，也可以将元数据和数据本身(“远程”转换)发送到 DWC 空间目标。

## 数据采集和分级

按照目前的设计，SAP BW Bridge 主要用作数据采集和暂存层，这一直是 BW 的支柱。其他能力，比如规划，不在 BW Bridge 的范围内；相反，SAP 将 BW 客户引向 SAP 分析云(SAC)规划，目前这是一项单独的服务；今年晚些时候，SAP 将在基于 DWC 的数据模型上增加对 SAC 规划的支持。

毫不奇怪，SAP BW Bridge 的场景围绕着将 SAP 遗留数据源引入现代云数据仓库世界。它可能包括那些希望在托管云环境中复制其现有 BW 环境和运营报告的客户。当然，这也可能涉及在本地迁移更现代的 BW/4HANA。如上所述，通过跨空间共享，BW 数据可以与 DWC 开发的其他绿地分析共享，并与其他来源的数据混合。

SAP BW Bridge 承认，为了让传统或遗留系统进入云中，云 SaaS 提供商必须提供更大的灵活性，以适应遗留系统中渗透的定制类型。这就是 AWS 最近推出的 [RDS Custom](https://aws.amazon.com/rds/custom/) 背后的基本原理，与 AWS 的标准服务相比，它让客户能够控制更多 Oracle 和 SQL Server 实施。

这是一个谨慎的平衡行为，因为定制在什么程度上损害了云中的规模经济性？这就注定了在 20 世纪 90 年代 ERP 应用浪潮中出现的史前一代应用服务提供商(ASP)的命运。当时流行的观点是，至少让企业的 ERP 系统托管起来，从而减轻企业为 2000 年问题进行系统现代化的压力。每个客户的定制程度使得服务提供商的成本太高，无法提供有意义的节约。

云中的解决方案是安装护栏，比如 SAP 在 DWC 采用的空间。如上所述，SAP 吸引了一直维护 BW 实例的 ABAP 开发人员。在不久的将来，SAP 计划使用一个工具集来检查 SAP BW 和 BW/4HANA 对象(如数据传输流程(DTP)、数据存储对象、信息对象、转换、流程链等)的准备情况，从而实现迁移的自动化。)然后自动移动那些绿灯。

这是一个好的开始。当然，因为我们从不满足，这里是我们的愿望清单。我们希望这样的工具也能排除故障，并尽可能地自动修复对象，使其符合法规。实际上，让我们更进一步。我们希望看到工具能够自动复制现有环境的快照，包括数据和代码，并自动执行必要的转换以使对象准备好移动。这是因为在未来，负责迁移和维护 BW 实例的开发人员可能不具备同样的 ABAP 技能。

无论哪种方式，SAP BW bridge 都是如何将这些顽固的遗留系统纳入 21 世纪云的一个很好的例子。
