# 云铸造、微服务和多云未来

> 原文：<https://thenewstack.io/exploring-future-microservices-multi-cloud-universe-cloud-foundry/>

“我认为微服务也给了你一个很好的实验和尝试的方式。你可以快速迭代和部署应用程序，这也意味着你没有花太多时间，”行业战略副总裁 [Abby Kearns](https://www.linkedin.com/in/abbygregory) 在最新一期 [The New Stack Makers](https://thenewstack.io/podcasts/makers/) 播客中提到。

[在法兰克福举行的最近一次 Cloud Foundry 峰会上，Kearns 与 Pivotal 高级软件工程师](https://www.youtube.com/watch?v=hWIHU54oIo8) [Mike Gehard](https://www.linkedin.com/in/mike-gehard-82672a4) 和 Cloud Foundry 技术副总裁 [Chip Childers](https://www.linkedin.com/in/chipchilders) 一起讨论了微服务、Cloud Foundry 和 Zipkin 监控工具。

[利用 Cloud Foundry 探索多云世界中微服务的未来](https://thenewstack.simplecast.com/episodes/exploring-the-future-of-microservices-in-a-multi-cloud-universe-with-cloud-foundry)

随着企业考虑将其应用迁移到基于微服务的架构，Cloud Foundry 等平台为开发人员提供了快速迭代新应用设计和框架的机会。随着[最近宣布支持 Google Cloud](https://thenewstack.io/google-cloud-now-become-cloud-foundrys-home-base/) ，Cloud Foundry 为跨多个云运行应用程序搭建了舞台。

虽然有些人可能会对多云的未来犹豫不决，但 Childers 解释说，这最终将成为大部分技术生态系统的事实。“现实是，世界上将存在多种平台。他们应该合作得很好。我们谈论的应用程序可能依赖于不同云中的某些东西。随着这些演变过程，我们正在考虑许多其他组织的需求。”

Zipkin 是一个开源的分布式跟踪系统，允许那些运行微服务架构的人更好地解决延迟问题。基于 2010 年的谷歌论文 [Dapper](https://research.google.com/pubs/pub36356.html) ，Zipkin 拥有一个活跃的社区，目前正在为项目的开发做出贡献，GitHub 上有超过 4500 颗星。

“运行分布式系统很难。随之而来的是大量的知识。Gehard 说:“像 Zipkin 这样的工具可以去除微服务的粗糙边缘。

Kearns 解释说，Zipkin 为开发人员提供了一种细粒度的方法来解决影响他们的微服务的网络问题，并补充说，“要解决的最后一个问题是，当一个微服务开始行为不端时会发生什么？Zipkin 让开发人员能够通过他们的微服务架构跟踪请求。他们可以开始以非常直观的方式查看呼叫时间，或者微服务需要多长时间。如果他们遇到问题，他们可以针对特定的微服务。”

Gehard 继续指出，Cloud Foundry 的容器管理系统 [Diego](https://docs.cloudfoundry.org/concepts/diego/diego-architecture.html) 的未来更新将为开发者提供从入站流量路线开始启动 Zipkin 跟踪的功能。
