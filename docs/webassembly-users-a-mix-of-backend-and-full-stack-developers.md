# WebAssembly 用户混合了后端和全栈开发人员

> 原文：<https://thenewstack.io/webassembly-users-a-mix-of-backend-and-full-stack-developers/>

[Web Assembly](https://blog.scottlogic.com/2022/06/20/state-of-wasm-2022.html)[2022](https://blog.scottlogic.com/2022/06/20/state-of-wasm-2022.html)的[状态调查数据表明，并非所有的 Web Assembly (WASM)用户都是一样的，后端开发人员和其他所有人之间出现了分歧。](https://blog.scottlogic.com/2022/06/20/state-of-wasm-2022.html)

[WASM 周刊](https://wasmweekly.news/)的编辑科林·埃伯哈特在 2022 年 6 月调查了 297 人，其中几乎一半(49%)有至少三年与 WASM 共事的经历[。总的来说，那些没有什么经验的人和其他人没有什么不同，这表明有机会改变市场上的观点。](https://thenewstack.io/what-makes-wasm-different/)

经验很重要的一个领域是现有的用例。网络开发仍然是最常见的使用 WASM 的方式，被 72%的受访者引用。一半的人提到了无服务器，集装箱化和插件环境排在第四位。

与 2021 年相比，音频/视频处理和游戏开发作为 WebAssembly 的目标显著下降。与研究平均值相比，拥有 5 年以上 WASM 经验的受访者中有两倍多的人拥有游戏开发用例(21%比 10%)。有了活跃的[《牛逼榜》](https://awesomeopensource.com/projects/game-development/wasm)，这个小众大概不会消失。

工作角色是如何使用 WASM 的关键决定因素。

*   五分之一的研究对象是后端开发人员，他们显然专注于云原生世界。54%的后端开发人员拥有无服务器 WASM 用例，42%的人将其部署为插件环境。
*   相比之下，全栈开发者，占调查的 47%，压倒性地关注 web 开发(80%)。
*   根据最新的[栈溢出调查](https://survey.stackoverflow.co/2022/)，三分之二的前端开发者也认为自己是全栈开发者。因此，全栈开发人员更加关注 JavaScript 和基于浏览器的问题也就不足为奇了。

当被问及 WebAssembly 的未来时，人们普遍认为调试、集成和工具可以改进，但受访者区分了什么是“真正需要”和只是“需要”

后端开发者特别关心与浏览器无关的 API，因为 86%的人说他们真的需要 WASI 提供的那种集成。全栈开发人员更加重视与 JavaScript 和浏览器 API 的集成。后端开发人员也更有可能渴望 WebAssembly 支持更广泛的语言，但他们中的大多数人不认为这是 WebAssembly 未来成功的必备条件。

## 语言使用和计划

这项研究从 WebAssembly 使用了哪些语言以及人们的路线图上有哪些问题开始。大约 67%的人有使用 Rust 的经验，其次是 C++或 Emscripten (47%)，以及使用 WAT (41%)、JavaScript (31%)、AssemblyScript (29%)和 Blazor 或 C# (25%)。

总体而言，56%的 WebAssembly 开发人员表达了在未来将 Rust 与 WebAssembly 一起使用的强烈愿望。后端开发人员和那些专注于无服务器的人尤其热衷于使用 Rust。他们也比一般 WebAssembly 开发者更有可能真正想使用 Go 或 TinyGo。

值得注意的是，47%认为 WebAssembly 是“黑箱”的受访者计划大量使用 Blazor/C#。现在，49%的人以前使用过 WebAssembly 语言，但是只有 39%的人有过 Rust 的经验！这似乎是锈病不占优势的少数领域之一。
