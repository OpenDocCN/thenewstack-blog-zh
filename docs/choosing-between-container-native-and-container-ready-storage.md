容器彻底改变了开发人员创建和交付应用程序的方式。影响是巨大的；我们不得不重新设想如何存储、保护、交付和管理数据。软件开发人员有很大的机会来应对灾难恢复(DR)、数据保护、恢复点目标(RPO)粒度、数据严重性以及其他数据存储和管理问题带来的挑战。

Kubernetes 和其他容器环境将所有资源视为服务。这意味着需要一个存储服务层，尽管称之为数据服务层可能更好，因为这个服务层必须提供比简单存储数据多得多的资源能力。

在关于新的容器数据服务层的讨论中，出现了两种主要的方法:容器原生的和容器就绪的。鉴于市场上混杂的信息和不同的观点，这两者之间的差异有些不清楚。然而，即使是高层次的比较也掩盖了关键的区别。这两种方法各提供不同的东西，评估哪种解决方案最适合您的企业需求非常重要。

## **中枢神经系统对中枢神经系统**

容器连接(CAS)或容器就绪方法很有吸引力，因为它使用现有的传统存储—通常是外部阵列—通过软件垫片连接到 Kubernetes 环境。因此，它保证了重用存储中现有投资的能力，并且作为到容器环境的初始桥梁可能是有意义的。对于那些正在尝试容器化、计划以较小的规模进行容器化或者将其作为传统整体 It 环境的附属品的人来说，它可能是有效的。

容器本地存储(CNS)的不同之处在于它是为 Kubernetes 环境构建的。CNS 是一个软件定义的存储解决方案，它本身运行在 Kubernetes 集群上的容器中。Kubernetes 是为容器编排而设计的。Kubernetes 内部的一切都是一种资源，每一种资源都由 Kubernetes 管理和编排。

当需要额外的资源时，Kubernetes 增加更多的存储容量、连接服务和计算服务。它复制和分发应用程序实例。如果出现任何问题，Kubernetes 会在其他地方重新启动。作为编排层，Kubernetes 通常保持事情平稳运行。CNS 是为编排而构建的，但是容器就绪或容器附加存储并不容易编排。

Kubernetes 提供了灵活性，降低了操作复杂性和成本，但集装箱附加存储增加了摩擦，限制了 Kubernetes 的全部优势。为了与 Kubernetes 航海主题保持一致，将传统存储附加到 Kubernetes 环境中就像是将锚抛到了船外。采用 Kubernetes 的理由是希望/需要统一、简单、自我编排的 IT。容器连接方法所需的单独存储和数据管理无法扩展、无法适应，也无法以 Kubernetes 所需的速度响应。这在两三个节点的小型集群中可能还可以接受，但是一旦你开始扩展，你就会意识到它的局限性。

可能更重要的一点是，传统方法将主要数据和次要数据分开。主要是实时数据。辅助数据是受保护的数据(备份或快照)，即过去某个时间点的数据拷贝。这整个概念在 Kubernetes 的世界里被颠覆了。Kubernetes 中二手数据的概念让人想起亨利·福特的“更快的马”的比喻。在实现容器的过程中，我们完全从头开始重新思考它。将备份和快照等旧的数据保护方法引入这个新世界是没有意义的。

## **彻底的转变**

Kubernetes-native storage 改写了存储时空连续体，它提供了一个统一的数据服务层，可提供高速、灵活的“主”存储，即当前存在的实时数据。它可以轻松流畅地实现这一点，就像它提供“辅助”存储一样(即时可访问的实时数据克隆，就像它在任何以前的时间点存在一样)。)数据在这里或那里、当前或以前的整个概念都消失了。容器原生存储层提供随时随地对数据的即时访问。Kubernetes 可以自由安排何时需要什么。

在你的头脑中掌握这种新范式似乎很难，但想象一下对于一个 19 世纪的农民来说理解联合收割机的想法有多难可能会有所帮助。但是不要让一个艰难的范式转变让你停留在过去。面对颠覆性技术，传统厂商长期以来一直采用一种常见的战术:认可创新，利用 FUD(恐惧、不确定性和怀疑)来推迟采用，然后承诺一旦创新经过充分测试和证明，就提供创新。

这种模式曾经成功地阻碍了创新，同时保护了老牌供应商的收入流。令人欣慰的是，今天的云优先、无所不包的 DevOps 方法挫败了这一诡计，容器原生现在被广泛接受为前进的方向。

## 你的使用案例是什么？

如今，组织有许多存储选项，他们比以往任何时候都需要更多的存储。随着容器的加入，决策会变得更加困难。哪种方法最适合您的用例？要回答这个问题，您需要了解容器附加存储和容器原生存储之间的区别。对于新的应用程序，Kubernetes 是理想的选择，因为它使容器间的协调变化变得轻而易举。仔细考虑你的需求和你的管理能力，做出明智的选择。
