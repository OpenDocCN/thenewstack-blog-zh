# 开放应用模型:模块化应用的厂商中立规范

> 原文：<https://thenewstack.io/open-application-model-a-vendor-neutral-spec-for-modular-apps/>

微软和阿里云新宣布的开源项目[开放应用模型(OAM)](https://openappmodel.io/) 旨在分离开发者、应用运营商和基础设施运营商的关注点。

这个项目是在[开放网络基金会](http://www.openwebfoundation.org/)下开发的，但是目标是找到一个厂商中立的基金会作为它的家。

这个名为 [Rudr](https://github.com/oam-dev/rudr/) 的项目最初的实现是建立在 Kubernetes 之上的，但是这个模型并不局限于那个环境。相反，根据项目负责人的说法，它可以在小型设备上实现，如边缘部署或无服务器环境，Kubernetes 可能不是最佳选择。

“将应用程序定义从集群的操作细节中分离出来，使应用程序开发人员能够专注于他们应用程序的关键元素，而不是部署位置的操作细节，”该项目负责人在宣布该技术的微软博客文章[中说。](https://cloudblogs.microsoft.com/opensource/2019/10/16/announcing-open-application-model/)

此外，关注点的分离还允许平台架构师开发可重复使用的组件，并允许应用程序开发人员专注于将这些组件与他们的代码集成，以快速轻松地构建可靠的应用程序

开发人员正花费越来越多的时间在他们的应用程序的基础设施工具上，在安全性、性能、配置和与应用程序将在哪里运行相关的问题上，而不仅仅是应用程序本身。

该模型提供了一个自以为是的工作流，将开发人员和运营商的角色分开，同时创建了一个跨云平台和边缘设备工作的标准方法。

在该模型中，应用程序开发人员负责将每个组件定义为一个**组件示意图** YAML。这些模块可以用多种方式重新打包。

应用操作员负责创建这些组件的实例，并在**应用配置** YAML 中应用操作特征，如副本大小、自动扩展策略、入口点和流量路由规则。各个实例由特征定义，这些特征可以根据独特的操作需求进行配置，例如特定环境的法规遵从性和安全性。

基础设施操作员负责声明、安装和维护平台上可用的底层服务，例如特定的负载平衡器。

“开放应用模型的力量来自于实现该模型的底层平台，它可以通过 OAM 以一种在支持该模型的任何平台上一致的方式展现使底层平台独特和有用的能力，”该文档指出。

“通过一个平台无关的规范，专注于将开发问题与运营考虑分离，开放应用模型为在 Kubernetes 等平台上构建和交付应用带来了模块化、可扩展和可移植的设计，”项目负责人说。

“我们的愿望是建立一个厂商中立的、社区所有的规范，随着时间的推移，最流行的 API 将被接受并添加到规范中。通过这种方式，演进将确保大多数用户能够通过开放应用模型成功构建云原生应用。”
