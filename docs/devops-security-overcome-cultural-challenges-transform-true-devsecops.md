类似于几年前移动设备在企业中的扩散，当时组织感到拥有移动战略的压力，但不知道从哪里开始，我们在开发方法上看到了同样的情况。为了加快开发速度，团队感受到了“做开发工作”的压力，并且在集成安全性时，感受到了“做开发工作”的压力但是很像在最初的移动浪潮中，许多公司说他们正在实施这些方法，甚至可能认为他们正在实施，但实际上，他们没有。还没有。

首先，重要的是要记住 [DevOps](/category/devops/) 和 DevSecOps 不是职位或角色。它们是思维和文化的范式转变。大转变。但是必要的。如果公司不接受它们，或者至少不接受一种更加敏捷和安全的开发方式，他们就会落后。据 Gartner 称，到 2018 年，90%试图使用 DevOps 而没有明确解决其文化基础的组织将会失败。

> 如果你在写《战争与和平》，你是愿意边写边修改，还是把所有的修改堆在最后？

第二，变革之船转向非常缓慢。在将安全性集成到软件开发生命周期方面，法规仍然要求一年只进行两次测试。在一个开发者不断做出改变，黑客不断寻找漏洞的环境中，每年扫描几次是没有意义的，而且实际上是鲁莽的。不仅一家公司一次暴露几天，而且由于漏洞在这些测试窗口之间累积，开发人员随后被大量修复程序淹没，其中一些很可能已经在生产环境中。这无疑会破坏开发速度，并强化安全警察是梦想杀手的观念。我经常打这个比方——如果你在写《战争与和平》，你是愿意边写边修改，还是把所有的修改都堆在最后？

第三，如前所述，开发人员将安全监管视为创新和速度目标的障碍。理应如此。安全性一直处于软件开发生命周期的最右边。我们如何让他们参与进来，并创造这种开发团队文化？发展和安全是两种不同的语言，两者都没有时间或意愿去拥抱大局。使这变得复杂的是，组织使用过多的测试工具来交付不同的结果，并且由不断缩减的员工手动部署。不同的人运行不同的测试得到不同的结果，与不同的开发人员一起解决这些问题。这造成了低效的信息口袋，而不是一种有凝聚力的理解文化。这无助于推动开发人员对全局的认识。

这种向 DevSecOps 的文化转变是如何开始发生的？

让协作成为关键要素。安全性需要培养更多的工程思维，而开发需要对安全性漏洞有更好的认识和理解，并开始考虑每行代码的安全性。将测试自动化并编排到软件开发生命周期中可以促进这一过程——从代码提交(最左边是静态代码分析和源代码合成)到部署(最右边是动态渗透测试和 [OWASP](https://owasp.org/) 漏洞检测)。采用连续测试和反馈循环。将已经使用的不同工具的结果标准化并关联起来，以便对可操作项目的结果进行优先排序。

开发并不反对部署安全代码。他们只需要更快地被引入，并有更好的沟通。持续的测试将会暴露出持续的问题——毫无疑问——但是这个过程对于开发来说更加有效和有针对性，并且对速度的影响也更小。通过标准化数据和反馈，用流程编排、自动化和沟通来标准化文化。

传统上，安全性被视为速度和创新的障碍。但是通过将它无缝地、持续地集成到软件开发生命周期中，它实际上可以加速创新。
