# 会见绘制 Kubernetes 路线图的项目管理团队

> 原文：<https://thenewstack.io/plotting-kubernetes-roadmap/>

[谋划着库伯的路线图](https://thenewstack.simplecast.com/episodes/plotting-the-kubernetes-roadmap)

曾经有一段时间，操作系统是平台，所有的软件都是为特定的操作系统编写的。应用程序间通信(IAC)意味着与操作系统的注册实体“对话”,用操作系统的本地语言来表达。组件对象模型及其后来的变体(如 DCOM)与 Windows 结合在一起。

在容器化的环境中，仍然有操作系统，我们仍然把大多数充当流程载体的单元称为“Linux 容器”但是 IAC 不再是由底层操作系统平台推动的东西。作为一名指挥者，Kubernetes 促进了组件之间的联网。从现在开始，这种促进将如何发生，是组装[Kubernetes 路线图](https://github.com/kubernetes/kubernetes/milestones)的人的一个关键话题。

对于特别版的 [The New Stack Makers](https://thenewstack.io/podcasts/makers/) 播客，谷歌——我们即将推出的电子书《Kubernetes 生态系统的状态》的赞助商——召集了三名监督和参与路线图项目的成员:

*   [**Aparna Sinha**](https://twitter.com/apbhatnagar?lang=en) ，谷歌云平台产品管理负责人。
*   [**谷歌基础设施副总裁埃里克·布鲁尔**](https://www.linkedin.com/in/eric-brewer-1031254/) 。
*   [**Ihor Dvoretskyi**](https://www.linkedin.com/in/idvoretskyi/en/?ppe=1)，Kubernetes 社区产品经理；米兰蒂斯项目经理。

Sinha 说:“从架构上来说，Kubernetes 社区所做的就是定义一个接口。容器运行时接口就是其中之一。我们有网络接口、存储接口和其他类型的接口。有了这个接口定义，就有多个不同的容器运行时可以符合这个接口定义，因此可以与 Kubernetes 集成。”

“Kubernetes 发行版没有必要集成许多不同的运行时，但这是一种选择。Kubernetes 定义了一个运行时接口，因为有许多运行时是用户希望使用 Kubernetes 的。因此，这些团队或用户组为特定的运行时创建运行时插件。”
