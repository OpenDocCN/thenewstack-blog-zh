# 用 DataDog 衡量成功

> 原文：<https://thenewstack.io/datadog-kubecon-infrastructure/>

谈论思想的奇妙会面——在这一期的新堆栈分析师中，主持人 Alex Williams 与 Datadog 技术社区总监 [Ilan Rabinovitch](https://twitter.com/irabinovitch) 和 [John Mathon](https://twitter.com/john_mathon) 在他的[cloud 漫谈](http://cloudramblings.me/)博客上分享了他对开源和企业软件的想法，他最近是 WSO2 的产品战略副总裁。

这个节目是在 KubeCon 录制的，只听听 Mathon 就很棒，他是 TIBCO 软件的创始人，也是发布-订阅的发明者，让 Rabinovitch 参与讨论 DataDog 托管监控平台的功能。

[#69:用 DataDog](https://thenewstack.simplecast.com/episodes/69-measuring-for-success-with-datadog) 测量成功

这个播客也可以在 YouTube 上看到。

当他的客人深入研究监控和指标时，Williams 询问了持续分解应用程序架构的普遍需求，以及这如何产生如此多的要跟踪的数据和要理解的复杂性，以便获得用于分析的整体视图。

“这就是今天的挑战，”拉宾诺维奇说。“不仅仅是你的基础设施在消失。我们已经脱离了单一应用程序的想法，人们正在构建这些非常专用的应用程序，它们之间的区别在于 API。无论你称之为“SOA”还是“微服务”，或者今天的新的、令人兴奋的名称是什么，这是作为一个行业，我们已经努力了一段时间的方向。总的来说，像 Kubernetes 和 containers 这样的东西，以及云的动态基础设施，最终使它变得可行。”

Rabinovitch 观察到，十年前，与一个人的工程和运营团队谈论拆分到微服务会导致一个巨大的管理痛苦移交，同时还面临着很高的进入壁垒。“现在，我们在 KubeCon 的展览大厅里走来走去，你可以看到像 Deis 和 OpenShift 这样的东西，或者只是 Kubernetes 本身，整个目标是让你更容易建立这些较小的服务，并加快上市时间，”他说。

“这是我所知道的许多公司面临的一个现实问题，”Mathon 说。“我曾经合作过的一家公司非常习惯于为他们构建的每个应用程序构建一个应用程序映像。当我们向他们展示如何使用多种服务(共有 24 种服务)构建这个应用程序时，他们会问，‘你是如何管理的？“你是怎么做到的，”这是一个很大的变化。"

“微服务的概念是拥有最小的足迹，”Mathon 继续说道。“有些人会说，监控对于微服务来说是无关紧要的，会降低它的速度——它不小，它不受管理。”

Rabinovitch 说:“从您的应用程序中暴露指标可能需要一点额外的代码。“在某些情况下，平台会为您揭示这一点，您可以查看系统或基础架构级别的指标。但是，您知道您的应用程序的关键事务是什么，您也知道您感兴趣的是什么。”

“如果你不对你的应用进行测试，”他说，“那么你就不知道你是否成功了，除非你进行测试，否则你无法提高。”
