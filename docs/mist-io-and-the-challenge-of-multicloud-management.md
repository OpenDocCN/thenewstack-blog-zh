# Mist.io 和多云管理的挑战

> 原文：<https://thenewstack.io/mist-io-and-the-challenge-of-multicloud-management/>

有时，多重云就这样发生了。例如，一些组织可能在一个部门有运行在 [Amazon Web Services](https://aws.amazon.com/?utm_content=inline-mention) 上的应用程序，而与此同时，另一个部门可能会依赖 Google Cloud 或其他云提供商服务。

如何让它们在一个统一的架构下工作？多云管理的困难是最新一集 New Stack Makers 播客的主题，我们采访了多云管理平台提供商 [Mist.io](https://mist.io/) 、 [Chris Psaltis](https://www.linkedin.com/in/chpsaltis) 的首席执行官兼联合创始人。在 TNS 主编 [Joab Jackson](https://twitter.com/Joab_Jackson) 的主持下，Chris 讨论了跨多个云服务运行操作的内在困难和可能的解决方案，以及 Mist.io 如何提供帮助。

[Mist.io 与多云管理的挑战](https://thenewstack.simplecast.com/episodes/mistio-and-the-challenge-of-multicloud-management)

Mist.io 的起源可以追溯到大约十年前，当时 Psaltis 正在合作经营一家 IT 咨询公司，专门从事基于开源项目的 web 应用程序。“在某个时候，我们的客户开始采用云—他们来自世界各地，包括澳大利亚、美国和欧盟。一些客户使用 AWS，在 AWS 上运行，一些人在托管环境中运行，或者在本地拥有基础架构，”Psaltis 说。“我们很难管理所有这些异构基础架构。我们就是这样开始建造 Mist.io 的。一晃几年过去了，我们仍然在创造东西。”

管理单个云环境已经是一项挑战，因此管理多个云变得更加复杂。公共云之间有相似之处，但仍然不尽相同 Psaltis 说，当向组合中添加更多平台时，如私有云、裸机和容器的虚拟机管理程序基础设施或 Kubernetes，“事情会失控”。

“复杂性如此之大，以至于甚至连跟踪单个虚拟机这样简单的事情都变得非常困难，”Psaltis 说。“这种复杂性导致了各种各样的问题，首先是可见性，但它也减慢了团队的速度，使其很难知道谁有权访问什么以及有什么类型的控制措施。”

“多云通常是 M&A 的结果或历史原因，例如当新一代的基础设施和云被引入但旧的没有被淘汰，或者可能是出于法律原因，”Psaltis 说。“例如，他们需要在特定的管辖区域运行特定的操作，但只是在最近，我们才看到组织更加关注多云，并更加正式地追求多云，以实现更好的业务成果。”
