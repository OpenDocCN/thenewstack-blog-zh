# 系统故障和安全事故有多严重？

> 原文：<https://thenewstack.io/how-bad-are-system-failures-and-security-incidents/>

根据第二次年度 Verica 开放事件数据库(VOID)报告，非常非常糟糕。

我们都知道，系统故障和安全事故越来越多。问问那些在 1 月 25 日依赖 Azure 或微软数十种服务的人就知道了。最近， [Verica](https://www.verica.io/) ，一家使用[混沌工程](https://thenewstack.io/chaos-engineering-business-value/)帮助公司使他们的系统更加稳定和安全的公司，宣布了其[第二次年度 Verica 开放事件数据库(VOID)报告](https://www.thevoid.community/report)的结果，以查看事件的当前状态。

## 10，000 起事故

VOID 基于不到 600 家公司的近 10，000 起事件，从 [FAANG](https://thenewstack.io/more-data-engineers-crave-faang-jobs/) 和财富 100 强到创业公司。与通常的收集公司事后报告和状态更新不同， [VOID 研究人员深入挖掘分散在互联网上的软件事故报告](https://www.verica.io/blog/announcing-the-void/)，从滚动的状态页面到隐藏在公司网站模糊角落的报告。

Verica 这样做并分享其发现，因为它可以“让我们领先于软件驱动世界的潜在灾难性后果。”他们没有错。每天都看到另一个问题，另一个失败。对于像网络安全管理软件产品和 Log4j 这样不可忽视的人来说，许多其他更安静的失败仍然需要关注。

## 最新发现

在这份最新报告中，他们发现:

*   没有一家公司能免于事故。从初创公司到财富 10 强，各种规模的组织都会发生事故。软件在每个可能的行业中都是至关重要的，包括银行、旅游、农业、商业等等。
*   事件持续时间并不像看起来那样一成不变:在一个事件中有许多有见地的指标可以衡量。事件的持续时间对事件本身没有什么意义，部分原因是很难确定事件何时开始或停止。
*   sre 和类似职位的其他人应将平均修复时间(MTTR)作为一项关键指标。为什么？因为许多原因，MTTR 不是复杂软件系统可靠性的可行度量，特别是因为持续时间数据的平均值是谎言。
*   随着组织转向更有意义的度量和分析，组织正在远离像根本原因分析(RCA)这样短视的方法。
*   企业应该花时间和精力来分析和记录事件。这种实践有助于组织更好地理解他们的系统，以及如何在将来使它们不那么麻烦。

## 研究细节

这有些令人惊讶。但是，以前没有人如此详细地研究过如此多的事件。当你仔细观察真实数据时，你会发现更多关于任何学科的真实情况。

正如 Verica 的首席研究分析师和虚空的创造者 Courtney Nash 所说，“我们惊讶地发现事件的长度和它有多‘坏’之间没有关系。我们听到许多人怀疑，时间越长的事件可能越难解决。相反，有些人认为，对于真正严重的事故，公司应该全员出动，更快地解决这些事故。公司可能发生或长或短的事故，这些事故可能非常轻微，也可能非常严重，以及介于两者之间的各种组合。持续时间不仅不能告诉团队他们有多可靠或有效，而且也不能传达任何关于事件影响或处理事件所需努力的有用信息。”
