# KrakenD API Gateway 作为 Lura 项目加入了 Linux 基金会

> 原文：<https://thenewstack.io/krakend-api-gateway-joins-the-linux-foundation-as-the-lura-project/>

Lura 项目 T1，以前被称为 KrakenD 的开源框架，已经加入了 T2 Linux 基金会 T3，根据一份新闻声明，“它将是唯一一个托管在中立、开放论坛中的企业级 API 网关。”

该项目现在每月活跃在超过一百万台服务器上，[KrakenD API Gateway](https://www.linkedin.com/in/alombarte/)的联合创始人兼首席执行官 Albert Lombarte 说。向 Linux 基金会的转移把技术放在第一位，而不是建立在它之上的企业公司的需求。

“我们是开源的忠实信徒，我们相信开源是这个项目的归属，”Lombarte 说。“我们已经看到，这项技术与 API 网关不太兼容，正在采取的实践对技术来说不是最好的，”而是对营销或销售产品和锁定客户的需求而言。“我们想释放它，”他说。

KrakenD API 网关建立在现在称为 Lura 项目的基础上，Lombarte 解释说，KrakenD 是一个固执己见的实现，因为它关注的是速度而不是其他功能。Lura 是一个构建 API 网关的框架，可以根据组织的需求进行定制。它是为速度和可扩展性而设计的。Lombarte 说，Lura 内置于 Go 中，是一个无状态的高性能 API 网关框架，设计用于云本地和本地设置，无状态是一个区别点。

“如果您使用传统的 API 网关，您会看到配置存储在数据库中。我们没有任何类型的数据库，所以所有的配置都在一个静态配置文件中，”Lombarte 说。“如果您想改变网关的状态，您必须重新部署网关的另一个版本。它就像当今任何公司的任何微服务一样。这是我们和有状态网关的主要区别。”

此外，Lombarte 说，有状态网关的问题是可扩展性。“因为它们通常连接到一个数据库，而该数据库是唯一的事实来源，所有节点都需要协调。因此，如果你去找大客户，他们不能适当地扩展，因为他们有一个瓶颈，那就是数据库。我们的节点，它们甚至不互相通信，所以它们可以线性扩展到无限。”

Lura 的另一个与众不同之处是，它的 API 网关超越了传统的 API 网关，充当多个微服务的聚合器，而不仅仅是一个扩展代理。Lombarte 说，市场上所有的 API 网关都只是“中间有共同关注点的代理”，API 调用到达一个单独的后端服务。相反，Lura 允许通过调用它创建的单个 API 端点来聚合大量后端服务，并且每个后端服务实际上也可以提供不同的 API 类型。

“当你泛泛地谈论 API 网关时，人们通常会认为 API 网关将授权请求，也许还有速率限制。这是每个人对 API 网关的理解，但这只是用其他东西代理，”Lombarte 说。“我们所做的是创建一个新的 API，所以你不再消耗你的后端，而是消耗我们的 API 网关。它就像前端的后端。我们更接近 GraphQL，而不是市场上的任何其他 API 网关。”

Lombarte 说，如果你正在寻找这些强调速度的功能，那么 KrakenD 可能是正确的选择，但如果你需要专注于其他功能，那么 Lura 可以帮助构建你需要的东西。他说，只需 20 到 30 行代码，你就可以使用 Lura 建立并运行一个 API 网关。易贝和 Mail.ru 是目前使用 Lura 的大型组织的两个例子。他说，该技术提供了速度和无限的线性可扩展性，但也允许他们根据自己的需要定制 API 网关。

在 Lura 捐赠给 Linux 基金会之后，Lombarte 表示，Lura 2.0 预计将在不久的将来发布，支持 GraphQL，并能够进一步扩展该项目，将定制插件作为路线图上的新功能。
