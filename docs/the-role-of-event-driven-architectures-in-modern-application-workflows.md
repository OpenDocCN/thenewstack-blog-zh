事件驱动架构(EDA)由通过异步事件连接的松散耦合的分布式服务组成。这些系统非常适合当今不确定的时代，因为新的服务和数据源可以随时添加或删除，而不会中断现有的应用程序流。

一些更常见的 EDA 用例包括流处理、数据集成和客户旅程映射。如果我们看一下数据集成，您可能在内部有一个主产品数据库，可能在 Oracle 中。但是，也许您已经将企业资源规划(ERP)迁移到了云，并且您正在运行 Salesforce。集成这些系统的事件驱动的应用程序流将使整个组织受益，因此每当产品数据库中出现新价格时，都会触发一个事件，由 ERP 系统进行预测，并由销售部门进行询价和报价。

由行业主导的[云事件规范](https://github.com/cloudevents/spec/blob/v1.0/spec.md#notations-and-terminology)将事件定义为:

*“表达事件及其背景的数据记录。事件从事件生产者(源)路由到感兴趣的事件消费者。可以基于事件中包含的信息来执行路由，但是事件不会标识特定的路由目的地。事件将包含两种类型的信息:表示事件发生的事件数据和提供关于事件发生的上下文信息的上下文元数据。一个事件可能会导致一个以上的事件。”*

EDAs 由三个组件组成:事件生产者、事件代理(或总线)和事件消费者。困难在于，新的研究发现，93%的公司在多云环境中运营，这使得整合来自不同云的事件源和消费者成为一大挑战。

## 无服务器

无服务器计算是指构建和运行不需要服务器管理的应用程序的概念。它描述了一个更细粒度的部署模型，其中应用程序被捆绑为一个或多个功能，被上传到一个平台，然后执行、扩展和计费，以响应当前所需的确切需求。无服务器经常与功能即服务(FaaS)相混淆，但是它们并不相同。FaaS 是一种云计算服务，允许用户执行功能或单一用途的程序，并消除了通常与服务器和软件堆栈相关的复杂性。FaaS 云提供商包括 Amazon Web Services Lambda、Google Cloud Functions、Microsoft Azure Functions、IBM Cloud Functions 和 Oracle Cloud Fn。

无服务器计算并不意味着我们不再使用服务器来托管和运行代码；也不意味着不再需要运营工程师。相反，它指的是开发人员不再需要在服务器供应、维护、更新、扩展和容量规划上花费时间和资源。相反，所有这些任务和功能都由一个无服务器平台来处理，并且完全从开发人员和 IT/运营团队中抽象出来。

无服务器计算的显著特征包括:

*   **自动扩展，包括扩展到零:**传统云或内部部署应用运行并消耗计算、存储和网络资源，即使它们不在使用中。使用无服务器时，如果不调用该函数，所有计算和其他资源都会闲置。
*   **基于使用量的定价:**携手扩展到零，当一个功能没有被使用时，你什么也不用付。无服务器提供商对每个函数调用收费。
*   **事件驱动:**无服务器使开发者能够专注于由事件驱动函数组成的应用程序，这些函数响应各种各样的触发器。
*   **用例**:常见的无服务器用例包括电子商务、点击流分析、联络中心、传统应用现代化和 DevOps 功能。

### Knative 是如何融入其中的？

Knative 是一个开源的无服务器平台，它提供了一组中间件组件来构建现代的、以源代码为中心的、基于**容器的**应用程序，这些应用程序可以在任何地方运行:内部、云中，甚至是第三方数据中心。正如前面句子中加粗的“基于容器”所暗示的，这是 FaaS 和基于 Knative 的无服务器方法之间的真正区别。

在他 2020 年伦敦 NDC 大会上的精彩演讲中，谷歌开发者倡导者 Mete Atamel 将 Knative 描述为帮助解决开发者之前必须在无服务器或容器之间做出的权衡。有了 Knative，您可以获得两者——容器的灵活性、零接触供应和无服务器的快速迭代。

Knative 做两件事:服务和事件。

*   **服务**建立在 [Kubernetes](https://triggermesh.com/ufaqs/what-is-kubernetes/) 之上，以支持无服务器应用和功能的部署和服务。服务提供了 FaaS 的自动扩展(包括扩展到零)功能，以及使用现代网络网关的细粒度流量控制。
*   **事件**为消费和生产符合 CloudEvents 规范(由 CNCF 工作组开发的规范)的事件提供了构建块。它包括对事件源的抽象，以及通过由可插入的发布/子代理服务支持的消息传递通道的解耦交付。
