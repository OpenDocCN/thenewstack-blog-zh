# 容器、微服务和云原生技术

> 原文：<https://thenewstack.io/containers-microservices-and-cloud-native-technologies/>

[微服务和云原生技术](https://thenewstack.simplecast.com/episodes/microservices-and-cloud-native-technologies)

[微服务](https://thenewstack.io/containers-microservices-two-peas-devops-pod/)是最近的热门话题，尽管这个术语和技术实际上已经存在了一段时间。为什么大家突然对微服务架构感兴趣了？有什么变化？为了理解微服务，我在加州索诺马的开源领袖峰会上采访了 Docker 的首席开发者倡导者 Patrick Chanezon。

Chanezon 提供了一个关于微服务架构演变的简短历史课程，以解释为什么这些天每个人都在谈论它。他说，在 90 年代末和 21 世纪初，开发人员正在将他们的应用程序构建成我们所说的整体。这些应用程序需要专用的专有服务器来运行。对于 Java 应用程序，需要 Java 应用服务器。与。NET 时，需要 IIS 服务器。

在整体架构中，一切都捆绑在一起。如果您只想升级或更新一项服务，则必须更新整个应用程序。独石有自己的缺点。最重要的是，它们降低了组织的速度，组织最终会花费更多的资源来管理运行这些应用程序的基础架构，而不是投资于应用程序本身。

随着 Docker 的到来和过去五年容器的兴起，人们已经开始将这些大块分割成我们所谓的微服务。其思想是，应用程序可以构建为一组松散耦合的服务，这些服务可以在容器基础设施下单独更新和扩展。

这些容器平台与 monoliths 不同，独立于底层基础设施。因此，它可以帮助客户开始他们的云原生之旅。Chanezon 谈到了微服务如何融入云原生讨论。事实上，他指出，CNCF(云计算原生计算基金会)正在考虑更新云原生的定义，以获得更大的清晰度。市场在不断发展，这个定义应该包含更多自这个术语诞生以来出现的新技术。

他还指出了新技术的出现，如服务网格和功能即服务，这是云原生技术的发展。

Chanezon 还谈到了微服务的一些用例，并讨论了它没有意义的领域。他提供了一些 Docker Inc .客户的例子，这些客户正通过微服务架构开始他们的云原生之旅。
