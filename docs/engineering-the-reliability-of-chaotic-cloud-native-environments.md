# 工程混沌云本地环境的可靠性

> 原文：<https://thenewstack.io/engineering-the-reliability-of-chaotic-cloud-native-environments/>

云原生应用在可扩展性和速度方面具有优势。然而，尽管它们具有弹性，但随着应用程序组件数量的不断增加，这些系统的复杂性也在增加。对这些组件如何组合在一起的理解已经超出了容易理解的范围，进一步挑战了组织为系统复杂性可能产生的技术问题做准备的能力。

上个月， [ChaosNative 举办了](https://www.chaosnative.com/)第二届年度工程活动，[混沌嘉年华](https://chaoscarnival.io/)，我们在会上讨论了混沌工程的原理，并利用它们来优化当今复杂 IT 系统中的云应用。

在大型分布式软件系统中，当试图快速移动而不破坏东西时，就会出现混乱。“在一个普通的软件堆栈中，有数不清的依赖关系。如今版本很多，而且变化太快。几年前的部署模式已经发生了变化，”Satchitanand 说。Satchitanand 补充说，通过验证假设来创建实验，以可控的方式打破事物，并重申关于系统的假设，都是关于“选择正确的实验，但在云原生世界中很难”。

但是当你在生产中打破东西时，“真的很难知道从哪里开始。琼斯说:“许多公司往往想变得更加自信，但却没有必要花时间去回顾他们以前的经历。

马赫说，采用快速的混沌工程方法，同时确保最佳实验需要“试图找出可能破坏它的东西，而不是走已经破坏的东西的路”。“一个非常简单的方法是从提出大量的混沌用例开始，”Moorthy 补充道。

通往混沌工程的旅程就是拥抱复杂性。根据 Satchitanand 的说法，“混沌工程是关于学习新事物的。我们期望事情以某种方式强烈地表现出来，因为你之前已经测试过了，你也看到了。实际上，有些事情并不一定要以某种方式运行，但是你没有机会看到它是这样运行的。然后，有些事情你可能根本没有想到，必须通过实验来检验。这些都是接近混沌工程和创建用例的好方法。”

混沌工程的一个关键挑战是将 KPI 和混沌实验映射到 ROI。这种价值通常体现在“无形的东西上，比如对系统的信任和希望更少的停机时间。但这是那种你无法正确标价的东西。因此，这就是为什么你必须找到其他方式来让人们相信它的价值，”琼斯说。

但是，尽管在提高云原生应用的弹性方面存在诸多混乱，但它会导致用户“创建新的指标，定义新的 SLO，而这些都是你可能从未想过的。也许这个场景迫使您寻找信息，所以您最终创建了一些度量标准。这是混沌工程的附带好处，”Satchitanand 说。
