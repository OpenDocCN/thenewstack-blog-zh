Kubernetes 之于容器，AWS Lambda 之于无服务器。换句话说，Lambda 是无服务器的主导平台。当然，有一个关键的区别:与 Kubernetes 不同，Lambda 不是开源的。它属于亚马逊网络服务公司，世界上最强大的技术公司之一。但这并没有阻止一个新兴的创业生态系统建立在 Lambda 之上。[桑德拉](https://www.thundra.io/)，一款可观察性和安全性的产品，就是其中之一。

我最近与桑德拉的产品副总裁 Emrah Samdan 进行了交谈，以了解他对 AWS 平台的看法——包括其局限性——以及他如何看待无服务器的未来。

Serverless 是一套为开发人员抽象后端的工具，允许他们只关注前端代码。AWS Lambda 无疑是无服务器平台的市场领导者，而桑德拉为基于 Lambda 的无服务器应用提供可观察性和安全性。

“它可以很容易地插入到你的应用程序中，”萨姆丹说。

然而，这并不是说桑德拉为 AWS 生态系统提供了全新的东西。[亚马逊 CloudWatch](https://aws.amazon.com/cloudwatch/) 是一款现有的可观察性和监控产品，还有[AWS X-Ray](https://docs.aws.amazon.com/xray/latest/devguide/aws-xray.html)——一款与 CloudWatch 集成的调试和追踪服务。

但是根据萨姆丹的说法，这些 AWS 产品并没有满足“社区的需求”。他说，速度慢是一个问题，但(讽刺的是)在处理无服务器功能时也无法扩展。他声称，CloudWatch 和 X-Ray 很难应对任何具有五种以上功能的应用程序。

“当你有 20 个以上的函数，并且它们都与(一个应用程序的)事件相互关联时，事情就变得复杂了，”他说。萨姆丹认为，这是桑德拉的市场机会。

桑德拉还提供了核心无服务器平台之外的应用程序可观察性。萨姆丹告诉我，桑德拉不仅对 Lambda 进行“端到端跟踪”,还对容器、虚拟机和应用程序涉及的任何东西进行跟踪。

他举了一个应用程序的例子，这个应用程序使用了来自亚马逊 Kinesis 的一些数据，这是一个用于管理实时流数据的 AWS 产品。如果 Kinesis 流被应用程序的 Lambda 函数使用，根据 Samdan 的说法，该事务“在 AWS 本地解决方案中”是看不到的。但是它被桑德拉捡起来了。

## 对 AWS 的依赖

在商业平台上构建第三方产品的一个缺点是，桑德拉在许多方面都依赖于 AWS。我问萨姆丹，它能提供给客户的东西有什么限制吗？原来是有的。

“[当]我们想要检索一些数据，从[Lambda]中取出这些数据时，我们仅有的时间就是客户函数的执行时间。因此，我们和所有其他第三方供应商不可避免地会给应用程序增加一些开销。”

问题是，桑德拉必须从运行在 AWS 上的客户应用程序中提取数据，以便能够应用自己的软件智能——测量请求和响应时间以及完成特定工作所花费的时间。

“为了做到这一点，”Samdan 继续说道，“我们正在与我们的 REST 端点建立 HTTP 连接，或者我们正在从 CloudWatch 日志中写入这些信息。”

第一种解决方案使用 HTTP 连接，增加了开销。第二种情况会给客户带来 AWS CloudWatch 成本。

萨姆丹说，也很难提供关于 AWS 的其他类型的见解，例如关于“Lambda 和其他服务之间的交互”的业务指标。目前，桑德拉必须编写自己的代码来提供这些见解，所以理想情况下，它希望 AWS 提供一个 SDK 来协助。

但这就是依赖商业平台的第三方供应商的生活。这不像在 Kubernetes，像桑德拉这样的公司可以简单地剥离一个新的开源项目来满足业务需求。

## 无服务器限制

同样值得注意的是，无服务器模型本身也有局限性。

“功能即服务[FaaS]有一些限制，这是注定要有的，”萨姆丹说。例如，一个功能最多只能运行 15 分钟，或者它最多只能有 3gb 的内存。因此，这自动消除了一些用例。”

例如，游戏应用程序并不适合无服务器应用，因为它们通常需要大量的计算工作。

无服务器应用的另一个技术限制是管理数据；到目前为止，无服务器应用趋向于无状态。然而，从上周的 KubeCon 来看，这种趋势似乎正在发生变化 KubeCon 举办了几场关于“有状态无服务器”的会议(我将在即将到来的专栏中写下这种趋势)。

萨姆丹说，将你的应用程序切换到无服务器模式也存在文化障碍。

“你不能简单地‘搬来搬去’一个应用程序，让它在无服务器环境下运行；你需要重新设计一下。”

## 无服务器的未来

作为一名无服务器供应商主管，Samdan 自然对无服务器的发展前景充满信心。但是他认为它成为云服务的“默认”还需要五到十年的时间。

“[过去]这一次，无服务器技术将继续解决我们目前面临的限制。与此同时，当人们考虑将[他们的应用]带到云上时，他们会习惯于这种想法，“嘿，如果我租了一台服务器，为什么不租计算能力呢”——这就是所谓的功能即服务。”

有趣的是，萨姆丹到目前为止还没有看到很多人采用多云——这是云计算的未来愿景，客户可以使用多个云平台。

“老实说，我们以前没怎么见过这种用法。所以人们没有同时使用 AWS Lambda 和 Azure 功能。”

尽管如此，Samdan 已经开始注意到“越来越多的人在使用”多云，因此公司的路线图上最终会支持这一点。

无论无服务器的未来如何，桑德拉似乎已经在拥挤的可观测性市场中为自己开辟了一个利基市场。虽然我们在新堆栈中涉及的大多数其他可观察性和安全性产品都集中在 Kubernetes 上，但桑德拉已经对新兴的无服务器生态系统提出了要求。尽管它希望 AWS 继续成为这个生态系统的好管家。
