# Koyeb 在其无服务器引擎中结合了函数和容器

> 原文：<https://thenewstack.io/koyeb-combines-functions-and-containers-in-its-serverless-engine/>

无服务器体验的核心思想是轻松运行工作负载以响应事件，而无需管理基础架构。基本上，您编写代码，决定它应该何时运行，然后无服务器平台从那里开始执行。

对于今年早些时候推出的 [Koyeb 无服务器引擎](https://www.koyeb.com/)，这一想法被提升到了一个更高的水平，工作负载以各种预先编写的数据处理功能的形式出现，用户可以从中选择，这使得像[调整图像大小](https://github.com/koyeb-community/image-resize)或在特定事件发生时发送松弛通知这样的事情变得快速而简单。现在，该公司已经扩展了它的功能，不仅可以运行用户编写的函数，还可以在同一个事件驱动的平台上并排运行容器。

[Koyeb 联合创始人兼首席执行官 Yann Léger](https://twitter.com/yann_eu?lang=en) 解释说，用户通常不得不选择一种服务用于事件驱动功能，另一种完全用于容器。随着新推出的在 Koyeb 无服务器引擎上运行两者的能力，Léger 表示，他们希望简化开发人员的体验，允许用户运行他们拥有的任何代码，而不必担心时间限制或兼容的运行时。

“这实际上是关于开发人员的体验，并为他们提供他们想要使用的工具，因为形势已经发生了变化。Léger 解释说:“我们真的不再需要 DevOps 能力，而是希望为开发人员提供一种方式，让他们能够拥有这些能力，而不必了解正在发生的事情。“我们真的希望简化开发体验，并抽象出您在部署这类应用程序时必须进行的复杂设置。我们消除了当前产品中的一些限制，最终用户必须在功能和容器之间选择特定的云技术，例如，当他们只想能够处理图像和视频时。我们让他们根据自己的需求选择最合适的技术。”

作为这些限制的一个例子，Léger 指出了亚马逊网络服务的 Lambda 无服务器服务，他说该服务不允许用户执行一个功能超过 15 分钟，使其不适合更繁重的数据处理。相比之下，Koyeb 允许用户运行函数和容器长达 24 小时。

“我们发现，大多数无服务器领域都集中在 FaaS(功能即服务)、‘轻量级’数据处理和简单的数据转换上。另一方面，容器通常用于繁重的处理，因为它们允许您嵌入所需的系统依赖项，但它们的设置更复杂，”Léger 说。“我们正在通过统一的功能和容器平台来弥合这一差距，这两种技术都具有相同的部署简单性和可扩展性。”

虽然这个版本的主要焦点是以各种形式运行用户创建的代码的能力，但 Koyeb 也开源了它在 [GitHub 资源库](https://github.com/koyeb-community)中提供的预写函数，其中还包括安装和配置 Koyeb CLI 的操作，以及帮助用户入门的 Hello World 函数示例。这个版本还引入了与 Git 和 GitHub 的集成，进一步简化了部署过程。用户不需要像使用 AWS Lambda 那样将功能打包成 ZIP 文件，只需要将 Koyeb 与他们的存储库连接起来，这样部署就像推送代码一样简单。

Léger 解释说，目前，Koyeb 专注于异步功能，使得对冷启动时间等问题的关注变得不那么相关。然而，展望未来，他说他们计划在明年早些时候进入同步领域，因为像使用 Python 提供 HTTP API 这样的事情是他非常感兴趣的领域。
