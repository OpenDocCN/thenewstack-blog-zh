# Split Software 的 Adil Aijaz 谈论敏捷部署

> 原文：<https://thenewstack.io/adil-aijaz-of-split-software-talks-agile-deployment/>

[Adil Aijaz 谈拆分软件的敏捷部署](https://thenewstack.simplecast.com/episodes/adil-aijaz-of-split-software-talks-agile-deployment)

大规模快速构建软件需要大量的基础设施、过程和管理。虽然像脸书和谷歌这样的一些公司可能会让持续集成/持续部署(CI/CD)看起来很容易构建，但实际上，这两家公司都花费了数十亿美元来优化他们的构建管道，并通过消除构建/测试/修复反馈循环中的障碍来提高开发人员的生产力。

对于我们其他人来说，有许多方法可以帮助改进 CI/CD 过程，而不需要像脸书曾经做的那样，实际上编写自己的构建系统或花费 10 亿美元将 PHP 编译成 C++。相反，众多的供应商、开源项目和 CI/CD 专家正在帮助您的团队更快地从编写代码到将代码部署到生产环境。

一家旨在帮助解决 CI/CD 加速问题的公司是 [Split Software](https://www.split.io/) 。 [Adil Aijaz](https://www.linkedin.com/in/adilaijaz/) 是 Split 的首席执行官和联合创始人，他和我们一起讨论了企业世界中敏捷部署的现状。他说，团队需要首先理解系统中发生的所有变化和更新，以便能够管理和优化这些环境，从而加快部署。

Aijaz 说，持续部署工具的市场不同于变更管理软件的市场，后者传统上帮助团队处理这个问题。“即使到今天，我们仍然处于这样一个世界，大多数工程师和开发人员；他们考虑的关键动词是速度和单位时间内的部署。公司就是这样奖励和衡量工程团队的。但我认为有新的曙光。你会看到文化的改变。我们开始将工程工作或产品变化与业务成果和产品成果联系起来，”Aijaz 说。

Aijaz 很早就看到了这些文化变化，因为他从 2009 年到 2012 年在 LinkedIn 工作。“我得到了前排座位。当我加入 LinkedIn 的时候，我是大约 200 名员工。这是一个古老而乏味的公司，坐在这个非凡的、可防御的业务的顶端。但是即使作为一个内部人员，我们所拥有的是，我们只是在软件开发方面有组织上的障碍。我们不能超过一个月发布一次。这真的很糟糕，因为我们会有这种发布培训模式来发布软件。它会每月发生一次，发布列车应该在 25 日发出。你在 18 号会有一个代码冻结，很多工程师会在 18 号开始为发布列车编码，这将导致发布列车的延迟，25 号变成了 1 号，这是一个永无止境的软件发布周期。
