# GitLab 找到了使用遗留系统持续交付的最佳途径

> 原文：<https://thenewstack.io/gitlab-finds-the-best-path-to-continuous-delivery-with-legacy-systems/>

GitLab 赞助了这个播客。

[GitLab 的技术宣传主管 Priyanka Sharma](https://www.linkedin.com/in/pritianka/) 最近发表了一篇博客文章，描述了 git lab 如何在没有首先将其底层 IT 基础设施完全转移到 Kubernetes first 的情况下开始其连续交付(CD)之旅，这让许多人感到惊讶。

在这一集的 New Stack Makers 播客中，Sharma 还在[云本地计算基金会(CNCF)](https://www.cncf.io/) 的董事会任职，他进一步讨论了 GitLab 如何在向 CD 迁移的过程中不一定选择最新最棒的工具集。相反，重点是找到最直接的 CD 途径，同时保持对 GitLab 遗留系统的投资和支持。

Sharma 写道:“我们正在向 Kubernetes 迁移，随着功能开发速度的提高，我们需要比过去更加频繁地部署。“但是，我们没有使用 Kubernetes 完全实现现代化，然后启动 CD，而是选择通过使用我们现有的传统工具和大量 smarts，将我们现有的 CI/CD 系统推向极限。”

[GitLab 的没有 Kubernetes 的 CD 之旅](https://thenewstack.simplecast.com/episodes/gitlabs-cd-journey-without-kubernetes)

目前，如上所述，GitLab 向 CD 的转变并不是以容器为中心的，而是依赖于虚拟机。“我认为我们的交付团队这样做是非常明智的，因为他们不需要做任何改变来创建一个管道，让我们可以进行许多、许多、许多连续的交付提交，以及许多、许多全天到达金丝雀的提交，”Sharma 说。“我们现在正处于一个我们将开始慢慢转向 canaries 的地方，因为现在我们已经用尽了我们可以用这个遗留系统做的事情。”

换句话说，文化、思维方式，当然还有工具集和底层基础设施都必须到位。夏尔马说，这个想法是“以对你有意义的速度前进，做对你的公司有利的事情，最重要的[元素]往往是工作流程的变化，所以要把它们分类:这就是我们给客户的信息”。“他们听得又清楚又响亮。”

Sharma 说，她观察到客户选择单一应用程序开发运维选项，“因为这给了他们比孤立工具更多的东西，”Sharma 说。Sharma 说:“单一应用程序之所以出色，是因为它实现了真正的 DevOps 工作流。

作为一种更直接接触客户的方式，GitLab 正在举办 GitLab Commit 会议。为了更好地参与 GitLab 社区，讲座和会议将包括 CD 和其他广泛的 DevOps 主题，从 9 月份在纽约布鲁克林举行的开幕活动开始。

大会”是为了我们更大的 GitLab 社区。Sharma 说:“我们所有人都是建设者、制造者，每天都在操作软件。“这很大程度上取决于有哪些做法？相对于谈论 DevOps，我们如何实际执行 DevOps？”

Sharma 说，在 GitLab 的采用迅速发展到 10 万个组织和“平台上数以百万计的开发人员和运营人员”之后，召开了这次会议。“所以我们意识到，现在是组织一次活动把这些人聚集在一起的时候了。”
