当我们想到 DevOps 时，我们想到的是速度、灵活性、创新和卓越的技术。但是生产力是导致它大受欢迎的另一个驱动力。IT 组织厌倦了开发人员和运营部门之间的孤岛，有时甚至是权力斗争。它们只是导致浪费时间，缺乏所有权和低劣的技术。

虽然 DevOps 带来了更多的创新以及更顺畅、更少中断的工作流，但有一件事仍然阻碍着体验 DevOps 生产力的全部潜力:云管理。

现在，开发运维与云几乎成了同义词，开发运维正面临着削弱工作效率和敏捷性的挑战。我们将在下面深入探讨其中的一些。

## 云监控和优化挑战

简而言之，持续监控和优化云环境是不可能手动完成的。这意味着 DevOps 团队不断执行琐碎的任务，照看他们的云基础架构，以预测未来规模、实时分析使用情况并管理动态环境。他们在使用这种方法时遇到的一些问题包括:

### 要分析的数据点太多

管理云需要 DevOps 工程师利用当前使用模式的实时数据以及未来预测来说明规模。这些预测需要提前很长时间达到 100%准确。不用说，在应对动态环境、波动的客户群和新技术的实施时，100%准确的预测是不可能的。

此外，为了实时采取行动，DevOps 工程师需要不断监控他们当前的云使用情况，以适应容量需求的变化。这意味着他们需要全天候值班，以接收来自其监控工具的警报。因此，很容易出现警报疲劳，这可能会使他们完全过滤掉这些通知。最后，所有这些数据都难以管理，非常繁琐，需要对电子表格进行持续的分析和无尽的计算。

### 实时做出太多决定

假设您的 DevOps 工程师能够轻松准确地收集这些数据。现在，他们必须立即决定这些数据意味着什么，以及前进的最佳方式。例如:假设您的应用程序规模显著扩大。适应这种规模的最佳实例是什么？解决这个问题需要时间、精力和手工计算。这些时间肯定可以更好地用于实施新技术、开发新功能或改进流程。

### 太多琐碎重复的任务

大多数 DevOps 工程师热爱他们工作的什么？创新能力——参与构建有价值的技术，为他们喜欢的开源项目做出贡献，当然，还有解决复杂问题的机会。他们讨厌什么？把时间浪费在重复的任务上，修复坏掉的系统，做一些很容易自动化的单调工作。不幸的是，管理云涉及太多的后者。

例如，假设您的 EBS 磁盘即将达到极限。半夜发出警报，无论哪个 DevOps 工程师待命，都必须起床手动扩展您的 EBS 磁盘，以防止应用程序故障。EC2 也是同样的过程。如果您希望在扩大和缩小规模时优化成本，DevOps 工程师将需要不断地购买和出售承诺，或者配置他们的应用程序以在现场实例上运行。根本没办法赢。

## 利用人工智能自动化提高 DevOps 生产力

正如我们已经确定的，DevOps 工程师不是保姆。他们是高度合格和有才华的工程师，通过构建新的和创新的技术而茁壮成长。因此，云管理的繁重工作通常被视为开发运维工作效率的障碍，因为它需要持续的监控、配置和调整。大部分工作不可能 100%有效地完成，这也于事无补。

谢天谢地，有更好的方法。

人工智能自动化非常适合处理重复性的日常任务，如分析实时数据、预测未来规模、调整基础设施以适应需求的变化等。此外，它可以非常精确地完成所有这些工作。

如果 DevOps 团队不断地扑灭云基础架构中的火灾，他们就无法达到预期的工作效率。通过自动化他们不喜欢做的任务，您的云保持完全优化，同时您的开发运维工程师能够更高效地从事他们最喜欢的工作。
