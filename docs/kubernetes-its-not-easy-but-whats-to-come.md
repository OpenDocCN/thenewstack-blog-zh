# Kubernetes:这并不容易，但接下来会发生什么呢？

> 原文：<https://thenewstack.io/kubernetes-its-not-easy-but-whats-to-come/>

贾纳基拉姆·MSV 是为数不多的分析师和作家之一，也是 Kubernetes 社区的认证成员。他能够将详细描述日常用户和开发人员面临的问题的分析带入他的作品。他也是 Kubernetes 和其他技术上新堆栈的[定期贡献者](https://thenewstack.io/author/janakiram/)。

在本期的[中，新的堆栈制造商](https://thenewstack.io/podcasts/makers)播客在 [KubeCon + CloudNativeCon 北美 2019，](https://events19.linuxfoundation.org/events/kubecon-cloudnativecon-north-america-2019/) Janakiram 讨论了为什么开发人员的快乐是一个如此大的挑战，配置器的新角色以及随着 Kubernetes 的成熟和隔离成为一个更大的问题而出现的问题。他还讨论了多租户和用户采用服务时面临的挑战，以及定义较少的集群隔离所带来的安全风险。库伯内特斯:这并不容易，但接下来会发生什么？

Janakiram 说，Kubernetes 困惑的根源可以追溯到“缺乏平台风味”。“Kubernetes 被认为是基础设施，并且仍然是现代应用程序的基础设施，但它缺乏应用程序的上下文，”他说。"这是开发人员采用 Kubernetes 的最大缺点之一."

Janakiram 说，另一个主要问题是 Kubernetes 如何“模糊了开发人员和管理员之间的界限”。Kubernetes 已经成为开发者和利益相关者可以合作的一层，这也是 Kubernetes 的魅力之一。然而，“对于这些人物角色中的每一个来说，都很难处理这个叫做 Kubernetes 的作品，”他说。

组织也开始意识到在 Kubernetes 上应用策略和管理安全性和合规性的挑战，以及 DevOps 团队如何处理多租户。Janakiram 说，对于开发者来说，与 Kubernetes 合作有“多个里程碑”要经历。例如，开发人员可能会处理 Docker 映像中的源代码，然后将 Docker 映像打包到 YAML 文件中，以便在应用程序转换为 Kubernetes 中的一系列对象时进行部署。

“它不仅仅是一个应用程序，而是一套服务，从一个非常一致的开发者体验中抽离出来，”Janakiram 说。“这对开发人员来说也是一个巨大的挑战，因为他们必须了解配置映射秘密、持久卷、网络策略或诸如此类的东西的本质，然后才能为最终用户发布您的 URL。这是一条真正的学习曲线。”

此外，DevOps、DevSecOps 以及网络、存储和平台管理员都使用相同的 Kubernetes 层。Janakiram 说:“对于他们如何处理这个叫做 Kubernetes 的项目，还没有明确的划分，这就是主要的挑战所在。”

Janakiram 说，退一步说，“开放应用模型”可以提供一些指导，帮助应对 Kubernetes 的学习曲线挑战。“这就是我所提倡和提议的。”
