# 从发布渠道引入管理

> 原文：<https://thenewstack.io/bringing-management-in-from-the-release-pipeline-cold/>

将企业过渡到不同的软件开发模型总是一个文化之旅，无论是从遗留瀑布到敏捷、精益、CI/CD 还是 DevOps 方法。

在遗留模型中，企业中的远见者看到了变革的需要，并将其传递给企业中的其他成员，这些成员根据定义的时间表开始旅程，如“计划”、“估计”、“进度”、“测试”和“交付”在这些团队中，团队成员拥有明确的角色，比如业务负责人、职能负责人、测试人员等等。然而，在过渡到 Scrum/Agile 世界的过程中，角色发生了巨大的变化；工作变得更加孤立，时间表变得更短，每个团队都独立工作，以便尽可能快地进行交付。

## 大寒

这不是一条畅通无阻的道路——曾经如此定义的通信线路现在变得模糊了。典型的与团队的管理关系突然消失了，向产品负责人更新项目状态的角色现在由 Scrum master 承担——一个技术角色。

敏捷团队是自治的筒仓；这是好处，同时也是坏处。每个团队有目的地独立工作，但是这使得真正知道事情在哪里是一个挑战，直到他们进入生产——并且这些团队仍然被期望互相交付。这里的问题是:管理在这一切中处于什么位置？

当在大型企业中进行扩展时，这个问题变得更加复杂。走向敏捷意味着将一堆固定的迭代编译到时间表中，但是他们仍然有一个必须遵守的整个功能集的整体交付时间表。仅仅因为方法改变了，并不意味着整个项目所依赖的主机也改变了。仍然有遗留组件和传统团队，新系统必须围绕它们进行协调，即使已经实现了，新系统也需要成功地与测试活动集成。例如，每个版本都有合规性问题；在向最终用户发布任何新代码之前，企业需要对运营和管理进行安全审核和批准。

更进一步，在敏捷之后是持续交付和开发:与传统模式相比，这实际上是文化变革的“西部荒原”。团队可以完全控制自己的方向；他们按需交付，没有更多的迭代，更广泛的业务甚至远离跟踪活动。实际上，管理层受到了冷落。

## 打破僵局

那么，一个组织如何克服这个管理问题呢？答案是将思维从“自上而下”转变为“自下而上”。关键是改变领导风格，将注意力从指挥和指导团队转移到授权他们自主工作，以及与更广泛的业务目标相关联。在这里，管理层在确保团队相互沟通和有效协作方面发挥着不可或缺的作用。

将管理带回管理开发是通过多个交付管道的协调来实现的，这并不容易。根据发布时间表以及其他因素(如变更顾问委员会和治理、法规遵从性和测试问题)协调发布的时间意味着管理层仍然有大量的工作要做。

本质上，发布经理坐在管理层和各个团队之间，寻找灌输持续改进的方法；然而，为了走得更快，这些部件仍然需要被组合在一起。

## 穿越暴风雪

为了使管理成功，经理需要查看所有工具的所有指标，并将它们汇总在一起。手动完成这项工作将会非常困难，因此能够实时整合系统中所有功能的工具是持续改进的真正推动者。

那么，改变的是发布管理的角色变得更像是一个“异常管理”,管理者就像一个调查报告者——寻找问题和可以优化的领域，以推动整个系统向前发展。这些工具可以帮助组织确定哪些团队结构在公司内工作得最好，提供一段时间内衡量指标的历史视图，以了解哪些地方正在进行改进——反过来，管理以及交付流程的质量也会变得更好。

本质上，发布经理充当“产品所有者的 scrum master ”,导航跨越所有交付管道的技术和商业问题。他们接管了协调业务、产品负责人和 scrum 大师的角色。为了帮助这个过程，企业发布经理也应该寻求测试环境经理来帮助支持自治团队。它们提供自助服务环境来管理内部和基于云的生产前环境，以及需求和供应、环境质量和供应，从而帮助实现整体应用交付增强。
