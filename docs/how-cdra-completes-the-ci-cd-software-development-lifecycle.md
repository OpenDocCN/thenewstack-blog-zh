# CDRA 完成 CI/CD 软件开发生命周期

> 原文：<https://thenewstack.io/how-cdra-completes-the-ci-cd-software-development-lifecycle/>

当谈到大规模软件开发时，持续交付和发布自动化(CDRA)是持续集成/持续交付(CI/CD)发展的下一步吗？

[Forrester Research](https://go.forrester.com/) 是这样认为的。该分析公司将 CDRA 描述为组织通过自动化数字管道和改善端到端管理和可见性来更快、更安全地交付更高质量软件的一种方式。

在本期 New Stack Makers 播客中，CloudBees 技术战略副总裁 [Anders Wallgren](https://www.linkedin.com/in/anderswallgren/) 讨论了 CDRA、支持工具以及开发运维团队在交付软件时面临的目标和挑战。CI/CD 系统提供商 [CloudBees](https://www.cloudbees.com/?utm_content=inline-mention) 在报告“[The Forrester Wave:Continuous Delivery And Release Automation，Q2 2020](https://www.cloudbees.com/analyst-report/forrester-wave-continuous-delivery-release-automation) ”中被评为领先的 CDRA 供应商

这一集由《新书库》的创始人兼发行人亚历克斯·威廉姆斯主持，TNS 执行主编约阿布·杰克逊协办。

[连续交付和发布自动化(CDRA)在 CI/CD 结束的地方拾取](https://thenewstack.simplecast.com/episodes/continuous-delivery-and-release-automation-cdra-picks-up-where-ci-cd-ends)

在许多方面，CDRA 非常类似于 CD，因为它是承诺的基础，然后交付软件，Wallgren 说。“价值流 CDRA 实际上是让你的价值流在持续的实时基础上变得可执行和可见。因此，它需要所有的东西、所有的活动、所有的工具、所有的平台、所有的软件——你为构建、测试、验证、部署和发布你的软件所做的一切——并使之自动化。

“理想情况下，我们的观点是关于一个平台，过度的编排，将你已经拥有的所有自动化 CI/CD 孤岛捆绑在一起，”他说。

许多组织在采用持续集成和自动化过程方面已经相当成功。“然而，在你的软件准备好放在网站上之前，你还需要做大约 58 件其他的事情，比如烧录到芯片里，放进盒子里等等，”瓦尔格伦说。“CDRA 实际上只是承认，大多数大型成熟软件公司仍在从事发布活动，我们还没有生活在一个我在笔记本电脑上编译软件，然后两分钟后，它就投入生产的世界里。”

在软件发布之前，DevOps 团队需要做的“58 件其他事情”包括:安全扫描、软件组成分析、静态分析、运行时分析、渗透测试以及“围绕软件用户验收的构建、测试和资格认证所做的任何事情，”Wallgren 说。

对于受到严格监管的行业，如金融、医疗保健、汽车、航空航天、国防和其他具有严格审计要求的组织类型，自动化至关重要。这些类型的组织“必须是可治理和可控制的，我的观点是自动化就是审计。”

“我认为，当你深入到审计和控制等所有细节之下时，它归结为记录你做了什么，并证明你做了你记录的事情，”瓦尔格伦说。“自动化定义是一种文档形式，尤其是在当今……所以它是关于真正收集所有这些来自其他组织、其他部门、其他职能部门的零碎信息，并将它们从筒仓中取出，放到我们都将使用的大工厂传送带上，以正确的方式将我们的软件运出门外。”
