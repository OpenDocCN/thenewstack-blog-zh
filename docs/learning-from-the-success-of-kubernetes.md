# Kubernetes 联合创始人谈 K8s 的过去、现在和未来(这并不都是美好的)

> 原文：<https://thenewstack.io/learning-from-the-success-of-kubernetes/>

[KubeCon + CloudNativeCon](https://www.cncf.io/kubecon-cloudnativecon-events/) 赞助本帖。

[Kubernetes 联合创始人就 Kubernetes 的过去、现在和未来提出了尖锐的问题](https://thenewstack.simplecast.com/episodes/kubernetes-co-founders-field-hard-questions-on-kubernetes-past-present-and-future)

许多人对 Kubernetes 的对错有自己的看法。在 5 月于巴塞罗纳 [KubeCon + CloudNativeCon](https://events.linuxfoundation.org/events/kubecon-cloudnativecon-europe-2019/) 现场录制的播客中，新的堆栈向 Kubernetes 联合创始人 [Tim Hockin](https://www.linkedin.com/in/tim-hockin-6501072) 、GKE Kubernetes 首席软件工程师[、Google Cloud](https://cloud.google.com/) 和 VMware[副总裁](https://www.vmware.com/)Craig McLuckie 提出了一些关于 Kubernetes 过去和未来的更难回答的问题。他是 Kubernetes 的创始人之一。新的堆栈还征求了微软合作伙伴项目经理 Gabe Monroy 的想法，他是微软 Azure 容器的首席产品经理，自称是“最积极参与的早期社区成员之一”。

虽然 Kubernetes 的创作者肯定对 Kubernetes 的创作结果感到高兴，但 Hockin 说它的受欢迎程度是“你永远无法预测的”“就吸取的教训而言，这不是你可以计划的事情，但话说回来，我们可以尝试，”霍金说。

但就 Kubernetes 的普及和采用的爆炸式增长而言，McLuckie 指出，在部署和运营 Kubernetes 这样的技术时，预测每一种消费和使用模式是不可能的，而且将继续是不可能的。

然而，从一开始，麦克卢奇就注意到项目创始人是如何考虑“社区的巨大力量”的。“这个项目实际上不是关于任何一个人或任何一个团体，而是创造一个机会，将一个非常强大和积极参与的社区聚集在一起，这个社区是根据一系列共同的理想建立的，以创造积极的结果，”麦克卢奇说。因此，如果有一条对其他项目的建议，那就是围绕任何事情建立一个有效的参与式社区都有很大的效用和效率。"

Kubernetes 社区确实看到了与 Docker 的分裂。例如，当时在 Deis 担任首席技术职位时，Monroy 说，“我们是社区中第一批对 API 进行公开押注的人之一，并在某种程度上放弃了像 [Docker Swarm](https://thenewstack.io/kubernetes-vs-docker-swarm-whats-the-difference/) 这样的事情。”“这可以归结为几件事:一个是社区背后的机器——不仅仅是有一个社区，而是有一个运转良好的机器，”蒙罗伊说另一个因素是，我认为直到今天(这确实是社区的一个重要因素)，Kubernetes 今天已经能够…做任何创建它的人都无法想象的事情。"

但另一方面，麦克卢奇称赞了 Docker 的“垂直整合的效用”，以及 Docker 如何利用这一点“为特定的结果创造完美的策划和调整的体验。”

“所以他们做了很好的工作，你知道，从代码到容器，并创建了一个真正可访问和可用的工具链，”McLuckie 说。“当我思考这个社区以及它为什么围绕 Kubernetes 而不是 Docker 发展时，我认为这实际上可以归结为一个问题，即效用是由垂直集成驱动的，还是由模块化驱动的，因为 Kubernetes 是一个本质上更加模块化的系统。这使得每个供应商都能看到它，并找到一些方法来创造独特的价值，”麦克卢奇说。他们可能会眯着眼睛说，‘我可以添加这个，然后创造价值，我可以创造巨大的价值。’因此，成为整个社区的天然聚集点，而由于 Docker 更关注垂直集成和封闭系统体验，它对许多希望带来资源和消费者的其他供应商来说并不那么感兴趣。"

霍金说，在接下来的五年里，Kubernetes 不应该“完全不同”。“我觉得会更成熟；会更稳定；它将更加健壮，将更加优雅地处理更多的用例，”霍金说。麦克卢奇说，事实上，这个计划是为了让“Kubernetes 变得无聊”。“但我不认为这将是一个比今天根本不同的事情，”麦克卢奇说。

除了大肆宣传之外，Kubernetes 应该代表“完美一致的基底，可以很好地缝合在一起——如果我们做好我们的工作，它将带来全新的创新浪潮，我们将要谈论的东西，而不是 Kubernetes，将是建立在 Kubernetes 基础上的下一代分布式系统，”McLuckie 说。

最终结果应该是“确保作为一个行业，我们可以重复我们能够用 Kubernetes 所做的事情，并确保我们在社区基础上构建的下一层和下一套系统也是多供应商开源的，并欢迎非常多样化的开源贡献者，”Monroy 说。
