# OASIS 将开放源代码和开放标准与开放项目相结合

> 原文：<https://thenewstack.io/oasis-combines-open-source-and-open-standards-with-open-projects/>

在开源软件的核心，存在一套共同的价值观和方法，帮助定义一种协作和共享的方法。根据为安全、物联网、内容技术和其他领域的开发开放标准提供平台的非营利性全球联盟组织结构化信息标准促进组织(OASIS)的说法，开放标准共享这些共同的价值观，并启动了 T2 开放项目，为组织提供一个新的空间，使他们能够更紧密地连接开放源代码和开放标准工作。

[OASIS 的首席开发官 Carol Geyer](https://www.linkedin.com/in/carolgeyer) 在接受 New Stack 采访时解释说，开放项目的出现是因为意识到开放源代码和开放标准具有这种共性，不应该再被分开对待。

“所有组织都意识到开源和标准之间的界限越来越模糊，统一的方法可以带来很多好处。通过开放项目，OASIS 带来了 25 年以上的标准专业知识，”Geyer 说。“我们知道如何生产得到政府和法律机构认可的产品，这些产品将被纳入公共采购和政策中。我们知道如何给予团队适当的流程和治理水平，以便他们的工作不偏离轨道，同时确保社区保留自主权和以对他们有意义的方式工作的能力。”

根据公司声明，通过开放项目，OASIS 打算为“任何开发代码、API、规范或参考实现的倡议”提供一个统一的家园，寻求“一条标准化和法律上批准国际政策和采购参考的道路”。声明接着说，新举措将赋予社区“在开源许可下，在一个地方开发他们选择的东西——API、代码、规范、参考实现、指南——的权力，并获得全球政策和采购的认可”。

本月早些时候启动的开放项目包括前两个开放项目——[生命周期协作开放服务(OSLC)](https://github.com/OSLC) 和 [OpenDocument 格式(ODF)倡导](https://github.com/ODF-Advocacy/odf-advocacy)——以及一个开放项目咨询委员会。该委员会将致力于提供“对开源项目需求的战略洞察力，确定当前行之有效的实践，探索新的方法，以改善开源的未来”，并由来自微软、Cloudbees 和 Apache Software Foundation 等公司的成员组成。

Geyer 解释说，开放项目还将为政府提供一个参考点，以确定开放源代码软件是否是根据某些原则创建的。

“我们的标准已经得到了 ISO、ITU 和类似机构的认可。政府，尤其是欧洲政府，依赖于此来告诉他们标准是以公开透明的方式产生的。当他们接触开源时，政府中有更多的困惑——他们更希望在他们知道可以信任的东西上盖个章。我们通过接受标准和开源，给他们一个可以做这两件事的地方，并给他们一条获得批准的道路。“政府希望确保事情是在一个开放的过程中完成的，每个人都有机会为标准提供信息，所以不只是三家公司聚在一起，把真正植根于他们自己专有的东西放在一起。开放是关键。绿洲流程确保所有这些事情都以符合国际期望的方式完成。”

根据 Geyer 的说法，此举不仅仅是为开放源代码和开放标准提供一个统一的空间，而是与咨询委员会一起，将提供一个重新思考创建开放源代码软件和开放标准的方法的机会。

Geyer 说:“我们将此作为一个机会，不仅将标准和开源结合起来，还将审视两者的工作方式——哪些是有效的，我们希望继续做什么，但也许我们还可以在治理和许可方面做得更好。“一切都摆在桌面上。”
