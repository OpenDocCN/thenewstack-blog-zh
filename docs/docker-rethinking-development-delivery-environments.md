# Scott Johnston，Docker:重新思考应用程序开发和交付

> 原文：<https://thenewstack.io/docker-rethinking-development-delivery-environments/>

[](http://www.docker.com/)

[Scott Johnston](http://www.docker.com/)

[拥有超过 25 年的行业经验，从软件开发和产品营销到 IT 运营和风险投资，Docker 产品管理高级副总裁 Scott Johnston 负责 Docker 的产品和技术战略。Johnston 曾在 Puppet Labs 担任营销、产品管理和战略副总裁，并在 Netscape、Cisco 和 Sun Microsystems 担任领导和运营职务。](http://www.docker.com/)

[](http://www.docker.com/)[](http://www.docker.com/)

在本播客中，我们采访了 Docker 产品管理高级副总裁 Scott Johnston，讨论了容器作为微服务和分布式计算的合理起点，以及 Compose 和 Universal Control Plane 等 Docker 工具如何提升微服务的使用。

Johnston 描述了软件交付的不太遥远的过去，那时大代码块需要几周甚至几个月才能交付。这是在编程基础设施、云供应商、配置管理以及 Docker 出现之前，Docker 帮助团队重新思考开发环境并缓解交付管道中的摩擦。

Johnston 说:“速度不仅仅由你的开发团队工作的速度来衡量，还由你的端到端应用交付管道的吞吐量来衡量。应用开发团队必须与他们的运营团队合作来部署应用、更新应用和更改应用。”

[斯科特·约翰斯顿，Docker:开发和交付的速度](https://thenewstack.simplecast.com/episodes/scott-johnston-docker-the-speed-of-development-and-delivery)

“编程基础设施绝对让人们尝到了运营方面的可能性，”他说，并补充说，使用容器“在开发人员、运营团队和系统管理员之间提供了一个预先的合同，或一个明确的界限。”

它立即给出了一个合理的起点，开发人员的外围应用是已知的，系统管理员的外围应用也是已知的

关于 Docker 的最新产品及其背后的理念，Johnston 承认，一些工具具有开发人员自助服务的方面，“这是有意的，我们希望促进团队的快速移动和快速迭代，并在应用开发管道中快速移动。”

“但是，这些工具的另一个非常重要的方面是为运营团队提供他们需要的工具，以便为基础架构配置 Docker 守护程序，并为其配置 Swarm 集群，然后(非常重要的是)设置策略和访问控制，同时仍然为他们提供一定程度的安全性、控制和治理，但允许开发团队自助服务并部署到这些配置和可用的集群中。”

“这种新的架构使这些团队能够更快地移动，我们的工具需要能够支持这一点，同时仍然给运营人员他们需要的治理控制，”他说。“找到平衡点，让运营部门获得控制权，但不减慢这一过程。”
