# 如何克服无服务器集成的困难

现在，我们大多数人都知道开发人员如何依靠无服务器平台在云中随意部署应用程序，而不必担心配置或管理硬件、操作系统或服务器。这种潜在的自由跳跃允许开发人员花更多的时间创建和编写代码，一个经常被忽视的潜在绊脚石是影响集成的问题。

我们通过研究无服务器集成的影响和结果来研究这个主题。基于我们的发现，我们得出结论，构建应用程序和主要管理集成的编程应用程序之间的差异正在迅速消失。然而，在这一重大转变发生之前，无服务器环境将需要一种集成语言来支持这些用例。

在本帖中，我们将讨论为什么组织需要采用一种专门用于集成 IT 环境和无服务器平台的编程语言。

## 未来应用程序编程

构建应用程序和构建集成之间的差异正在迅速消失。云、API 和无服务器架构的兴起使应用程序开发更像集成，有效地使集成无处不在。应用程序由 API 组成，编写这些应用程序所需的逻辑类似于编写集成。因此，在未来的系统中，对类似集成的技术的需求将更加普遍。这一发展将大大拓宽集成的可用市场，同时推动现有的编程语言和平台为编写集成提供一流的支持。

我们可以使用 Java、Go 和 Python 等编程语言编写简单的集成程序。然而，无服务器架构中的复杂集成也继承了企业集成环境中出现的大多数问题。例如，这些集成需要通过组合来自不同供应商的 API 来构建，这会导致时间、数据格式和上下文不匹配。使用编程语言(如 Java)来实现这一点需要大量的详细代码。此外，随着微服务和 API 的广泛采用，集成中涉及的服务数量可能会增加，从而增加集成和服务的复杂性。为了处理这样的场景，企业服务总线(ESB)供应商，如 [WSO2](https://wso2.com/) ，Mulesoft 和 IBM，已经提供了集成语言。云中的集成可能也需要类似的工具。

历史上，早期的集成是使用经典的编程语言编写的，如 COBOL、C、Java 和 C#。然而，大多数集成用例涉及服务、API、数据接收器和数据源之间的通信，然后匹配它们的输入、输出、定时和控制流。经典编程语言没有一流的服务调用结构，也不处理 XML 或 JSON 之类的数据格式。因此，开发人员需要从基本原则重新实现逻辑，这导致代码冗长、编写复杂且难以维护。为了应对这一挑战，内部集成供应商已经支持用于编程集成的特定领域语言。

目前，无服务器平台用户依赖语言，如 Java，来实现他们的集成用例。然而，随着更复杂的用例转向无服务器架构，我们可能会重新发现多年前我们面对这些语言的局限性。因此，缺乏支持集成的高级语言一直是无服务器计算面临的挑战。

这可以通过几种方法解决:

*   用户可以在基础设施即服务(IaaS)解决方案中运行集成中间件，并使用它来集成无服务器应用程序。
*   云提供商可以支持集成中间件，将其作为他们的平台即服务(PaaS)服务之一。
*   云提供商可以支持集成语言，例如，[芭蕾舞演员](https://ballerina.io/)或特定领域语言(DSL)，作为 PaaS 支持的语言。
*   解决方案提供商可以扩展对一般集成的可视化编程支持。

## 结论

在探索无服务器对集成的影响时，我们得出结论，构建应用程序和构建集成之间的差异正在迅速消失，无服务器环境将需要一种集成语言来支持这些用例。我们相信这一观察将有助于组织规划其系统的长期战略。

我们发现，需要大量集成支持的无服务器平台具有显著的影响，表明这些平台是可行的。假设无服务器平台将支持一种集成语言，我们期望在未来三年内看到许多新的应用程序采用无服务器架构。由于这些应用程序使用 API 并与外部系统交互，随着应用程序迁移到云中，集成用例也将随之迁移到云中。我们调查结果的完整报告可以在[这里找到。](https://github.com/wso2/ETAC/blob/master/outlooks/serverless4integration_outlook.md)
