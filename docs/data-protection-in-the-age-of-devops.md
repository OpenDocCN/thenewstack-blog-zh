随着组织开始接受数字化转型(通过改进分析、自动化业务流程或创建新的业务模式)，这需要通过采用微服务架构和使用敏捷方法来彻底转变应用程序的开发方式。通过分离开发团队，IT 团队获得了惊人的灵活性。我们现在看到许多公司每月(甚至更频繁)发布应用程序。

然而，这需要一个与规避风险的传统 IT 运营方式截然不同的系统。DevOps 是一种文化心态，通过让每个开发团队负责他们自己的代码部署和失败时的恢复，帮助组织实现了这一点。为了适应这种快速的变化，每个版本签入都包括应用程序代码版本、应用程序数据架构更改以及关联的应用程序定义和配置的组合。

Kubernetes 已经成为能够实现所有这些目标的唯一系统。因此，大多数组织都采用 Kubernetes 作为实现数字化转型目标的标准。

为了理解灵活环境下的内容，让我们讨论一下存储层在 Kubernetes 时代是如何发展的。

容器允许抽象。但是，在某些时候，环境需要与存储系统进行对话，存储系统可以是外部存储或超融合存储系统。在 Kubernetes 早期，实现这一点的方式是，环境需要以不同的方式与每个存储系统进行交互。对于那些拥有多个存储系统选项的人来说，它变得非常专有和笨拙。

Kubernetes 社区提出了一种称为 CSI(容器存储接口)的东西，这种接口使环境能够以统一的方式与所有这些不同的存储系统进行对话，以便自动调配、连接和装载数据块和文件存储。此外，Kubernetes 存储兴趣小组认为快照操作是许多有状态工作负载的关键功能，因为它使工具能够备份数据。

[Kubernetes 卷快照—推出 beta 1.17](https://kubernetes.io/blog/2019/12/09/kubernetes-1-17-feature-cis-volume-snapshot-beta/) 是为了应对这一挑战。这是一个标准 Kubernetes 部署默认启用的功能，并在多个 OEM 和第三方发行版中广泛可行，如 Openshift、迪亚曼蒂、Anthos 和 TKG。随着 Kubernetes 1.17 逐渐被用于在文件级系统拍摄快照，这增强了与 CSI 驱动程序集成的数据保护解决方案。

这对许多企业来说是一件好事，他们终于能够将其企业级数据保护解决方案扩展到 Kubernetes 环境。

在如此快速的变化中，发生故障的可能性很高，拥有良好的数据保护解决方案对于确保同样快速地恢复到稳定状态至关重要。

但是，大多数数据保护解决方案主要关注应用程序数据；并且这些会导致回滚期间的配置漂移。保护跨各种 Kubernetes 对象的相关环境配置是很重要的，包括配置映射、机密等。

此外，让数据保护解决方案在自助服务模式下可用也很重要，这样开发人员和应用程序所有者就可以针对 Kubernetes 环境管理他们自己的回滚和数据保护策略。IT 系统管理员仍将使用管理整个基础架构(包括裸机、虚拟化和集装箱化环境)的通用界面保留完全访问权限，以帮助应对更广泛的系统故障。

总之，Kubernetes 环境正在被各组织广泛采用，因为它们正在转向微服务、敏捷和 DevOps 文化，以帮助实现快速转型。创建这些环境的公司以及拥有更好体验的最终用户都从中受益。随着 Kubernetes 1.17 和 CSI 标准的成熟，现在是企业采用它们并寻找企业级数据保护解决方案来帮助保护这些环境的时候了。
