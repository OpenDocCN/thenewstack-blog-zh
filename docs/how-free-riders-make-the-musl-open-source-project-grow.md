在标准的开源[祝福](https://en.wikipedia.org/wiki/Beatitudes)(“有福了…因为他们将…”)，基本上没有人记得[里奇·费尔克](https://twitter.com/RichFelker)的补充:“开源用户是有福的，那些不贡献一行代码的人，因为他们将使你的项目繁荣。”对于 Felker 来说，musl libc 项目的创始人，他的项目的用户可能会胜过所有其他人——包括代码贡献者——通常被认为是开源社区的必要条件。

一些开源用户以“免费搭车者”或“索取者”的身份出现，但对 Felker 来说，他们是让开源工作的团队。毕竟，开源软件的广泛使用鼓励其他人努力确保互操作性，以及其他好处。

## 概念证明

 [马特·阿萨伊

Matt 是 AWS 的负责人，参与了开源及其支持的所有领域(云、机器学习、数据基础设施、移动等。)近二十年来，为各种开源公司工作，并定期为 InfoWorld 和 TechRepublic 撰写文章。你可以在推特上关注他(@mjasay)。](https://www.linkedin.com/in/mjasay/) 

大多数人不会听说过 musl，它是构建在 Linux 系统调用 API 之上的 C 标准库的实现。令人讨厌的东西。正如 Felker 在采访中解释的那样，musl 是用户空间中最基本、最底层的库代码。换句话说，它是“粘合剂”，或者，如果你愿意的话，是应用程序和 Linux 内核之间的桥梁。Musl 提供了这些应用程序需要与 Linux 内核对话的接口，这样它们才能运行。

当 Felker 在 2005 年作为一名学生开始修补时，开发人员可以使用其他库来与 Linux 内核接口，包括 uClibc 和 glibc。前者缺乏稳定的 API，后者难以升级，当时的社区“对用户的批评或错误报告不太友好。”几年后，Felker 决定建立 musl libc，不是为了与现有的图书馆竞争，而是作为一个展示 glibc 未来概念的地方。

“当时我真的没有想到它会流行起来，被很多人使用，”费尔克说。(*旁白:Musl libc 实际上会迅速发展，并被许多人使用。*)

尽管 musl 很受欢迎，但对 Felker 来说，它一直是一项热爱的工作，而不是一项“大的创收活动”。偶尔会有短期的约定来以某种方式改进 musl，但这通常只发生在客户想要 Felker 和 musl 社区已经打算建造的大东西的时候。因此，在很大程度上，Felker 花在开发 musl 上的大部分时间是由 Patreon 或 T2 GitHub 赞助商 T3 资助的。费尔克说:“这两者都非常有助于让我专注于此。”。

## '比写任何代码都有价值'

尽管 Felker 做了大量的 musl 开发工作，但多年来，围绕它已经形成了一个社区。费尔克说，这是一个欢迎所有人的社区。这不是一个人们感觉会受到攻击或骚扰的地方。这不是意外。可能是为了回应 glibc 对贡献过敏的名声和 Linux 内核有点有毒的名声，Felker 特意去做简单的事情，包括最简单的事情。“重要的是要真心感谢别人。他说:“传达出你有多重视他们所做的事情。

甚至，事实证明，那些停下来使用代码的人也没有任何回报。

除此之外，在费尔克看来，他们*做*简单地通过使用 musl 回馈一些东西。费尔克说，除了感谢人们，当人们问“谁是真正的贡献者？”时，他还会特意公开纠正他们的错误或者，“谁在写代码？”

为什么？因为费尔克说那些写代码的人不是真正的贡献者。“那只是很小的一部分，”他解释道。“真正的贡献者是那些让这一切成为可能的人，他们告诉我们什么时候出现了问题，或者在利益可能冲突的不同人之间进行协调，以便在问题出现问题时找出解决问题的方法。”

再说一遍？“用户、测试人员、采用者、错误报告者比编写任何代码都有价值。”

## 间接代码贡献

虽然这可能看起来违反直觉，但在 musl 的世界里却非常有意义。尽管 C 语言很小，但它很可能是所有软件库中界面最大的。想一想[所有用 C](https://www.reddit.com/r/C_Programming/comments/43c42c/most_famous_programs_written_in_c/) 写的软件，或者用另一种语言写的但是运行时是建立在 C 上的软件，包括 50000 多个开源程序。他们中的每一个人都对像 musl 这样的标准接口应该如何工作有所期待。Felker 说，如果没有很多用户要求 musl 的上游兼容性，“我将不得不花费 10 到 100 倍的时间来为数百甚至数千个其他软件产品安装补丁和修复程序，否则就会出现 musl 构建问题。这是不可能的。但是随着许多人使用它，希望事情正常运行，并且不希望用他们自己的构建维护他们自己的外部树补丁，这一切都发生了。人们很早就接触到了上游。”

从某种意义上说，这些用户是“免费搭车者”,当然，他们没有贡献代码，但是他们使用 musl 的行为促使其他人编写确保兼容性的代码。难怪 Felker“考虑那些基于 musl 构建发行版并报告他们在 musl 中发现的每一个问题的人，当他们发现需要修复的可移植性问题时，向其他项目的上游报告事情，做所有在不同方之间协调的工作；对 musl 最重要的贡献。”这种贡献在 musl 的早期就开始了，并导致了今天 musl 几乎一切都开箱即用的局面。

或者，正如费尔克后来[强调的](https://twitter.com/RichFelker/status/1258509698778624001)，用户创造了对其他软件互操作的需求，并可能修复与 musl 的代码贡献者可能永远不会看到的其他软件的互操作性。毕竟，这种“使用你的软件的贡献者”可以简单地使用其他的东西，迫使一个项目去做令人难以置信的艰苦工作，自己做所有的兼容性测试。相比之下，费尔克感谢那些仅仅因为使用 musl 而为 musl 做出贡献的人的政策不仅仅是仁慈的。这是明智的。

那么你如何为 musl 做贡献呢？嗯，当然要用。但是还有其他方法。Felker 希望花更多的时间在 musl 上，并参与 C 语言和 POSIX 的标准进程。如果你或你的公司想为 musl 的持续成功做出贡献，考虑通过 [Patreon](https://www.patreon.com/musl) 或 [GitHub 赞助商](https://github.com/sponsors/richfelker)成为赞助商。
