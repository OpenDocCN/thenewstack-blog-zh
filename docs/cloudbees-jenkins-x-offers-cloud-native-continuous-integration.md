# CloudBees 的 Jenkins X 提供云原生持续集成

> 原文：<https://thenewstack.io/cloudbees-jenkins-x-offers-cloud-native-continuous-integration/>

今年是玛丽·雪莱的《弗兰肯斯坦》出版 200 周年，因为这个标志性的怪物继续在我们的集体心理和想象中扮演着非常活跃的角色。

雪莱怪物的幽灵也是软件开发世界的一部分，就像“詹金斯”这个臭名昭著的术语指的是构建在开源 Jenkins 持续集成服务器上的大型 CI/CI 平台，现在扩展到 Kubernetes 上的部署，以及最近的无服务器平台

在最近的[devo PS World | Jenkins World 2018](https://www.cloudbees.com/devops-world/nice)，[cloud bees](https://www.cloudbees.com/)CEO[Sacha Labourey](https://www.linkedin.com/in/sachalabourey/)也提到了“弗兰肯斯坦·詹金斯”以及该公司的一个新项目 Jenkins X 如何能够提供帮助。在一次主题演讲中，他讨论了 Jenkins X 在帮助自动化和改善从连续交付管道到 Kubernetes 和云原生环境的平稳过渡的开发体验方面将发挥的积极作用。

在对新堆栈的一对一采访中， [CloudBees](https://www.cloudbees.com/) 的高级架构师 [James Strachan](https://www.linkedin.com/in/jstrachan/) ，Jenkins X 的项目负责人，讨论了 Jenkins X 对 Kubernetes 以及现在的无服务器平台的支持如何反映了 CloudBees 以及 Jenkins X 正在变得更加以云为中心。他详细介绍了 CloudBees 和 Jenkins 社区将如何通过这种方式，继续推动生产管道从内部部署向云本地部署的大规模转移。

显而易见的目标是让原生云开发世界摆脱臭名昭著的 Jenkinsteins，这种情况在过去组织开始依赖 Jenkins 来快速大规模自动化持续集成和持续部署时经常发生。

“我认为我们都在努力变得更加云原生，并以云原生的方式转变我们正在做的事情。这就是 CloudBees 如何在很大程度上使 Jenkins 更加云原生，并将 Jenkins 转变为一个更加云原生的系统，”Strachan 说。“这也反映了我们大多数人在我们的产品中构建自己的内部应用程序时所做的事情。”

Strachan 说，这个想法是扩展经典的 Jenkins 插件模型，使其更接近云，“所以你可以使用更多的微服务，使用 Kubernetes 等的客户资源，这样任何人都可以用任何语言扩展这个平台。”

“Jenkins X 的主要目标之一是尽可能地重复利用一切有益的东西。Strachan 说:“Kubernetes 做的任何事情，我们都使用它，而且在 Kubernetes 生态系统中有一大堆开源项目。"这是一个围绕 Kubernetes 的令人难以置信的多样化和充满活力的开源生态系统."

分析公司 [Redmonk](https://redmonk.com/) 的联合创始人 James Governor 没有谈到他是否认为弗兰肯斯坦的幽灵，甚至 Jenkinsteins，会不会成为云原生部署中通常不受欢迎的一部分——尽管他对 CI/CD 与容器上的云原生部署集成的未来持乐观态度。

感谢开源社区，以及包括 Jenkins 在内的许多工具，Jenkins 应该会继续看到改进，尤其是通过帮助企业“更好地测试和左移”，Governor 说。“作为组织利用自动化测试和构建过程进行的转型的一部分，现在的问题通常是他们如何将所有这些整合在一起，”Governor 说。“我们现在看到的是堆栈的重建，这正在云原生技术中全面展开。”
