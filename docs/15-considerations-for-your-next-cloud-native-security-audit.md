在云环境中，安全审计变得更加复杂和微妙，组织通常在多个云提供商上运行应用程序。在本帖中，如果你是一个在多云基础设施上运行应用的组织，我们来看看首要考虑事项；尤其是对这些应用程序进行安全审计的重要性。

让我们从评估您当前合规状况的 15 大考虑事项开始:

1.  我们是否实时跟踪系统的变化？
2.  我们是否在添加新资源时就发现了它们？
3.  我们是否持续监控威胁和漏洞？
4.  我们是否有适用于新资源的合规性标准列表？
5.  我们合规流程的哪些部分可以自动化，哪些需要手动完成？
6.  我们对核心 DevOps 工具(如 GitHub 和 Jenkins)的使用有合规性吗？
7.  我们是否有容器图像使用的合规措施？
8.  我们审计我们使用的开源项目的安全实践吗？
9.  我们是否有跨所有云位置的法规遵从性状况的集中视图？
10.  我们是否收集所有正在使用的云服务和云资源的审计日志？
11.  我们能够从审计日志的噪音中收集信号吗？
12.  我们是遵循 RBAC 还是更细粒度的基于属性的访问控制？
13.  有多少用户对我们的云资产拥有管理员或超级用户权限？
14.  我们如何处理许可中的冲突？
15.  我们是否有办法让身份请求执行关键任务所需的额外访问权限？

[DNSStuff](https://www.dnsstuff.com/rbac-vs-abac-access-control) 正确地简化了关于安全性的讨论，他说，“安全性的两个基本方面是认证和授权。”认证是关于用户是谁。授权包括他们可以访问哪些资源、资产和数据。虽然这两者都很重要，但是让我们在这里详细讨论一下授权。

## **基于角色的访问控制(RBAC)已经过时**

RBAC 早在 1992 年就被引入，此后几乎没有什么变化。这是一个很好的方法，可以减少每个人的手动配置，只给一组人相同的访问级别。但是，对于今天多样化的云原生栈来说，它太初级了。通过宽松地给予许可而不是限制它们，RBAC 使得最小特权原则难以实施。更好的选择是 ABAC。

## **输入基于属性的访问控制(ABAC)**

[据 TechTarget](https://searchsecurity.techtarget.com/definition/role-based-access-control-RBAC) 报道，ABAC 基于属性组合来控制访问——用户属性可能包括姓名、国籍、组织、ID、角色、安全许可等等。资源属性的例子包括所有者、名称、数据创建日期等等；而环境属性包括访问位置、访问时间和威胁级别。

实施 ABAC 的两个重要组成部分是策略和元数据。策略定义了谁受访问规则的控制。这些策略基于如上所列的属性或元数据。

政策的范围可能有所不同。有些可以是适用于多个云位置的通用策略，而有些只能适用于特定的云实例。与角色不同，策略可以重叠。它们更具延展性和灵活性，这使得它们非常适合系统、身份和权限总是在变化的云原生设置。策略可以跨云环境、应用程序和数据集进行复制。通过这种方式，他们可以节省时间并改善安全状况。

为了让策略正常运行，它们依赖于底层元数据的质量。这需要对数据进行准确分类，并随着系统的变化进行持续维护。元数据标签的一些例子包括公共、机密和绝密。还可以有特定于团队、应用程序和服务的标签。随着数据生命周期的继续和数据与其他数据的结合，元数据应该保持不变，甚至变得更加细粒度。

最初，用元数据进行标记的过程会涉及大量的人工工作；但是应该尽可能自动化。像 Prisma Cloud 这样的系统，可以[自动发现并标记](https://www.paloaltonetworks.com/prisma/cloud/visibility-governance-compliance)新获取的资源和数据，可以节省大量时间并确保更好的安全性。

## **审计庞大的多云堆栈**

《2020 年云原生安全状况》[调查显示,“94%的组织使用不止一个云平台。大多数人——60%——使用 2 到 5 次。”](https://www.paloaltonetworks.com/state-of-cloud-native-security)

然而，每个云供应商都忙于构建自己的围墙花园，很少考虑与竞争对手的互操作性。在这种矛盾的环境中，组织需要确保其多云运营得到充分的保护和审核。这需要超越云供应商自己提供的普通工具，寻找其他[供应商无关的工具](https://www.paloaltonetworks.com/prisma/cloud)，这些工具可以扩展多云设置的复杂性。

更具体地说，安全策略需要跨云供应商平台统一应用。尽管有特定于供应商的术语，但节点-主机配置在实施中应该是相似的。就像网络请求穿越多云网络一样，问题的故障排除也应该跨越整个多云网络。为了全面和有用，监测应包括所有环境。

日志记录和警报在促进云原生安全审计方面发挥着关键作用。默认情况下，为所有云服务打开 API 日志记录非常重要。安全监控应该提醒您像这样重要的最佳实践中的失误。

最后，对于在高度监管的行业中运营的行业，如金融和医疗保健，云原生审计涉及额外的考虑因素。应在[云原生应用生命周期](https://blog.paloaltonetworks.com/2020/05/cloud-secure-cloud-native-applications/)中实施 PCI、HIPAA 和 GDPR 的法规合规性控制。虽然同样的原则适用，但这些原则在这些部门的执行将更加严格，对 PII 和 ePHI 数据的处理有更多的检查和平衡。

但是，即使接近这个成熟度级别，也需要采用云原生安全审计的基础知识，并从基本的 RBAC 转向更高级的策略和标记控制。
