# Kintaba 为现代事故管理引入自动化

> 原文：<https://thenewstack.io/kintaba-introduces-automation-for-modern-incident-management/>

对于 [Kintaba](https://www.kintaba.com/) 来说，当您的公司遇到一个“事件”——例如服务器和服务宕机——解决这个事件不仅仅是简单地提醒您的站点可靠性工程师(SREs)并让他们重新启动和运行。相反，Kintaba 管理所有步骤，从声明事件到处理响应，再到进行事后分析，帮助实现所有相关过程的自动化。

现在，Kintaba 正在推出一项自动化功能，将业务逻辑插入到其流程中，以帮助自动将这些 sre 之外的组织的其他适当成员纳入事件管理流程，从而在某人试图在忙碌的时刻处理事件时，不再由他来决定包括谁，而是预先确定。

“自动化实际上意味着去做我们认为是‘繁忙的工作’的事情。”“我用这个词的方式相当激进，因为我们不希望人们浪费时间去做一些事情，比如遍历组织层级来决定应该增加谁，”金塔巴的首席执行官兼联合创始人[约翰·埃根](https://www.linkedin.com/in/john3gan)在接受采访时解释道这是我们应该能够捕捉到的一条业务逻辑。“这项技术是由前脸书工程师开发的。

Egan 举了一个个人身份信息(PII)暴露的事件的例子，而不是简单的服务器宕机。在这种情况下，解决方案不仅仅是让服务重新上线，还涉及更多的 SREs 人员，事实上还可能包括法律团队、公共关系和其他人员。这里自动化的作用是确保响应者简单地用标签(如“PII”)标记事件，然后在 Kintaba 中触发一系列事件来通知所有需要的各方。伊根说，这只是自动化所能做的事情的开始。

“它真的给了我们一个平台，在这个平台上，我们可以继续扩展决策能力，让 Kintaba 自动介入并做出决定是合适的，而不是在那些更适合人类做出这些决定的地方，”Egan 说。“它让你的公关人员去做对外回应。它带来了你的客户成功的人。它带来了解决这个问题的工程师。然后是记录根本原因分析的人类时间表的过程，并确定核心问题在哪里以及对此将采取什么措施，允许所有者和响应者在这种背景下反思公司允许失败发生的过程。”

虽然这种工作流可能看起来很明显，Egan 解释说，当前的解决方案通常是将许多工具与脚本或流程缝合在一起。Kintaba Automations 旨在将这些人工流程自动化，并将它们统一在一个地方。

“自动化很好地表明了我们的发展方向——能够简化人类在处理重大停机或事故时不得不处理的所有令人痛苦的开销。我们所做的一切都是基于去除那些开销和繁忙工作的原则，”Egan 说。“当我审视 Kintaba 的发展方向时，我会通过流程来审视组织内部的弹性——这些事件会带来哪些变化。Kintaba 应该在这方面帮助你，它应该帮助你找到团队，组建团队，尽可能有效地解决问题，然后实际更新你的组织所拥有的知识，以便它可以在未来更有效地运作，并且永远不会再次发生那种事件。我们行业的成功在于没有重复。发生一次的事件不会再发生。”
