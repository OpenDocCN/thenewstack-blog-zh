# OutSystems 推动低代码超越其 Visual Basic 遗产

> 原文：<https://thenewstack.io/outsystems-pushes-low-code-beyond-its-visual-basic-legacy/>

不手工编码可以做开发者吗？这听起来像是一个亵渎神明的问题，就像问你不演奏乐器能成为音乐家吗？但是，随着“低代码”运动在云原生时代的日益流行——更不用说它的“无代码”表亲了——我们看到越来越多的开发工作在可视化、基本无代码的环境中进行。

去年八月，Gartner 将 [OutSystems](https://www.outsystems.com/) 评为“企业低代码应用平台”市场的领导者，与微软、Salesforce 和 Mendix 齐名。因此，为了更好地了解这一趋势，我与两位 OutSystems 高管进行了交谈:[产品营销高级总监迈克·休斯](https://www.linkedin.com/in/mnjhughes/)，以及前微软高管[巴里·戈夫](https://www.linkedin.com/in/barrygoffe/)，他最近开始在 OutSystems 担任平台战略高级副总裁。

我提到 Goffe 的 Microsoft connection 是因为有些人会说低代码运动始于 20 世纪 90 年代微软的 Visual Basic 编程语言——尽管当时它被称为“快速应用程序开发”，或 RAD。Visual Basic 有一个图形用户界面，它本质上对用户隐藏了代码(但是您可以选择在任何时候查看它)。这个想法是为了让编程更容易学习和使用。事实上，我自己在那段时间也使用过 Visual Basic，这证明了它有多简单。

然而，当我问 Hughes 和 Goffe 今天的 low-code 是否比得上 Visual Basic 时，我得到了两种截然不同的回答。

“我认为这很不一样，”休斯说。“我认为这项技术已经发展了很多。我的意思是，你不会在 Visual Basic 上运行你的业务。”

他继续说，低代码“更加健壮”，并指出你可以在外部系统上运行关键业务应用程序。

Goffe 曾在 Visual Basic 的全盛时期在微软工作，他有不同的看法。

“市场的趋势完全相同，”Goffe 说。“微软在投资 Visual Basic 时所做的反应是，客户面临着令人生畏的复杂程度，这使得他们很难完成日常工作。”

在 Goffe 看来，低代码为开发人员做了与 Visual Basic 在 20 世纪 90 年代和 21 世纪初做的一样的事情:通过隐藏复杂性，使他们的工作更容易。

低代码作为一种趋势，涵盖了一系列不同的用例，从专业开发人员到所谓的“公民开发人员”(业余爱好者，就像我以前用 Visual Basic 一样)。但是 OutSystems 本身非常关注前者。它希望真正的开发者使用它的平台。

OutSystems 将自己标榜为构建企业应用程序的全面平台。除了低代码工具，它还提供部署、扩展和与外部服务的集成。

休斯说，“我们认为低代码只是我们平台的一个属性”。

## 常见的低代码应用

那么什么样的应用程序是使用低代码构建的呢？

“关键任务应用程序，”休斯说，“对我们的平台来说是相当独特的，因为我们已经研究了 20 年；我们相信，你应该在我们的平台上经营你的业务。”

面向员工的应用程序，如那些支持劳动力创新和优化的应用程序，是企业最常见的应用程序类型。有时，他们会扩展到面向客户的应用程序，例如银行和保险应用程序。

“对于组织来说，最大的机会是解决严重的问题(用低代码)，”Hughes 说。“通常，当组织构建新的应用程序时，如果它是一个相当大的应用程序，在过去有两种选择:让我们找到一个[软件]包并对它进行定制，[…]或者开始一个大的 Java 项目或一个大的。NET 项目。

根据 Hughes 的说法，组织现在可以在外部系统上构建相同类型的“严肃”应用程序。

Goffe 补充说，OutSystems 有一种“工业强度方法”来构建应用程序。除了支持低代码开发方法，OutSystems 还致力于简化业务建模、安全管理和应用程序生命周期的其他部分。

“随着你进入越来越大的组织，”Goffe 说，“不同的个人甚至不同的团体将拥有拼图的某些部分，我们已经做了工作来建立优化的工具和基础设施[……]使他们的工作更快[和]更容易。”

## 编码还是不编码，这是个问题…

这听起来不错，但是开发人员自己对低代码有什么感觉呢？进步软件公司[在](https://thenewstack.io/low-code-platform-adoption-gets-a-boost-from-digital-transformation/) [2018 年对 5565 名开发者的调查](https://www.progress.com/docs/default-source/kinvey/low-code-platforms-what-developers-think-and-why-whitepaper.pdf)显示“67%的受访者对无/低代码平台持否定态度。”

休斯承认，对于一些开发者来说，“他们的人生使命就是成为某个框架的专家。”但他说，这些开发商没有抓住重点。

“事实是，这些框架的寿命正在缩短。我认为，我们看到越来越多的开发人员明白，关注结果、成为企业的英雄比过多关注某项特定技术更好。”

这种逻辑对许多开发人员来说是行不通的。我个人认识的那些人对他们辩论 0 和 1 的能力感到自豪——这是一种思维方式，而不仅仅是一份工作。我们也不要忘记，能够用 C 或 Java 之类的编程语言编码会让你更接近机器，而低级代码必然会在你和机器之间增加一层抽象。因此，C 或 Java 的程序员无疑在他们构建的东西上有更大的能力和灵活性(当然，这取决于他们有多有才华)。

但是 Visual Basic 如此成功是有原因的——事实上，*仍然是*(令人难以置信的是，它目前在 [TIOBE 指数](https://www.tiobe.com/tiobe-index/)中排名第六)。Visual Basic [坚持](https://www.techrepublic.com/article/zombie-programming-languages-could-visual-basic-be-the-next-cobol/)是因为企业希望保持他们的软件系统简单。

简单也是低代码的吸引力所在，尽管 OutSystems 正试图改变这种看法。其网站目前打出的口号是“创新无极限”，该公司在案例研究中使用了“转型”和“重建”这样的词。

Mike Hughes 坚持认为一个外部系统开发人员可以与任何其他开发人员的工作相匹配，尽管他承认需要一些手工编码。

“开发人员不能指着[out systems]说，它不能做 x。它 T10 可以做 x，对吧，而且大部分 x 都是用少量代码编写的——可能是 80/20、90/10……但你可以做到。问题是有多少是自动化和抽象化的，而有多少是诉诸于代码。”

这确实是个问题，尽管大多数开发人员并不认为这是“求助于”手工编码。他们会认为这是**对编码的沉迷。至于低代码和手工代码哪个能产生最好的结果，这最终要由市场来决定。但是只要 Visual Basic 还在十大编程语言索引中徘徊，你就会觉得低代码有大量的潜在客户。**
