如果你以前听过这句话，请打断我:IT 领导者越来越担心他们的团队能否跟上数字化转型的步伐。尽管有 DevOps 革命，组织通常还没有集成 DevOps 文化，导致开发人员、运营团队和其他利益相关者仍然各自为政。

换句话说，虽然 DevOps 实践使组织能够以前所未有的速度惊人地提高灵活性以及部署和更新应用程序的能力，但要在整个组织中应用这些实践并解决阻碍 IT 创造业务价值的挑战，仍有许多工作要做。

Dynatrace 委托进行的调查证实了这一矛盾:

*   89%的首席信息官表示数字化转型已经加速。
*   58%的人预测这一速度将在未来几个月继续加快。
*   93%的受访者表示，IT 实现业务价值最大化的能力受到孤立的业务和 IT 团队等挑战的阻碍。
*   IT 团队 16%的时间花在与业务部门会面以确定问题的原因和解决方案上，每年平均会给组织造成 170 万美元的生产力损失。

正如调查的定性分析所显示的，孤立的 BizDevOps 团队的传统运营模式以及不同的监控和管理解决方案被证明是无效的，并且不适合云原生架构。因此，IT 团队可能会浪费无数时间手动将不同的数据拼接在一起，作为被动寻找和修复问题的一部分。如上所示，这一艰苦的过程是以时间和资源为代价的，而这些时间和资源本可以更有意义地用于协作和创新工作。

## 数字化转型加速，但孤立的开发运维团队仍然成本高昂

组织中有两种孤岛正在阻碍 IT 团队最大化业务价值的能力:围绕团队自身的运营孤岛，以及使用不同监控解决方案自然产生的数据孤岛。

这些问题有多普遍？

49%的首席信息官表示，他们的 IT 和业务团队各自为政。这种跨团队协作的缺乏使得组织很难快速准确地识别严重的性能问题并采取补救措施来减轻其影响。此外，从 DevSecOps 和 AIOps 到云基础架构监控、数字体验管理和利用业务分析改善结果，缺乏跨团队协作会带来全面的负面影响。当 IT 和业务团队不在同一个页面上，并且不为共同的目标协同工作时，他们最终会在不同的目标上工作。因此，您的 IT 团队陷入了效率低下的困境，在作战室浪费时间，并在出现问题时相互指责。

与此同时，IT 团队正面临着前所未有的工作压力。近一半的首席信息官表示，他们的 IT 团队比以往任何时候都更加捉襟见肘。这种资源的缺乏限制了他们对用户体验以及组织的数字服务表现的洞察力和观察力。可悲的是，这种现状只会变得更糟，近 90%的人承认数字化转型在过去 12 个月中加速了，并且在未来只会继续加快步伐。

因此，在没有资源的情况下匆忙进行数字化也会给企业带来巨大的成本后果。如上所述，由于团队浪费时间来确定影响业务成效的问题的原因和解决方案，组织每年平均损失 170 万美元的生产力。

## 通过跨团队 BizDevOps 协作打破孤岛

随着数字化转型的加速和现代动态云环境变得越来越复杂，IT 团队越来越需要共同努力，做出推动业务发展的数据驱动型决策。业务、开发和运营团队联合成的筒仓使得这种协作几乎不可能。

消除这些孤岛需要组织转变团队的工作方式。目前，只有 14%的首席信息官表示，他们的组织利用单一平台实现跨团队协作，并了解 IT 对业务成果的影响。对于尚未采用单一平台方法的 86%的组织来说，前进的道路在于采用 BizDevOps 流程和自主云运营——这一切都是为了将 BizDevOps 团队带到同一页面上，从单一的事实来源进行运营，并通过团结而非竞争的努力朝着相同的目标前进。

传统的“我的部分工作正常”方法每年浪费数百小时和数百万美元。四分之三的首席信息官表示，他们对试图将通过多种监控工具收集的数据拼凑在一起感到沮丧，只是为了了解他们的 IT 投资对业务产生了什么影响。此外，BizDevOps 之间的有限协作使 IT 无法快速响应业务需求的突然变化，从而无法跟上数字化转型和多云复杂性的步伐。由于缺乏跨团队的可见性，这导致了工作和费用的浪费。

组织可以通过采用更具协作性和整体性的业务模式来应对这些挑战，这种模式以跨多个团队和多个学科(如 AIOps、用户体验、安全性和云基础架构)的统一监控和管理方法为基础。采用单一平台方法进行监控和管理使团队能够实时洞察其环境。反过来，这些见解使 BizDevOps 能够更有效地利用资源，将更多时间用于真正的创新而不是手动救火，推动业务成果的改善，并从长远角度最大限度地提高 IT 对业务的价值。
