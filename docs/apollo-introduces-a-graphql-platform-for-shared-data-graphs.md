# Apollo 为共享数据图引入了 GraphQL 平台

> 原文：<https://thenewstack.io/apollo-introduces-a-graphql-platform-for-shared-data-graphs/>

GraphQL 工具提供商 [Apollo](https://www.apollographql.com/) 发布了对其 [Apollo 数据图平台](https://www.apollographql.com/platform/)的重大更新，使团队能够为组织操作单一的共享数据图，据该公司称，这有望显著改善“应用交付和用户体验”。

这项工作是今年早些时候发布的架构 [Apollo Federation](https://blog.apollographql.com/apollo-federation-f260cf525d21) 的后续，用于构建在微服务架构中实现 [GraphQL](https://graphql.org/) 的分布式图。根据公司博客文章，Apollo Federation 提供了“两全其美:一个完整的模式，将我们所有的数据与一个分布式架构连接起来，这样团队就可以拥有他们在图中的部分”。

在接受 New Stack 采访时，Apollo 的首席技术官兼联合创始人 Matt DeBergalis 解释说，GraphQL API 查询语言加快了开发团队提供丰富关联数据的速度，而在此之前，他们将忙于编写无法提供价值的粘合代码。

“如今，企业正面临着构建卓越数字体验的挑战。基本的难题是，在越来越多的平台上提供高质量的应用程序存在巨大的压力，不仅仅是网络和移动平台，而是从物联网到手表到 CarPlay 再到语音的所有平台，而且这个列表还在不断增长。这些应用程序体验与越来越多的数据相关联，”德贝加利斯说。“推动 GraphQL 崛起的根本原因是，团队发现构建连接到他们拥有的所有这些惊人数据的高质量应用程序非常缓慢和困难，因为他们必须编写大量管道来将所有数据显示在所有屏幕上。这导致了各种各样的自定义 API、后端和前端，UI 开发人员不得不编写这些东西，但没有一个创造任何价值。人们不得不煞费苦心地编写这些点对点的 API，而团队能够推出一个数据图。”

虽然开源的 Apollo Federation 的初始版本提供了分布式数据图的基本功能，但最新版本增加了许多功能，包括直接连接到持续集成系统的验证检查，以确保不会发生重大变化和冲突。类似地，更新包括托管服务部署，它协调对共享图的更改，确保它被正确部署并集成到 CI/CD 流程中。最后，Apollo Data Graph 平台提供了对图形性能的分析，因此团队可以根据性能指标重写查询。

“数据图带来了非凡的好处，有助于组织快速构建具有丰富数据驱动体验的高质量应用，”DeBergalis 在一份声明中说。“面临的挑战是在不引入会造成发展瓶颈和主要失败点的整体结构的情况下运营这一新层。这就是我们引入托管联盟的原因；实现和管理数据图的工具和基础设施，这些数据图由许多不同的团队共同操作。”

DeBergalis 解释说，GraphQL 的采用通常始于企业中的单个团队，然后从那里扩散开来。随着时间的推移，图开始承担越来越多的角色，成为共享资产——这就是联合图的用武之地。随着 Apollo Data Graph 平台的最新更新，开发人员可以拥有自己的图形，而不必担心他们正在进行突破性的更改。虽然一个组织的单一图表提供了好处，但它也带来了困难，这就是 DeBergalis 所说的这些更新旨在解决的问题。

DeBergalis 说:“如果我们从企业的规模来考虑，就会出现这样的情况，即您的组织需要一个单一的图表，但您需要一种实现分布式实施的方法，这样就不会有一个庞大的代码库位于您拥有的每个服务之上，并为您拥有的每个应用程序提供动力。”“联邦图的理念是它作为一组 GraphQL 微服务的一部分来实现。这让我们可以将图表分割成单独的关注点。在联邦图中，我们可以将所有这些组合在一起，这样最终得到的就是一个通用图。它给你所有的信息，真正丰富的联系。”
