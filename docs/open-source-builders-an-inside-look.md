## 什么样的贡献算？

在与许多这样的项目创始人和领导者的交谈中，看到他们对开源贡献的不同观点令人惊讶。例如，Musl 创始人 Rich Felker 认为“用户、测试人员、采用者、bug 报告者比编写任何代码都有价值。”毕竟，给定一个特定项目的足够多的用户，其他软件产品或项目被迫良好地互操作——当然，除了集成软件之外，这也为原始项目的用户创造了价值。

相比之下, [fio](https://github.com/axboe/fio) 项目的创始人 Jens Axboe 认为有能力贡献代码的用户是最好的贡献者。“他们发送一个(或多个)补丁，修复他们遇到的问题，或者添加他们需要的功能。爱这些用户！”

与此同时，[Wireshark](https://www.linkedin.com/in/geraldcombs/)[的创始人 Gerald Combs](https://www.wireshark.org/) 强调了软件用户所谓的“驱动贡献”的有用性。“我们的路过通常以新的或更新的协议剖析器的形式出现。每一个新的剖析器都使 Wireshark 变得越来越有用，并使我们的社区增长了一点，这意味着对长期开发者来说更有用的反馈。”

但是，即使是地理空间数据抽象库( [GDAL](https://gdal.org/) )项目的首席维护者 Rouault 也强调，驱动贡献也有不利的一面:开发人员对对他们重要的代码部分进行更改，然后离开，这使得“建立一个粘性核心贡献者的骨干来帮助进行错误分类、修复、发布版本、审查拉请求、确保持续集成保持运行以及所有其他无数维护任务”变得很困难

这些开发者并没有真正争论彼此的观点，而是指出了开源软件开发中固有的复杂性。虽然有一些通用原则或多或少适用于开放源码，但也有一些细节可能适用于某一类项目(比如一个复杂的数据库项目)，而不适用于其他项目。

## 在战壕里

在这个系列中，我们将深入探讨这些不同的观点，同时也试图揭示元主题。随着我们的进展，我希望你会欣赏这些开源构建者所做的令人印象深刻的工作，在某些情况下已经做了几十年——通常没有太多的认可，而且经常是免费的。在每一种情况下，他们都说他们这样做是因为好玩。让我们找出原因。
