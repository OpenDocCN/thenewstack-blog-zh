# 为什么云原生基础架构自动化对所有参与者来说都是双赢

> 原文：<https://thenewstack.io/why-automating-for-cloud-native-infrastructures-is-a-win-for-all-involved/>

开发中的圣杯是创建和维护安全的应用程序，这些应用程序会产生很高的 ROI 和满意的客户。但是如果这种开发不是高效、高速和可扩展的，这个圣杯很快就会变得不可企及。如果您发现自己对当前的基础架构有更高的期望，那么是时候考虑云原生了。它不仅检查所有这些框，而且自动化云原生基础架构可以提高效率和效果。下面来看看它帮助组织更快地前进并保持高质量的一些关键方法。

## 让你的花费与你的用途相匹配

Robert Hopson 是 Fairwinds 公司的现场可靠性工程师。他在大学地下室的机架服务器上完成了他的基础架构，并领导了几家波特兰地区公司的云迁移项目。](https://www.fairwinds.com/) 

即使在云中，你仍然要运行资源。例如，你可以向亚马逊支付电脑处理器、内存和存储空间的费用。但是，你怎么知道你需要多少呢？对于许多组织来说，这有点像瞎猜。如果您的公司需要 100 台服务器来处理当前的高峰需求，那么它们很可能在非高峰时段处于闲置状态。事实是，你还不知道。

一旦在云中部署了一些东西，就可以自动收集关于性能的信息。您的服务器崩溃了吗？他们对用户请求的响应速度快吗？进出的数据量是多少？当您了解所有这些后，您就可以实时地将其直接整合到您的基础架构扩展中。因为信息完全在云中，所以没有理由不能关闭这个循环，并使其成为基础架构自动化的一部分。

这样，你就不会入不敷出。需求总会有高峰(和低谷)。但是如果你有了自动反馈回路，你就可以随时满足任何地方的需求——并相应地消费。

## 大幅减少停机时间

需求激增会导致额外的问题，即停工期。如果你突然发现自己面临着用户数量的激增——就像新冠肺炎·疫情出现时世界的急剧变化——停机是很常见的，通常是由不可预见的基础设施瓶颈造成的。云原生技术中的相同自动化知道如何根据需要扩展和缩减，也知道如何修复这些类型的问题。

以一个崩溃的服务器为例。自动化意识到这意味着你没有足够的服务器，该技术可以旋转新的服务器来处理它。通过利用云基础架构提供的内省功能，您将能够更灵活地处理问题，并大幅减少停机时间。

## 提高开发速度

开发速度对成功至关重要，当工程师可以编写代码、按下按钮并看到它运行时，他们就如鱼得水了。您可以通过围绕这些操作实现自动化，在云中实现这一点。将您现有的首次展示剧本转化为一系列自动化的行动——然后在开发和面向客户的系统中使用这些行动。然后，自动化剧本成为另一个需要迭代和改进的核心组件。当您启用自动化时，一旦代码准备就绪，就可以投入使用并进行测试，从而提高开发速度。您能够快速推出，这也意味着您将更容易满足客户，并获得收入。

云原生基础设施自动化的另一个好处是，多个开发人员可以同时推出特性。由于它是自动化的，他们可以同时在云中单独测试这些功能，而不会互相阻碍或依赖。这有助于高速开发，并提供显著的生产力回报。

## 建立自动化的响应系统

当您使用云原生基础架构时，您还可以受益于为您标记可能问题的其他技术。例如，如果某些东西工作不正常，监控软件可以提醒你。将它与能够识别特定类型错误的软件结合起来，你就可以在问题出现时自动处理它们。

如果您看到一个错误，您可以通过修复它来解决问题的根源，或者自动解决问题(无论是重新启动服务器还是通过添加额外的服务器来扩大规模，等等)。).随着时间的推移，错误列表变得众所周知，您可以通过开发人员或系统管理员以脚本形式使用的相同实践，在自动化中解决这些错误。这可以节省您的时间，并防止您的团队不得不手动响应。

## 启用服务所有权

云原生基础设施非常适合微服务方法，在这种方法中，小团队处理孤立的功能。这样做的好处是服务所有权和安全性。这样，不需要访问整个云基础设施的开发人员可以在有限的一组服务器上部署一小段代码。如果有人在前端工作，例如做设计工作，他们不需要访问充满客户信息等的后端数据库。因此，您可以——也应该——自动确定您的客户中谁可以访问哪些部分。

以前，开发人员必须等待可能没有资格的人的批准，才能启动某个功能。使用微服务方法，当开发人员进行更改并看到他们的工作在现实世界中成形时，摩擦被消除，动力被建立。这不仅让你的团队在他们的核心专业领域拥有自主权，而且还提高了安全性。您可以在需要的地方添加控件，同时消除障碍并提高效率。

当您针对云原生基础架构实现自动化时，您可以享受众多优势，最终帮助您节省时间、提高性能并获得更多利润。换句话说，你将把你的开发团队、客户和公司放在第一位。每个人都赢了。
