# 火的洗礼:NS1 INS1GHTS 调查当前 DevOps 挑战

> 原文：<https://thenewstack.io/baptism-by-fire-ns1-insights2020-surveys-the-current-challenges-of-devops/>

[NS1](https://www.ns1.com/) 赞助本帖。

DevOps 包含许多内容:它是以不断增长的节奏部署应用程序和更新的需求。有必要通过在软件生产流水线的最开始集成安全过程和检查来左移。但是，随着应用程序越来越多地部署在高度分布式的多云和容器化的环境中，确保代码保持安全也是不可忽视的。然后是最后一英里，这取决于运营团队如何管理底层基础架构，从而以最佳方式提供最终用户体验。

也许 DevOps 以及组织的命脉所基于的最重要的元素，包括不断努力改善所有团队成员的福祉。这意味着尽一切努力促进内部和社区的多样性，而技术社区在这方面做得还远远不够。

但是现在，尤其是新冠肺炎疫情以前所未有的规模加剧了这些挑战，因为远程工作人员的安全和技术需求激增。

NS1 的首席执行官兼联合创始人 [Kris Beevers](https://www.linkedin.com/in/beevek) 表示:“我们为支持网络和传统应用交付而构建的传统基础设施基础突然面临巨大压力。“这一戏剧性的重大事件似乎一下子把所有东西都推到了网上。”

上周，在 NS1 的[NS1 insights](https://ins1ghts.ns1.com/)在线峰会上，这些以及这些动荡时代的其他主题得到了关注、讨论和辩论。

## 巨大的冲击

冠状病毒疫情引发了远程工作者的激增，尤其是在 IT 部门。Beevers 在他的主题演讲中说，由此产生的对连通性要求的重大影响给互联网和网络基础设施带来了前所未有的压力。

“我们通过这次冠状病毒了解到，我们的基础设施并不是真正为支持所有技术的交付和访问而建立的，”比弗斯说。

比弗斯说，在冠状病毒疫情之前，许多组织当然已经在经历数字化转型。“所有这些相关的突然转变都在充当催化剂，加速这种从多年时间线到几乎立即发生的事情的转变。这突出表明，每个企业都需要想方设法加强其技术姿态，以保持弹性和可扩展性，”比弗斯说。“但是，当然，鉴于目前的经济形势，企业也需要提高成本效益。因为客户和用户并不真正宽容，所以即使在这种环境下，他们也要求并应该获得真正出色的数字体验。”

## 最后一英里

无论是应用交付还是流媒体视频游戏，出色的客户数字体验都需要完美的最后一英里交付。例如，在今年的超级碗比赛中，NS1 的任务是控制直播视频流量。“如果你想一想那个基础设施看起来像什么，它是巨大的和实时的，有很多机会让一切出错，”Beevers 在 INS1GHTS2020 炉边聊天会议上说。“在这种情况下，我们的工作首先是确保不出任何问题，确保任何人去看超级碗比赛时，都不会看到无休止的缓冲。”

与超级碗生产商相比，安全和软件部署需求规模小得多的 DevOps 团队现在比以往任何时候都更加依赖疫情之后的流量导向能力。“正如你所想象的那样，许多企业并不一定准备好让所有员工突然转向远程办公，尤其是不一定准备好保护这种足迹，”比弗斯说。这些组织通常“使用 VPN 为 5%到 10%的员工提供服务，而突然之间这一比例就达到了 100%。”

## 智能云

NS1 首席技术官兼联合创始人 Jonathan Sullivan 在主题演讲中表示，寻求通过迁移到云并利用微服务和容器环境提供的机会来实现基础设施现代化的组织“正在为他们公司的未来奠定基础”。沙利文说，随着“软件继续占据一切”，企业将更具弹性，并为技术不可避免的持续发展做好准备。

然而，明智地规划和构建云至关重要。Sullivan 说:“您将需要现代应用交付堆栈服务，以便利用和利用您已经做出的投资。“如果没有这一点，您可能会有这种奇妙的混合云战略，如果您没有办法智能地协调流量，您就不会看到潜在的投资回报。”

当然，迁移到具有容器化和微服务环境的云环境也绝非易事。沙利文说，数字化转型过去涉及“迁移到云”并确定“这对你的业务意味着什么”。“我们今天发现的是，一切都比这复杂得多，”他说。

例如，沙利文在 INS1GHTS 炉边聊天会上说，今天的组织可能通常在谷歌云平台上拥有资源。“但这永远不会简化为，‘我们将能够把所有东西都迁移到内部，阅读剧本，然后把它放到一个云中，’”他说。“因此，你必须准备好像 Kubernetes 这样的工具或像[VMware 的 Tanzu](https://tanzu.vmware.com/tanzu)这样的框架，以便将你的东西放在任何地方，并弄清楚如何利用这种复杂的基础设施和复杂的基础设施。”

虽然使用云资源调配来增加容量变得越来越容易，“因为我们可以出去购买更多的计算，但这也让我们更容易忽略一些事情。” [Heidi Waterhouse](https://www.linkedin.com/in/heidiwaterhouse) ，高级开发者倡导者，[launch blackly](https://launchdarkly.com/)，在她的演讲《突破压力:一个关于能力和测试的故事》中说道“我们没有太多的早期预警信号表明我们的计算能力即将耗尽，如果每个人都需要同时购买计算能力，那么我们的能力就只有这么多。沃特豪斯说:“当我和为主干网做网络供应的朋友交谈时，我感觉‘我们仍然在使用仍然来自制造地的计算机，我们仍然需要电缆、光纤和交换机来运行云’。“云只是别人服务器房里的服务器。因此，当我们考虑增加容量时，我们需要有一个合理的预期，即我们要么建设它，要么购买它。”

Waterhouse 说，弹性容量云供应可以解决一些，但不是所有的容量问题。“当你在建立一个强大的系统时，你还需要能够灵活地做出反应，你可以拐弯真的很重要，因为如果你有一个大型系统，只有在道路上有东西时才能前进，你会有很多麻烦，”沃特豪斯说。

## 生产流水线

“今天所有的公司都是软件公司”这句格言仍然适用，这意味着任何组织的成功或失败都取决于它是否用最终用户想要或需要的应用程序和代码填充软件管道。 [Rookout](https://www.rookout.com/) 的联合创始人兼首席技术官 [Liran Haimovitch](https://www.rookout.com/author/liran-haimovitch) 在他的演讲“与 Jenkins 和 Kubernetes 的 CI/CD-the Ugly Bits”中，谈到了他的公司在决定选择哪些工具来开发其 SaaS 平台时所面临的挑战。对于 CI 来说，Rookout 依靠 Kubernetes，以及 Docker、Helm 等工具和平台，实现了能够一天多次交付应用和更新的目标。

例如，为依赖 Rookout 平台的用户提供强大的体验需要使用 Jenkins 管道进行大量工作。海莫维奇问道:“你为什么要把像詹金斯这样复杂的东西拿到像库伯内特这样复杂的野兽身上运行呢？”例如，很难将配置作为代码来配置 Jenkins，这意味着“你可能必须进行一些手动配置，”Haimovitch 说，并补充说“这些日志可能有助于跟踪。”

然而，Jenkins 为 Haimovitch 的团队提供了“可编程管道，本质上是作为代码的配置，但具有实际的代码，”Haimovitch 说。

## DevOps vs. NetOps vs. SecOps

NS1 INS1GHTS2020 的一个反复出现的主题是在 DevOps、SecOps 和 NetOps 之间保持协作精神的重要性。在峰会之前，[NS1 渠道和联盟副总裁 Warren Mead](https://www.linkedin.com/in/warren-mead-9803342/) 表示，NS1“很好地弥合了开发运维、网络运维与安全运维之间的差距”

“我们在这三者之间如此纠结，”他说。

然而，在他的主题演讲中，Sullivan 说 DevOps、NetOps 和 SecOps 可能会有冲突，并描述了如何解决这些冲突。“我们的软件定义和容器化管理的私有 DNS 和 DDI 软件能够很好地与所有这些群体合作。Sullivan 说:“这些团队都在从事高速部署持续集成/持续交付(CI/CD)工作，而所有这些团队在管理 VDI 或 DNS 基础架构方面都有些分歧，因为他们都想从 it 部门获得不同的东西。我们拥有的集装箱化平台“能够很好地与所有这些群体合作，并为他们提供拉动和共存的杠杆。”

## 多元化工作

在最近的事件中，多样性和[黑人的命也是命运动](https://thenewstack.io/black-lives-matter-how-the-tech-community-can-provide-support/)成为 2020 年 NS1 的主题。事实上，在乔治·弗洛伊德被杀后，最初的峰会被推迟了，以支持正在进行的示威游行，要求回归生活和平等。

还讨论了保持适当的工作与生活平衡的重要性。事实上，例如，制定支持 DevOps 团队福祉的政策不仅对组织的运作能力至关重要，还可以作为技术社区的榜样，被视为一种道德责任。

“如果你建立一种基于同情和支持的文化，这是企业价值观的核心，那么你就在团队和组织中为真正每个人的需求创造了空间和支持，”比弗斯说。

对于整体健康来说，如何避免警觉疲劳和最终的倦怠，特别是在待命团队中，是 NS1 的技术运营经理[贝瑟尼·阿博特](https://www.linkedin.com/in/bethanyross)的演讲主题。她描述了她作为 NS1 和其他地方的技术运营经理所学习和实施的流程。

在她的演讲中，阿博特问峰会的在线参与者——他们大多都是随叫随到的，她说“想想你的随叫随到的转变，现在想想你现在是否感到筋疲力尽，你是否在职业生涯中感到筋疲力尽，或者如果你是一名经理，并且有团队成员告诉你他们感到筋疲力尽，”阿博特说。“所以，想想如果这种情况发生在你的生活和你的技术生涯中，”艾伯特说。

Abbott 传达了一些团队可以采取的防止倦怠的策略。除了促进同理心文化之外，这还包括我们描述职责的语义变化，例如在提及项目责任时删除了“四分卫”一词。“在我们的事件管理过程中，我们不需要体育隐喻，”Abbott 说。让我们称之为“所有者”、“联络点”或“事故指挥官”。"

注意到警报“本质上是基于中断的”，Abbott 描述了 [runbooks](https://en.wikipedia.org/wiki/Runbook) 如何通过简洁地概述每种类型的警报需要做什么来帮助减少压力和焦虑。Abbott 说，当收到警报时，应该可以点击操作手册和仪表板来查看相关的监控信息，以便知道“这是我在收到传呼时需要做的事情”。

尽管人工智能(AI)取得了重大和持续的进展，但软件也不仅仅是自己运行。它当然也不会自我创造，正如 DevOps 中的任何人都非常清楚的那样，它一旦建立起来就经常崩溃。许多团队成员，如可靠性工程师(SRE)，也会在半夜代码或网络崩溃时接到可怕的电话。

在她“培养生产卓越”的演讲中， [Honeycomb.io](https://www.honeycomb.io/) 的首席开发人员 Liz Fong-Jones 利用了她作为 SRE 或系统工程师超过 17 年的经验。她讨论了“我学到的一些经验……它如何真正影响了我们对我们的社会技术系统的看法，以及我们必须如何设计这些技术系统以及我们的人力系统，以确保我们能够可持续和可靠地运行它们，”她说。“所以，我认为我们试图创造代码来解决我们试图解决的问题，例如世界上正在发生的某种问题，无论是提高生产力还是商业，但问题是代码并不存在于真空中，”Fong-Jones 说。

Fong-Jones 说，最终目标不应该只是让系统更可靠，而是“对操作系统的人更友好”。Fong-Jones 说:“你不会偶然到达那里——我们真的需要制定一个路线图和计划，找出我们如何从今天的位置到达我们未来想要的理想世界。”
