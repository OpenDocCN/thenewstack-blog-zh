# 安全大转移到了左边

> 原文：<https://thenewstack.io/the-great-security-shift-to-the-left/>

Twistlock 赞助了这个播客。

[【国安】向左移动](https://thenewstack.simplecast.com/episodes/the-great-security-shift-to-the-left)

通过采用 DevOps 实践，一个组织可能已经实现了将以前孤立的团队集成到其软件开发和部署过程中的巨大飞跃。对于一些安全团队来说，虽然这种情况可能会发生，但安全代码集成和流程类似于网球比赛:开发人员添加代码，然后将代码交给安全团队，然后将现在已安全化的代码发送回开发人员，开发人员再将其发送回安全部门进行最终检查。或者更糟的是，安全团队直到部署前才给出输入，当获得正确的安全性所需的时间比预期的长时，就产生了潜在的瓶颈。(当然，更糟糕的是，应用程序带有明显的安全漏洞，只有在出现重大漏洞后才会被修补)。

如果你愿意，可以称之为 DevOps 成长的烦恼的一个例子，但在持续集成/持续交付(CI/CD)过程中，安全经常需要“左移”，[twist lock](https://www.linkedin.com/in/sonyakoptyev)的福音传道总监 Sonya Koptyev 在这一集由新堆栈的创始人兼主编 Alex Williams 主持的[新堆栈制造商](https://thenewstack.io/podcasts/makers)播客中说道。

改善构建和部署过程中的安全性集成的主题也有望成为本周举行的 RSA 会议的主题。“我对这次会议感到非常兴奋。我听说过很多关于它的事情，在我职业生涯的前几个职位上，我也听到过业内人士谈论它——所以，我很高兴能有一个前排座位，我想说，甚至比前排座位更兴奋，”科普特耶夫说。“我认为，随着这些人以各种不同的方式交流，我希望我们在一些内容和讨论主题中看到的一些趋势确实是，我认为，从安全(一般而言)、云原生安全和应用安全主题开始向左转移，我认为，我们已经讨论这个问题有一段时间了，至少一年半了。”

Koptyev 说，在 DevOps 中，一些工作流程正在成为一种更加“全手工操作”的安全方法，当然，直接涉及 CI/CD 集成。“随着人们构建他们的管道，从一开始就将自动化纳入实际的应用程序开发生命周期，这一切都是关于如何将安全性集成到这些管道中。因此，不仅要构建并推向生产，还要确保在部署过程中设置了适当的安全阈值，以便不允许应用程序退出开发，除非它通过了特定的安全阈值，否则不要部署它——这是一个特别重要的工作流程，”Koptyev 说。“为了确保在运行时，您拥有所有正确的阈值和警报集，以及所有潜在恶意进程的黑名单集，同时立即定义它们并将其自动化到构建流程中。”

Koptyev 说，以前，在企业领域，“你(经常)会遇到生活方式非常瀑布式的人。“所以，他们开发应用程序，然后交给安全部门，告诉他们要修复什么，然后投入生产，”科普特耶夫说。“但是现在，随着这种持续不断的构建和部署，应用程序的可用性和不断更新正在发生；你在一个过程中有一个工作流程，也确实需要更新，”科普特耶夫说。“正如我们已经更新了我们的构建、持续部署、应用的可用性和持续更新一样。因此，正如工具更新以自动化构建流程一样，我们(作为一个行业)也需要更新我们如何考虑将安全性集成到流程中。”
