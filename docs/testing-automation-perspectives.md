# 加起来:测试自动化不是一个工具的故事

> 原文：<https://thenewstack.io/testing-automation-perspectives/>

测试自动化工具不经常使用。根据[世界质量报告(WQR) 2018-201](https://software.microfocus.com/en-us/assets/application-delivery-management/world-quality-report-2018?) [9](https://software.microfocus.com/en-us/assets/application-delivery-management/world-quality-report-2018?) 的调查，只有 16%的性能测试案例是用测试自动化工具执行的，安全测试也是以同样的频率完成的，该调查调查了员工超过 1000 人的公司的 1700 名 IT 决策者(ITDMs)。尽管 QA 和测试工作角色已经适应了敏捷开发实践，但是请记住，即使一个测试是自动化的，大多数测试仍然是手工完成的。

当然，自动化不一定要通过专门的测试工具来实现。无论您关注的是软件 QA 还是信息安全，自动化的实际水平实际上更高，因为通常自行开发的流程的离散组件是以编程的方式处理的。例如，同样的调查发现，40%的人用集成的测试用例来自动化他们的交付管道(自动化部署)。由 [DZone](https://dzone.com/guides/automated-testing-your-end-to-end-ecosystem) 进行的另一项针对开发人员的调查发现，他们组织中 77%的管道已经进行了自动化集成测试。自动化可用性测试要困难得多，并且只发生 25%的时间。

人们一致认为测试不可能完全自动化。这部分是因为人类仍然是需要的，但也有其他障碍。当被问及测试自动化的挑战时，61%的 WQR 受访者表示他们的应用程序在每个版本中都有太多的变化。因此，虽然由于发布和管理自动化，发布频率增加了，但是测试可能是未来 SLDC 自动化的瓶颈。测试自动化的第二个最常见的障碍是测试数据的可用性和环境的可用性。

可以自动生成测试环境，这可以解决日益显著的测试环境成本和由于测试环境的不准确测试配置而产生的缺陷数量的挑战。像 [Plutora](https://www.plutora.com/platform/plutora-environments) 和 [Enov8](https://www.enov8.com) 这样的供应商提供了自动化工具，可以减轻这些类型的测试挑战。然而，它们没有解决在这些环境中进行的测试的数量和类型的具体问题。
