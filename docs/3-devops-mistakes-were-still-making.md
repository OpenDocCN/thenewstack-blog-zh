DevOps 现在是一个青少年。就像一个人类青少年，它超越了婴儿，但有点笨拙，还没有完全成熟。不过，我们正在实现这一目标，虽然缓慢，但却稳扎稳打。我们必须这么做。更快发布更多软件的压力已经很大了，只是新冠肺炎加快了速度。

我们已经掌握了许多对 DevOps 成功至关重要的关键要素。在关键原则上没有分歧:协作是王道，自动化是根本，而“持续一切”包括持续集成/部署/测试，以及持续改进。

持续改进的一个组成部分是积极寻找阻碍成功的错误，并努力避免它们。以下是我见过的数百家全球 2000 强公司犯的三个最大的 DevOps 错误，以及你可以做些什么来避免它们。

## 1.开发者负担过重

随着“数字化转型”成为每位首席信息官 2021 年议程的重中之重，企业希望以创纪录的速度推出源源不断的改变游戏规则的功能，从而击败竞争对手。当然，这必须是一个团队的努力，包括每个人，从开发人员，到产品负责人，到测试人员，到运营和 sre。但是，每当迫切需要更多的功能，或者有些东西达不到预期时，谁来承担责任呢？几乎总是开发商。然而，除非你是 GAFA(谷歌、苹果、脸书、亚马逊)，否则你可能不会有顶级开发者敲你的门。吸引和留住有价值的开发人员是一场持续的斗争。

当您的开发人员只专注于开发时，很难满足企业对软件的贪得无厌的需求。这就是为什么你“左移”测试和安全等事情的策略必须超越仅仅将更多的责任推给开发人员的普遍反应。不要误解我的意思:从一开始就在应用程序中构建质量和安全性是至关重要的。但要以一种不会给开发人员带来沉重负担的方式来做。否则，你的顶尖开发人才可能会跳槽到一个让他们专注于他们渴望的创造性工作的组织。

## 2.期待每个人都成为超级巨星

通过正确的方法和适当的培训/支持，每个组织和每个团队成员都可以为组织的 DevOps 成功做出贡献。但是不要期望每个人都以同样的方式做出同样的贡献。

任何运动、流程或技术的早期采用者往往是你的超级明星，你的超常发挥者。他们对自己的工作充满热情，他们跟踪所在领域的所有新兴趋势，他们天生就有动力去做任何需要做的事情。这变成了一种个人使命——常常近乎痴迷。从不断地重新设想和重塑提议的解决方案，到寻找解决下一个棘手问题的方法，到构建新的方法和扩展以回馈给更广泛的社区，他们都参与其中。在很大程度上，他们做到了，带来了令人印象深刻的结果。

但是扩大这种成功并不容易。大多数应用程序交付专业人员不是这样工作的。即使许多在工作中表现出色的人也没有这种冒险精神和给生活带来新事物的原始冲动。他们已经花了几年，通常是几十年的时间来完善他们的工艺，他们并不总是急于转变成一种截然不同的工作方式。

此外，不同的企业团队有不同的技能组合、不同的舒适区、不同的优先级，并且通常有非常不同的应用程序堆栈和法规遵从性/治理需求。DevOps 方法和工具集非常适合处理您的移动体验的初创团队，但对于负责后端的专注于 SAP 的团队来说可能就不那么好了。

然而，一切都是相互联系的，如果你真的想加速企业范围内的创新，每个人都需要被置于更广泛的 DevOps 保护伞之下。通常，实现可扩展成功的途径包括:

*   坚持最初采用的核心理念。
*   在工具集和实践方面提供额外的企业友好选项。
*   引入编排、可见性和治理的总体层(是的，就是这个，讨厌的 g 字！)在从事相关系统和项目的各个小组中。

## 3.没有充分了解用户体验就发布

今天的用户对任何他们认为是问题的东西都有极高的期望和很低的容忍度。 [Forrester 最近发现](https://go.forrester.com/blogs/improving-customer-experience-by-1-point-can-drive-more-than-a-billion-dollars-in-revenue-in-2019/)即使是客户体验的微小改善也会对公司的底线产生惊人的影响。它估计，CX 指数仅增加一个点(从 0 到 100)就可以显著提高年收入——例如，11 亿美元(汽车)、4.96 亿美元(零售)或 3.88 亿美元(电信)。可以合理地假设，在相反的 CX 方向上的一个小失误将导致类似的收入损失。

当然，团队中的每个人都想创建和发布用户喜爱的软件。这是一个团队，一个奋斗到终点的团队。但是不同的角色会带来不同的观点、优势和劣势。从计划，到测试，到发布，再到监控，你需要让真正了解业务的团队成员参与进来，并顽强地捍卫整体用户体验。

许多 DevOps 团队仍然仅仅依赖于底层的技术检查(例如，单元测试)来决定一个候选版本是否可以安全发布。这种测试可以很好地捕捉编码错误，但是还需要更多的测试来交付出色的体验，最终为您的组织带来优势。这包括:

*   基于风险的测试，这样您可以立即决定一个新的测试失败是否真的是一个阻碍。
*   交易的端到端功能测试(例如，从移动到 API，到 SAP 和 Salesforce 等打包应用，一直到定制应用和大型机)。
*   负载/性能测试，以确保应用程序能够扩展以满足需求的激增。

**奖励**:阅读 [Forrester 关于持续测试实践的研究](https://www.tricentis.com/resources/forrester-devops/)，该研究将开发人员和敏捷领导者与落后者区分开来。

如果你的组织正在犯这些错误，请放心，你肯定不是唯一一个。它们在企业开发人员中很常见，而且不像人类青少年，它们很容易纠正并步入正轨。
