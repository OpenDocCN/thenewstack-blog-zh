# 新电子书:新堆栈的云原生存储快速指南

> 原文：<https://thenewstack.io/new-ebook-the-new-stacks-quick-guide-to-cloud-native-storage/>

The New Stack 今天发布了最新的电子书《国家现状:云原生存储的新方法》。“这份快速指南比之前的 TNS 电子书要短，它展示了我们根据原始报告和分析得出的关于云如何改变存储格局的观点。

随着云计算和容器架构的出现，企业应用程序中的状态管理发生了变化。以前，这意味着将数据存储在安装在硬件上的数据库中，并管理运行有状态的应用程序，这些应用程序通常由严格的服务级别协议(SLA)管理。企业已经想出了如何在内部世界中处理状态。当许多公司继续以他们过去在 monoliths 上使用的基本相同的方式处理状态时，挑战就出现在向云的迁移中——至少直到他们艰难地学会不这样做。

为了了解我们现在在云原生应用中管理状态的情况，新堆栈发布了一系列[专题文章](/tag/cloud-native-storage-ebook-series/)，比较云原生存储供应商、传统存储供应商和云服务提供商如何处理云原生架构中的状态和存储问题。

此后，我们通过对“现状”的概述完善了报告，并将该系列编辑成由 NetApp 赞助的关于云原生存储的综合电子书。我们旨在帮助开发人员和 DevOps 了解他们在云原生应用中的存储的差异和选择。

在这本 48 页的云原生存储快速指南中，您将了解到:

*   如何最好地将云原生应用连接到存储。
*   Kubernetes 如何使用储物件。
*   云原生存储供应商、传统存储供应商和三大云提供商之间的差异。
*   DevOps 调配和管理存储的注意事项。

## 云原生服务、传统服务和云服务

记住一些关键事实很重要:容器被设计成无状态的。Kubernetes 旨在编排无状态、短暂、不可变的容器。起初，这些应用程序需要的任何状态都只是存储在筒仓式存储设备中，并通过卷插件来访问。

但随着公司开始看到 Kubernetes 和容器化架构如何提高应用程序的灵活性和速度，越来越多的公司开始将越来越多的应用程序打包到容器中，并使用 Kubernetes 来管理计算和存储资源。

现在可以通过存储服务或不同种类的存储系统在有状态容器内部处理状态。

该快速指南解释了容器连接、软件定义的存储、传统存储硬件供应商提供的云存储以及云服务提供商提供的本机选项之间的差异。在许多情况下，企业将使用至少两个(通常是所有三个)类别的选项来构建在容器中运行有状态应用程序的策略。下载“现状:云原生存储的新方法”以了解更多信息。
