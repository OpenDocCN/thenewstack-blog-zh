# 2022 年:人工智能开始编码的一年

> 原文：<https://thenewstack.io/2022-the-year-ai-came-to-coding/>

2022 年是人工智能真正走向编码的一年。

这一年见证了 GitHub Copilot 从 2021 年首次推出的 Jetbrain 上的一个插件转移到 3 月份广泛提供给 T2 的 Visual Studio IDE。随后，亚马逊在 6 月发布了代码完成服务 Code Whisperer，并在 10 月发布了 Replit 的 Ghostwriter。[代码生成的人工智能初创公司 Tabnine](https://thenewstack.io/code-stays-behind-firewall-with-copilot-alternative-tabnine/) ，获得了[1550 万美元的资金](https://techcrunch.com/2022/06/15/tabnine-raises-15-5m-for-ai-that-autocompletes-code/)，而另一家代码完成初创公司 [Kite，随着 Copilot 的流行而消亡](https://www.kite.com/blog/product/kite-is-saying-farewell/)。

然后，也是在年底，当 GitHub 因在 Copilot 中使用开源库而陷入诉讼时，这一切都变成了一个大问号。

## AI:对开发者来说，不仅仅是代码完成

尽管 2022 年的大部分焦点是自动化编码和代码完成，但事实证明，人工智能技术在过去一年中以更微妙的方式改变了代码。

利用人工智能的 DevOps 自动化初创公司 [Coherence](https://www.withcoherence.com/) 的联合创始人兼首席执行官[扎克·扎罗](https://www.linkedin.com/in/zacharyzaro/)说:“我们不相信我们会看到人工智能取代 DevOps 工程师或平台工程师，而是真正增强他们的能力。“应用层发生了很多事情——人工智能开始帮助开发人员编写应用程序代码，而不是基础设施代码。”

Zaro 补充说，对于那些用例，这将使 [DevOps](https://thenewstack.io/automate-devops-with-an-everything-as-code-approach/) 更加重要，因为将有更多的代码需要更频繁地测试。

[Coherence 在 12 月的一篇博客文章中确定了 DevOps 的许多人工智能用例](https://www.withcoherence.com/post/ai-and-the-future-of-devops)，包括:

*   提高了代码质量；
*   加强监测和警报系统；
*   更好的安全措施；和
*   提高工程生产率。

“我们更多的是关于机器学习模型，关于找到有趣的日志线或指标，或者找到不需要运行的测试，这样你就可以减少你的构建时间，”Zaro 说。“我们已经看到了这样做的公司。所有这些都是增强用例。这不是威胁工作的用例。”

## 超越代码完成

开发者想要的是更快，人工智能可以帮助他们，David DeSanto 说，他领导 GitLab 的产品组织，专注于为整个 DevSecOps 生命周期提供单一平台。 [GitLab 调查了 5001 名 DevSecOps 专业人员](https://about.gitlab.com/developer-survey/)，发现 31%的受访者现在使用人工智能/ [机器学习](https://thenewstack.io/key-concepts/machine-learning/)作为代码审查的一部分，近一半的人说他们已经实现了完全的测试自动化。

GitLab 预测，在接下来的一年里，AI/ML 将进一步推动开发，同时帮助进行安全补救，改进测试自动化，以及更好的可观察性。

DeSanto 说，调查还发现，70%的受访者谈到了加快发布代码的压力。他说，人工智能助手可以通过改善代码审查和帮助开发人员监控和分类生产事件来提供帮助，以便 CI/CD 可以更快更有效。

“组织正在试图自动化和改善他们的测试自动化，这种转变的一部分是为了更快地交付，这意味着你必须找到优化你正在做的事情的方法，”DeSanto 说。“如果你要持续运输，你就要把人类从这个过程中抽离出来。你如何将人类从这个过程中抽离出来？您开始在代码评审、代码质量和部署方面构建一个更智能的代码验证流程。”

## 可靠吗？

DeSanto 警告说，作为 DevOps 的一部分，AI 仍处于蹒跚学步阶段。他说，它仍在成长，尽管当以正确的方式使用时，它非常有效——如果不是完美的话。

DeSanto 提醒我们，即使在人工智能出现之前，技术也一直需要培训:就在大约十年前，web 应用防火墙必须见证大量的生产网络流量，才能识别异常和正常行为。

“不要只是假设人工智能第一次就会是正确的。就像其他东西一样，它需要一段时间的训练才能变得更准确，”DeSanto 说。“我们在测试版中还有另一个功能，那就是建议标签来帮助实现[工作流自动化](https://thenewstack.io/5-workflow-automation-use-cases-you-might-not-have-considered/)。当我们第一次为自己推出这个功能时，我认为，准确率不到 30%。但随着时间的推移，我们了解了我们的项目，并看到我们如何给它贴标签，它变得更加有效。”

当被问及开发者是否应该谨慎使用人工智能和机器学习时，他说，这同样适用于所有人工智能和机器学习。

“如果你意识到像任何新技术一样，它必须变得更智能，[……]今天没有什么事情是违反直觉或不应该做的，”他说。

他提供了一个正确用例的真实例子。GitLab 去年收购了一家[公司](https://about.gitlab.com/press/releases/2021-12-14-gitlab-acquires-opstrace-to-expand-its-devops-platform-with-open-source-observability-solution.html)，在此过程中发现 GitLab 在为代码变更指派合适的编码员方面并不尽如人意。

“这家公司有能力利用你的项目知识来确定正确的代码审查者，以保持你的团队更快地前进，”他说，“当我们将它应用到我们自己的代码时，在收购之前，作为改善我们自己的一部分，我们发现它比我们更有效。”

该计划通过学习项目和代码，然后将正确的编码分配给任何需要进行的代码更改，来帮助加速该过程。

“这使得开发人员可以继续加快开发速度；他们不仅仅是在试图找人审查他们的代码时受阻，”他说。

## 人工智能和可观测性

人工智能还可以在提高软件供应链中的[可观察性](https://thenewstack.io/continuous-integration-observability-explained/)方面发挥作用，Hans Dockter 说，他是 [Gradle](https://gradle.com/) 的创始人兼首席执行官，该公司专注于通过 Gradle Enterprise(一个大规模软件开发的统一平台)和其用于多语言软件开发的开源 Gradle Build 工具来提高开发人员的生产力。

“工具链是不可测量和不可观察的，这在某种程度上是疯狂的，对吗？达克特说:“这个行业让所有其他行业都变得如此可观察、如此高效——每个工厂都配备了全套仪器，你完全知道发生了什么。”。“当谈到开发人员每天与大多数人交互的工具时，组织没有基本数据。你今天做了多少测试？你多久回来一次？花了多长时间？它多久失败一次？为什么会失败？对于软件行业来说，这是一种奇怪的状态。”

在开发人员调查中，Gradle 询问了这种洞察力会在多大程度上提高生产率，绝大多数人说至少会提高两倍，但也可能是三倍或四倍，同时更好地观察过程。

“想象一下你可以用这些数据做什么，”达克特说。

他以 LinkedIn 为例。LinkedIn 每天运行 600，000 次构建测试——这意味着每天可能有数千万次测试。通过分析这些数据，LinkedIn 了解到某些测试不需要针对新代码运行。这是达克特在其他公司看到的模式。

“为什么我们总是运行它们——如果你改变代码的那个区域，我们基本上可以向你保证 99.999%的可能性这个测试不会失败，”他说。“在这背后，我们有机器学习模型。所以现在我们有了一种叫做预测测试选择的东西，你运行你的构建和测试，通常 80%或 90%的测试根本不需要运行。”

他补充说，这是大量浪费的计算，现在可以节省。
