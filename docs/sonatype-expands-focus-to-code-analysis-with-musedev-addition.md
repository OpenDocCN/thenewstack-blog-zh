# Sonatype 通过 MuseDev 扩展了对代码分析的关注

> 原文：<https://thenewstack.io/sonatype-expands-focus-to-code-analysis-with-musedev-addition/>

软件供应链管理和安全工具公司 Sonatype 已经收购了代码分析平台 [MuseDev](https://www.muse.dev/) ，将公司和团队整合到其现有的 Nexus 平台中，以将其范围扩展到第三方依赖之外。

借助 Muse，软件团队将获得 24 个预配置的代码分析器，这些分析器将评估每一个拉请求，并在代码审查期间提供反馈。除了添加 Muse 之外，Sonatype 还添加了代码形式的基础设施合规性功能，使用 Nexus Container“虚拟修补野生容器”的能力，以及即将添加的高级法律包，该包将提高软件开发和法律团队对开源许可义务的可见性。

凭借所有这些新增功能，该公司表示，其平台现在提供了“对云原生软件开发生命周期的全方位控制”

[Sonatype](https://www.sonatype.com/?utm_content=inline-mention) 联合创始人兼首席技术官 [Brian Fox](https://www.linkedin.com/in/brianefox/) 将公司的最新发布视为市场最终成熟的一部分，其中许多安全任务的重点已经转移到开发团队。

“我们一直在构建这个平台来做这些事情。福克斯说:“现在开发部门真正参与进来，真正拥有了端到端的依赖性管理问题，这是我们能够真正扩大我们向他们提供的东西的范围的恰当时机。“因此，您通过 Muse 获得的源代码、通过 Nexus Container 获得的容器、Terraform 以及通过基础架构获得的这些类型的脚本都是代码，Advanced Legal Pack 利用我们现有的法律分析，并通过允许组织管理许可证的义务，使其更进一步。”

Muse 的加入特别增加了对内部代码创建的关注，Muse 不仅提供预配置的代码分析工具，还将这些工具的结果插入到拉式请求和代码审查流程中，创建一个反馈循环来确定哪些错误被忽略，哪些被开发人员修复。Muse 还帮助大规模推出代码分析工具，因为代码库可能由需要多种工具的多种语言组成，但 Fox 表示，配置代码分析工具并不是团队经常面临的最困难的问题。

“更困难的问题是弄清楚如何以这样一种方式配置它，即你鼓励开发者前进并生产更好的软件，而不是向他们倾销大量以前看不见的技术债务，结果只是被完全忽略，”福克斯说。“通过集成到“拉”请求中，他们基本上能够在新创建的代码片段或在特定时间被触及的代码部分中引入相关发现，因此开发人员可以轻松自然地进行更改。该工具能够监控和学习各种工具的发现，并了解哪些工具最有可能得到修复。当他们这样做时，团队实际修正所展示的发现的可能性是其他团队的 70 倍，而不是把所有的发现都扔给他们，然后指望他们自己解决。”

虽然 Sonatype 正在增加对内部代码创建和分析的关注，但 Fox 表示，他们也期待将这种类型的工作流(如 Muse 所介绍的)引入其现有工具。例如，关于依赖和许可的问题可以以类似的方式呈现给团队，这样就不会让人不知所措，也不会导致警报疲劳。

“我们设想利用 Sonatype Nexus Lifecycle 提供的政策发现，并使用这种同样的开发人员第一类型的功能，在工作流程中向他们提供关于依赖性和法律问题以及其他事情的信息，并以同样的方式建立学习反馈循环，”Fox 说。“我们认为，这种方法可以将我们已经非常好的政策结果具体化、有针对性和可操作性，但将它们纳入反馈循环并从中学习会使它变得更好。”
