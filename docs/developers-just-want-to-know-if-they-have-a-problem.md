# 开发人员只想知道他们是否有问题

> 原文：<https://thenewstack.io/developers-just-want-to-know-if-they-have-a-problem/>

开发人员只想知道他们的代码在投入生产之前是否有漏洞。但是通常，来自安全组的回答——“需要更多的分析”——并不是开发人员想要听到的。

在新堆栈的创始人兼发行人 [Alex Williams](/author/alex/) 主持的最新一期《新堆栈制造商》播客中， [Synopsys](https://www.synopsys.com/software-integrity.html?utm_content=inline-mention) 的产品管理高级总监 [Meera Rao](https://www.linkedin.com/in/meerasubbarao) 说:“我一直在大声而明确地说:只要告诉(开发人员)出了什么问题，而不是说，‘我们从 SAS 发现了这个问题，我们从渗透测试中发现了这个问题，我们从人工审查中发现了这个问题。’。“他们不应该关心这个，他们需要知道我们的发现是至关重要的，这就是你需要解决的问题。他们只需要知道这些。”

[开发者只想知道自己有没有问题](https://thenewstack.simplecast.com/episodes/developers-just-want-to-know-if-they-have-a-problem)

Rao 是一种新的智能编排技术的创造者，这种技术可以帮助开发人员在没有长时间等待的情况下解决问题。依靠 Synopsys 的系统让开发人员知道出了什么问题，以及特定的安全漏洞是否需要立即修复，从而弥补了这一漫长的等待。

作为最近“左移”安全趋势的一部分，在某些情况下，发现和修复代码中的安全问题的繁重任务错误地落在了开发团队身上。除了给他们经常超负荷的工作负荷增加更多的任务之外，开发人员因此有更少的时间投入到包括开发和更新应用程序在内的主要工作中。在许多情况下，他们不得不成为事实上的安全专家。

Rao 说:“每个人都意识到，你不能指望你的开发人员学会每一个工具、每一项技术、每一个仪表板。“现在事情已经开始慢慢改变了。”

最近，随着运行良好的 DevOps 中的运营团队开始为持续集成/持续交付(CI/CD)提供更多支持，以帮助加快从开发到部署的周期，承担纯安全职责的团队开始出现。不幸的是，在许多情况下，这种结构产生了一种动态，在这种动态中，当安全问题出现时，通过减慢开发周期，安全团队被视为“坏人”。

“但是我们——就像在安全行业一样——制造了所有的摩擦，”饶说。

很快就发现安全团队和工具需要做得更好。

“你能一直花钱，改造同样的管道，却不提供开发者需要的抽象吗？开发人员不在乎你用什么工具，做什么分析，分析有多深入，”饶说。“他们只想知道:我是否有需要立即修复的严重或高度漏洞？是还是不是？”

实际上，这意味着允许开发人员使用 Synopsys 的智能编排技术来确定安全漏洞是否是关键的，如果是，如何修复它，然后如何将修复与主分支合并。这是通过使用一个“简单的 API”来实现的，这个 API 可能只是对发布管道进行一个“一行代码的更改”,而不会造成任何中断。

Rao 说，该工具有助于确定“运行什么活动，跳过什么活动，以及需要发生的活动的深度应该是什么”。"因此，我们在自动化活动和手工活动之间取得了平衡."
