# 与 Stoplight.io 的 Taylor Barnett 讨论微服务和 API

> 原文：<https://thenewstack.io/discussing-microservices-and-apis-with-stoplight-ios-taylor-barnett/>

[与 Stoplight.io 的泰勒巴内特](https://thenewstack.simplecast.com/episodes/discussing-microservices-and-apis-with-stoplight-ios-taylor-barnett) 讨论微服务和 API

在今天的 [The New Stack Makers](https://thenewstack.io/podcasts/makers) 播客中，TNS 创始人兼主编 Alex Williams 在今年早些时候于波特兰举行的 OSCON 2018 上与 [Stoplight.io](https://stoplight.io/) 的首席社区工程师 [Taylor Barnett](https://www.linkedin.com/in/taylormbarnett/) 进行了讨论。

对话一开始，Williams 就提到了服务网格的话题，对此 Barnett 很快指出，“很多地方甚至还没有出现。他们仍处于起步阶段，现在我们已经谈论微服务几年了，以及如何发展它。还有很多工具有待开发。”

Williams 接着问如何解释工程师们今天面临的 API 问题。特别是，设计第一原则，如测试驱动开发，Barnett 指出，现在正转向设计驱动开发。“很多时候你看到，你在使用一个 API，而两个端点没有任何意义。你就像是，‘他们没有想过这些是如何相互关联的吗？’像这样修理东西。”Barnett 接着指出，她所做的很多工作都涉及到让人们“后退一步”，在规范的层面上工作。

Williams 随后询问了 Barnett 在 Stoplight.io 的工作，指出她的角色非常面向社区，涉及到与最终用户的大量工作。她帮助他们编写文档、内容策略，领导 Stoplight 的一些技术支持，并且经常是产品反馈回到工程团队的联系人。Stoplight.io 帮助公司“建模和设计他们的 API，然后从中自动生成文档，它还托管模拟服务器。回到不只是先设计的想法，在迭代周期中把 API 放在用户面前。然后，也帮助内部团队。比方说，前端团队需要开始在内部实现一个 API，而后端团队实际上还没有完成开发，这是很正常的事情，Stoplight 帮助团队更多地并行工作，这样一旦他们能够定义该规范并对其感到满意，前端团队就可以通过模仿来开始实现该 API。”
