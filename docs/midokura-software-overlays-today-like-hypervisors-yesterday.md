# Midokura:覆盖网络将挤掉软件定义的网络

> 原文：<https://thenewstack.io/midokura-software-overlays-today-like-hypervisors-yesterday/>

容器编排平台本身不容易扩展。去年早些时候，当 Docker Inc .、CoreOS 和新的参与者如 [Weaveworks](https://thenewstack.io/connecting-containers-weaveworks/) 将覆盖网络移出实验阶段时，容器化真正火了起来。通过提供对网络资源的查看和控制，[覆盖网络](http://www.idc-online.com/technical_references/pdfs/data_communications/Network_Overlays_An_Introduction.pdf)可以为基于容器的操作提供一种扩展到数百个节点的方式。

但也许不是成千上万。在三月中旬于圣塔克拉拉举行的[开放网络峰会](http://events.linuxfoundation.org/events/open-networking-summit)上，网络专家就主要电信运营商所称的资源协调进行了大量讨论，从他们的角度来看，其节点扩展模型在服务协调所需的水平上是如何不可行的——以高质量的流同时处理数千个客户。电信公司坚持要一个完整的[软件定义网络](https://thenewstack.io/defining-software-defined-networking-part-1/) (SDN)基础设施。

这是一个与 Midokura 关系非常密切的问题，Midokura 是一家旨在实现资源协调的覆盖网络制造商。在周四于旧金山举行的英特尔云日活动上，[mido kura 的创始成员兼业务联盟副总裁亚当·约翰逊](https://www.linkedin.com/in/adjohn)告诉 New Stack，他认为[我们从网络专家那里听到的关于“真正的 SDN”与覆盖网络的许多摩擦](https://thenewstack.io/opendaylight-onos-sdn-really-need-two-controllers/)，与这些专家在虚拟化工作负载的前景第一次摆在他们面前时表现出的摩擦类型相同。

“如果你认为覆盖层很像一个计算虚拟机管理程序，人们在早期也对虚拟机管理程序说过同样的话，”Johnson 说。“他们说，‘你不可能在虚拟机管理程序上运行这些延迟敏感、资源敏感的应用程序。’但是现在，我们面临着许多工作负载都运行在虚拟机管理程序上。因此，我认为这也将发生在网络上，你将获得硬件技术的进步，你能够将一些工作转移到硬件上，从而获得更多的改进。"
