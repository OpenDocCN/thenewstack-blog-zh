在 [{code}](http://web.archive.org/web/20180220231349/https://thecodeteam.com/) ，[我们一直在撰写](https://blog.dellemc.com/en-us/csi-updates-and-call-to-action-for-you/)关于云原生模式和容器技术给我们带来的好处，以及我们(作为技术专家、用户和行业领导者)需要解决的挑战。在某种程度上，这些文章是我们一直在发布的关于标准化[容器存储接口](https://github.com/containernetworking/cni) (CSI)的公告的背景，而{code}在这方面投入了大量资金。

现在，我想我们为什么需要这样的东西已经很清楚了。容器通过动态地管理从它们消耗的资源中抽象出来的应用程序，使我们能够解决一些真正困难的问题。容器生态系统中的人们、社区和组织正在认真地工作，以创建越来越强大的互操作工具来帮助实现这一目标。然而，在管理存储方面，我们都遇到了“每个人的做法都不一样”的阻碍创新受阻，我们从未停止关注基础。

有了以前的竞争、创新和快速发展的技术，计算机行业已经认识到创建标准界面是一个好主意，它可以作为基本规则并最大限度地减少用户的困惑。没有它们，这条道路就会变得复杂和支离破碎。制定标准就像为家庭装修绘制蓝图。通过记录电源插座应该放在哪里，居民可以知道在哪里插入硬件，并自己决定安装什么小发明，而不必担心如何供电或它可能有什么限制。

[CSI](https://github.com/container-storage-interface/spec) 是一个[社区驱动的计划](https://groups.google.com/forum/#!forum/container-storage-interface-community)，由容器编制者——Mesos、Kubernetes、Cloud Foundry 和 Docker——领导，定义一个接口来满足他们管理的应用程序的存储需求。我们都想要由容器支持的更智能、更强大的应用程序。

我认为 CSI 很重要，主要有三个原因:

### 1:犯罪现场调查提供了一个吻和社区的方法

这并不是说其他人以前没有尝试过与存储全面集成。在某些方面，他们提供了更多的细节和功能。然而，早期的项目最终更加固执己见，关注面更广，如果不能实现的话，也是一个完成的负担。因为有太多的决策，让参与者参与进来甚至合作都变得更加困难..咳嗽..分心。

所以 CSI 采用了最简单的方法:捕捉人们今天如何使用容器存储，但是以接口规范和可插入代码的形式。简单——“保持简单，笨蛋！”(KISS)方法—使 CSI 首先关注基础。结果——我们希望如此！CSI 将在容器编排器和存储平台之间提供更好的集成，我们可以对其进行迭代改进，并依赖它来构建更好的应用程序。

我们一直在构建 CSI，发布了几个关键驱动程序:一个用于裸机的 [ScaleIO 驱动程序](https://github.com/codenrhoden/csi-scaleio)和一个用于虚拟化云原生环境的 [vSphere 驱动程序](https://github.com/thecodeteam/csi-vsphere)。

### 2: CSI 为更多的用例提供了一个可预测的功能接口

构建任何类型的界面的挑战在于，它需要为开发人员和数据中心专业人员提供易于掌握的用户体验，同时支持使用它的应用程序的深层功能集。

例如，Docker 为它的音量驱动程序创建了一个规范，但是(说得客气点)它还有一些改进的空间。当故障发生时，它的界面不能优雅地处理情况。如果一个组件出现故障，很难重新同步到正确的状态。

我没有直接指责 Docker 的意思。我们已经定义了很多这样的界面，但没有一个是用户界面精彩的完美范例。

CSI 团队正在尽最大努力从别人的不完美中学习。我们希望[扩展与存储相关的设计模式，以支持更多应用](https://blog.dellemc.com/en-us/understanding-the-container-storage-interface-project/)。

例如，我们目前只引入具有文件系统的卷。当应用程序需要存储数据时，它必须获取一个带有文件系统的卷。有了 CSI，我们有了原始数据块设备支持，这是一项基本功能，但目前还没有。用户可以定义没有文件系统的卷。这使得新的应用程序能够从容器中运行，当与逻辑块寻址存储配对时，容器表现优异。实际上，需要高性能或主机间共享设备的应用程序可能更喜欢这样。

### 3.CSI 是由用户需求和社区驱动的，而不是由存储公司驱动的

CSI 自顶向下的设计来自容器编排者和社区。出于简单的目的，团队成员不会到达月球；相反，这是一种“做一点点，但要做得非常非常好”的老派方法。对我来说，这可能是 CSI 最重要的一个方面。

每个从事 CSI 工作的人都在努力使规范与将要使用它的人相关。团队不会根据存储公司能提供什么以及他们想向您销售什么来做出决策。

相反，CSI 团队考虑的问题包括:“如何解决这个问题？谁是解决它的合适人选？理想的解决方案是什么样的？这将如何实现？”

## 这将如何改变你的生活？

如果有什么是技术人员讨厌的，那就是被迫选择一个工具，因为它是唯一一个具有合理连接性或与您公司使用的工具相接口的工具。如果 CSI 取得成功，它将确保组织不会仅仅因为支持特定的存储或云提供商而需要选择容器平台。它将提高所有平台的交互质量。这方面的成功将曾经复杂的任务带给了大众，并进一步巩固了动态管理和存储可移植性的云原生模式。这会让每个人的工作变得简单一点。
