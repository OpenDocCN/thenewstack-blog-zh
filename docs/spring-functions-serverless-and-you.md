无服务器很酷，但是你已经知道了。开发人员都对这种可能性感到兴奋——无服务器所释放的原始潜力——这是理所当然的。简单，灵活。令人陶醉。这是一种解放。我应该什么都用它，对吗？

嗯，事实上，没有。可能没有…

无服务器只是一长串抽象中最新的一个。IaaS、CaaS、PaaS 以及最近的 FaaS 已经让我们离应用开发的技术管道越来越远了。每个人都把“价值线”向前向上移动，把我们从辛劳中解放出来。

但是，就像早期技术的一般情况一样，我们在无服务器方面保持脚踏实地很重要。重要的是要记住，无服务器只是多了一个选项，当然是一个强大的选项，但它并不适合所有工作负载。

我们中最聪明的人已经在问:“对于我和我的特定工作负载来说，无服务器是正确的方法吗？”

在最近在 [SpringOne 平台大会](https://springoneplatform.io/) 上录制的一段 [视频中，Pivotal 的开发者倡导者](https://www.youtube.com/watch?v=8tOj4A7jgWg&feature=youtu.be)[NathanielSchutta](http://www.ntschutta.io/)带我们穿越最近的历史，以回答这个最紧迫的问题。

一路上，舒塔经过 IaaS、CaaS、PaaS 和 FaaS 设置的中转站。他着眼于使这一切成为可能的技术，并为我们带来了无服务器领域的最新创新(包括 Knative 和 riff)。最后，我们发现像 Kubernetes 这样的技术如何帮助我们在任何有 K8s 拨号音的地方“无服务器工作”—无论是在内部还是在公共云中。

下面是纳撒尼尔 S c 赫塔关于“弹簧、函数、无服务器和你”的演讲的一些摘录

### 一种尺寸不适合所有人

“我们有责任真正理解何时应该使用每种工具。IaaS、PaaS、FaaS、无服务器等。”

### 现在，快速交货就是一切

“我从事这项工作已经很长时间了，我非常清楚地记得服务器是这种本土产品的时候。它们是定制的。它们是手工制作的。虽然我们真的很欣赏三明治中的这一点，但这可能不是我们想要的基础架构。我曾经为了一个开发数据库等了一年。这可不好。我不得不填写大量的表格，写电子邮件，建立支持票，参加会议，然后更多的后续工作，最终，我终于有了一台服务器。这些服务器对我们来说是如此珍贵，以至于我们用辛普森一家的角色给它们取了一个简洁的名字。这需要时间。这花了一大笔钱。我们真的需要缩短从代码到产品的周期。”

### 平台很棒

“[平台即服务]的美妙之处在于，你不必了解它的所有内容。我的运行时可以来自一个 buildpack。你只需推送一个应用程序，它就会说“好的，我知道这是哪种应用程序。“然后，它继续为它获取适当的构建包，并将您的应用程序部署到该构建包中，上传它，然后路由器会说“好的，我有这个家伙的空间”，将它放到服务器上，然后您就可以走了！”

### 一种范式并不适合所有人

“生活节奏很快。我们已经从 IaaS 到 CaaS 再到 PaaS，现在我们有了无服务器和 FaaS。实际上，几个月前有人找到我说:“我们将把整个应用程序重构为一组函数”。我咬住自己的舌头，不让自己笑出声来。这可能不是一个好主意。”

### 这完全是关于延迟敏感性

“很多人喜欢说我们不能在无服务器环境中使用 Java，因为冷启动太伤人了。这不相关。如果你担心冷启动，问题不在于你选择的语言，而在于你试图把它作为一个函数。您必须后退一步，问自己“这个特定的用例是否对延迟敏感？“如果是，那很可能不是正确的选择。你应该找到另一种解决方法。你有其他的抽象可以选择，你应该选择其中的一个。”
