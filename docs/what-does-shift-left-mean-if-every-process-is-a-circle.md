# 如果每个进程都是一个圆，那么“左移”意味着什么？

当谈到软件开发生命周期(SDLC)中的安全性时，对于是“左移”还是“右移”一直存在困惑为了消除这种混淆， [Synopsys 在成熟度模型中构建安全性](https://www.synopsys.com/software-integrity/software-security-services/bsimm-maturity-model.html) (BSIMM)引入了一个新术语来优化 DevOps 生命周期中的安全性测试:“无处不在的转变”

术语“左移”起源于大约 15 年前，几乎立即被误解为在 SDLC 中更早地实现安全测试。这完全没有抓住要点。“左移”总是意味着在 SDLC 的每个阶段尽早执行安全测试*。*

软件交付链中的左移使得能够尽早且经常地管理安全问题，减轻与在生产中发现的安全缺陷相关联的风险，并降低修复安全漏洞的成本。转移权利意味着在生产中立即测试、识别和响应安全问题。

“无处不在的转变”包含了这两种方法。

## **为什么到处移位？**

组织不再能够在分隔的阶段中执行所有传统的 SDLC 安全活动。相反，安全活动需要作为一项持续的工作扩展到所有阶段。这意味着一旦安全活动所依赖的工件可用，就以最高的保真度尽可能快地进行安全活动。在某些情况下，这意味着左移—到 SDLC 的开头。但在其他情况下，它意味着向右或向中间移动。

例如，动态应用程序安全测试(DAST)应该在代码运行后立即执行。一旦定义或运行了环境，就应该执行配置审查。一旦您部署了代码，就应该从生产代理中收集复合分析事件，这些事件显示了动态合并到运行系统中的依赖关系。

同样，有时这是您的组织今天正在做的事情的左边，但通常是右边——可能一直到生产中。

## **何时向左移动**

将安全测试转移到软件交付链的左侧，使组织能够尽早地、经常地管理安全问题，作为管道的一部分，并减少与在生产中发现的缺陷相关的风险。这种方法降低了修复安全漏洞的成本。

## **何时右移**

虽然左移方法是有益的，但它无法降低关键漏洞进入生产环境的风险。评估缺陷发现工具和服务的组织越来越倾向于在整个价值流中进行连续的基于事件的安全遥测(而不是单一的时间点分析)。

一些常见的右移测试方法包括利用蓝/绿、滚动部署、运行时应用程序自我保护(RASP)、运行时分析和跟踪工具——所有这些都提供了连续的基于事件的遥测技术，从而优化或强化应用程序并改善安全状况。

来自基于事件的遥测技术的反馈回路支持为管道阶段设置安全挂钩和触发器。这些触发器利用漏洞数据、攻击数据、威胁情报等。，以加强和扩充日志和事件数据。这使得在从构建到生产再到运营的整个开发工作流程中，可以快速识别和修复生产漏洞。

Shift right 还允许利用生产中发现的漏洞的数据，了解开发工作流程和管道中安全控制的差距。这些差距可以通过改进工具、流程和培训，或者通过增加右移运行时保护来缩小。

## **到处转移的主要好处**

*   解决技术安全债务的能力
*   安全软件设计
*   提高代码和软件质量
*   减少误报
*   安全证明和不可变遥测
*   自动化合规保障
*   提高事件响应能力

## **结论**

DevOps 和 DevSecOps 正在推动 IT 文化变革，包括转变组织处理安全性的方式。这种不断变化的格局包括工程主导的安全集成、软件定义的安全、云配置管理、现代应用框架、编程语言的改进、容器编排、微服务架构的采用、站点可靠性工程(SRE)等等。所有这些都导致了对“无处不在的转变”安全方法的需求——利用遥测技术、知识驱动的智能管道和工具，使持续的安全活动能够在从构建到运营的整个开发工作流程中传播。
