专业人士经常发现自己与他们的安全运营(SecOps)兄弟撞头——或者更确切地说，撞代码。这通常是因为将安全实践集成到 DevOps 工作流中是一种“试火”，SecOps 和 DevOps 的角色和职责是不固定和不明确的。在这种背景下，出现了一些阻碍凝聚力、合作和协作的神话。以下是五个最令人震惊的误解:

**谬论:安全专家介于对安全的迷恋和痴迷之间。**

事实:有时看起来像是三级控制狂的案例，实际上是合理担忧的例证。SecOps 知道，当代码出现安全漏洞、漏洞百出或被最终用户视为不合规而拒绝时，手指、眼睛和指责都会直接指向它们。在发布之前检查代码并确保它是严密的和经过测试的，这不仅是避免这种痛苦的最好方法，也是唯一的方法。

**谬论:只有 SecOps 负责确保端到端操作和代码的安全。**

**事实:**安全性不是代码的一个方面或特征，就像可用性、兼容性、灵活性或可伸缩性一样。安全性是整个产品的基本定义特征。将安全性推给 SecOps 的组织(但没有为他们提供合适的资源)正在走一条危险且最终不可持续的捷径，因为违规行为爆发或愤怒的客户退出(或通常两者同时发生)只是时间问题。事实上，头条新闻充满了违规后的恐怖故事，领导者犯下了代价高昂(在某些情况下是灾难性的)的错误，未能将安全视为公司范围的义务和目标。

在 [Alcide](https://www.alcide.io/) 上，我们还发现，在这方面最成功的组织都知道安全性跨越所有层面:用户、基础设施、应用程序以及它们之间的一切。

**误解:SecOps 总是想降低发布周期的速度。**

**事实:**从表面上看，赛科普斯似乎只想把速度降得慢一点；令 DevOps 沮丧的是，他们承受着巨大的压力，要求更快地行动并加速反馈循环。但现实却截然不同:SecOps 只是在努力做好自己的工作，那就是提高安全性，确保合规性。此外，当组织实施一个网络安全平台，提供跨所有传统和新兴计算系统的鸟瞰和粒度视图时，审查、强化和验证代码的速度会显著提高。

使用为敏捷性、速度和阶段性可见性(即，在不妨碍 CI/CD 的情况下观察事物)而天生构建的工具也是至关重要的。

神话:SecOps 不理解也不关心软件开发的商业方面。

事实: SecOps 通常比一些高管更了解业务方面——因为他们知道不安全的、松散的、满是 bug 的代码不会在市场上赢得很多人的喜爱。相反，这将引发广泛的批评和争议。我们可以回到引人注目的数据泄露场景:有多少首席执行官希望他们有一台时光机回到过去，并听取他们的 SecOps 团队的反复警告？他们不是生活在象牙塔里。相反，他们比任何人都更脚踏实地！

**谬论:DevOps 从 SecOps 那里学不到任何重要的东西。**

**事实:**相反，DevOps 专业人员可以也应该从他们的 SecOps 表亲那里学到许多明智的东西，包括:

*   敏捷和速度是达到目的的手段，但不是目的本身。不要把过程速度误认为产品卓越。
*   密切注意安全访问级别和权限。如果没有适当的安全治理，协作可能会适得其反。
*   针对漏洞时，不要只关注可用性和功能性。与优化、集成和 UX 相比，最终用户对安全性的关注程度不亚于(有时甚至更高)。

一位 Alcide 客户有效地弥合了 DevOps 和 SecOps 之间的差距，并促进了相互尊重和理解，他发现这种比较特别有帮助:正如 DevOps 使用层来使操作适应规模，SecOps 使用层来使操作适应违规。

## 底线

DevOps 和 SecOps 永远不会做相同的事情或有相同的授权，当前形成新的“DevSecOps”规程的趋势更多的是思维方式的转变，而不是新团队结构的企业配方。但是越来越清楚的是，DevOps 和 SecOps 之间的不同并不像将他们联系在一起那样重要。最终，他们要么都成功提高了速度、敏捷性和安全性，要么他们的组织失败了。这真的可以归结为。
