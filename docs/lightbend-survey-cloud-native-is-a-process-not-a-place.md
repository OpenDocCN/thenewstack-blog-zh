# Lightbend 调查:云原生是一个过程，而不是一个地方

> 原文：<https://thenewstack.io/lightbend-survey-cloud-native-is-a-process-not-a-place/>

[Lightbend](https://www.lightbend.com/) 赞助了这篇文章。

 [克林·芬利

Klint Finley 是一名专业作家和分析师，专门研究企业计算和开发人员技术。他在《连线》杂志担任了近 8 年的特约撰稿人，报道了人工智能、云计算和数据科学等主题。他的作品也出现在读写网、TechCrunch 和 BoingBoing 上。Klint 位于俄勒冈州波特兰市。](https://www.linkedin.com/in/klintron/) 

随着云计算的成熟，越来越清楚的是，重要的不是你的应用在哪里运行，甚至不是什么特定的技术支撑着云服务。重要的是云计算让客户能够以不同的方式做事。

云计算和相关技术正迅速成为主流。根据云原生软件公司 Lightbend 在[对 1，000 名开发人员和 IT 决策者](https://info.lightbend.com/Cloud-Native-Adoption-Trends-Report.html)进行的调查，38.5%的组织表示他们正在积极采用云原生战略，另有 33.8%的组织处于采用云原生战略的早期阶段。与此同时，59.2%的人说他们已经在容器和/或 Kubernetes 集群上运行大多数新应用程序。

但是，在调查中接受采访的高管很少从他们的应用程序在哪里运行的*方面考虑云原生，而是从如何构建这些应用程序的*方面考虑。对于这些 IT 领导者来说，原生云不仅仅是一个地方或一件事情，而是他们做的事情。**

一些高管还希望通过将工作负载转移到公共云来节省资金或提高可用性。但几乎所有接受调查的人都表示，他们采用云原生技术的核心目标是更快地将应用和功能推向市场。一家制造公司的 it 副总裁说:“云原生的好处是构建软件的速度和在构建过程中进行协作的能力。

这意味着充分利用云提供商提供的独特技术(如机器学习 API)，使客户无需自己构建这些功能。一家大型金融软件公司的 CISO 表示:“如果我们想使用自然语言处理或人工智能，如果我们可以从亚马逊或 Azure 租赁这些功能，我们就不想构建它们。”。“当我们利用这些云服务时，我们可以更快地进入市场。”

但也许更重要的是，高管们说，走向云原生是关于采用允许开发人员采用持续集成(CI)和持续交付(CD)的过程。

对于外行来说:CI 是让程序员频繁地将他们的工作提交给中央存储库，以便存储库总是有最新的变化。CD 是一种随时准备部署这些变更的实践。

不像传统的“瀑布”式代码发布那样，一次对一个软件做很多大的改变，开发人员对软件做增量的改变。这个想法是，这使得整个过程更快，因为小的变化可以更快更有效地得到验证。

一家大型电子商务网站的首席技术官说:“当你做一个非常简单的改变时，你应该能够点击一个按钮，这些功能应该在大约 30 分钟内运行在云上。”“我不想花几个小时或几天的时间。”

这是一种云原生开发方法，尽管它不适用于在物理介质上大规模生产的软件。

有许多不同的软件包被设计来处理 CI/CD 工作流，但是 CI/CD 不仅仅是挑选一些现成的软件。它要求开发人员改变他们的过程和实践。“作为云迁移的一部分，我们转向了纯敏捷项目管理，”另一家金融服务公司的 CISO 说。

CI/CD 还鼓励一种不同的、更加云原生的架构方法。一家科技公司的首席技术官说:“将事情分解成微服务，可以更容易地进行持续交付/部署。这是因为当功能被隔离到不同的微服务中时，单个开发人员或团队可以专注于他们自己的代码，而不太担心对应用程序的另一部分进行更改。它还可以更容易地整合云原生功能，如 API 或无服务器。

是使用公共云服务还是私有云服务，是在容器还是虚拟机中运行工作负载，以及是否全力以赴实现功能即服务，这些都是当今企业面临的重要决策。但是，相对于如何构建云技术的问题，你使用哪种云技术是次要的。
