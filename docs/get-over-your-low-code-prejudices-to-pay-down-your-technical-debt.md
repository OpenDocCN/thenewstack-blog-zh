任何文化都不可避免地发展出一个内部的分类系统，反映其等级中的感知状态和相关价值，开发者文化也不例外。伴随着互联应用程序开发的增长而繁荣起来的“努力编码，经常编码”编程风气的荒谬的虚拟捶胸是这个过程如何表现的一个明显的例子，但它也有更微妙的形式。比如你最喜欢用什么 IDE？当然，你可能是 IntelliJ 或 Visual Studio 的粉丝——这两个环境都很好——但是真正的程序员使用像 TextMate 或 Atom 这样的文本编辑器，对吗？除非你真的是铁杆，用 EMACs。或者你真的，*真的*铁杆并且只用 vi。

在一天结束时，最重要的是按时完成工作，但当谈到在同事面前炫耀自己时，我们许多人都控制不住自己。在这种情况下，采用一些要求我们做更少实际编码的东西的想法——或者，天意不允许，*不*编码——对我们中的许多开发人员来说是一种诅咒。

但是这种态度与影响每个开发者的另一个问题背道而驰——技术债务。你可以称之为 backlog 或 legacy 代码或任何你想称之为的东西，但事实是，每个期望编写产品级代码的开发人员都处于交付的持续时间和资源约束之下，这导致了关于什么能做什么不能做的困难决定。我们想做所有的事情，但是现实世界的压力要求必须牺牲一些东西。在特别不正常的团队中，这可能会产生一种“不文化”，最好的例子是经常听到的权衡，“不，我们不能添加那个特性，除非你愿意扼杀这个同等或更大价值的特性。”

需要快速交付业务应用程序的新方法，这是低代码和无代码系统可以大放异彩的地方。对于开发人员来说，它们提供了非常适合特定任务的便利工具。团队可以通过利用低代码环境中固有的限制来缩短构建-测试-部署周期，这可以将开发人员的注意力集中在手头的任务上。你不会看到木匠哀叹钉枪的使用，只是因为他们知道如何挥舞锤子——开发人员应该以同样的眼光看待低代码设置。您可能不使用低代码开发工具来构建核心内容管理应用程序，但是当用于设计应用程序所依赖的内容批准过程时，它可以节省时间。

低代码工具也有一个包容性的元素，最终有利于负担过重的开发团队。低代码环境允许非开发人员以最少的努力或经验设计他们自己的交互。这使得针对特定业务流程的应用程序开发更接近最了解它们的主题专家。使用前面的例子，内容管理系统的开发人员会将他们所有的技能和注意力放在允许创建、存储和分发内容的核心部分上，但是他们可以将内容审批过程的构建交给负责实施它的业务团队。移交控制这些流程的控制权允许业务团队准确地构建他们需要的东西，而无需等待开发人员腾出时间。

低代码和无代码环境并不能取代稳固的内部开发团队，但是它们可以提供一种更好地利用开发人员生产的服务的方法。这又产生了另一个动机，将每个服务公开为完全解耦的 web API，这样它们不仅可以被应用程序开发人员使用，还可以被业务专家在他们自己的交互中使用。

低代码和无代码解决方案应该在开发人员工具箱中占有一席之地。它们可以帮助组织克服技术债务，腾出时间专注于重要的事情——对开发人员身份没有威胁的积极好处。是时候重新审视这些系统了。
