# 加起来:DevOps 安全性需要更多的工具

> 原文：<https://thenewstack.io/add-it-up-devops-security-needs-more-tooling/>

DevOps 团队参与了安全性工作，但他们需要做得更多。最近的两项研究表明，工具是不够的，安全性没有适当地集成到整个 DevOps 过程中。

由 [Logz.io](https://logz.io) 对 1000 多名 IT 专业人员进行的 [DevOps 调查](https://logz.io/devops-pulse-insights-2018/)发现，55%的组织都由 DevOps 处理安全问题。这一发现并不令人惊讶，因为 1)将安全性向左转移的努力通常是由负责与整个组织互动的开发运维团队实施的，2)32%的受访者是开发运维工程师。不管这是否意味着这些组织实际上做了开发，56%的人说他们至少开始实施这种做法。

就像 DevOps 一样，DevSecOps 可以被视为一种文化或一套工具。文化是通过培训开发人员来接种的，在软件方面，DevSecOps 有时通过使用自动化安全测试和代码依赖测试来定义。作为当前软件可以做得更多的标志，57%的 Logz.io 受访者表示，没有足够的工具可以让他们在 DevSecOps 中获得成功。

自由动力公司的另一项[调查](https://info.checkmarx.com/hubfs/2018-Managing-Software-Exposure-Time%20to%20fully%20embed%20security%20into%20your%20application%20lifecycle.pdf),《登记与检查》( Register and Checkmarx)提供了更多的视角，让人们了解不同的团队必须如何合作才能让 DevSecOps 发挥作用。当被问及软件安全挑战时，62%的受访者强烈同意开发人员、测试人员、安全专家和运营人员需要合作。这一愿望尚未与现实相匹配。超过一半(56%)的人认为，将安全性集成到整个开发运维流程中要么做得不好，要么根本不存在。

集成到 CI/CD 管道中的软件可以解决将安全性集成到整个开发运维流程中的一些需求。然而，如果是这样的话，这是否意味着 DevOps 的工程师将会被信息安全和临时开发的安全需求所困扰？请继续关注正在进行的 DevSecOps 传奇中新堆栈的下一部分。
