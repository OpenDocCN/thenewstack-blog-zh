# 开源领导者:慈善专业人士监控微服务的使命

> 原文：<https://thenewstack.io/open-source-leaders-charity-majors-mission-monitor-microservices/>

下面的采访是“开源领导者”系列的一部分，在这个系列中，我们描述了开源 IT 社区中的项目领导者，以了解他们如何开发他们的软件，以及运行开源项目带来的挑战和好处。

[C](https://github.com/charity)[harity Majors](https://github.com/charity)第一手了解每个软件开发人员在发布一个精心构建并经过艰苦测试的项目时的痛苦——也就是说，在那个时刻，性能监控变得几乎不可能。因此，她共同创立了 [Honeycomb.io](https://honeycomb.io/) ，提供监控和调试工具，为应用程序的“真实世界”性能创建实时窗口。

梅杰斯在移动应用程序开发公司 Parse 磨练了自己的技能，她在 Parse 的移动后端即服务项目中担任基础设施技术主管。当脸书[在 2013 年收购](https://techcrunch.com/2013/04/25/facebook-parse/) Parse 时，梅杰斯成为一个生产工程师团队的经理，负责超过 500，000 个移动应用的维护和供应——基础设施、数据库管理、自动化、事件响应、持续部署管道、脸书集成。

Charity Majors 采访了 New Stack，谈论了离开脸书与合伙人/首席执行官(也是前解析工程师)Christine Yen 一起创建 Honeycomb.io，以及开源生态系统的一些帮助。

**你是什么背景？**

我是音乐专业的辍学生，后来成为了技术人员。我总是被 ops 和 DBA 工程角色所吸引——我对抽象的计算机科学的容忍度很低，我必须做需要做的工作。所以我做过系统工程师和工程经理，但不知何故，似乎最终也要负责数据库——我从 17 岁起就随叫随到了！我在工程管理和集成电路角色之间来回转换已经有一段时间了。2016 年 1 月，我和另一位前 Parse 工程师 Christine Yen 一起创立了 Honeycomb.io。公司还没有倒闭。

**你是怎么开始做 Honeycomb.io 的？**

当我在脸书工作时，我爱上了潜水。水肺是脸书的内部观察工具，它有着巨大、快速和灵活的用户界面。当我准备离开脸书时，一想到我将回到一个充满老式指标和仪表板的世界，我就感到非常震惊。我意识到，我只是一个更强大的工程师，拥有丰富的、基于事件的交互式工具，我再也不想像过去那样在黑暗中摸索，从一个工具跳到另一个工具。所以我们建立了蜂巢，给小公司同样的实时应用性能分析，像脸书这样的大型企业组织都可以使用。

**蜂巢帮助解决了哪些商业或技术问题？**

一个字:复杂。我们看到了不久的将来，每个人都必须成为分布式系统工程师来管理爆炸式增长的复杂性。LAMP stack 这一代工具旨在快速有效地回答已知问题。它做得很好，石墨，新遗迹，麋鹿/弹性等。但是，在未来，我们将没有已知未知的奢侈。我的意思是，在这个系统中，如果你知道问题是什么，你可能会回答它。然而，现在你越来越没有一个明确的问题。看看[微服务](/category/microservices/)和分布式系统:找出症状(可能存在，也可能不存在，由不可靠的叙述者或同样不可靠的工具报告)来源于哪个(哪些)组件是非常具有挑战性的。哪些是原因，哪些是结果？

> “大多数仪表板都是过去失败的产物，如果你认为它适合你当前和未来的情况，那你就是在蒙蔽自己。”—慈善专业

Honeycomb 拥有大量的 Scuba DNA，是一种激进的、从根本上不同的解决问题的方式，而且是完全必要的。我在脸书和其他地方一再看到这一点:如果我们坚持旧的方式，我们就会迷失。大多数仪表盘都是过去失败的产物——如果你认为它适合你现在和未来的情况，那你就自欺欺人了。

如今，不管公司是否知道，他们都需要蜂巢这样的东西。幸运的是，人们越来越认识到运行大型多租户平台和开发深度数据工具不是免费的，也不是他们核心业务的一部分，因此他们应该雇用像我这样的人来为他们处理这些事情。Honeycomb 能够很好地处理高基数数据，没有其他系统能做到这一点。你可以在那个有蜂窝的干草堆里找到每一根针。

**开源在这方面发挥了什么作用？**

到目前为止，Honeycomb 的[开源组件是一个集成库，它是将数据传输到我们的 API 的软件。如果要我猜的话，到目前为止，至少有 50 人已经提交了与蜂巢相关的代码，我们将在接下来的两个季度中大幅提升。](https://github.com/honeycombio)

让蜂巢的前端开源是巨大的。将客户端数据传输给我们的所有连接器的所有代码——我们不可能为所有类型编写所有连接器，这将是一个非常长的尾巴，我们根本没有这些资源。但是有了开源的 Honeycomb 集成，现在任何人都可以去阅读代码，用 Erlang 或其他我们永远不会支持的语言写点东西，因为没有足够多的人使用它们。开源服务于所有人，在这种情况下。

我们还没有开源蜂巢的后端。主要是因为它还没有准备好——我们专注于作为一个企业的生存，现在没有资源去那里。我们希望有一天。

在我的日常工作之外，我也为开源做出了贡献。我最近做的最大的是我的 [Terraform 配置文件](https://charity.wtf/tag/terraform/)。 [Terraform](https://www.terraform.io/) 是一个开源工具，它将 API 编译成声明性的配置文件，可以在团队成员之间共享，被视为代码，进行编辑、审查和版本控制，其 GitHub repo 有超过 1000 名贡献者。我还写了 [apiary](https://github.com/lexelby/apiary) ，这是捕获/重放 MySQL/HTTP 流量基准测试的原始工具。现在由 Lex Neva 维护。

**货币化的潜力如何？**

我们讨论过让整个蜂巢开源，并通过出售企业许可、定制和支持等模式来赚钱的想法。但同样，我们是一家初创公司，专注于生存，让自己在市场上站稳脚跟。

我是超级开源专家，我的整个职业生涯都在使用操作系统软件，并热情地支持它。但是我也很务实——根据我的经验，开源有助于你更快地扩张，并吸引人们。但是你需要从一个稳定的、已建立的基础上成长。除非你有世界上所有的风险投资，否则几乎任何人都做不到开源整个创业公司。因此，我们现在尽我们所能，并且随着我们的发展，一定会扩大我们的参与。

**你既是一名经验丰富的开源贡献者，也是一家软件公司的创始人。在您看来，开源和企业如何才能最好地融合在一起？**

作为一个消费者和现在做相对小规模企业的人，在那个领域，我觉得我们都应该倾向于开源。如果有争论，倾向于公开。除非你有不透明的理由，否则就去做！

从更大的角度来看，我们应该让那些在核心业务的关键部分使用操作系统软件的大型公司感到羞耻，但他们不会通过指派一名或两名或三名工程师或开发人员来回报，让他们为帮助他们盈利的操作系统资源做出贡献，并为其他使用该资源的人提供更好的资源。
