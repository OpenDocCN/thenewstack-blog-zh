# 凯尔西·海托华和本·西格曼就微服务与整体服务展开辩论

> 原文：<https://thenewstack.io/kelsey-hightower-and-ben-sigelman-debate-microservices-vs-monoliths/>

欢迎来到[新的堆栈环境](https://thenewstack.io/podcasts/context)，在这个播客中，我们将讨论云计算领域的最新新闻和观点。在本周的节目中，我们采访了谷歌的开发者倡导者[凯尔西·海托华](https://github.com/kelseyhightower)，以及可观察性服务提供商 [LightStep](https://lightstep.com/) 的首席执行官兼联合创始人[本·西格曼](https://www.linkedin.com/in/bensigelman/)，讨论在构建云原生应用时，团队是否应该倾向于使用整体架构而非[微服务方法](https://thenewstack.io/microservices-vs-monoliths-an-operational-comparison/)。

TNS 编辑和营销总监 Libby Clark 主持了这一集，旁边是 TNS 创始人兼出版商 Alex Williams 和 TNS 执行主编 Joab Jackson。

[第 110 集:凯尔西·海托华和本·西格曼辩论微服务 vs 独石](https://thenewstack.simplecast.com/episodes/episode-110-kelsey-hightower-and-ben-sigelman-debate-microservices-vs-monoliths)

Hightower 最近[在推特上](https://twitter.com/kelseyhightower/status/940259898331238402?lang=en)预测“在人们发现分布式单片应用的缺点之后，单片应用将会重新流行起来。”对于那些一直倡导[微服务](/category/microservices/)运营优势的人来说，这是一个相当大的惊喜。为什么要回到一块巨石上？

正如 Hightower 在播客中解释的那样:“有很多人从未离开过一块巨石。所以真的没什么可回去的了。所以这实际上是关于采用微服务架构的挑战。从设计的角度来看，就像极少数公司谈论的那样，这里是我们如何设计我们的整体。”

另一方面，Sigelman 认为微服务对于快速开发是必要的，而快速开发反过来对于维持业务也是必要的。“与其说你应该使用微服务，更像是，如果你没有比竞争对手更快地创新，你的公司最终将被抹去，就像，这就是实际问题。为了做到这一点，你需要建立许多差异化的技术，”他说。微服务是维持大型软件团队同时保持有竞争力的开发速度的最合理的方法。

在节目的稍后部分，我们讨论了本周的一些顶级 TNS 播客和新闻帖子，包括[对 IBM 的孙林关于服务网格重要性的采访](/why-an-ibm-master-inventor-calls-istio-service-meshes-storage-boxes/)，以及 [Sysdig 提供的分布式、可扩展的普罗米修斯](/sysdig-offers-full-prometheus-capability-to-monitor-cloud-scale-workloads/)，一群首席技术官[希望帮助美国政府](/ctos-seek-volunteers-to-help-the-u-s-governments-covid-19-efforts/)应对当前的新冠肺炎疫情，以及[开源安全带来的隐藏漏洞](/what-happens-to-your-app-when-an-npm-module-retires/)。
