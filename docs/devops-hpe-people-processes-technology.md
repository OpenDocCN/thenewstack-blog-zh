# DevOps 与人、流程和技术息息相关

> 原文：<https://thenewstack.io/devops-hpe-people-processes-technology/>

DevOps 的核心是将效率付诸实践。正如凤凰计划 中程序员小说 [*的作者*](http://itrevolution.com/books/phoenix-project-devops-book/) *[Tripwire](http://www.tripwire.com/) 创始人 [Gene Kim](https://twitter.com/RealGeneKim) 所指出的， [DevOps](/category/devops/) 是，“关于能够更频繁地发布，同时能够保持质量的结果。”*

在这一集的[中，我们将探讨 DevOps 的核心是什么，以及企业可以实施哪些基本原则来使 DevOps 流程取得成功。《新书库》创始人亚历克斯·威廉姆斯采访了格雷·格鲁弗咨询公司总裁加里·格鲁弗。](https://thenewstack.io/podcasts/)[IDC 项目服务副总裁 Cushing Anderson](https://twitter.com/cushinganderson) 和 [HPE](https://www8.hp.com/us/en/solutions/business-solutions.html) DevOps 高级总监 [Ashish Kuthiala](https://twitter.com/kuthiala) 听听他们的想法。这段播客是在 2016 年 HPE 探索大会上录制的。

[什么是 DevOps:对 HPE 来说，一切都是关于人、过程和技术](https://thenewstack.simplecast.com/episodes/what-is-devops-for-hpe-its-all-about-people-processes-and-technology)

虽然 DevOps 已经成为下一个新的业务实践，但开发人员通常选择避免在企业中设置自制 DevOps 工具的繁琐任务。库西亚拉开始研究这背后的原因。“当你开始在一个更大的企业中扩大规模时，我认为对来自开源的大量下载，或者将自己开发的工具拼接成一个工具链来实现目标的容忍度会降低。您希望在整个公司范围内实现标准化，并提供可扩展性。”

Anderson 随后签署了一份坚定的协议，指出“开发人员必须做一些事情，然后把它扔给其他人”的日子已经一去不复返了，因为组织已经开始培养他们自己的团队成员，使他们了解开发过程的所有部分。

Kuthiala 继续解释说，将 [DevOps 流程](https://thenewstack.io/tns-research-devops-factions-software-development-world/)实施到一个人的业务中并不像安装一套工具或一个平台那么简单，而是从整体上改变公司结构。“在我看来，如果你希望在一家公司或企业中实施 DevOps，它与三件事有关:人员、流程和技术，就像凳子的三条腿一样。该技术能够实现跨文化和流程的变革。我们在 HPE 开始考虑的是，我们如何利用我们的用户群已经在使用的工具来实际执行开发运维，我们如何补充这些工具并填补这些空白？”

Gruver 强调了这样一个事实，即专注于组织可以快速解决的问题通常会导致其大多数问题的解决，而不是在实施 DevOps 战略时为公司设置五年的转换过程。“这不是三年计划，也不是两年计划。我们试图让他们明白自己在未来 30 天内能做些什么，”格鲁弗说。

格鲁弗继续解释说，开发运维没有“唯一正确的方法”，库西亚拉和库欣指出，开发运维必须是可扩展的、标准化的和高效的，才能取得成功。随着讨论接近尾声，Gruver 向考虑将 DevOps 实践引入其工作流的企业提供了最后一条建议。

“这一切都是为了应用这些理念和原则。我不知道是否有正确的方法。这完全取决于部署代码的瓶颈是什么。”
