# 为什么你应该考虑建立一个开源项目办公室

SAP 对开源软件的贡献由来已久，无论是以 OpenJDK 项目和我们的 OpenJDK 版本的形式，还是以我们多年来对 Eclipse 基金会的贡献 T10 的形式(最近，我们的漏洞评估工具变成了 Eclipse stable)，我们正在进行的贡献[Kubernetes 贡献 T15 和](https://thenewstack.io/why-sap-wants-to-back-these-open-source-projects-for-kubernetes/)[Knative 贡献 T17，或者是像 CLA Assistant](https://knative.dev/community/contributing/annual_reports/Knative%202019%20Annual%20Report.pdf)这样的单个项目

然而，即使在 SAP 内部，这些贡献有多完善和广泛也是一个秘密，SAP 在 GitHub 上的开源商业贡献者中排名第九。这是 2018 年初决定在 SAP 内部建立开源项目办公室的一个因素。

“在过去，我们在共享我们的开源项目和活动方面可能不够积极，”SAP OSPO 主管 Peter Giese 说。

公司内部的目标是建立一个联系人，以便更好地支持员工对开源软件的贡献，同时通过增加我们与开源基金会的联系(例如，通过增加我们在开源活动中的参与)，使这些贡献在外部更加显眼。

在这里，我们将分享我们建立 OSPO 后 SAP 的一些变化，以及由此带来的好处，以及如果您正在考虑建立一个开源程序办公室，您应该考虑什么。

## 汇聚专家

OSPO 峰会带来了一个非常具体的变化，也带来了切实的好处，那就是对外流程的改进。2020 年初，项目团队成员在慕尼黑举办的 GitHub Enterprise Roadshow 上谈到了这一点。正如[法比安·哈格在她的博客](https://blogs.sap.com/2020/02/19/saps-open-source-outbound-process-from-scattered-and-tedious-to-streamlined-and-automated/)中对此次活动的报道，自 OSPO 创立以来，申请的批准时间减少了 70%。在法比安看来，“在正确的时间把正确的人召集到一起，问正确的问题”，才是让整个过程变得更快、更精简的秘诀。

先前存在的流程和政策会随着时间的推移而扩展，但这些更改仅由各个团队不断变化的需求指导。“我们有几个团队负责开源的特定方面，比如安全扫描、许可证扫描和构建我们自己的开源工具，”Giese 说。“但 SAP 没有专门的职能或角色来全面负责所有开源项目。”

“这种情况现在已经改变了，”他继续说，“SAP 的首席技术官负责 SAP 的开源。”

自 SAP OSPO 问世以来，这些改进可以通过一个中心组织来实现，该中心组织从这些不同的内部利益相关方那里提供必要的支持和专业知识，以协作方式进行这些更改。

## 使用熟悉的方法

OSPO 在处理开源工具、过程管理和治理的所有其他方面时采用的方法是上面描述的敏捷 scrum 工作模型 Fabienne，它使用来自不同责任领域的灵活项目团队。当然，并不是所有需要专业知识的人都曾在开发组织中工作过(敏捷方法和 scrum 工作模型在这里已经非常成熟)，但使用这样的工作模型意味着您可以(再次)利用现有的知识来铺平道路。

使用 GitHub 作为管理出站流程本身的工具是受到了同样的想法的启发:当一个现有的、熟悉的工具已经足够时，不要让开发人员学习或适应一个新的工具。使用熟悉的工具和工作模型的好处是，您不需要讨论将使用哪些工具，也不需要花时间学习新方法，您可以直接致力于您要实现的目标。

## 合作

当然，学习新的工作模式和工具对于一些参与者来说可能是必要的，但是在像 SAP 这样的大型组织中，我们发现这对于给定角色的典型日常工作来说是一个令人耳目一新的变化；它也有助于在那些无法基于共同角色或经历而相互认同的人(和部门)之间建立共鸣和信任。

互利的项目合作是开源方法的一个关键原则和优势，如果您想增加您的开源贡献或建立自己的 OSPO，学习以不同的方式合作和工作可能是您面临的挑战之一。通过拥有一个更大的“熟悉内情”并建立了工作关系的人才库，新的想法或潜在的流程改进可以更快、更非正式地得到处理。

## 改变观念

如上所述，即使在 SAP 内部，也可能对我们的开源贡献以及开源在我们产品中的作用缺乏认识。但是 SAP 100%的解决方案都以这样或那样的方式触及开源。拥有一个 OSPO 意味着意识到开源是标准的实践，它也有助于吸引过去使用开源的外部人才。

开源开发让 SAP 能够在许多不同的技术上利用合作创新，这是我们自己永远无法开发或推动采用的。“不是在这里发明的”反而成为一个机会，可以在他人的想法基础上协作构建，更有效地开发您的产品——有时也通过 [InnerSource](https://www.youtube.com/watch?reload=9&v=2nhKMcv5STc) 开发，开发人员遵循开源原则和方法在您的公司内协作。

## 持续改进

例如，出站流程仍在改进和完善中。通过定期召集各种开源专家，已经开发了新的工具来自动化流程的其他方面，并确保我们开源项目的安全性(如 SAP 的[开源项目安全评级框架](https://github.com/SAP/fosstars-rating-core))。自从引入这些改进之后，我们已经看到新的外向开源项目显著增加。

在某种程度上，有一种滚雪球效应，工具和流程的持续改进只会进一步扩大圈子，让更多的人参与进来并做出贡献；沟通渠道意味着想法和成功可以被分享和放大。这反过来意味着有更多的话题可以对外分享和推广，因此增加了在[开源峰会](https://events19.linuxfoundation.org/wp-content/uploads/2018/07/OSLS_2019_SAP_OSPO_Peter_Giese.pdf)、[、](https://conferences.oreilly.com/oscon/oscon-or-2019/public/schedule/detail/79361) [FOSDEM](https://fosdem.org/2020/schedule/event/npes/) 、 [EclipseCon](https://www.eclipsecon.org/europe2019/sessions/developing-openjdk-java-libraries-eclipse-jdt) 等的出席率。“这就是为什么我们还加入了 TODO 小组，以分享经验，共同开发最佳实践，并致力于共同的工具，”吉斯说。这种更大的互动反过来为学习和提高开辟了新的途径。

## 新冠肺炎和电晕警告应用程序

SAP 内部已经建立的 OSPO 的好处的一个最近且非常具体的例子是与德国电信为德国的罗伯特科赫公共卫生研究所共同开发的 Corona-Warn-App。联邦政府要求创建一个应用程序，提醒用户之前接触过后来新冠肺炎病毒检测呈阳性的人。

我们的意图是使用一种分散的方法来开发应用程序，使用开源原则并最大限度地尊重用户的隐私，这在德国是一个绝对基本的要求。只有以开源项目的形式开发和发布这款应用，才能确保安全和数据隐私的清晰性。随之而来的早期反馈和错误报告也在改进应用程序本身方面发挥了重要作用，并且[有助于确保在发布后成功采用](https://www.linkedin.com/pulse/corona-warn-app-wo-stehen-wir-nach-dem-ersten-monat-first-mueller)(德语和英语文章)。

SAP 的 OSPO 已经建立的工具、流程、专业知识网络和协作渠道意味着这一艰巨的挑战可以在短短 50 天的开发时间内以非凡而成功的方式得到解决。

根据我们在 SAP 的经验，建立 OSPO 的好处和影响无疑为我们的开源工作提供了新的焦点和动力。拥抱定义和驱动开源方法的开放和协作精神也可能给你的组织带来好处。
