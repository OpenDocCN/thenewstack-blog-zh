[无服务器计算](https://thenewstack.io/category/serverless/)框架开始获得关注，因为它们提供了另一个抽象层，使 IT 基础设施资源在需要时动态可用。对于许多开发人员来说，这种能力代表了类似于 it 涅槃的东西，因为至少在理论上，这意味着 IT 管理员不再需要在他们和他们的代码之间进行干预。

在某种程度上，那是真的。无服务器计算框架基于事件驱动的架构，这使得开发人员无需担心任何潜在的资源限制。这并不意味着没有任何服务器。这只是意味着额外资源的供应对开发人员来说是即时和透明的。

最广泛使用的无服务器计算框架是 AWS Lambda。但这几乎不是唯一的一个。如今，每个主要的云提供商都提供无服务器计算框架，在不久的将来，无服务器计算框架将成为任何应用程序开发和部署框架的自然延伸。

开发人员采用无服务器计算的主要原因是，它支持更细粒度的方法来构建微服务。不需要持续运行的应用程序的无状态元素现在可以处于休眠状态，直到需要的时候。每当发生需要微服务的事件时，就会向无服务器框架发送警报，从而使计算资源立即可用。随着应用程序的无状态元素转移到无服务器计算框架上，应用程序变得更小，运行成本更低。最重要的是，应用程序环境变得更加灵活，这反过来导致更多的应用程序以更快的速度被开发。

虽然这种方法非常适合无状态工作负载，但在真实的 IT 世界中，大多数企业应用程序会大量使用需要访问持久存储的有状态工作负载。有状态工作负载不适合无服务器计算框架。随着应用程序开发的不断发展，长期运行的有状态应用程序将继续在现有的框架上运行，这些框架将通过无服务器计算框架得到增强，以处理无状态工作负载。这些无状态工作负载本质上是主应用程序工作负载的子进程。

未来真正的挑战是让开发人员更容易在他们正在使用的平台环境中调用无服务器计算框架来构建和部署他们的企业应用程序。今天，无服务器计算框架是一种前沿模式，用于使用开发人员难以掌握的晦涩的函数式编程模型来处理特定类别的工作负载。而无服务器框架只适用于无状态的工作负载，这些工作负载往往会在应用程序中激增。长期运行的工作负载将在基于容器或虚拟机的现有应用程序开发和部署框架上运行，这些框架可以部署在内部或公共云中。

限制无服务器计算框架使用的其他因素包括缺乏健壮的管理工具，以及监视和调试无服务器计算环境需要高度专业知识的事实。除非组织拥有成熟的应用程序开发和部署流程，否则添加无服务器计算框架对一般企业 IT 组织来说是一项复杂的挑战。无服务器并非没有痛苦。

同样值得注意的是，无服务器计算框架来自任何接近行业标准的东西。像云本地计算基金会(CNCF)这样的组织正致力于创建一组接口，使多个无服务器计算框架能够作为 Kubernetes 集群的自然扩展被调用。但目前，AWS Lambda 等无服务器计算框架是专有平台，将 IT 组织锁定在特定的云服务中。然而，随着调用无服务器计算框架的接口被定义，更快的公共云计算服务将成为商品。用不了多久，如果 IT 组织愿意，云服务提供商甚至可能会出价处理每秒只需几分钟 CPU 周期时间的工作负载。并非每个 IT 组织都准备好接受毫秒级粒度的成本模型，并从 PaaS 的成本稳定性中获益更多。重要的是能够跨云内外的多种类型和类别的抽象保持灵活性。

无服务器计算框架将不可避免地经历困扰其他新兴技术的炒作周期。非理性繁荣之后很快就会幻灭。毫无疑问，无服务器计算框架将在未来的 IT 中发挥至关重要的作用，但它们只是众多抽象类型中的一种，包括虚拟机、容器以及同样重要的 PaaS 环境。事实上，可以说是通过 PaaS 环境，无服务器计算框架的大部分优势将最终作为任何全面的多云计算策略的扩展来实现。

云计算原生计算基金会是新堆栈的赞助商。
