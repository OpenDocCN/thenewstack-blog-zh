# 测量开发人员生产力的精神病任务

> 原文：<https://thenewstack.io/the-psychotic-task-of-measuring-developer-productivity/>

测量开发人员的生产力是一项纯粹的、完全精神错乱的任务。没有比工程师和开发人员更好的职业来开发一个本质上是游戏的系统了。设定一个衡量标准，让它变得公平，变得合理，在某种程度上，它将成为一个游戏。真正的问题是，“我们是否在牢记公司使命的情况下，稳步、可靠、负责任地朝着我们的目标前进？”这能被测量吗？在战术层面上衡量这些东西合理吗？几乎在每种情况下，这都是相当疯狂的，但每天都在努力。有时这些努力比其他的更成功，但是在几乎每一种情况下，除了少数情况，这些努力会导致更替和奇怪的进展水平，不可靠的软件，有时甚至是对正在构建的东西缺乏责任感。然而，有些公司在留住员工、稳步发展以及将责任赋予客户、构建者和其他参与者方面取得了非凡的成功。我已经为其中几家公司工作过，其他人也为我知道的其他几家公司工作过。在这篇文章中，我不打算说出这些公司的名字，因为那样会分散注意力，因为人们会无休止地争论它们的优点和缺点。相反，我将写一些具体的战术努力，以及他们如何取得进展，可靠性和责任。谁有哪些弱点，谁有哪些长处？找到优势，将它们与劣势配对。建立和维持强大团队的最有效策略之一。以下是我对如何少关注生产力，多关注组织的整体实力的一些观察。

*   将优势与劣势配对，至少 20%的时候将劣势与优势配对。如果有人真的精通服务器端的 JavaScript，那就找一个客户端的 JavaScript 编码器一起做一些节点编码。翻转它们，因为它们都有优点和缺点，让客户端 JavaScript 编码人员领导 Node.js 编码。
*   塑造个性，尤其是在年轻和没有经验的开发人员中。人们做得好的时候需要提醒。荣誉徽章之类的东西很棒，但真正的奖励也很壮观，比如金钱或其他。尤其是如果这部电影是一部价值百万美元的电影，那就给他们一部分真钱。他们不会离开，他们会为他们的成就感到骄傲！
*   把失败说成失败，把它们公开，讨论它们。但最重要的是，把失败当成失败，而不是把团队中的一个人当成失败。一次失败是团队的失败，而不是个人的失败。攻击个人等同于不尊重——消除不尊重，立即将其从任何文化中移除。
*   包容，发现那些不是主流的人。找到那些能给团队带来新思想、新观点或新观察的人。帮助他们帮助我们所有人成功地维持我们的努力。不要因为人们看起来古怪、不同或者不是标准的“白人男性程序员”而忽略他们。这意味着领导力必须真正与他们圈子之外的人交谈，领导力必须扩大其个人关系网，领导力必须站在这些努力的最前沿，并树立理想的榜样——而不仅仅是谈论崇高的目标。否则，领导层就会失败，让这种失败永久化，并毁掉公司成为伟人的任何机会。
*   停止消极攻击，它不会帮助任何人以提高他们编码能力的方式维护自己。

作为一名开发人员或工程师，与任何销售、运营或营销工作一样，有许多“生产力”问题。实际上，开发人员生产力的真正含义在很大程度上仍然是一个猜谜游戏。但有一点是肯定的——警惕不成文的规则，当一个好的猜测只能让你到目前为止时，就会出现问题。
