# 开源项目的动力:需要什么

> 原文：<https://thenewstack.io/open-source-project-momentum-what-it-takes/>

[KubeCon+CloudNativeCon](https://www.cncf.io/kubecon-cloudnativecon-events/) 赞助了这个播客。

许多项目都是为了解决组织或用户遇到的问题而启动的。由于开源的魔力，社区可以帮助解决问题，并且在理想情况下，提供比创造者最初希望的更好的解决方案。维护人员的主要任务主要是帮助确保软件平台或工具持续改进，并确保贡献得到适当的维护和管理。

在本期 [The New Stack Makers](/podcasts/makers) 播客中，来自 Red Hat 和 VMware 的嘉宾讨论了开源项目如何发展、变化并变得可持续。嘉宾有:[VMware 产品总监 Michael Michael](https://www.linkedin.com/in/mimichael)；[特拉维斯·尼尔森](https://www.linkedin.com/in/travis23)，高级首席软件工程师，[红帽；](https://www.openshift.com/) [安妮特·克莱维特](https://www.linkedin.com/in/annette-clewett-2101379)，红帽首席建筑师；以及 Red Hat open shift 产品管理高级经理 Rob Szumski 。《新书库》的创始人兼发行人亚历克斯·威廉姆斯主持了这次对话。

[开源项目的动力:需要什么](https://thenewstack.simplecast.com/episodes/open-source-project-momentum-what-it-takes)

开源用户的需求和约束当然会有所不同，同时为开源项目的发展提供了主要的推动力。例如，Clewett 描述了她如何使用 Rook 将它带到下游，成为 Red Hat 称之为[open shift Container Storage](https://www.openshift.com/products/container-storage/)的产品。

“我在上游做了很多测试和工作，但我的主要任务是尝试让 Rook 在下游运行。我们所做的一切都是先上游后下游，”Clewett 说。“所以这真的不存在同步问题。”

作为最初的 Rook 维护者和创造者之一，Nielsen 解释了当 Rook 创建时，他的团队是如何看到对存储和 Kubernetes 的需求的。今天，Nielsen 仍然是一个项目维护者，并随着用户需求的变化不断寻求改进 Rook。

尼尔森说:“我们有一些功能，当客户要求时，当我们看到有需要时，我们就会带来这些功能。”。“在库伯内特，人们需要各种各样的储物用品。”

VMware 在其业务部门中拥有许多开源工具和技术，例如 Harbor(一个开源注册表)和 Contour(一个用于 Kubernetes 的[入口](https://www.ingress.com/)控制器)。“从项目的角度来看，我的主要目标之一是促进和确保我们有一个健康的生态系统来发展我们的项目，”迈克尔说。

开源治理政策尤其有助于确保 Michael 领导的项目的目标得以实现。“我们希望每个人都有发言权、机会和席位，最重要的是，帮助我们发展这些项目，”他说。

红帽的 [Operator SDK](https://github.com/operator-framework/operator-sdk) ，一个改进操作符编写方式的框架，以及 [Operator Lifecycle Manager](https://github.com/operator-framework/operator-lifecycle-manager) ，一个帮助用操作符扩展 Kubernetes 的框架，最近在 Cloud Native Computing Foundation 成功实现了[孵化状态。虽然 CNCF 身份是一个主要目标，但 Szumski 解释说，该团队的“长期目标”是“让运营商成为一个东西”](https://www.cncf.io/projects/)

“我们教人们如何思考编写这种类型的软件，以及它与他们可能编写的其他软件有何不同，然后如何向客户推销这种软件，”Szumski 说。“所以这是大量的开源软件。”
