# 新的安全之路:开发优先安全的优势

> 原文：<https://thenewstack.io/a-new-path-for-security-the-benefits-of-a-dev-first-security/>

Snyk 赞助了这篇文章。

 [盖伊·波德贾尼

Guy Podjarny 是 Snyk 的联合创始人兼总裁。](https://www.linkedin.com/in/guypo/) 

“数字化转型”对许多人来说是一个时髦词，但事实上，它是一种自然力量，从根本上改变了企业的运营方式。它鼓励创新，并带来了全面的成功结果。

然而，随着整个组织寻求根据最新的数字化转型趋势(软件、云或开发运维)重塑自身，安全实践与二十年前相比仍基本未变。

数字化转型企业对安全性的需求与以前一样强烈，甚至更强烈。但是，当组织的其余部分通过自治团队不断变化时，安全团队通常保持孤立状态，并且在做出决策时通常不被咨询，这导致了不安全的应用程序。

因此，这种不知情的安全团队会强制执行手动审查和安全检查，从而在企业寻求加速发展时减慢速度。这一切都无助于十年来安全领域人才的严重短缺，导致安全团队人手不足。

这不是前进的道路。

我们需要一种新的方法来实现持续和集成的安全性。一种将安全性融入新技术和方法并从内部构建的模式。It 需要支持自给自足的团队，加速业务发展，而不是减缓业务发展。换句话说，它需要以开发者为先。

## **用 DevSecOps 取得控制权**

这种方法通常被称为 DevSecOps，需要一种更具协作性的安全方法，从而带来更高的效率并鼓励数字化转型，而不会增加安全风险。

但是在实践中是如何运作的呢？

最终，它需要开发人员和安全团队一起工作，并依次对安全性负责。

对于开发人员来说，这意味着首先要考虑安全性，并感觉有能力掌控它。而对于安全团队来说，更多的是不再把自己看作是一个控制者，而是一个支持性的功能，任务是让开发人员能够发现并解决安全问题。

## **授权开发者进行切换**

要让开发人员完全接受安全性，需要对安全性当前的运行方式进行重置。本质上，安全团队需要自下而上的思考，而不是自上而下的思考——这不是规定开发的步骤，而是帮助他们自己做出安全的决策。

要有效地做到这一点，需要安全团队投资于建立同理心；在你开始强加你自己的观点之前，花时间真正站在开发者的角度，尝试更好地理解他们的世界观。

> 要让开发人员接受安全解决方案，它必须看起来像是开发人员的工具，而不是安全工具。

这一同理心之旅的关键一步是理解当你“缩小”时所看到的差异。例如，假设您正在查看某个给定应用程序的开源依赖项中的已知漏洞。对于安全人员来说，缩小范围意味着查看所有应用程序中的已知漏洞，以便更好地了解风险。对于开发人员来说，这意味着查看同一应用程序的所有缺陷——包括功能性、可操作性等——以更好地了解应用程序的质量。从全局角度来看，这种区别需要重新思考如何呈现安全缺陷，将它们置于*应用*环境中，而不是*风险*环境中。

团队之间的另一大区别是他们使用的其他工具看起来像什么。对于安全团队来说，邻近的工具本质上是核心(它们关注风险和治理)，并且它们的目标是发现和管理缺陷。然而，开发人员身边都是自助式的工具(帮助他们构建和创建，而不是破坏)，并且专注于修复问题而不是记录问题。要让开发人员接受安全解决方案，它必须看起来像是开发人员的工具，而不是安全工具。

除了用户体验，我们还必须承认，开发人员不具备——也永远不会具备——与安全团队同等水平的安全专业知识。开发人员有不同的全职工作，安全是他们承担的多重责任之一。因此，面向开发的安全解决方案必须具有内置的安全专业知识，并引导开发人员做出安全决策。越容易做出正确的决定，应用程序就越安全。

## **反思安全团队**

尽管如此，这种转变需要的不仅仅是工具和开发人员的参与。安全团队也需要适应。

在组织内部，您应该考虑将您的安全组——传统上倾向于反映中央 IT——与开发团队保持一致。例如，您可以将一个 AppSec 人员分配给几个开发团队，允许他们更好地融入团队，并为开发人员提供一个明确的合作伙伴。组织越一致，合作就越容易。

另一个你应该反省的是你团队中的技能；并考虑通过招聘或培训来缩小差距。例如，希望构建工具并具有更好的开发同理心的安全人员将从了解如何编码中受益。更重要的是，建立一个支持性的安全团队需要成员兴奋地这样做，你可能需要处理那些仍然希望独裁控制。

不管你喜不喜欢这个流行词，数字化转型正在发生。它带来的变化是真实的，安全实践的发展速度远远赶不上开发和运营。我们需要对此做些什么。我们需要首先将安全性转变为开发。
