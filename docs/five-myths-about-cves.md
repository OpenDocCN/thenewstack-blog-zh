# 关于简历的 5 个误区

> 原文：<https://thenewstack.io/five-myths-about-cves/>

任何在软件行业工作的人都知道，CVEs 或“常见漏洞和暴露”任务可能会带来污名。这是由误解造成的，误解使得供应商和开源维护者不愿意请求 CVE，这降低了透明度，并最终将软件安全置于风险之中。

不情愿是可以理解的，因为仅仅提到 CVE 这个词就可能引发恐惧、不确定和怀疑。这会给我的项目带来坏名声吗？这个问题是否严重到足以证明获得任务的努力是正确的？我应该如何与提交漏洞的安全研究员合作？

然而，每个 CVE 不需要引起那种程度的焦虑。毕竟，它们只不过是分配给特定漏洞的唯一标识符或跟踪号。那么，为什么供应商和开源维护者对请求 CVEs 犹豫不决，即使漏洞很小？

多年来，出现了一系列关于 CVE 的神话，给它们带来了坏名声，这滋生了对报告的犹豫，并增加了漏洞未被发现和修补的可能性。揭穿这些误解是至关重要的，因为 CVE 对于确保最终使软件更加可信和安全的透明性是必不可少的。这里有一个关于 CVE 的五个主要误解的简要分析，如果不解决这些误解，将会对软件安全的进步产生不利影响。

1.  CVE 总是一个非常严重的问题。 这并不总是正确的，因为严重性不能从 CVE 标识符的存在来推断。负责任的供应商和维护者，实际上，会请求 CVE 并发布建议，甚至是针对小的漏洞。
2.  **CVE 有损软件及其供应商或维护者的声誉*。*** 完全相反。对潜在漏洞保持透明表明供应商和维护者认真对待安全性，并增加对他们项目的信任。与此同时，忽视潜在问题可能会带来可怕的声誉后果。
3.  **低严重性漏洞不需要 CVE *。*** 不真实。咨询应该描述技术细节，以便用户可以自己对漏洞做出决定。如果供应商和维护者认为漏洞很小，他们可以在严重性评分和公告本身中解释这一点。
4.  **CVE 是发现漏洞的安全研究员收集的战利品*。简历都是透明的，给用户做出明智决定的机会；它们不是需要大量收集的低质量奖杯，也不是“明白了！”时刻。研究人员为其报告的问题寻求 CVE 奖是健康安全实践的一部分。此外，如果简历被认为不公平或不正确，就会受到质疑。***
5.  获得 CVE 奖是一个痛苦的长达数月的过程。 不如此。事实上，CVE 编号机构(CNAs)在几天内分配简历的情况并不少见。一些提供编辑服务，以确保请求符合程序规则，甚至编写可以与其他 CVE 数据库共享的摘要。

显然，认为 CVE 会破坏项目的想法是不正确的，而对漏洞保持沉默并抱乐观态度可能会导致声誉上的后果。维护人员可以通过 CVE 编码机构(CNA)请求 CVE。例如，GitHub 充当平台上任何利用 GitHub 安全咨询(GHSA)的开源项目的 CNA。通过 GHSA，维护人员可以将漏洞通知他们项目的社区，这样用户就可以更新包的依赖关系，并研究安全漏洞的影响。(披露:我是 GitHub 的员工。)

CVE 是在项目中建立信任的重要元素，也是确保漏洞信息可被广泛访问且准确的最佳方式，这允许用户基于事实对风险做出自己的评估。在克服这五个误解的过程中，供应商和维护者可以将他们的注意力转移到围绕他们的软件项目建立和维护透明性和信任上来。
