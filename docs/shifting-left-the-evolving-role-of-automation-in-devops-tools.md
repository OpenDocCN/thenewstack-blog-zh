# 左移:自动化在 DevOps 工具中的角色演变

> 原文：<https://thenewstack.io/shifting-left-the-evolving-role-of-automation-in-devops-tools/>

安全和开发团队目前面临的最大挑战之一是在速度和安全性之间取得平衡。利用自动化产生高质量结果的应用安全测试(AST)工具必须继续发展，以帮助组织将这两个组件结合在一起。目标应该是通过自动化漏洞检测和分类来缩短安全软件的上市时间，从而转变为真正的 DevSecOps 模型。

根据[分析公司 Gartner](https://info.checkmarx.com/gartner-mq-2020) :

*“…现代应用程序设计和 DevSecOps 的持续采用正在扩大 AST 市场的范围。安全和风险管理领导者将需要通过在软件交付生命周期中无缝集成和自动化 AST 来满足更紧迫的期限和测试更复杂的应用。”*

然而，目前市场上的遗留 AST 解决方案倾向于在所使用的 CI 工具之外运行，并且扫描通常在合并/构建已经发生之后执行——进一步到软件开发生命周期(SDLC)的右侧。

## **说真的，是时候左移了**

另一方面，较新的 AST 工具允许组织将该功能转移到左边——最具创新性的工具具有编排层，简化了现代开发环境中安全测试的实现和自动化。随着 DevOps 和安全测试的发展，扫描现在可以自动触发，将结果直接嵌入到 GitLab 等 [DevOps 工具](https://thenewstack.io/category/devops/)的 [CI/CD 管道](https://thenewstack.io/category/ci-cd/)中。结果呢？简化的 DevOps 工作流、更早的软件漏洞检测(也就是尽可能向左移动)，以及更高的开发人员满意度，因为他们能够一直呆在自己喜欢的环境中。

## **是的，端到端自动化是一件事情**

通过在 SDLC 的早期自动执行扫描代码所需的步骤，它不再需要耗时的手动扫描配置。它还允许开发人员根据代码管理工具中预先配置的策略发布和更新扫描结果。在初始配置之后，除了开发人员发起的拉取请求之外，AST 扫描活动无需人工干预即可执行。

似乎再简单不过了，现代自动化工具还允许开发人员:

*   在编码阶段(开发的最早阶段)捕获并修复漏洞。
*   照常工作，没有中断，没有新工具，不需要额外的安全审查，等等。
*   同等对待安全错误和功能错误，并允许他们立即在他们当前工作的代码分支中解决这些错误。
*   减少手动打开、验证和关闭安全票证的开销，而无需在错误跟踪或票证管理系统上花费大量时间。

## **抹去沙地上的线导致真正的发展**

高级自动化工具消除了 DevOps 中每个项目的手动和耗时配置，从而消除了开发人员和 DevOps 团队在需要将扫描步骤添加到所有 CI 管道的作业中时的摩擦。使用传统的 CI-scan 模型添加扫描代码的作业或步骤很有挑战性。先进的自动化工具最终打破了团队之间的障碍，使他们能够更好地合作，实现真正的 DevSecOps 集成。

最终，将 CI/CD 管道左移和自动化将显著提高 SDLC 内的安全性集成。组织可以立即加入其开发、安全和运营团队，并简化其安全策略和开发运营流程的治理。传统的 AST 解决方案提供商将开发人员抛在了后面，因为没有在您的环境中直接扫描源代码的能力，您就不得不手动处理扫描——这为边际错误留下了很大的空间，并为您的最终交付日期增加了很多时间。

如果我能告诉你一件事，那就是集成是自动化的关键，你使用的工具应该支持最左移的方法，自动化可以在 SDLC 中发生，改变 AST 解决方案在所有 DevOps 环境中的嵌入方式。
