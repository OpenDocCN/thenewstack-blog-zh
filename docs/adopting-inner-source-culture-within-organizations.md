# 在组织内采用“内部资源”文化

> 原文：<https://thenewstack.io/adopting-inner-source-culture-within-organizations/>

开源软件的整个伦理有一个未言明的含义:它旨在被更广泛的社区共享。不是所有的软件组织都可以使用，*应该*在它自己的边界之外共享——物理的或虚拟的。但是，对专有服务的保护和保障的存在是否一定会消除应用*某种*开发社区伦理的任何机会，仅仅是在那些受保护软件确实适用的人中？

Capital One 解决这个问题的方法是开发自己的内部应用程序框架——一个仅适用于自己员工的平台，但它鼓励所有团队采用社区建设方法。在最近一次 OSCON 会议上，Capital One 软件工程高级总监[与 New Stack 的 Alex Williams 交谈时，Kranthi Dandamudi](https://twitter.com/KranthiDandamud) 介绍了 Chassis，它的内部应用程序框架旨在促进公司所称的“内部来源”工作道德。

Dandamudi 解释说:“Capital One 需要收集商业事件，以便进行法律辩护。“围绕支付或余额转移有一些功能需求。人们需要对应用程序进行换肤和主题化。”Dandamudi 解释了这些类别中的每一个，作为他所谓的“横向交叉”的指示——适用于多个部门的功能类别。

但是，当平台已经存在于更广泛的开源社区中时，一个资本规模的机构真的需要播种它自己的内部“内部源代码”社区，以便主动创建一个应用程序平台吗？换句话说，如果 Capital One 既做了内部共享，也做了外部共享，难道它不会完成更广泛的社区已经完成的基本相同的事情吗？

不见得，Capital One 软件工程副总裁玛格丽特·梅尔解释道。她的团队确实调查了外部框架的可能性，但最终意识到平台应用消费者的要求(在该公司的情况下，这些要求都是内部的)适用于公司的核心道德和目标，没有任何外部平台可以完全解决。
