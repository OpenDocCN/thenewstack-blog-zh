随着原生云成为构建和部署现代应用的事实上的标准，以及微服务风格应用架构、敏捷开发运维流程和持续交付工作流的激增，对高度扩展的原生云业务自动化平台的需求不断涌现。

虽然传统工作流平台提供了运行、管理和优化贯穿各种业务功能的长期运行业务流程的能力，但它们在 Kubernetes 或 OpenShift 等容器平台上运行和编排应用的能力限制了它们编排云原生应用和微服务的潜力。

考虑到这一点，出现的最大问题是:云原生业务自动化/工作流平台由什么组成，它与传统的流程自动化平台有何不同？下面是我对云原生业务自动化引擎的特征的看法。

### 状态管理和持久性

传统的工作流自动化平台严重依赖关系数据库来保持流程实例的状态，而云原生业务自动化平台倾向于依赖内存中的数据网格来进行状态管理。使用内存中的数据网格(带有合适的后备存储)通常会减少通常引入状态管理/业务自动化引擎的延迟，同时在关系数据库中保持流程实例状态。

### 活动采购

云原生业务自动化平台通常会通过事件源为事件提供一流的市民支持。在这种情况下，事件源是指在执行工作流的过程中本机发出流程及其组成部分的生命周期事件的能力，以便其他应用程序(包括微服务)可以根据业务需求订阅这些事件并对其做出反应。

此外，该平台需要能够输出流程模型中定义的一组业务事件，以便微服务和业务应用程序的编排在本质上更具反应性，从而使流程自动化更加灵活和松散耦合。

### 特定领域上下文

提供原生业务领域和工作流特定 API 的能力，而不是一组通用 API，是云原生业务自动化平台的一个关键区别因素。最常见的是，工作流引擎提供了一组 API，这些 API 通常会导致工作流引擎的抽象泄漏(如引擎特定的数据模型、接口定义等)。)集成到与工作流引擎集成的应用程序中。提供特定于领域和工作流的 API 将使与工作流的交互更加符合微服务和云原生风格。

### **在容器平台上本地运行**

利用容器平台的核心功能，更具体地说是可扩展性、弹性、服务发现、容错、无缝部署、安全性、可追溯性、可观察性和监控，是云原生工作流引擎相对于传统工作流引擎的最大优势之一。云原生业务自动化平台通常提供将每个工作流应用程序(微流程流)单独原生容器化并在任何容器平台上运行的能力(可能使用[操作符](https://coreos.com/operators/) ) —非常像任何其他微服务，从而提供混合云支持。

### **无服务器**

对集成了几个应用程序的工作流(短期流程)的处理进行直接编排(包括管理它们的状态和重试，以及故障处理)可能极具挑战性。借助对无服务器计算的本机支持(主要通过 [Knative](https://cloud.google.com/knative) 组件)，云本机业务自动化平台能够将错误处理、重试和故障处理逻辑(从编排逻辑)卸载和分离到服务网格。这使它成为基础设施问题，而不是工作流问题。

### 最小占地面积

惊人的快速启动时间和难以置信的低内存占用是运行在容器平台上的应用程序的当务之急。云原生业务自动化平台就是这种情况。这些平台通常在 Kubernetes 等容器编排平台中提供近乎即时的纵向扩展和高密度内存利用率。

### 结论

目前，没有多少业务自动化引擎非常适合云原生业务自动化领域——我想到了 [AWS Step Functions](https://aws.amazon.com/step-functions/) 和 [Lambda](https://docs.aws.amazon.com/step-functions/latest/dg/tutorial-creating-lambda-state-machine.html) 、 [Zeebe](https://zeebe.io/what-is-zeebe/) 和 [Kogito](https://kogito.kie.org/) 。虽然这些业务自动化平台中的每一个都提供了上述许多功能，但 Kogito——一个由 Red Hat 赞助的项目，源自一些著名的开源项目( [Quarkus](https://quarkus.io/) 、 [Drools](https://drools.org/) 、 [jBPM](https://jbpm.org/) 、 [OptaPlanner](https://optaplanner.org/) 和其他一些项目——似乎是构建云原生工作流的新兴平台。
