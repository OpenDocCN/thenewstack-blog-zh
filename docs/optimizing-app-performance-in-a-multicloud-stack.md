云越来越成为企业生存的地方。这一进程在最近几年有所加快。不仅企业越来越依赖云，许多企业使用云的方式也发生了变化。

无论是作为多供应商战略的一部分，避免锁定的一种方式，额外的安全冗余，开发人员的偏好，还是出于其他原因，多云正在成为新的常态。例如，如果您正在使用 Salesforce 或 Office 365 等 SaaS 应用程序，那么您已经依赖于多个云。

多云策略有很多优势。但它也给企业带来了一些挑战:更加复杂的监控环境、缺乏可见性，以及跨区域和云架构的性能差异。所有这些挑战都会影响最终用户体验。

现在，客户关系的质量和组织员工的生产力取决于他们数字体验的质量。这种数字体验反过来又依赖于由多个云、分布式应用架构和复杂的 API 和第三方服务网络组成的连锁系统。

这意味着有许多活动部件，许多部件可能会出错。一旦出现这种情况，迅速解决问题是当务之急。但是，支持我们所依赖的服务和功能的复杂性，加上缺乏端到端的用户体验视图，可能会使正确升级和解决问题成为一项挑战。

## **云迁移流程呈现出独特的可见性问题**

你的情况可能会不同。也许正是因为这些顾虑，您拒绝将重要的公司功能和数据迁移到云，因为依赖公共互联网会导致缺乏控制，并且缺乏对云提供商基础架构的可见性。

对于许多公司来说，迁移到云是一个高风险的操作。这就像把皇冠上的珠宝交给别人保管。仅仅能够在转移后监控一切是不够的，您需要在转移开始前获得可见性和测试能力。

我们需要的是一种在生产前环境中测试性能影响的方法。这样，尽管放弃了对应用程序的控制，企业也可以轻松地进行迁移，因为他们能够在迁移过程开始之前对其进行监控和评估。问题是:你如何做到这一点？

## **运行中的多云监控**

多云监控的挑战本身是多方面的。需要安全、可靠和顺利的迁移。多个应用程序基础架构的复杂性相互重叠。需要考虑地区性能差异，以及您不拥有的环境中的可见性挑战。

与此同时，对整个供应链的端到端可见性对于查看、预测和优化客户和员工所依赖的数字体验至关重要。

传统的监控工具在企业边界之外，为多云部署创造了一个可见性盲点，因此原生云监控是不够的。网络是所有云通信的粘合剂——从终端用户到云、云内和云之间。因此，在多云环境中校准性能需要了解支持应用的公共和私有生态系统之间的数百种依赖关系。

这意味着一个成功的多云监控解决方案需要一些关键功能。它应该能够:

*   在一个云中、跨云以及跨区域对测试应用性能。
*   通过公共互联网和云提供商的私有主干网络测试连通性。
*   在预生产环境中，使用综合监控来查看和评测端到端应用性能。

多云架构非常复杂，运行于多个托管环境、地区和提供商。为了管理多云环境中的应用性能，必须跟踪相互依赖的连接并访问实际性能数据，以避免错误假设的风险并优化数字体验的最终交付。
