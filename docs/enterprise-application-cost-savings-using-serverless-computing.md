数据中心时代的容量规划是一门艺术。它部分是科学，部分是猜测，部分是与供应商就硬件和软件进行的谈判。一般来说，企业能力规划周期比实际需求提前六个月到一年开始。该流程包括预测特定应用程序的需求，然后采购和设置服务器以扩展应用程序。对于企业来说，早在半年前就预测其应用能力是一种非常常见的做法。

毕竟，由于担心错过新的业务/用户，企业总是购买过剩的容量，以防万一。更糟糕的是，在使用了这种过多的、不必要的容量之后，项目团队通常倾向于囤积服务器，担心无法再次获得用于项目的容量。正因为如此，企业数据中心过去常常得不到充分利用。我有时看到 10-20%的计划容量的使用统计。即使在这一切之后，额外的容量请求也可能需要长达三周的时间来满足，即让服务器映像化、打补丁、设置好并准备好投入使用。

云计算带走了这一切。供应在几个小时内完成，并立即发布。这种弹性 IT 供应模型可以扩展到无服务器模型中的微服务。这消除了采购、供应、管理、升级或支付服务器基础架构的需求。这些服务现在可以在几分钟内扩展。

## IT 成本高，利用率低

VMware 和 IBM 发布了一项联合研究调查和一份名为“[使用虚拟化提高数据中心效率](https://www.semanticscholar.org/paper/USING-VIRTUALIZATION-TO-IMPROVE-DATA-CENTER-Talaber-Brey/e6e121fbd960ff542c0bcc40cb977ee9b5985512)”的白皮书，根据来自全球数千家企业中运行的 300，000 台生产服务器的信息，发现 20%的服务器运行时容量利用率低于 0.5%，约 75%的服务器运行时利用率不超过 5%。这种利用不足的情况通常会增加 IT 成本，使其超过支持应用程序所需的成本。

## 关于应用程序的业务决策

当企业决定推出新的应用程序、新的业务模式或平台时，他们通常会将决策建立在经济影响分析的基础上，而 IT 只是经济影响分析的一小部分。大多数情况下，“假设”分析基于使用场景和牵引模型，但从不基于 IT 成本分析(固定成本是一个决定性因素)。因此，从技术上讲，当 IT 成本包含在业务规划模型中时，适当的成本应该包括这些未充分利用的事实，而这几乎从来没有做到过。

IT 成本往往要么是固定的，要么没有得到充分体现。

例如，如果一个应用程序预计将带来 100 万美元的收入，而 IT 支出预计为 10 万美元，那么就应该设置上限。如果收入达到 200 万美元，那么应该使用支出上限中的任何因素。当他们的决策基于业务影响时，就很容易证明按存储容量使用计费的合理性。但现在，它根据实际支出进行退款，有时会让业务部门大吃一惊。

如果事情不顺利，就没有获得所购买设备的回报的追索权。

## 首席信息官的噩梦——根据业务需求平衡应用成本

随着基础架构的解放，以及自动化的即时基础架构扩展，很难知道要扩展哪个应用程序以及何时扩展。任何应用程序的扩展都应该基于其背后的业务需求，而不是因为应用程序过载。首席信息官们总是害怕“价值有限的应用程序应该放在规模和成本无限的平台上吗？”在没有完全理解应用程序提供的商业价值和它有价值的时间的情况下，这对于 IT 主管来说是一个很难回答的问题。

例如，福利登记应用程序在登记期间应该几乎没有上限支出限制，而在非登记期间或测试期间几乎没有支出批准。换句话说，在注册期间，应该允许注册应用程序使用无限的资源进行无限扩展——尤其是在最后几天，每个人都试图加入进来——而不是在没有人使用它的非注册期间。

任何技术主管都可以通过展示需求和商业价值来迅速证明额外支出模式的合理性。辅助应用程序或实验性应用程序可以获得非常有限的支出预算，并且可以得到严密的管理，而订单收取应用程序必须保持“始终开启”，以便无上限地扩展以满足任何需求。

然而，无服务器仅解决基础设施问题。它帮助企业根据需要扩展基础架构。首席信息官仍然面临着准确预测每个应用程序的云使用量的问题，如果不知道应用层的可扩展性，这是很难做到的。因此，即使在云环境中，过度调配基础架构的情况也很常见。有状态无服务器可以帮助解决这个问题。

## 基于业务需求而非基础设施成本的 IT 预算

这对首席信息官来说，解决了一个非常重要而又痛苦的问题。在数据中心容量规划模型中，询问运行业务关键型应用程序的具体预算始终是猜测。更糟糕的是，退款更加复杂。当仅以 20%的容量运行时，向阿布(业务部门)收取全部容量的费用可能会过高，并且会被推后，但是只向他们收取已用容量的费用会增加 it 支出的浪费成本。但是，几乎实时地了解准确的使用成本，可以让首席信息官灵活地与公共汽车进行谈判。由总线和业务应用程序所有者来决定是否值得为一个应用程序花费那么多钱。

更有说服力的是基于特定的需求。BUs 现在可以为任何特定的应用程序设置支出限制或上限。这种细粒度的成本计算在过去即使有云计算也几乎是不可能的。巴士和其他行政人员可以很容易地证明基于机会成本的支出成本。如果使用或业务机会尚未成熟，或者如果业务部门决定开展额外的活动，他们可以提前几天或几小时通知首席信息官，调整他们的支出和额外预算审批。这也使得首席信息官和其他首席信息官能够决定在应用程序之间合理分配开支，而几乎不会浪费 IT 成本。

## 企业批准的实时 IT 支出

这种基于业务功能和需求来控制平台支出的想法对企业来说是非常新颖的。我看到这件事越来越受关注。业务应用程序可以根据需要扩展到基于需求设置的最大允许限制。该限制也可以是非常动态的，并且也可以是基于日历的。使用 AI/ML 模型，可以预测应用程序的使用需求，当发现趋势时，应用程序的使用可以得到应用程序业务所有者的预先批准，而不是在几个月后让他们大吃一惊。

尤其是在无服务器的情况下，很容易在几毫秒内按需启动服务。这将允许应用程序仅在需求激增时请求额外的资源，而不是根据之前对预测需求的猜测随机增加容量。
