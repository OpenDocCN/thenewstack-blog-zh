# 可编程基础设施由开发人员驱动

> 原文：<https://thenewstack.io/tns-analysts-show-58-programmable-infrastructure-is-developer-driven/>

新堆栈的煎饼早餐和播客电路上周在阿姆斯特丹 Roest 加入了[软件马戏团](http://softwarecircus.eu/)。New Stack Analysts 主持人 Alex Williams 与一个小组探讨了“可编程基础设施”的会议主题，该小组包括 Apcera 的技术布道者 David Blank-Edelman，以及会议的三位主题发言人:Battery Ventures 的阿德里安·科克罗夫特；思科系统公司首席技术官云服务 Ken Owens 和 Kelsey Hightower，CoreOS 的产品经理和首席律师。

更多剧集，请查看新堆栈的播客部分[。](https://thenewstack.io/podcasts/)

[#58:可编程基础设施是开发者驱动的](https://thenewstack.simplecast.com/episodes/58-programmable-infrastructure-is-developer-driven)

这个播客也可以在 YouTube 上找到。

Alex 向小组成员提问，关于可编程基础设施，他们的首要想法是什么。

Adrian 建议“开发者驱动的基础设施”是看待这一现象的另一种方式。这将对话推进到开发人员的当前条件和考虑事项，默认情况下，开发人员已经拥有了开发过程的不断扩展的广度，以及它的安全挑战和它在市场中的关键作用。

对速度的需求、开发管道日益增长的复杂性以及对安全性日益增加的关注，意味着人们对平台的看法出现了新的现实。

> “开发人员现在关心的是他们做事的速度、效率和安全性，”Adrian 说。“他们需要能够向他们表达这些想法的平台。”

“他们需要帮助。他们需要系统。”他说，这些系统应该是 API 驱动的平台。

David 提到，Apcera 在其混合云操作系统中的策略优先解决方案提供了充当安全“护栏”的规则集——开发人员可以随心所欲地工作，因为他们是在与网络打交道。

从他多年系统管理员的角度来看，Kelsey 的观点是基础设施“一直是可编程的，但是接口很糟糕。”他引用 AWS、Apcera、CoreOS 和 Docker 的例子说，标准库是最近改进的库。

“对开发者来说，事情一年比一年复杂，”Ken 说。“以编程方式定义构建应用程序的环境非常困难，因为您必须了解网络部分、存储部分，并且必须在一定程度上了解虚拟化环境。”

在肯定他是 API 的忠实粉丝的同时，Ken 说现在开发人员需要“快速访问服务”

“你需要网络服务:你需要存储服务。为什么要定义一堆需要开发人员去搞清楚的 API 呢？”肯说。“让我们只写代码，把代码放在那里，靠它赚钱。如果网络断了，打电话给我，我们会修好的——这就是我们要做的。”

Ken 对 API 的立场为接下来的对话奠定了基础，David 和 Kelsey 提供了反驳。Adrian 指出了两个问题之间的区别，一个是 API 的长期稳定性，另一个是它们快速增长的数量。Ken 重申，开发人员只需要一个服务层来帮助他们避免陷入 API 管理的困境。

在观察了关于 API 的这种反复之后，Alex 认为这只是这些体系结构的巨大复杂性的一部分，讨论转向了将所有东西放在一个帐篷中。

Apcera、Cisco、CoreOS 和 Docker 是新堆栈的赞助商。
