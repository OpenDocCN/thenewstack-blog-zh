# 采用 Jenkins X 的云原生 DevOps

> 原文：<https://thenewstack.io/cloud-native-devops-with-jenkins-x/>

[云土著德沃普斯与詹金斯](https://thenewstack.simplecast.com/episodes/cloud-native-devops-with-jenkinsx)

CloudBees 赞助了这个播客。

在向云原生开发运维的转变过程中简化 CI/CD 非常困难，而且这条路可能充满危险。但是潜在的回报是巨大的。

自动化是 DevOps 的重要组成部分，Jenkins 在帮助简化 CI/CD 流程方面处于领先地位。但是 Jenkins 本身，虽然对于许多组织的开发过程仍然是必不可少的，但是众所周知很难掌握和管理，特别是当项目被扩展时。这实际上为那些可以作为内部詹金斯专家的人提供了机会，但这样的专业在许多方面违背了 DevOps 的精神。

随着我们进入云原生世界，将 Jenkins 扩展到新平台，以及改进 CI/CD 流程，这些流程在过去可能对整体部署相当有效，这又增加了另一项技术挑战。

CloudBees 创造了“Jenkinsteins”一词，用来比喻当组织试图解决 Jenkins 的笨拙时所面临的斗争的最终结果。

“詹金斯 X 工具”旨在让这种隐喻成为历史。它开发于一年前，有助于标准化如何将 CI 和 CD 与事实上的云原生平台 Kubernetes 集成。这个想法还允许组织使用他们自己的源代码，同时 [Jenkins X](https://jenkins.io/projects/jenkins-x/) 在移植到云原生环境时自动化这个过程。

CloudBees 的高级架构师、 [Jenkins X，](https://github.com/jenkins-x)的项目负责人 James Strachan 说:“安装和管理 Jenkins，配置 Jenkins，弄清楚如何创建管道，使用哪些插件，甚至如何进行 CI 和 CD，都需要相当大的努力。“我们想用 Jenkins X 做的是反其道而行之，让 CI 和 CD 几乎成为一种你可以消费的设备。”

作为“云原生 DevOps 的新堆栈指南”电子书和播客系列的一部分，CloudBees 的一些关键团队成员讨论了 Jenkins X、CI/CD 和 DevOps 如何完美地结合在一起。CloudBees 的联合创始人兼工程经理[迈克尔·尼尔](https://www.linkedin.com/in/michaelneale?originalSubdomain=au)和 CloudBees 的首席软件工程师[詹姆斯·罗林斯](https://uk.linkedin.com/in/james-rawlings-14b9311)与斯特拉坎一起分享了他们的观点。

在许多方面，詹金斯 X 正在成为詹金斯的管弦乐队，斯特拉坎说。Strachan 说:“我们希望用 Jenkins X 将 CI、CD 和 Kubernetes 标准化，这样你就可以带着你的源代码，而 Jenkins X 会自动完成剩下的所有工作。”
