# “安全”对开源软件意味着什么

> 原文：<https://thenewstack.io/what-security-means-for-open-source-software/>

这个故事是最新的

[in a series](https://thenewstack.io/securing-the-software-supply-chain-with-a-software-bill-of-materials/)

我们将在文章中探讨安全在开源软件供应链中的作用。请在本月随时查看更新。

任何现代企业都不可能拥有完全免费的开源软件。开源是帮助公司快速创新和使用软件创造竞争优势的重要部分。但是，与自定义应用程序或来自供应商的软件相比，它确实带来了一些独特的安全挑战。

开源软件的广泛使用，加上大多数软件供应链的复杂性，增加了开源安全和许多依赖开源的公司的风险。安全公司 [Snyk](https://snyk.io/) 的创始人 [Guy Podjarny](https://www.linkedin.com/in/guypo/?originalSubdomain=uk) 说:“突然之间，开源组件中的安全错误产生了巨大的影响。

开源安全与闭源企业软件或定制应用程序有何不同？我们与几位安全专家进行了交谈，以了解开源带来的独特安全挑战，以及组织如何减轻这些风险。

## 谁上钩了？

开源软件面临的最基本的挑战是，没有人明确负责帮助企业解决安全问题，解决安全问题的过程也不同。容器安全公司 stack rox stack rox 首席产品官[希拉里·本森](https://www.linkedin.com/in/hillary-benson/)说:“如果你发现一个安全问题，你不会去找供应商并要求他们修复，你会在开源存储库中提交一个拉请求，或者你会希望其中一个维护者查看回购的安全性，并报告和解决安全问题。这是一个非常不同的责任模型，意味着企业需要在内部承担更多的管理软件安全的责任，而不是由供应商为他们提供和维护软件。

Podjarny 解释说:“维护者免费做这件事。“他们真的不欠你什么，但这确实带来了所有权挑战。”

在大多数企业中，开源足迹的规模和复杂性使这一切变得复杂。软件越来越多地被组装而不是从零开始构建，任何开源项目都可能包括许多其他开源组件。“一个典型的企业中使用了成千上万的开源组件，”Podjarny 解释道。“所有这些都需要跟踪漏洞。”

组织甚至可能不知道在复杂的依赖关系网络中存在哪些开源组件。“你真的需要有工具来说，好吧，我包括了这个库，然后看着这个库说，‘这个库包括什么库？’”本森说。

现实情况是，通常没有人明确跟踪许多开源项目的漏洞。开源项目有很大的变化，从做一件非常特殊的事情的简单项目，有两个日常工作不相关的维护人员，到像 Linux 或 Kubernetes 这样有强大社区和支持的项目。鉴于开源软件的使用范围，大多数企业都有软件组件，从超小型到大型用户社区。

当一个供应商的软件有漏洞时，它总是被报告给[常见漏洞和暴露](https://thenewstack.io/cvss-struggles-to-remain-viable-in-the-era-of-cloud-native-computing/) (CVEs)的数据库。“大多数时候，当你在开源软件上遇到问题时，他们最终并没有获得 CVE 奖，”本森解释道。因此，基于 CVE 数据库扫描已知问题的工具看不到这些漏洞。

当然，有一些公司“拥有”开源安全的例子——也许最著名的例子是 [Canonical](https://canonical.com/) 和 [Red Hat](https://www.redhat.com) ，这两家公司都通过为开源组件提供安全和补丁支持而拥有一些所有权。

## 黑客的高额回报

如果你是一名黑客，攻击开源软件也很有吸引力。云原生安全公司 [Aqua](https://www.aquasec.com/) 的战略和产品营销副总裁 [Rani Osnat](https://www.linkedin.com/in/raniosnat/) 解释说:“如果我是一名攻击者，并且我知道成千上万的组织使用 MySQL，那么根据统计数据，编写一个试图基于已知漏洞侵入 MySQL 的漏洞利用程序更有可能成功，因为它在任何地方都被使用。

这些漏洞通常是众所周知的，这也很有帮助——即使它们不在 CVE 的数据库中，它们也可能在 GitHub 中被报告为 bug。“很少有坏人真的努力去寻找软件中的漏洞，”奥斯纳特说。开源也更容易检查和使用，因此开发它所需的复杂程度更低。

开源也是…开放的。因为任何人都可以做出贡献，攻击者可以通过故意在代码中插入漏洞来毒害软件供应链，目的是在将来利用这些漏洞。这是一个威胁向量，虽然在闭源软件项目中并非完全不存在，但对于内部开发的自定义代码或来自供应商的闭源软件来说，这基本上不太可能是一个问题。

## 开源安全的成功

这是否意味着开源软件本质上不如定制软件或闭源软件安全？绝对不行。

Osnat 说:“运行良好的开源项目比你在其他地方得到的软件质量更好。"高质量的软件缺陷少，缺陷少意味着漏洞少."

Osnat 补充说，就安全性而言，开源项目的主要好处是，通常有这么多人在关注和审查它。

尽管如此，使用开源软件的企业必须承担大量的安全责任。这必须包括超出扫描已知 CVE 范围的工具。“这是关于知道你在使用什么，然后知道其中存在哪些漏洞，”Podjarny 说。嵌套依赖的复杂性也意味着工具是绝对必要的。

“这不是你可以手动完成的事情，”本森说。“我不认为这是可能的。”

开源当然带来了独特的安全挑战，但安全专家显然认为，与开源提供的优势相比，这些挑战是可管理的，也是值得的。

“开源为组织提供了巨大的价值，是我们今天看到的创新步伐的核心驱动力，”Podjarny 说。“我们必须掌控它，不让它失控。”
