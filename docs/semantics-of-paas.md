# 平台、容器和 PaaS 的语义

> 原文：<https://thenewstack.io/semantics-of-paas/>

PaaS、CaaS、容器管理、云平台，您如何称呼您正在使用的技术？在本期[新堆栈分析师](https://thenewstack.io/podcasts/analysts)播客中， [IDC](http://www.idc.com/) 平台即服务研究经理兼首席分析师 [Larry Carvalho](https://twitter.com/robustcloud) 解释了为什么术语 PaaS 实际上已经过时。

Larry 没有谈论平台即服务，而是使用了术语“云应用平台”这个云平台提供计算能力、开发者服务以及后端服务。在其核心，它必须提供一个传统上作为虚拟机提供的计算引擎，但计算越来越多地通过容器甚至功能来完成。换句话说，无论应用程序如何打包，它仍然可以在平台上运行。

[#134:平台、容器和 PaaS 的语义](https://thenewstack.simplecast.com/episodes/134-platforms-containers-and-the-semantics-of-paas)

所有云平台的共同主题是用户正在寻求自动化。拉里描述了三种类型的平台:

1.  低代码或无代码(如[外系统](http://outsystems.com)、[Force.com](https://www.salesforce.com/products/platform/products/force/)、[门迪克斯](https://www.mendix.com/))。)
2.  以部署为中心。这些可以采用捆绑或固执己见的方法，如(例如 CloudFoundry 或 OpenShift 等)，也可以采用更自由的方法，对基础架构提供更多控制(例如 AWS ECS、自行部署的 Kubernetes 等)。)拉里表示，大公司的初创企业和创新实验室倾向于采取自由放养的方式，而大型企业则倾向于固执己见的平台。
3.  事件驱动的功能即服务(无服务器)—除了承认它不太适合模型之外，没有详细讨论这个主题。

当我们[与 Chip Childers](https://thenewstack.io/pies-platforms-paas-vs-caas/)谈论容器即服务(CaaS)时，他将 CaaS 与 PaaS 区分开来，说 CaaS 往往是非结构化的，并且大多只是提供运行容器所需的基础设施。Childers 认为 PaaS 实际上有助于容器的构建和部署。虽然这种区别很有价值，但 Larry 提供了一个更简单的解释，他说，“CaaS 只是打包应用程序的另一种方式，更快。”

对话的后半部分(链接至 26:00 分钟标记)涵盖了首席信息官对提升转变、云迁移和应用程序组合审查流程的看法。拉里说，有两种首席信息官:1)依赖顾问的首席信息官，2)将咨询方法引入内部的首席信息官。第二种类型的首席信息官越来越普遍。有时，他们会依靠内部创新实验室来发展广泛的技能。

基础架构的抽象推动了对全栈知识的需求。首席信息官们正在寻找拥有许多不同技术知识的人来评估技术决策，并帮助引入新的平台功能。在评估平台时，他们希望吸收 CI/CD 和测试等开发人员服务，以及引入机器学习和物联网相关服务等新功能的能力。为了帮助首席信息官，厂商越来越多地在售前过程中引入解决方案架构师。这些供应商方工程师与企业合作，解决围绕战略、治理和架构的问题。

Alex 在节目结束时说，PaaS 过去只是一个梦想，但现在有了容器的第二次机会。
