# 如何在不造成伤害的情况下规避技术债务和实现现代化

> 原文：<https://thenewstack.io/how-to-get-around-technical-debt-and-modernize-without-causing-harm/>

尽管名称有些误导，但技术债务不是货币债务，尽管它会产生财务影响。实际上，[技术债务](https://thenewstack.io/security-engineers-problem/)指的是一个快速或不全面的软件开发和更新的副产品。

令人惊讶的是，技术债务几乎是 IT 运营的一个自然组成部分。据估计，如今处理技术债务[占据了大约 25%的开发时间](https://www.sciencedirect.com/science/article/pii/S0167642318301035)。这没什么可怕的；然而，它确实需要计划。

尽管在将新工具推向市场时可能会出现技术债务，但在遗留系统更新期间会变得更具挑战性。

在这种情况下，由于遗留软件带来遗留问题的事实，问题变得更加严重。此外，试图“掩盖”困难以创建可行的解决方案通常只会增加原始负载之上的技术债务。

对于您的组织来说，这意味着软件可能不会像它应该的那样有效地运行，错误可能会更频繁地发生。久而久之，这将变得更加明显，尤其是如果负载和使用增加。

为此，区分技术债务的类型是有帮助的。

## 明知故犯的债务

在这种情况下，工程师故意造成技术债务，以便更快地解决问题或更快地将产品推向市场。他们知道自己走的捷径，但认为这些是实现目标和期望所必需的。越来越受欢迎的敏捷过程的一个可能结果是，这种类型的技术债务可以使用以下步骤来管理:

1.  创造意识:确保整个团队意识到更快的上市时间可能会带来不必要的后果。
2.  开发一个跟踪过程:确保所有重要的变更都被记录下来，因为它们的发生使得追踪 bug 变得更加容易。
3.  **留出解决债务的时间:**这意味着回到已经发生的问题和解决这些问题的工程解决方案。

## 未能适应未来

未能考虑到未来的发展或试图预测每一个可能发生的事情会导致软件出现缺陷和迟缓，达不到其目标。以下步骤可能有助于首先减少问题:

1.  允许系统内部的更新:不要把你的软件架构看作是最终版本，而是一个开放的、可以改变和改进的版本。

2.  **不要被诱惑去设计每一个可能发生的事情**:想想网飞原创剧集《[黑镜:banders natch](https://www.netflix.com/title/80988062)】中的警世故事，剧中主角试图将几乎无限数量的选择编入视频游戏。虽然选项是可能的，但试图考虑每一种可能性会给你留下不可行和低效的软件。

## 现代化过程中管理技术债务的技巧

对于计划更新软件的团队来说，为了避免在更新时造成伤害，这里有一些重要的提示。这些建议源自处理[定制企业软件](https://www.iflexion.com/services/enterprise-software-development)的最佳实践，旨在减少现代化项目中的技术债务。

1.  **从一开始就进行有效的规划**:在开始开发之前，坐下来首先规划项目路线图是至关重要的。不要被炒作所左右；如果你只受传言的引导，去追求市场上最新的技术可能会失败。虽然在某些情况下，在你的项目堆栈中包含新兴技术可能有经济意义，但情况并非总是如此。在决定之前，根据你的目标评估选择的工具和行动。

2.  **让利益相关者参与规划**:虽然你的工程师可能会也可能不会很好地掌握如何管理企业，但有一点他们比任何人都清楚——实现既定目标所需的时间和精力。让所有相关部门的主要利益相关者参与规划过程，作为一个团队工作以确保更准确的估计，并防止在不切实际的截止日期之前匆忙将其推向市场。

3.  **保护您的系统安全**。随着系统的更新，可能会出现更多的薄弱环节，这对您的业务构成风险。

在开始现代化之前，不仅要考虑预期的最终结果，还要考虑网络安全措施，以确保您和您的用户尽可能地安全。当处理基于云的软件和在以前不需要的地方实现新的安全级别时，这是最重要的。
