# GraphQL 的李拜伦从零开始重建一个项目

> 原文：<https://thenewstack.io/graphqls-lee-byron-on-rebuilding-a-project-from-scratch/>

该播客由 [InfluxData](https://www.influxdata.com/) 和 [KubeCon+CloudNativeCon](https://www.cncf.io/kubecon-cloudnativecon-events/) 赞助。InfluxData 提供了一个完整的开源平台，专门为指标、事件和其他基于时间的数据而构建，是一个现代的时间序列平台。KubeCon + CloudNativeCon 将于 5 月 20 日至 23 日在巴塞罗那举行，将邀请 Kubernetes、Prometheus、gRPC、Envoy、OpenTracing 等热门项目背后的主要维护者以及其他领域专家、采纳者和最终用户参加，以推动云原生计算的教育和发展。

[graph QL 简史与李·拜伦](https://thenewstack.simplecast.com/episodes/a-brief-history-of-graphql-with-lee-byron)

在这一集的 New Stack Makers 播客中，我们将与 [Lee Byron](https://github.com/leebyron) 对话，他是 [GraphQL](https://graphql.org/) API 查询语言的创造者之一。他曾在脸书开发 GraphQL，现在是投资应用 Robinhood 的网络工程团队经理。他也是 GraphQL 基金会的创始人，该基金会是 GraphQL 社区支持 GraphQL 和周围生态系统扩展的中立场所

脸书在 2012 年委托 GraphQL 在移动设备上更有效地提供脸书。脸书最初是基于客户端-服务器架构设计的，因此他们显然需要一个本地移动数据平台。

到 2015 年，React 开源社区获得了关注，并召开了第一次会议。Byron 说，开源社区使用的扩大导致了从头开始重建 GraphQL 的必要性。

重建您投入生产的系统的想法听起来像是开发人员中很少有人能负担得起的奢侈品，Byron 说这伴随着很多风险。他观察过其他项目尝试这种方法并失败，并从他们的错误中吸取教训。他说，第一次，开发商非常愿意考虑所有的选择。第二次，你开始认为你已经知道什么是有效的，并希望解决自第一版发布以来你想到的每一个可能的问题。但是当重新设想一个更好的系统时，这些可能是绊脚石。

GraphQL 团队为重新设计建立了结构，以帮助解决这些问题。他与另外两位开发 GraphQL 和 React 的开发人员建立了作者-编辑关系。对于每一部分，他都写了一份提案，必须通过编辑，他们会问一些有用的问题，比如“这是在解决一个实际问题吗？”很快这个被缩短为“YAGNI ”,代表你不会需要它。

他们设定了最后期限。他们为下一次 React 会议提交了一份演讲提案，因此他们知道他们必须有一些东西向观众展示。Byron 说，最后，也是最关键的一点，他们创造了一条增量路径，展现在开发人员面前。他说，他们需要考虑如何从 A 点到达 B 点。一旦他们看到了这条路，他们可以看到自己在使用它，它就以一种大的方式起飞了。
