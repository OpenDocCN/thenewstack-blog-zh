# SUSE 对 Rancher Labs 的收购:整合是否扼杀了云原生承诺？

> 原文：<https://thenewstack.io/suses-rancher-labs-acquisition-is-consolidation-killing-the-cloud-native-promise/>

SUSE 以 6 亿美元收购 Rancher Labs 的交易是个大新闻。这也是另一家企业软件供应商 T2 收购 T4 独立的 Kubernetes 供应商 T5。这些频繁的收购清楚地表明，这个行业非常活跃，Kubernetes 是一项即使是大公司也在押注的技术，我们可以预计未来几年市场将充满活力。

但是这对用户来说意味着什么呢？毕竟，云原生技术的关键价值主张之一是其模块化、互操作性和整体灵活性。

## **范式转换:长(短？)实时模块化、灵活的架构**

在整个企业技术领域，我们见证了一场范式转变，从:

1.  一体化企业解决方案将公司(通常由 IBM、Microsoft 或 Oracle 等公司提供)锁定在
2.  模块化、灵活的 IT 系统，由多个可互操作的云原生模块或平台组成，这些模块或平台以技术不可知的方式将它们捆绑在一起(例如 Rancher 或 [Kublr](https://kublr.com/) )。

虽然云原生选项增加了实施的复杂性，但它也提供了前所未有的灵活性。如果独立供应商(这是一个很大的假设)兑现了云原生承诺，你可能会获得与独立供应商*相同或相似的灵活性。灵活性是关键，因为它实际上是您明天和以后每天都需要的唯一业务需求。所有其他要求都难以预见——形势变化太快。*

这些整合意味着采用更不可知方法的平台正在被合并到更传统的业务模式中，从而抵消了云原生的优势。尽管他们通常承诺不这么做，但历史告诉我们，他们最终会这么做。否则他们为什么要花这么多钱收购那些初创公司。他们的投资者期望投资回报率，这通常意味着利用它来销售更多自己的产品。

虽然 SUSE 的目标是成为一家真正的开源公司，但它面临着快速增长的压力，毕竟它被 EQT 收购已经快两年了，他肯定期待未来几年的投资回报。

至于这是如何发生的，Rancher 用户可能会像 Red Hat 一样被绑定到某个操作系统上。毕竟 SUSE 的 CaaS 和 App 平台只运行 SUSE 的产品。那么 SUSE 能够兑现他们的云原生承诺吗？我们必须等等看。

## **尽管合并，仍保持供应商独立性**

对于现在面临这种可能的现实，但希望保持供应商独立性的企业来说，有什么选择呢？

首先，要确保你选择的 Kubernetes 发行版不会束缚你。市场上有多种发行版，其中一些会将您绑定到一个技术堆栈或基础设施上，这使得切换非常困难(做您的研究)。

虽然不能保证某个供应商不会被收购，甚至从市场上消失，但要确保平台本身不会把你束缚住。如果真正开放，你将能够迁移到一个不同的平台，如果这两种情况发生。

## **四个主要的 Kubernetes Distro 锁定警告标志**

当你选择一个不会锁定你的发行版时，这里有四个警告信号要记住:

1.  **分叉的 Kubernetes:** 在这种情况下，平台包括与上游 Kubernetes 或其他供应商不兼容的专有特性。如果使用，应用程序会被锁定。
2.  **供应商提供的 Kubernetes 集群依赖于供应商平台:**如果没有特定于供应商的附加组件或有效许可证，Kubernetes 集群将无法“生存”和/或使用。
3.  **在目标环境**或目标操作系统(例如，仅支持一个来自云提供商的托管 Kubernetes 云)方面没有可移植性(例如，仅支持来自供应商的特殊操作系统)
4.  **在不同的环境中缺乏集群部署的一致性:**在这种情况下，供应商利用由其他工具提供的托管集群。

总之，虽然你不能控制市场动态，但你可以追求一种策略，让你保持灵活性，围绕自己的市场需求，而不会被挟持。云原生技术为最终摆脱供应商束缚提供了巨大的机会。获得的灵活性太重要了，不能让它溜走。这就是为什么在 [Kublr，我们建立了一个完全可定制的平台，](https://kublr.com/)使用普通的上游 Kubernetes 组件，并与任何 Kubernetes 兼容的工具包一起工作。无论你选择 Kublr 还是任何其他 Kubernetes 平台，确保它不会把你锁在里面。
