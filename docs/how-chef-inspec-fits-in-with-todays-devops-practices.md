# Chef Inspec 如何适应今天的 DevOps 实践

> 原文：<https://thenewstack.io/how-chef-inspec-fits-in-with-todays-devops-practices/>

[Inspec:谁在采用它，为什么，它如何适应今天的 DevOps 实践](https://thenewstack.simplecast.com/episodes/inspec-whos-adopting-it-why-and-how-it-fits-with-todays-devops-practices)

在今年 5 月举行的 ChefConf 上，TNS 创始人亚历克斯·威廉姆斯(Alex Williams)与 Chef 的高级产品经理[张秀坤·里希特](https://www.linkedin.com/in/dominikrichter)和咨询公司 [10th Magnitude](https://www.10thmagnitude.com/) 的云自动化工程师[安妮·黑德佩斯](http://www.anniehedgie.com/)坐在一起，了解 Chef [InSpec](https://www.inspec.io/) 的扩展使用，这是一种合规性和安全测试工具。在今天的 [The New Stack Makers](https://thenewstack.io/podcasts/makers) 播客中，三人组深入讨论了 InSpec 如何填补当今 DevOps 工作流程中的空白，以及该项目的未来目标。

随着 2 月份 InSpec 2 的发布，增加了对云环境(如 Microsoft Azure 和 AWS)中测试配置的支持，以及速度和工具的改进。开发人员现在可以使用 [Chef Automate](https://www.chef.io/products/automate/) 作为合规性配置文件的来源，为合规性和安全性审计存储 InSpec 报告。InSpec 网站还指出，这些结果可以以 JUnit 格式导出，以便集成到 Jenkins 等 CI/CD 工具中。

“除非你接受检查，否则你没有把握。直到你全部测试过，你才真正知道。你也可以不断地测试它。Hedgpeth 说:“如果你想要的状态是代码，代码的合规性，并且你不断地测试它，你知道它总是停留在那个想要的状态，它是安全的。

Richter 继续指出，随着今天的技术堆栈继续发展，InSpec 旨在通过在一个人的基础设施中使用不同的工具来实现空间的清晰性，并解释说，“它旨在简单，旨在描述那里有什么。”

随着 InSpec 在安全领域的发展，Richter 继续指出，社区对 InSpec 的贡献和使用非常有趣。“社区将采用这样的东西，并以意想不到的方式使用它。我们有一些用户从纯粹的运营角度出发，因为他们希望了解他们的基础架构的运行情况，以及他们在某些棕地环境中运行了什么，而他们多年来对此没有任何见解。”
