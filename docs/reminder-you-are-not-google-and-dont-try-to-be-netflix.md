# 提醒:你不是谷歌，也不要试图成为网飞

> 原文：<https://thenewstack.io/reminder-you-are-not-google-and-dont-try-to-be-netflix/>

你不是谷歌或脸书。见鬼，你可能也不是网飞或埃西。很可能，让开发团队尝试复制这些环境的想法阻碍了进展。

DevOps 运动通常是由那些做着惊人事情的大型软件公司推动的。“普通人的”DevOps 的两个海报孩子变成了网飞和 Etsy。但它们一点也不常见。当我看到关于这个主题的话题和聊天时，它被“持续交付”和“持续部署过程”高度主导——哦，还有“Docker”他们抢尽风头，给新的采纳者一种感觉，如果你不做 CD，你就不做 DevOps。但是所有这三个对于典型的组织来说都是不实际的。

如果让我来猜的话，假设 90%的公司都在进行持续集成。减去那些只为业务线应用程序开发的人。那么交付占百分之九，部署占百分之一。原因和问题都在于 CDs 意味着特定类型的应用。具有高交易量和/或地理上分散的用户群的公司。

但是，即使你不符合这些类别，这并不意味着 DevOps 不适合你。因为 DevOps 毕竟不是个东西。这是一个框架。它永远不会结束。甚至谷歌也不能声称他们创造了终极开发环境。如果他们有，他们将停止创造新的语言和框架。

所以，深呼吸，为自己争取一些成功，不要试图成为脸书或谷歌。你应该从领导者那里得到的是证明这是可能的。但是你自己需要关注的是:

1.  自动化一些东西。
2.  释放更多。
3.  在管道中提前引入质量。
4.  打破沟通障碍。
5.  简单的做更好的软件。

所有这些自然导致了 DevOps 环境。

不幸的是，当我推销 DevOps 时，我经常听到人们说“我们不是脸书”所以他们得到了消息，但他们的解释是错误的。这个想法是 DevOps 以人为本；推动流程的人；由工具执行的过程。只要你没有退步，那么你就在迈出下一步。

供应商没有帮忙。供应商们认为，客户做 CD 的一个实例就意味着整个世界都在做 CD，这是有罪的。因此他们提前进行了营销。出血边缘。当他们的理想客户的复制品没有突然出现时，他们总是感到惊讶。供应商有忽视大众的倾向。他们拒绝那些认为他们太超前的潜在客户。但是实际上，软件发布的自动化对所有公司都是有价值的，自动化功能测试也是如此。

在我看来，自动化测试和持续集成都是任何公司都可以接受的起点。它们利用所有与 CD 相同的工具和类似的方法，因此是一种融入现有环境以及未来的方式。问题是，这些新流程意味着组织变革。这将是艰难的，也是必须的。要么你做得快，要么在很长一段时间内痛苦地慢下来。

所以，基本上，市场是自己更大的敌人。早期的大多数人有一种管道自卑情结，供应商们不断地将消息传递推到太早的边缘，但在技术上是准时的。

Docker 是新堆栈的赞助商。
