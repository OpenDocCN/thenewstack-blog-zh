# 一个开源基金会可以修复持续的输送管道吗？

> 原文：<https://thenewstack.io/can-an-open-source-foundation-fix-continuous-delivery-pipelines/>

[一个开源基金会可以修复持续输送管道吗？](https://thenewstack.simplecast.com/episodes/can-an-open-source-foundation-fix-continuous-delivery-pipelines)

在过去的几年里，云原生开发工具和平台选项的选择呈指数级增长。但是，如此广泛的选择带来了不确定性，可悲的是，在许多情况下，当决定哪个选项最适合软件生产管道时，会感到困惑。

[Chris Aniszczyk](https://www.linkedin.com/in/caniszczyk/) ，云原生计算基金会(CNCF)的首席技术官和 [Linux 基金会](https://www.linuxfoundation.org/)、[的开发人员关系副总裁，最近描述了](https://thenewstack.io/continuous-delivery-foundation-launches-with-jenkins-jenkins-x-spinnaker-tekton/)云原生领域的现状，其中“至少有 20 到 30 种工具，而且还在不断增长，包括初创公司和云提供商，”这是“供应商和用户为这个领域带来理智的大好时机。”

这种急需的“理智”很大程度上涉及到连续交付(CD)的公共标准的可用性。为此，创建了[持续交付基金会(CDF)](https://cd.foundation/) 来促进使用 [Jenkins](https://jenkins.io/) 、 [Jenkins X](https://jenkins-x.io/) 、 [Spinnaker](https://www.spinnaker.io/) 和 [Tekton](https://github.com/tektoncd) 进行部署的协作。

在[New Stack @ Scale](https://thenewstack.io/podcasts/at-scale)系列的这一期播客中，New Stack 的创始人兼主编 Alex Williams 讨论了 CDF 的使命以及开发社区可以期待什么。

在开源领袖峰会上录制的播客讨论也涵盖了对这样一个基金会可能代表什么的希望和担忧。发言者包括:

当在开源领袖峰会上宣布 CDF 时，Linux 基金会的执行董事 Jim Zemlin 描述了“快速创建代码、测试代码和持续交付的能力对于现代开发速度来说是非常重要的。

“我的意思是，持续交付的概念在开发人员创造非常非常好的代码的速度方面创造了巨大的回报。为此，Linux 基金会非常兴奋地宣布我们对持续交付基金会的支持。“这是一次合作，为 Jenkins 创建一个中性的家… Jenkins X、Spinnaker 和 Tekton:这些将成为持续交付基金会的基石项目。”

泽姆林还说“我们非常感谢像网飞和谷歌这样为这些项目贡献代码的人。”

Zemlin 说:“这将是一个厂商中立的开源项目之家，用于在发布管道过程中的连续交付和与连续交付相关的规范。”

然而，Lyft 的 Klein 质疑 CD 的另一个基础的潜力。他承认在 CI/CD 工具的丰富性方面普遍存在混乱。例如，CNCF 已经试图解决其中的许多问题。

“在 CNCF，我们已经遇到了这个问题，即帮助最终用户了解哪些技术适合他们的使用案例。我担心的是，通过开始另一个密切相关的基金会，”克莱因说。

“就像我们看到的那样，有大量的重叠——我不认为没有 CI 和 CD 就能构建一个云原生架构，”Klein 说。“所以现在，我们有一些关键的联锁技术，它们跨越了几个不同的基础，本身并没有被打破，但我认为我们需要小心，以避免混淆用户。”
