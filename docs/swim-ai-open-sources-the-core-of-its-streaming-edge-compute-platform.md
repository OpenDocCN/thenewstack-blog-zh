# Swim.ai 开源了其流媒体边缘计算平台的核心

> 原文：<https://thenewstack.io/swim-ai-open-sources-the-core-of-its-streaming-edge-compute-platform/>

Swim.ai 联合创始人兼首席架构师[克里斯·萨克斯](https://www.linkedin.com/in/cdsachs/)表示，边缘计算和云中计算的区别远不在于计算的位置，而在于方法。萨克斯说，这个定义可以归结为你处理数据的方式和时间。

“边缘和云的区别与数据库有很大关系。在我们看来，云应用是以数据库为中心的应用，”萨克斯在接受采访时说。“对我们来说，边缘不是。所谓边缘，我们不是指设备，虽然我们确实运行在物理边缘设备上，我们实际上根本不是指一个地方。我们指的是在数据流动时由数据驱动的应用程序，而不是在数据存储在数据库后，无论是在云中还是在物理边缘。我们不在乎我们在哪里跑步。”

Swim.ai 自称“世界上第一个用于构建有状态、实时、分布式应用的应用平台。”周四，它在 Apache 2.0 许可下以开源形式发布了该平台的核心。该平台为开发人员提供了一个单一的集成堆栈，使他们能够“开发与现有硬件和软件兼容的应用程序，而不依赖于数据库、消息代理、应用服务器或云存储，”根据该公司的声明。

萨克斯说，SWIM 的边缘计算平台与传统的云计算模式形成对比，它在到达时进行实时处理，而不是在存储到数据库后进行处理。他将应用程序分为两大类——以数据库为中心的和数据驱动的，Swim 支持后者。

“云的无状态 REST 范式对于云来说很棒，但对于边缘来说就不那么棒了。当你一切都依赖于数据库时，你只能使用过去的数据。最终的限制是你有多少内存。以数据库为中心的应用程序本质上是非常集中的，”萨克斯说。“数据驱动是指分析正在发生的事情。限制你的不是记忆，而是时间，是你处理的速度。如果你有比你能储存的更多的数据，你最好快速处理它。这些数据驱动的应用非常水平分散。移动每天产生的万亿字节的数据非常困难。”

Swim 通过重新设计软件架构本身解决了这一问题，用其“持续一致的多路复用流协议”WARP 协议取代了 [REST](https://en.wikipedia.org/wiki/Representational_state_transfer) ，据称该协议创建了“世界上唯一的流优先 API 的基础”。Sachs 解释说，WARP 协议使得处理 edge 场景中出现的大量数据成为可能。该公司的 go-to 示例显示了交通灯的实时状态，Sachs 解释说，所涉及的数据使传统大数据场景的数据相形见绌。

“CPU 以纳米级时间运行，而数据库以毫秒级运行。当您处理连续生成的边缘数据流时，数据量大得惊人。它让大数据看起来像是沧海一粟，”萨克斯说。“我们在美国七个城市的交通应用每秒处理 50 万个数据点。Twitter 是每秒 3000 条推文。边缘的规模令人震惊，现有的数据分析方法无法解决这个问题。”

相比之下，Sachs 认为 REST APIs 无法接近实时，而 WARP 只能在网络延迟时运行。

萨克斯在一份声明中说:“Swim 是为了解决从多种异构数据流中构建实时应用程序的挑战而创建的。”“无状态的、基于 REST 的架构很快就被高数据量和存储成本所淹没。通过使用 WARP 协议，Swim 提供了一种高效、有状态的方式来管理流数据，并构建与现实世界持续同步的应用程序。”

有兴趣的开发者可以访问 [developer.swim.ai](http://developer.swim.ai/) 和 [GitHub](https://github.com/swimos/swim) 上。出于好奇，请访问 [ripple.swim.ai](http://ripple.swim.ai/) 观看 swim 的实际操作。
