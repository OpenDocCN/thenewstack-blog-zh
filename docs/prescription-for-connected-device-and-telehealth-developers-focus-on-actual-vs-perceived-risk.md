# 互联设备和远程医疗开发者的处方:关注实际风险与感知风险

> 原文：<https://thenewstack.io/prescription-for-connected-device-and-telehealth-developers-focus-on-actual-vs-perceived-risk/>

[Rezilion](https://www.rezilion.com/) 赞助了这篇文章。

 [塔尔·克莱因

Tal 是行业领先的自主云工作负载保护平台 Rezilion 的 CMO。他在 IT 和信息安全行业拥有 20 多年的经验，与云安全、客户端虚拟化、网络和数据通信领域的领导者和令人兴奋的新兴供应商合作。](https://www.linkedin.com/in/talmklein/) 

今年早些时候，[云安全联盟](https://cloudsecurityalliance.org/)发布了一份非常有趣的[白皮书](https://cloudsecurityalliance.org/artifacts/telehealth-data-in-the-cloud/)，阐述了增加云中远程医疗数据审查的需求。这让我想起了 2017 年在这个网站上发布的一篇文章，关于围绕移动健康应用开发的无数合规和隐私问题。在新冠肺炎时代，医疗应用和设备开发商面临着有效管理和处理敏感健康数据的日益多样化的需求。

尽管 HIPAA 和 GDPR 为以安全为中心的思想提供了有用的合规框架，但远程医疗应用程序开发人员和医疗设备制造商自己必须为他们提供的服务提出保护方案。他们必须确保远程医疗应用程序和连接设备生成、存储和传输的数据保持合规。

由于这些服务将大量患者数据传输到云服务，应用安全工程师必须将端到端的安全性和合规性融入到架构中。例如，HIPAA 安全规则要求互联医疗设备制造商和远程医疗服务提供商维护合理和适当的管理和技术安全措施，以保护患者健康数据。具体而言，提供者必须证明通过其服务创建、存储、处理或传输的所有患者健康数据的机密性、完整性和可用性，并识别和防范对患者健康信息的安全性和完整性的合理预期威胁。

为了确保安全性，HIPAA 要求持续的安全威胁风险分析。该评估包括对云计算的威胁。作为评估的一部分，互联设备制造商和远程医疗应用程序开发商必须解决患者和医疗保健提供商对治理、合规性、保密性、完整性、可用性以及事件响应和管理的担忧。

## **顺应性便秘**

对于开发人员来说，所有这些法规遵从性限制带来的挑战是它们抑制了创新，并且在某些情况下，实际上增加了攻击面。除了不能以非医疗行业同行享受的速度推送代码和更新之外，甚至安全补丁本身也带来了摩擦和负担。医疗保健市场受到严格监管，新的解决方案必须通过严格的审批流程，这将减缓采用速度，并给竞争对手带来优势。

漏洞扫描器使开发团队不堪重负，并且困惑于堆积如山的结果，这些结果不可能一次全部修补。在拥有成熟漏洞管理计划的组织中，现有的优先级实践(如 CVSS)并不能显著减少违规。

## **更卫生，推动更顺畅**

Rezilion 最近发表了一项案例研究，发现[漏洞扫描器识别的互联设备云服务中 67%的已知漏洞从未加载到内存中](https://www.rezilion.com/blog/the-intersection-of-devops-and-security-in-med-tech/)。至关重要的是，首先验证关键应用程序是否安装了最少的必要资源来最大限度地减少其攻击面，然后测量关键应用程序使用的可利用攻击面，以便可以降低不可利用的相关漏洞的优先级，从而显著提高运营效率。

这里有两个要点:

1.  关注实际风险与感知风险可以帮助开发人员更快、更安全地推送代码。
2.  由于互联医疗设备制造商平均每年在漏洞管理活动上花费 140 万美元，因此将这一支出减少 60%以上不仅对利润，而且对组织的敏捷性都具有重要意义。

随着健康遥测在云中的使用越来越多，互联设备和远程医疗服务开发人员必须充分主动地解决数据、隐私和安全问题。通过关注重要的风险，医疗应用提供商甚至可以以 DevOps 的速度进行扩展，同时保持安全可靠的服务。
