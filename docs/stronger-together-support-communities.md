# 一起变强:Cloud Foundry 如何支持其他社区

> 原文：<https://thenewstack.io/stronger-together-support-communities/>

孤独的树不是森林。在一个网络生态系统中，没有其他树木，一棵树的生活会更加艰难——它缺乏其他树木的庇护，更难抵御风暴和干旱。树木通过相互连接的根系网络相互强化。

聚集在森林中的树木还有另一个优势:通过它们的根，树木通过一个被称为“木材万维网”的网络相互帮助和支持，交流威胁，甚至相互传递养分。树木和森林中的其他植物之间还有更深、更广的联系——菌根真菌通过称为菌丝的分支细丝形成的网络，菌丝丰富了土壤，并使维持森林生态系统的互惠关系成为可能。

## 一棵孤独的树对一片森林

开源 Cloud Foundry 应用程序开发平台是在六年前公开发布的，在这一过程中，我们与其他项目建立了联系，采用了其他开源社区成熟的技术。例如，在 Docker 成为一家公司甚至一个项目之前， [Cloud Foundry](https://www.cloudfoundry.org/) 平台使用 Linux 容器将部署的应用程序相互隔离。我们的容器实现不像 Docker 那样是以通用的方式构建的；它不是为解决容器运行时的所有潜在用例而设计的。它是专门为支持 Cloud Foundry 最初打算支持的无状态 web 应用程序而设计的，并且是以一种安全的、多租户的方式实现的。

随着 Docker 的成熟，它分离出了较低级别的容器运行时库， [runC](https://github.com/opencontainers/runc) ，并最终将其捐赠给了[开放容器倡议](https://www.opencontainers.org/)。这是 Cloud Foundry 技术社区审查其架构的信号， [Cloud Foundry 成为了 runC](https://www.cloudfoundry.org/garden-and-runc/) 的早期采用者。在一段时间的并行支持之后，团队最终[宣布【runC 支持已经完成，我们将放弃之前的容器后端。](https://github.com/cloudfoundry/garden-runc-release/releases/tag/v1.0.0)

如果你在技术行业工作，特别是如果你关注所有来来去去的令人兴奋的开源项目，很容易被下一件令人兴奋的事情分心。现实情况是，仅仅因为它是新的东西就接受它很少会有回报，事实上这对于像 Cloud Foundry 这样的项目来说是一个主要的风险。我们的社区关注最终用户的工作效率，高于一切。这意味着我们与其他开源社区合作，首先观察，然后评估，最后参与我们认为对双方都有利的活动。当项目发展的时机成熟时——当技术足够成熟并在生产中得到验证时——我们会考虑是否以及如何在云计算平台中采用一些东西。

Cloud Foundry 社区对其 Linux 容器运行时的方法是我们方法的一个完美例子。容器网络接口也是如此。我们希望改变[容器网络](https://www.cloudfoundry.org/meet-new-container-networking-stack-cloud-foundry/)在 Cloud Foundry 中的工作方式。规范很好，我们既能利用另一个社区的成果，又能找到支持他们的方法。

和任何开源技术社区一样，有许多声音和观点。在健康的社区中发生的是围绕项目技术发展的团体哲学的出现。特别是 Cloud Foundry 社区，他们理解为整个生态系统提供一个可靠的、有主见的平台的责任的严重性。我们寻找合适的时机来采用已经在云铸造社区内和由其他社区孵化和开发的技术。当我们需要一个新的组件时，我们首先寻求采用。如果不存在稳定的、可伸缩的、安全的和适当的功能，我们构建“刚好够”来解决我们的用例。

但是技术不会停滞不前。随着时间的推移，其他项目将会出现或以开始符合我们需求的方式发展。当这种情况发生时，我们会把他们推到允许我们收养他们的那条线上。

不同开源项目之间的关系是双向的。我们已经通过上游贡献帮助了那些我们已经接受的社区。即使是 Cloud Foundry 支持的语言和框架社区(通过我们的 buildpacks)也可以利用 Cloud Foundry 平台提供出色的开发人员体验——不仅出色，而且是最新和安全的，因为我们密切跟踪上游语言和框架版本。我们让开发人员致力于一切事物的现代版本。这为开发人员提供了更好的体验，有助于加强该语言或框架的价值，甚至向用户介绍了不同的编程语言，否则他们可能从未尝试过这些语言。

我们的 [Open Service Broker API](https://openservicebrokerapi.org/) 计划是开源生态系统可以相互提供双向(在这里是多向)支持的另一个例子。一个最初仅用于 Cloud Foundry 平台的 API 已经提升到了多生态系统的地位，为采用多种平台技术的最终用户提供了一种统一的方式来展示和使用应用程序的支持服务。

我们采用成熟、深思熟虑的协作方法，在有意义的时候将技术整合到云铸造平台中，而不是在技术成熟之前引入任何东西。如果成熟后有机会更换核心组件，我们会这样做。这种协作理念使得开源社区作为一个整体能够进一步分摊单个组件的开发成本。

并不是每项技术都被引入平台，但是我们有很多系统的扩展点。开源社区，甚至是单一厂商的开源项目，都有大量的双向机会很好地集成到 Cloud Foundry 平台中。这样做可以让你接触到一个庞大的应用程序开发人员社区，他们可以使用你的技术，加入你的社区，并提高你在世界上的地位。

一棵树单独存在不如一群树联合起来创造一个可持续的生态系统强大。就像森林中的树木一样，开源项目可以相互滋养和支持。

[云铸造基金会](https://www.cloudfoundry.org/)是新堆栈的赞助商。
