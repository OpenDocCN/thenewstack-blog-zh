# 使用最佳工具进行软件交付的重要性

> 原文：<https://thenewstack.io/the-importance-of-using-best-of-breed-tools-for-software-delivery/>

[CloudBees](https://www.cloudbees.com/) 赞助了这篇文章。

 [安德斯·瓦尔格伦

Anders 是 CloudBees 的技术战略副总裁。他有 30 年构建商业软件的丰富经验。](https://www.linkedin.com/in/anderswallgren/) 

当他们完成在企业中实现 DevOps 的过程时，公司与许多最终决定他们项目成功的问题搏斗。他们将做出的最重要的决定之一是如何设置他们的工具链。他们应该采用端到端的“围墙花园”方法吗？还是依靠同类最佳的工具？

虽然任何一方都有自己的观点，但钟摆已经明显向同类最佳的方向移动，这提供了更大的灵活性和更强的根据个人需求定制实施的能力。端到端工具链可以提供更多的一致性和标准化。但是，由于企业面临着创新和快速创新的压力，如果他们的平台方法不起作用，他们无法承担拆除整个工具集并替换它们的费用。为软件交付过程找到一个“一刀切”的方法来很好地满足*的所有*需求总是具有挑战性的。企业需要在团队、技术*和工具*之间进行选择，以安全有效地开启创新。

在 2019 年 7 月的 Gartner 报告《如何为云原生应用构建持续交付渠道》(需要 Gartner 订阅)中，分析师 Traverse Clayton 指出:

“考虑您选择的所有工具都是彼此松散耦合的，这样它们就可以很容易地组合起来与其他工具链一起工作。选择同类最佳的工具，并确保它们与其他同类最佳的工具很好地集成。”

## **围墙花园的利与弊**

DevOps 工具链的一站式方法可以帮助正在努力实现流程标准化并处理大量独特应用的大型组织。它可以让运营工程师更容易地在项目中跳来跳去，而无需了解特定应用程序的复杂性。

如今，围墙花园方法的最佳论据是苹果生态系统。史蒂夫·乔布斯对集成 iPhones、iPads、HomePods、MacBooks 和一系列苹果专有服务(包括支付、音乐和流媒体电视)的愿景呈现出令人羡慕的用户体验。但这很难实现，一体化方法很少最终兑现承诺。它不可避免地涉及到妥协。

任何组装过 DevOps 工具链的人都知道有多少移动部件必须一起工作才能创建高质量的协调流程。工具有助于在漫长而复杂的管道中移动应用程序，执行从版本控制、持续集成、工件管理、测试自动化、环境自动化和配置、发布管理、日志聚合和搜索、指标跟踪、团队通信和报告等功能。

HDMI 标准使得更换电视或流媒体设备变得更加容易。软件工具不是那么容易换入换出的，但是行业在最近几年已经取得了很大的进步，建立了互操作性标准，比如 REST(表述性状态转移)和 JSON (JavaScript 对象符号)。这些新格式为我们提供了一种消费 API 的方式，而这在几年前还是没有的。工具格式的标准化提供了灵活处理资源的能力——添加新工具、切换其他工具、实施微服务、在 Docker 容器中运行工作负载、在 Kubernetes 中部署——并使其在我们的 DevOps 管道中工作。

鉴于软件交付变化如此之快(用于加速和自动化软件交付的工具也是如此),选择专门为手头工作构建的最佳工具非常重要。它们还必须能够轻松集成到您的环境中。这样，当一项新技术或新功能出现时，您就能够将它添加到您的管道流程中，而干扰最小；它将真正为您的软件交付过程增加价值。对于试图采用和发展其 DevOps 环境的组织来说，被单一的、轻量级的工具集所困，只能做好某些事情，而不能让您充分利用新工具或技术，这可能会带来巨大的痛苦。

最佳方法具有以下优势。

## **避免供应商锁定**

为不同的工具集选择多个供应商会使采购过程变得复杂，但是在引入新技术时利用它们是值得的。关键是能够持续采用同类最佳的工具，而不仅仅是预先采用。明天总会有更好的方法，如果你买了一个供应商的股票，你就要对这个供应商的产品负责。采用的成本可能很低，但是如果供应商有一个固定的受众，它就不会面临改进产品的竞争压力。如果一年后的平台看起来和今天一样，那么你就只能使用同样的工具。

## **选择不限制其他工具使用的工具**

使用单一平台解决方案，如果您想要升级到新版本，您通常必须升级大量其他工具和/或平台的自定义集成，以确保平台对您仍然有效。人们就是这样被无法进化的工具链卡住的。就变成了技术债。做出改变变得越来越难，加速软件交付变得越来越难。

## **选择链上每个点的功能级别**

在您的工具链中包含一些具有深度功能的优秀产品和一些稍显平庸的产品是完全可以的。同类最佳并不总是等同于最昂贵或最先进的解决方案。对您来说，最好的解决方案可能是包含更少的附加功能，但更适合特定应用程序的解决方案。你可以控制这个决定。当您购买一个有围墙的花园时，供应商决定哪些功能是“同类最佳”的，而不是让您决定哪些工具是完成工作的最佳选择。

## **避免冗长和破坏性的整合**

每当你用一套工具替换另一套时，你必须确保结果证明它是合理的。这是因为转换的成本不仅仅是安装新工具集的价格，也是干扰员工的价格。例如，如果你从一个团队中拿走两个工具，你将会破坏他们的生产力。然后，如果您用一个平台解决方案替换掉 10 个同类最佳的整套工具，您必须重新培训—和重新定位—所有员工。培训 800 个人更换 10 个工具比培训 50 个人更换两个工具更难。您最好能够小批量地、一个团队一个团队地切换单个工具或者采用一个新工具。根据您的进度，升级您今天需要升级的工具或在明年集成一套新工具。对于单一平台解决方案，要么全有，要么全无。你可能不得不抛弃所有正在使用的工具，一次转换所有 800 名开发人员。这是对软件的重大破坏。

## **它允许您扩展您的 DevOps 实施**

有人可能会说，购买一个“盒子里的 DevOps”让一个不成熟的组织能够快速无缝地启动一个 DevOps 项目。但是这种方法极大地简化了组织应该用 DevOps 做什么。这并不是说要尽快将一堆工具准备就绪。DevOps 是关于理解你的过程和你想要达到的目标；了解你在文化、流程和工具方面的弱点；制定一个计划来创造一个未来的状态，在这个状态下，你的工作效率会更高，更能满足业务需求。一下子做出所有这些决定就像创建一个过时的“瀑布式”交付系统:你做了就完事了。事实是，你永远不会结束。工具集应该不断发展，融合今天、明天和后天的技术。

## **结论**

今天的商业环境变化很快。企业正在采用新技术和新流程来保持同步，并创建系统来交付跨传统、传统和现代应用程序的任务关键型创收应用程序。拥有灵活、实用的 DevOps 工具链对他们的使命至关重要。相互集成并响应未来需求的一流技术为企业提供了满足其不断发展的业务需求的最佳机会。
