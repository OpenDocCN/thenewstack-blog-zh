# DevOps 死了。拥抱平台工程

> 原文：<https://thenewstack.io/devops-is-dead-embrace-platform-engineering/>

开发者不想再做运营了，这对 DevOps 来说是一个不好的信号，至少根据斯科特·凯里的这篇文章和 T2 的这篇 Twitter 帖子来看是这样。

平心而论，这取决于你问的是谁。这个 [Reddit 帖子](https://www.reddit.com/r/devops/comments/wusvc9/devs_dont_want_to_do_ops/)强调了开发者对“你构建它，你运行它”范例的强烈且经常相互矛盾的观点。一些开发人员认为他们接管操作任务是有益的，甚至有时是必要的。还有的根本不想碰运营。尽管如此，其他人还是无动于衷。

Luca Galante 是平台工程社区的主要贡献者，也是 Humanitec 的产品负责人，他在 Twitter 上发起了一个关于这个话题的非正式投票。他收到的回复很好地说明了这种分歧，41.8%的受访者对运营任务表示肯定，42.1%表示否定，16.1%表示无所谓。

这是问题的一部分。当团队中的开发人员对他们应该或可以在多大程度上完成运营任务没有达成一致时，强迫每个人以一刀切的方式完成 DevOps 会产生灾难性的后果。

主要的后果是增加了开发人员的认知负荷。这迫使许多团队重新考虑如何平衡来自开发人员自助服务的自由和通过抽象减轻认知负荷。两者都是必要的:自助服务能力对于快速高效地移动至关重要。但随着现代云原生世界的复杂，没有适当边界的自由造成了太大的压力，变得适得其反。对于许多组织来说，找到正确的平衡是一项艰巨的任务。

顶级组织已经发现，答案就在平台工程中，这种技术越来越受欢迎。第一次平台工程师会议 PlatformCon 吸引了 6000 多名与会者。Gartner 将平台工程添加到其 2022 年软件工程的[宣传周期](https://twitter.com/TheMarkONeill/status/1557531379746881536)中。

但是什么是[平台工程](https://platformengineering.org/blog/what-is-platform-engineering)？按照加兰特的说法，是:

“设计和构建工具链和工作流的学科，在云原生时代为软件工程组织提供自助服务能力。[平台工程师](https://thenewstack.io/vmware-targets-the-platform-engineer/)提供一个集成产品，通常被称为“内部开发者平台”，涵盖应用整个生命周期的运营需求。”

[平台工程使用产品方法](https://thenewstack.io/how-is-platform-engineering-different-from-devops-and-sre/)支持适量的开发人员自助服务，并为单个组织和团队找到合适的抽象级别。成功的平台团队将用户研究、定期反馈和营销最佳实践结合起来，以了解他们的开发者，创建一个解决常见问题的平台，并获得关键利益相关者的内部支持。

他们的平台提供了[黄金路径](https://humanitec.com/blog/what-developer-self-service-shouldnt-look-like)，这是开发人员可以用来完成日常任务的阻力最小的路径。这些黄金路径带有推荐的工具和内置的最佳安全实践，减少了认知负荷，同时保留了开发人员在需要时偏离路径的自由。所有这些都确保了该平台减少了认知负荷，并在开发人员的自助服务和支持需求之间取得了恰当的平衡。

这就是 DevOps 死的原因。平台工程万岁。
