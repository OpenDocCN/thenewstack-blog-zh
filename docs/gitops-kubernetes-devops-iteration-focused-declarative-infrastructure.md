# 面向 Kubernetes 的 GitOps:一个专注于声明性基础设施的 DevOps 迭代

> 原文：<https://thenewstack.io/gitops-kubernetes-devops-iteration-focused-declarative-infrastructure/>

[用于 Kubernetes 的 GitOps:一个专注于声明性基础设施的 DevOps 迭代](https://thenewstack.simplecast.com/episodes/gitops-for-kubernetes-a-devops-iteration-focused-on-declarative-infrastructure)

[GitOps](https://www.weave.works/blog/gitops-operations-by-pull-request) 是 DevOps 的迭代，适用于云原生，特别是 Kubernetes，它非常强调声明式基础设施， [Weaveworks](https://www.weave.works) 首席执行官 Alexis Richardson 在本期[新堆栈制造商](https://thenewstack.io/podcasts/makers)播客中说道。Richardson 表示，Weaveworks 团队通过持续管理 [Weave Cloud](https://www.weave.works/product/cloud/) 对 GitOps 的概念感到兴奋，该公司为容器和微服务部署、监控和管理 SaaS。

Richardson 说，该团队注意到有多少 Weaveworks 系统是在 Git 中捕获的，并开始探索如何开发一个更具声明性的环境来捕获状态，并允许在 Kubernetes 架构上进行持续的实验。他们发现在 Git 中管理的代码状态，当独立于 Kubernetes 管理时，允许更好的跟踪和恢复。Kubernetes 的持续交付方法通常依赖于 Jenkins 等技术。

值得注意的是，Weaveworks 的团队是技术上最精通 Kubernetes 和云原生架构的一些人。他们已经开始依赖 GitOps 来保持他们自己的系统和服务在 Kubernetes 上的 Google Cloud 中运行。在亚马逊网络服务去年秋天开始支持 Kubernetes 之前，Weaveworks 使用 GitOps 来管理 Kubernetes。

Weaveworks 已经建立了自己的 diff 工具，例如 Slack updates，它表达了 Weaveworks 录制播客的主题。即使在声明性环境中，协作也是至关重要的，以允许在微服务环境的多个组件之间查看任意数量的更新。
