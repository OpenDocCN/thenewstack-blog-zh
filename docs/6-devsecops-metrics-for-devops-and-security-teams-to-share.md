如果您在 DevOps 中工作，很容易感觉到安全团队在那里让您的工作变得更加困难。同样，如果您是一名安全工程师，您可能会感觉到 DevOps 与您没有相同的优先级，并且永远不会像您希望的那样认真对待安全性。

幸运的是，不一定非要这样。通过设定和追求共同的目标，您组织的安全和开发团队可以加强彼此的成功，而不是相互矛盾。

这是一本入门书，介绍了安全和开发运维团队可以共同追求的目标类型，以及他们可以使用哪些指标来衡量他们的共享进度。

## **为什么安全部门和开发部门要共享目标和指标**

DevOps 和安全性之间的概念差异很容易理解。这两个团队传统上有着根本不同的目标——devo PS 想要快速、有效的发布，而 security 想要消除所有的漏洞，即使这意味着减慢软件开发生命周期——他们之间几乎没有直接的重叠。

此外，建立目标的方式几乎没有给共同目标感留下空间。每个团队定义自己的优先事项，然后要求其他团队支持它们。

这远非理想，更好的方法是可能的。在一个运行良好的 IT 组织中，开发运维与安全运营应该通过识别和追求互利的目标来相互加强(有些人可能称之为[开发运维](https://blog.paloaltonetworks.com/2020/05/cloud-devsecops/))。这样做可以让每个团队感觉到他们分享了其他团队的成功。它还以共享度量的形式提供了一种通用语言，两个团队都可以用它来度量他们朝着集体目标的进展。

## **开发运维团队和安全团队的共享指标**

您组织的开发运维团队和安全团队共享的最佳目标和指标将因您交付的软件类型、您的应用程序托管方式等而异。但是一般来说，以下目标和指标是一个很好的起点。

### **减少了总开安全票**

减少在给定时间段内打开的安全票证的数量是安全团队的一个明显目标。然而，DevOps 团队也从减少安全罚单中受益。安全问题通常意味着软件交付的延迟，或者(在严重事件的情况下)甚至回滚到早期版本——这对 DevOps 团队持续发布速度的目标是一个巨大的打击。

这两个团队都有助于减少每月或每季度打开的安全票证总数。[集成到 CI/CD 管道中的安全工具](https://blog.paloaltonetworks.com/prisma-cloud/cloud-devops-plugins/)可以帮助安全团队改进他们对漏洞的审查，同时帮助自动化开发人员在开发和测试期间找到并修复安全问题。

### **缩短部署时间**

部署时间是 DevOps 团队传统上致力于最小化的指标。部署每个版本的速度越快，就越接近连续交付。

但是安全性也受益于较短的部署时间，因为这意味着新版本可以更快地纠正安全性问题。安全团队可以通过自动化候选版本的审查流程，并努力[将安全问题转移到左边](https://blog.paloaltonetworks.com/2019/07/4-practical-steps-shift-left-security/)，从而在安全问题通常更容易解决的时候，在管道中更早地发现安全问题，从而帮助最大限度地缩短部署时间。

### **发现生产前漏洞**

说到安全性的左移，在软件投入生产之前被识别的安全漏洞的数量改善了开发操作和安全性的结果。对于 DevOps 团队来说，这意味着降低了部署后安全问题触发回滚或对连续交付周期造成严重破坏的风险。对于安全性而言，这意味着进入生产环境的严重漏洞更少，而这些漏洞可能会造成最大的破坏。

通过合作识别预生产代码中的错误，开发人员和安全人员可以互相支持对方的成功。

### **缩短修复时间**

补救安全问题需要安全团队和开发运维团队之间的协作。安全团队带头找出问题所在，DevOps 团队负责实施修复。

由于该指标固有的共同责任，共同跟踪(并寻求最小化)修复时间是开发运维团队和安全团队共同的有效目标。

### **减少失败的安全测试**

当一个版本由于未能通过安全测试而被拒绝时，不仅安全工程师不高兴地发现 DevOps 试图推出一个包含漏洞的版本，而且 DevOps 团队也被迫重写代码并面临交付过程的延迟。如果 DevOps 觉得安全测试不必要的严格或者关注了错误的项目，两个团队之间的关系也会变得紧张。

然而，通过让两个团队设定一个减少失败的安全测试的共同目标，他们获得了对这个指标的集体所有权感。反过来，他们更有可能一起解决问题，而不是浪费精力互相指责。

### **通过安全审计的百分比**

对于 DevOps 团队来说，将安全审计视为他们必须应付的事情可能很有诱惑力，但最终基本上可以忽略。如果安全审计在 DevOps 过程中发现缺陷，他们可能会面临一些批评，但是当审计失败时，锤子主要落在安全团队身上。

然而，现实是，失败的安全审计将两个团队都置于风险之中，不管失败的责任在哪里。反复出现的安全审计失败损害了整个 it 组织的声誉，并最终可能引发对两个团队的彻底检查。

另一方面，安全审计成功的稳定记录对安全工程师和 DevOps 工程师都有积极的影响。这两个群体的成员都可以为自己是团队的一员而感到自豪(并向他们的下一个潜在雇主吹嘘),这个团队在实现安全目标方面取得了巨大的成功。

## **结论**

谈论弥合开发运维团队和安全团队之间分歧的重要性很容易，但在实践中让这些团队一起工作通常要困难得多。通过建立每个团队共同拥有的共享目标和度量标准，组织可以改善结果并减少倾向于将开发与安全分开的紧张关系。
