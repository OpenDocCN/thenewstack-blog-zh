# 为什么是时候将数据收集与监控分析分离了

监控是有时刻的。由于人工智能和实时分析的加速发展，IT 运营已经成为人们关注的焦点。下一代监控的潜力似乎是无限的。但是，即使新的模型被集成到我们的工具中，仍有一些东西在阻碍我们。我们试图监控的是越来越复杂的东西。

由于数据不一致或不完整，即使是最好的监控平台也难以发挥其全部潜力。我们不得不选择一个监控平台，而不是另一个，这是基于一个尴尬的横截面，即它可以用它可以收集的数据做什么。数据收集在传统上是次要因素。

喜欢预测，但希望它能监测 X？好吧，也许你可以写一个定制的集成，或者找一个开源的——但是你不应该这么做，而且这通常是不切实际的。当您有几个定制集成时会发生什么？谁让它们保持最新？发现 bug 会怎么样？你如何推出更新？在任何情况下，您的堆栈都会发生变化，一个非常适合的分析解决方案在一个决策的时间跨度内可能会变得不适合。

那么我们如何跟上呢？监控如何才能像我们的需求一样快速发展？我们需要将数据收集与数据分析分离开来。让我们让数据分析平台专注于从数据中获得更好的洞察力。同样，让我们为数据收集专家开辟一个空间，以改进我们监控堆栈的方式。

在 [Blue Medora](https://bluemedora.com/) ，我已经在近十几个监控平台上帮助将一百多个不同的监控集成推向市场。过去是一系列硬连线(1:1)集成，现在演变为可重用(1:N)库，现在正汇聚到一个用于整个 IT 数据收集部门的通用解决方案(M:N)。

为了说明这一点，考虑所有支持 PostgreSQL 的监控平台。传统上，每个平台都编写一些代码来连接 PostgreSQL，收集信息，将数据组装成预期的格式，最后发送出去供使用。每个平台通常实施自己的 1:1 解决方案。

如果您想将 PostgreSQL 监控引入各种平台，您需要做得更好。您需要一个独立的解决方案来监控 PostgreSQL——一个可以路由到任何监控平台的解决方案。这是一个 1:N 的解决方案。

最后，如果您想监控的不仅仅是 PostgreSQL，也许是 100 种不同的技术，但仍然将这些数据带到各种平台，该怎么办呢？这是 M:N 问题。这里的关键是要有一个通用和独立的数据模型——一个代表你的栈的真实状态的模型，并且可以被转换以服务于任何平台。

这就是我们如何更好地收集数据，但是还有一个相关问题的完整缩影，从集成模式到生命周期管理，效率的提高到最佳实践。这些年来，我越来越清楚地认识到，数据收集是其自身的一流空间，有其自身的一系列挑战，这些挑战需要比我们将收集与消费混为一谈时得到的解决方案更好的解决方案。

行业需要各种不同技术的标准标准化，但我们也需要 SME 来确定每个端点环境中的重要因素。我们需要监控集成，以便:

*   自动保持最新状态。
*   最大限度地减少堆栈和您的开销。
*   让单一观察在多个分析平台中发挥作用。

这些都是可以解决的挑战，但是它们显然属于它们自己的领域——它们自己在工具链中的一环。

历史已经向我们表明，不同问题空间的抽象是创新的倍增器。在硬件方面，考虑虚拟化。在软件方面，考虑容器化。从我的角度来看，我们目前在监控市场正处于类似的关头。他们说历史不会重演，而是押韵。我很兴奋地说，新的节奏响亮而清晰，IT 监控正在向新的方向进军。
