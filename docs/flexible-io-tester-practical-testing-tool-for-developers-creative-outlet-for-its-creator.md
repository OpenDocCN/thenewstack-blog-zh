对于一些人来说，开源是一份全职工作。对其他人来说，这是一份兼职。对于[延斯·阿克斯博](https://twitter.com/axboe?lang=en)，两者兼而有之。

Axboe 是 Linux 内核的块 IO 子系统的主要开发者和维护者。与这项工作相关，他已经编写了一系列令人印象深刻的代码，从不同的 IO 调度程序到开发和维护 ATAPI，到 SATA NCQ，再到(最近的) [io 期间](https://kernel.dk/io_uring.pdf)。哦，他还开发了[灵活 IO 测试器](https://github.com/axboe/fio) (fio)，这已经成为存储工作负载建模和压力测试的行业标准。

正如最好的开源项目经常出现的情况一样，Axboe 并没有试图创建一个行业标准。他只是需要一种简单的方法来重现存储工作负载，以便查找 Linux 代码的问题。开发人员正在记录错误，但他正在努力模拟产生这些错误的工作负载。Fio 使开发人员能够描述工作负载，以便 Axboe 可以构建和运行工作负载，而不必编写简单的应用程序来测试各种配置或内核代码的更改。

费奥做的另一件事是对阿克斯博个人而言——这给了他一个创造性的出口，一个玩乐的地方。作为[开源构建者系列](/open-source-builders-an-inside-look/)的一部分，我采访了 Axboe，以收集对有效的开源社区管理、最有价值的贡献者等的见解。

## 一个创造性的附带项目

虽然 Axboe 很幸运地因为分发开源软件而得到了一份薪水，但他从来没有因为 fio 而得到过报酬。他说，这一直是他的副业。在最初的几年里,《费奥》是一部充满爱的作品——一部 Axboe 独自完成的作品。即使他在 fio 中构建了新的功能来帮助他的 Linux 内核工作，它仍然与他的全职工作(为多家公司做系统工程，目前是脸书)分开，这一直很适合他。

根据 Axboe 的说法，fio 是一个创造性的出口，他控制着一切。这并不是因为 Axboe 是一个无法与他人相处的控制狂，而仅仅是人类表达自我的自然欲望。“当你在其他人的项目上工作时，Linux 内核最终就是这样，有一个严格的审查过程，这是我们所需要的，”他解释道。“此外，其他人需要签署您的代码，您必须说服 Linus 接受它。再加上内核编程只是稍微难一点，调试更难。所以 fio 是我在用户空间中创造性地做事情的一种方式，这很简单。每当我对内核方面的东西感到有点厌倦，或者如果我需要休息一下，我就会在 fio 上开发新的东西，因为这很有趣，也很自由。”

尽管现在 fio 中的大部分代码都不是他创作的，但 Axboe 仍然坚持这种创造性的出路。例如，他主要是合并其他人的代码，同时处理可能由新的存储硬件产生的更复杂的任务。无论如何，fio 仍然是 Axboe 的一个创造性渠道，类似于他创建的其他开源项目，以帮助他完成工作。

## 吸引贡献者阶层

如果 Axboe 今天能够通过合并其他人的代码来维护 fio，那是因为他在吸引有才华的开发人员从事 fio 方面做得很好。在他看来，任何想要在几年内蓬勃发展的项目都需要吸引新鲜血液。另一种选择是项目被搁置。为什么会这样？有时，最初的开发人员可能忙于其他事情，或者可能只是厌倦了这个项目。如果有一个充满活力的社区，项目的风险就会最小化。

分担领导项目的负担也很重要。对于 Axboe 来说，开源项目不会成为代码的负担，而是成为代码贡献者的负担，换句话说，就是人的负担。这个副业项目吸引了可能变得苛刻的用户。他说，“这应该是一项充满爱的工作，是你为了乐趣而做的事情，因为你觉得这很令人兴奋，”但突然之间，这个兼职项目让你被支持问题淹没了。

这也许是为什么 Axboe 认为贡献者世界分为给予者和接受者。后一类包括仅仅使用软件的人。如果有些东西不工作，他们可能会也可能不会提交错误报告。“与这些用户一起工作会耗费大量时间，”Axboe 说，“这也是你有时会收到稍微苛刻的请求的地方。”从 Axboe 的角度来看，他参与这个项目不仅仅是为了拥有大量的用户，因为他认为指标本身毫无用处。“这是一个很好的副作用，但我不会把它放在我的清单上，”他说。简单地提交 bug 报告的用户让他陷入问题中，却没有提供帮助。

(或许值得注意的是，Axboe 与这些“免费搭车者”的经历与 musl libc 的创始人 Rich Felker 的经历之间存在明显差异。Felker 喜欢各种类型的用户，因为采用这种方式可以确保其他人需要做与 musl 集成的工作。因此，也许在他的案例中，软件用户不会像 Axboe 那样制造麻烦。我的感觉是 Axboe 的反应更常见。)

然而，第二类用户则不同。这个小组使用软件，*和*能够做出改变。他们发送一个(或多个)补丁，修复他们遇到的问题，或者添加他们需要的功能。

“我爱这些用户！”阿克斯博说。“本着真正的开源精神，他们通过回馈帮助推动项目向前发展。软件和项目因此变得更好，这也让他们的生活变得更轻松。”而且这个班不仅仅包括编码员。“fio 最大的漏洞——可能也是其他开源项目中的一个*漏洞——是有人编写质量文档。我的文档不是很好。”*

所以，我问，一个项目怎样才能吸引更多同样做出贡献的用户——代码、文档等等？

## 论领导力

Axboe 说，一个项目中首要的需求是一个强有力的领导者。一个不太害怕做决定的人，一个拥有做决定的“技术能力”的人。

“人们对项目领导者有一定程度的技术尊重是非常重要的，”他解释说，“所以每当他或她不得不做出困难的决定时，他们通常会被认为是一个好决定。”在他看来，如果没有这样的领导，没有什么能拯救这个项目。它会漂移，然后，很有可能，死亡。

开源成功的另一个关键因素是沟通。“拥有一个欢迎新贡献者的环境是非常重要的，”Axboe 说，这样新来的人就可以放心地问问题或对项目提出修改。

关于这一点，如果你想参与 fio，从[阅读文档](https://fio.readthedocs.io/en/latest/fio_doc.html)开始总是一个好习惯。希望文档更好？Axboe 也是！他很乐意看到你的贡献。一旦你准备好开始调整代码并提交一个拉请求，在 GitHub 上找到 [fio。](https://github.com/axboe/fio)
