# 为什么云原生数据保护不再被忽视

> 原文：<https://thenewstack.io/why-cloud-native-data-protection-can-no-longer-be-ignored/>

[戴尔科技](https://www.delltechnologies.com/en-us/index.htm)赞助了这次播客。

在匆忙部署到云原生环境中时，组织现在需要担心另一个警告。随着微服务和容器提供 CI/CD 流程的灵活性，应用部署后保护数据的需求没有得到应有的重视。在一系列高级别的漏洞之后——当然包括 [Equifax 漏洞](https://www.theverge.com/2019/7/22/20703497/equifax-ftc-fine-settlement-2017-data-breach-compensation-fund)——基础设施层面的数据漏洞如今仍然是黑帽黑客相对容易的目标。

在本期[节目中，New Stack Makers](https://thenewstack.io/podcasts/makers) 播客在[kube con+CloudNativeCon North America 2019 现场录制，](https://events19.linuxfoundation.org/events/kubecon-cloudnativecon-north-america-2019/)戴尔 EMC 的[数据保护解决方案营销总监 Nelson Hsu](https://www.linkedin.com/in/nelsondhsu/) 和[产品营销顾问 Alexander Almeida](https://www.linkedin.com/in/alexanderalmeida/) 描述了数据保护和云原生部署的交集，以及 DevOps 在保护任务关键型工作负载方面的作用。TNS 出版商和创始人亚历克斯·威廉姆斯主持了这次播客。

[为什么云原生的数据保护不能再被忽视](https://thenewstack.simplecast.com/episodes/why-data-protection-for-cloud-native-can-no-longer-be-ignored)

Hsu 说，如今，许多组织的 DevOps 的主要目标是“利用构建在云原生平台上的工作负载，并考虑到敏捷性和效率”，同时管理“这些任务关键型工作负载，并推动它们更多地进入生产环境”。

“这是一个拥有特殊数据保护的好时机，我认为在 KubeCon，我们开始听到更多关于保护公司工作负载的需要，”Hsu 说。最初，一切都是为了将 DevOps 世界从[基础设施]世界中分离出来，因为存在容器、Kubernetes 和无服务器的原则。但是现在，随着我们将这些应用迁移或转换为围绕云、多云和混合云的分布式应用，我们必须了解如何保护这些数据，因为这些数据最终非常非常重要。"

最终，为云原生部署添加数据保护的任务落在了 DevOps 身上；在团队成员之间制造孤岛只会损害数据保护。Almeida 说:“是团队成员周围的角色将这些新技术带到了任务关键型环境中，你不能将开发运维管理员与 It 基础架构管理员对立起来。”“他们都参与其中，从数据保护的角度来看，他们都有自己的角色。”

Almeida 说，持久的容量和数据最终是“商业的价值”。Almeida 表示:“随着这些源自 DevOps 文化的新 DevOps 应用在更多任务关键型环境中实施，满足任务关键型基础设施的 SLA 变得越来越重要。“这正是您需要数据保护解决方案的地方，该解决方案将为团队的双方提供这种控制和可见性。”

具体到戴尔技术，他们一直在与 VMware 合作，同时在持久数据卷保护和应用其 [PowerProtect 产品](https://www.dellemc.com/en-us/data-protection/powerprotect-software.htm)方面“处于领先地位”。Almeida 说:“您有自己的基础架构，其中有您的传统应用程序和虚拟化应用程序，现在有您的容器化应用程序。“因此，我们从数据保护的角度涵盖了所有领域，同样，您不仅需要运行所有三种类型的基础架构，还需要保护所有三种类型的工作负载。”
